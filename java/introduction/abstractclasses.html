Chapter 13

Abstract Classes

This chapter gives "a class that only exists to be extended" its own keyword, and lets a method itself be a promise instead of an implementation.

§13.1

The Problem — two different gaps, one shared shape

Suppose we're building a vehicle rental system. We have several kinds of vehicles: Cars, Motorcycles, Vans, SUVs, etc. Although they're different, they also have a lot in common. Every vehicle has: a make and a model. We can capture those shared characteristics in a superclass named MotorVehicle. At this point, we have an interesting situation.

  • No reason to exist alone: An Incomplete Class The MotorVehicle class is useful. It provides shared state. However, nobody rents "a MotorVehicle" in the abstract — a customer rents a specific Car, a specific Motorcycle , a specific Truck. MotorVehicle by itself was never a complete, rentable thing; it's just the shape every one of those specific kinds happens to share. Every real vehicle available for rental is some kind of vehicle — MotorVehicle on its own is only the shared part.
  • No sensible default behavior. A separate problem shows up the moment MotorVehicle needs a startEngine() method. Starting one is not a fact about vehicles in general — it's specific to the kind of engine a particular vehicle actually has: a Car's starts with a push of a button, a Motorcycle's needs a kick or a thumbed starter, a Truck's diesel engine needs its glow plugs to warm up first. No shared sentence describes all three correctly — any default MotorVehicle.startEngine() body would either be wrong for two of the three kinds, or a sentence written just to satisfy the compiler.

Two different gaps — one in the class itself, one in a single method — but Java closes both with the same word, applied at two levels: abstract on the class says it can never be instantiated on its own, only extended; abstract on a method says it has no body at all, only a signature, binding every concrete subclass to supply one.

§13.2

abstract Classes — a shape that can't stand on its own

The word abstract reflects the role of the class. Vehicle represents a general concept rather than a concrete thing. You can rent a car. You can rent a motorcycle. You cannot rent "a vehicle" without knowing what kind of vehicle it is. It doesn't contain enough information to behave like a real object. An abstract class models a concept that is useful to describe and reuse, but too incomplete to create directly. Therefore, an abstract class cannot be instantiated.

abstract class MotorVehicle {
    String brand;
    String model;

    MotorVehicle(String brand, String model) {
        this.brand = brand;
        this.model = model;
    }

    @Override
    public String toString() {
        return "MotorVehicle[brand=" + brand + ", model=" + model + "]";
    }
}
MotorVehicle v = new MotorVehicle("Toyota", "Corolla"); // compile error — MotorVehicle is abstract

Subclasses are untouched by any of this. Car extends MotorVehicle, Motorcycle extends MotorVehicle, and Truck extends MotorVehicle can still be instantiated exactly as before (§10.3) — abstract only blocks new on the class it's written on, not on its concrete subclasses.

Stronger than any access modifier

§11 covered four access levels a constructor could carry, from public down to private — but every one of them only ever restricts which callers can invoke it; none of them can say "no caller, anywhere, ever." A private constructor would even block Car and Truck themselves from calling super(...), which isn't what's wanted here at all — MotorVehicle's constructor still needs to run, just never on its own. abstract on the class is a different kind of rule entirely, orthogonal to access modifiers: it doesn't say who may call new MotorVehicle(...) more narrowly, it says the expression can't exist at all, for anyone, while still leaving the constructor fully callable via super(...) from any subclass.

§13.3

abstract Methods — a signature with no body

An abstract method takes the same idea one level deeper. Abstract methods describe behavior the superclass expects but cannot implement itself. The superclass defines what every subclass must be able to do. Each subclass decides how to do it.

abstract class MotorVehicle {
    String brand;
    String model;
    private boolean engineRunning;

    MotorVehicle(String brand, String model) {
        this.brand = brand;
        this.model = model;
    }

    @Override
    public String toString() {
        return "MotorVehicle[brand=" + brand + ", model=" + model + "]";
    }

    boolean isEngineRunning() {
        return engineRunning;
    }

    protected void markEngineStarted() {
        engineRunning = true;
    }

    abstract void startEngine();
}

Every motor vehicle, regardless of kind, genuinely has an engine that's either running or not — that's true of a Car, a Motorcycle, and a Truck equally, so it belongs on MotorVehicle itself, right alongside brand and model. isEngineRunning() can be entirely concrete for the same reason: reading that shared state doesn't depend on what kind of vehicle it is. What can't be shared is how engineRunning becomes true in the first place — a push-button start for a Car, a kick-start for a Motorcycle, and a Truck whose diesel engine needs a moment before it's ready. That part stays abstract:

Every concrete (non-abstract) subclass is now required to provide a real startEngine() — the compiler checks this the same way it checks that every abstract method gets answered, somewhere, before an object of that type can ever exist. Each override is responsible for the same two things: describing its own way of starting, and setting the inherited engineRunning flag once it has:

class Car extends MotorVehicle {
    Car(String brand, String model) {
        super(brand, model);
    }

    @Override
    void startEngine() {
        System.out.println("Push-button start.");
        markEngineStarted();
    }
}

class Motorcycle extends MotorVehicle {
    Motorcycle(String brand, String model) {
        super(brand, model);
    }

    @Override
    void startEngine() {
        System.out.println("Kick-start the engine.");
        markEngineStarted();
    }
}

class Truck extends MotorVehicle {
    double capacityTons;

    Truck(String brand, String model, double capacityTons) {
        super(brand, model);
        this.capacityTons = capacityTons;
    }

    @Override
    void startEngine() {
        System.out.println("Turn key — glow plugs warming up first.");
        markEngineStarted();
    }
}
MotorVehicle v = new Car("Toyota", "Corolla");
System.out.println(v.isEngineRunning()); // false

v.startEngine(); // Push-button start.
System.out.println(v.isEngineRunning()); // true
A simplification, on purpose

Real cars don't all share one ignition system — plenty still start with a turned key, not a button. Tying "push-button start" to the Car class itself, as a fact true of every Car, is a simplification, the same way area = width × height simplified real rectangles back in §2. It's a deliberate choice for this fleet, in this example: good enough to show what abstract buys, without needing a whole separate hierarchy of ignition mechanisms to get there.

Leave startEngine() unimplemented in a subclass, and that subclass simply inherits the same obligation MotorVehicle had — it becomes abstract too, whether or not the word is written on it, and it can't be instantiated either, for exactly the same reason. The only way to end the chain is to actually write the method.

Dynamic dispatch, still ← §10.5

v.startEngine() above runs Car's version, chosen at runtime by the object's actual type, exactly the way overriding send() worked for EmailNotification and SmsNotification in §10.5. Nothing new is happening mechanically — an abstract method is simply one with no inherited default to fall back on, so every concrete class in the hierarchy is forced to supply its own version rather than optionally overriding one.

Scope of this chapter

A class can mix abstract and concrete methods freely, as MotorVehicle does above (toString() and isEngineRunning() concrete, startEngine() abstract) — there's no rule requiring all-or-nothing. Multiple levels of abstract subclassing, and how abstract classes interact with the next chapter's interfaces, are left for §14 to build on.

§13.4

Bringing It Altogether — a second example, end to end

The same shape shows up anywhere a family of things shares a field but not a way of acting on it. A notification always has a message — but how it actually gets sent depends entirely on what kind of notification it is:

  • Notification
    message: String
    send(): void
    • EmailNotification
      emailAddress: String
      send(): void
    • SMSNotification
      phoneNumber: String
      send(): void
abstract class Notification {
    String message;

    Notification(String message) {
        this.message = message;
    }

    abstract void send();
}

class EmailNotification extends Notification {
    String emailAddress;

    EmailNotification(String message, String emailAddress) {
        super(message);
        this.emailAddress = emailAddress;
    }

    @Override
    void send() {
        System.out.println("Emailing \"" + message + "\" to " + emailAddress);
    }
}

class SMSNotification extends Notification {
    String phoneNumber;

    SMSNotification(String message, String phoneNumber) {
        super(message);
        this.phoneNumber = phoneNumber;
    }

    @Override
    void send() {
        System.out.println("Texting \"" + message + "\" to " + phoneNumber);
    }
}

Notification.send() is a strong candidate to be abstract. The reason is: a generic notification does not actually know how to send itself. Sending requires something Notification itself doesn't have — an email address, a phone number, whatever the specific channel needs — so Notification can declare that every notification must know how to send itself, without being able to say how, in exactly the same way MotorVehicle could declare startEngine() without being able to say how an engine starts.

A Main class ties the pieces together — it creates a few notifications of different concrete types, then sends every one of them through the same Notification-typed reference:

public class Main {
    public static void main(String[] args) {
        Notification[] notifications = {
            new EmailNotification("Your order has shipped.", "alex@example.com"),
            new SMSNotification("Your table is ready.", "+1-555-0142")
        };

        for (Notification n : notifications) {
            n.send(); // which send() runs depends on the object, not the array's element type
        }
    }
}

// Emailing "Your order has shipped." to alex@example.com
// Texting "Your table is ready." to +1-555-0142

Every element of notifications is declared as a Notification — the array itself doesn't know or care whether a slot holds an EmailNotification or an SMSNotification. Yet each call to n.send() prints something different. That's dynamic dispatch again: at runtime, Java looks at the object n actually refers to, not the type of the variable holding it, and runs that object's own send(). It's the same rule that made v.startEngine() work earlier in this chapter, and the same rule behind overriding in general (§10.6).
