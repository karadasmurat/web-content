Chapter 14

Abstract Classes

§11's protected constructor trick narrowed who could instantiate a class, but couldn't close the gap entirely — same-package code could still do it. This chapter gives "a class that only exists to be extended" its own keyword, one that closes that gap completely — and lets a method itself be a promise instead of an implementation.

§14.1

The Problem — two different gaps, one shared shape

No reason to exist alone. §11.6 used a protected constructor to keep new Vehicle("Generic", "Model", 2019) from being written outside Vehicle's own package — a real restriction, but not a complete one: protected still means "same package, plus subclasses anywhere" (§12.7), so any other class sharing that package could still call it directly. What's actually true about Vehicle is stronger than "most callers can't construct this": Vehicle was never meant to be a complete, standalone thing at all, for any caller, anywhere. Every real vehicle is some kind of vehicle — a Car, a Truck, something more specific — and Vehicle on its own is just the shared shape they all happen to have.

No sensible default. A separate problem shows up the moment Vehicle needs a startEngine() method. describe() has always had a real implementation — every vehicle has a make, a model, a year, so one shared sentence works for all of them (§11.3). Starting an engine has no such shared sentence: a Car's starts with a push-button, a Truck's diesel engine needs its glow plugs to warm up first. No default Vehicle.startEngine() exists that isn't either wrong for someone or an outright lie, a body written just to satisfy the compiler.

Two different gaps — one in the class itself, one in a single method — but Java closes both with the same word, applied at two levels: abstract on the class says it can never be instantiated on its own, only extended; abstract on a method says it has no body at all, only a signature, binding every concrete subclass to supply one.

§14.2

abstract Classes — a shape that can't stand on its own

Marking a class abstract makes instantiating it, directly, a compile error — not a convention, not a note in a comment, an actual rule the compiler enforces:

abstract class Vehicle {
    String make;
    String model;
    int    year;

    Vehicle(String make, String model, int year) {
        this.make  = make;
        this.model = model;
        this.year  = year;
    }

    String describe() {
        return year + " " + make + " " + model;
    }
}
Vehicle v = new Vehicle("Generic", "Model", 2019); // compile error — Vehicle is abstract

Subclasses are untouched by any of this. Car extends Vehicle and Truck extends Vehicle can still be instantiated exactly as before (§11.3) — abstract only blocks new on the class it's written on, not on its concrete subclasses.

This replaces the protected constructor

§11.6 used a protected constructor to keep most unrelated code from writing new Vehicle(...), while still letting Car and Truck call it through super(...) — but same-package callers were always an open gap in that scheme. An abstract class doesn't have that gap — instantiation is blocked outright, for every caller, everywhere, regardless of what access level the constructor itself declares or what package the caller is in. Vehicle's constructor above is package-private and that's fine: abstract is doing the actual work now.

§14.3

abstract Methods — a signature with no body

An abstract method takes the same idea one level deeper: no body at all, just a signature ending directly in a semicolon.

abstract class Vehicle {
    String make;
    String model;
    int    year;

    Vehicle(String make, String model, int year) {
        this.make  = make;
        this.model = model;
        this.year  = year;
    }

    String describe() {
        return year + " " + make + " " + model;
    }

    abstract String startEngine();
}

Every concrete (non-abstract) subclass is now required to provide a real startEngine() — the compiler checks this the same way it checks that every abstract method gets answered, somewhere, before an object of that type can ever exist:

class Car extends Vehicle {
    Car(String make, String model, int year) {
        super(make, model, year);
    }

    @Override
    String startEngine() {
        return "Push-button start.";
    }
}

class Truck extends Vehicle {
    Truck(String make, String model, int year) {
        super(make, model, year);
    }

    @Override
    String startEngine() {
        return "Turn key — glow plugs warming up first.";
    }
}
Vehicle v = new Car("Toyota", "Corolla", 2022);
System.out.println(v.startEngine()); // Push-button start.

Leave startEngine() unimplemented in a subclass, and that subclass simply inherits the same obligation Vehicle had — it becomes abstract too, whether or not the word is written on it, and it can't be instantiated either, for exactly the same reason. The only way to end the chain is to actually write the method.

Dynamic dispatch, still ← §11.5

v.startEngine() above runs Car's version, chosen at runtime by the object's actual type, exactly the way overriding save() worked in §11.5. Nothing new is happening mechanically — an abstract method is simply one with no inherited default to fall back on, so every concrete class in the hierarchy is forced to supply its own version rather than optionally overriding one.

Scope of this chapter

A class can mix abstract and concrete methods freely, as Vehicle does above (describe() concrete, startEngine() abstract) — there's no rule requiring all-or-nothing. Multiple levels of abstract subclassing, and how abstract classes interact with the next chapter's interfaces, are left for §15 to build on.

A second example: Notification

The same shape shows up anywhere a family of things shares a field but not a way of acting on it. A notification always has a message — but how it actually gets sent depends entirely on what kind of notification it is:

  • Notification
    message: String
    send(): void
    • EmailNotification
      emailAddress: String
      send(): void
    • SMSNotification
      phoneNumber: String
      send(): void
abstract class Notification {
    String message;

    Notification(String message) {
        this.message = message;
    }

    abstract void send();
}

class EmailNotification extends Notification {
    String emailAddress;

    EmailNotification(String message, String emailAddress) {
        super(message);
        this.emailAddress = emailAddress;
    }

    @Override
    void send() {
        System.out.println("Emailing \"" + message + "\" to " + emailAddress);
    }
}

class SMSNotification extends Notification {
    String phoneNumber;

    SMSNotification(String message, String phoneNumber) {
        super(message);
        this.phoneNumber = phoneNumber;
    }

    @Override
    void send() {
        System.out.println("Texting \"" + message + "\" to " + phoneNumber);
    }
}

Notification.send() is a strong candidate to be abstract. The reason is: a generic notification does not actually know how to send itself. Sending requires something Notification itself doesn't have — an email address, a phone number, whatever the specific channel needs — so Notification can declare that every notification must know how to send itself, without being able to say how, in exactly the same way Vehicle could declare startEngine() without being able to say how an engine starts.
