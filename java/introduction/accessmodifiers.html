Chapter 12

Packages & Access Modifiers

Every class so far has quietly lived in the same unnamed place, and every field and method has been reachable from anywhere in the program, the moment you have a reference. This chapter names both: the packages classes are organized into, and the access modifiers that use them to control who's allowed to reach in.

§12.1

Packages — organizing code into named groups

A package is a named group of related classes — Java's way of organizing a codebase into folders instead of one flat pile of files, and of keeping two unrelated classes from colliding if they ever happen to share a name. A class declares which package it belongs to with a package statement, as the very first line of the file, above even its imports:

package com.example.shapes;

class Rectangle {
    ...
}

Every class in this tutorial has omitted that line. Omitting it doesn't mean "no package" — it places the class in Java's single, unnamed default package. Every class shown so far has quietly shared that one package with every other class. Reaching a class declared in a different, named package requires an import statement naming it — a detail this tutorial hasn't needed, precisely because everything so far has lived in the same one.

Why this chapter starts here

Packages matter to this chapter specifically because two of Java's four access levels are defined directly in terms of them: package-private (§12.6) means "same package," and protected (§12.7) means "same package, plus subclasses anywhere." Neither rule means anything without already knowing what a package is.

§12.2

Why Restrict Access — a class with no way to say no

Every field declared so far — width, height, make, doors — has been directly accessible from outside its class, with nothing in between. Accessible covers both directions at once: any code holding a reference can read the field's current value, and can just as freely write a new one:

Rectangle r = new Rectangle(10, 20);
System.out.println(r.width); // read — anyone can see it
r.width = -5;                // write — compiles fine, nothing stops this either

Nothing about Rectangle's definition says outside code shouldn't be able to see width, or that a width can't be negative — the field is just an int slot, and any code holding a reference to the object can dereference it, in either direction, and do exactly that. The class has no way to say "outsiders shouldn't be allowed to do that," because every field has been, by default, exactly as accessible as a field can be — visible to read, and open to write. Access modifiers are Java's way of giving a class control over both.

§12.3

private — visible only inside the declaring class

Marking a field private restricts it to code written inside that field's own class — not other classes, not even a subclass:

class Rectangle {
    private int width;
    private int height;
}
Rectangle r = new Rectangle(10, 20);
System.out.println(r.width); // compile error — width has private access in Rectangle
r.width = -5;                // compile error — same reason, either direction

The field still exists — every Rectangle object still has a width slot, laid out exactly as before (§3). private doesn't change the memory layout at all; it changes what the compiler lets other code write. Code inside Rectangle's own class body — its constructors, its methods — can still read and write width freely. Everyone else is locked out.

§12.4

Encapsulation — exposing behavior instead of state

Locking a field with private is only useful if there's still some way to reach it — otherwise no code, including Rectangle's own users, could ever set a width at all. The standard idiom is a pair of methods: a getter to read the field, and a setter to write it — with the setter free to enforce rules the field alone never could:

class Rectangle {
    private int width;
    private int height;

    public int getWidth() {
        return width;
    }

    public void setWidth(int width) {
        if (width < 0) {
            throw new IllegalArgumentException("width cannot be negative");
        }
        this.width = width;
    }
}
Rectangle r = new Rectangle(10, 20);
r.setWidth(15);  // fine
r.setWidth(-5);  // throws IllegalArgumentException

setWidth is an ordinary method — the same shape as every method since §6 — with one detail worth noticing: this.width = width; only runs when the check above it passes. There is no longer any way to give a Rectangle a negative width, because the only door into width is a method that refuses to let one through.

Throwing one yourself ← §4.2

§4.2 introduced exceptions the way you first meet them: happening to your code, automatically, when Java hits a problem it discovers on its own — dereferencing null raises a NullPointerException without you ever writing the word throw. throw is the statement that raises one deliberately: throw, followed by an expression that produces an exception object, ends the statement there. new IllegalArgumentException("width cannot be negative") constructs that object exactly the way new Rectangle(...) constructs any other (§4.1) — it's throw that then hands it off, instead of an ordinary assignment. Once thrown, execution stops at that line and jumps out looking for a handler, same as any exception. IllegalArgumentException itself is just a different name for the same kind of thing NullPointerException is — Java ships many exception types; this one specifically means "an argument was unacceptable."

Bundling data (width) together with the code that guards it (setWidth) — so the only way in or out is through methods the class itself controls — is called encapsulation. It's the reason private fields with public getters and setters are the default shape of a Java class, rather than the plain public fields this tutorial started with.

§12.5

public — visible everywhere

public is the opposite end from private: no restriction at all — any code, anywhere, holding a reference to the object, can call it. It's why getWidth and setWidth are declared public above: they're the class's deliberate, controlled doorway, so they need to be reachable from outside. Constructors are almost always public for the same reason — nothing else can create the object otherwise:

public Rectangle(int width, int height) {
    setWidth(width);
    this.height = height;
}

Notice the constructor can call setWidth instead of assigning width directly — the validation rule lives in exactly one place, and even Rectangle's own constructor goes through it.

§12.6

Package-Private — the default, when no modifier is written

§12.1 covered what a package is; this section names what happens when a field or method is declared with no access modifier at all. It isn't unrestricted — it's a fourth, distinct level, usually called package-private: visible to every class in the same package, and invisible to everything outside it.

class Rectangle {
    int width; // no modifier — package-private
}

This is stricter than public (outside code can't reach it) but looser than private (other classes in the same package still can). Every earlier chapter's field declarations, before this one, were package-private by default — reachable from anywhere in this tutorial's examples because every class happened to share one package (§12.1), which is exactly what made the unrestricted access in §12.2 possible in the first place.

§12.7

protected — visible to subclasses too

protected sits between package-private and public: visible within the same package, exactly like package-private, plus visible to subclasses even outside the package.

Here's a case public genuinely can't handle. Vehicle only exists to be extended — nobody should ever write new Vehicle(...) directly, only new Car(...) or new Truck(...). A public constructor can't prevent that; a private one would block subclasses too, since Car's constructor needs to call super(...). protected is the one modifier that says both things at once — closed to arbitrary outside code, open to the family:

class Vehicle {
    String make;
    String model;
    int    year;

    protected Vehicle(String make, String model, int year) {
        this.make  = make;
        this.model = model;
        this.year  = year;
    }
}

class Car extends Vehicle {
    int doors;

    Car(String make, String model, int year, int doors) {
        super(make, model, year); // fine — Car is a subclass, protected is reachable
        this.doors = doors;
    }
}
Vehicle v = new Vehicle("Generic", "Model", 2019); // compile error — constructor is protected
Car c = new Car("Toyota", "Corolla", 2022, 4);      // fine — Car's own constructor is public

The restriction lands exactly where it's wanted: unrelated code can no longer create a bare Vehicle, but every subclass still can, through super(...), because that call happens from inside a subclass — precisely the case protected exists to allow.

Fields can use the same idea

A field can be protected for the same reason a constructor can: a subclass sometimes needs to reach or adjust inherited state directly, as part of its own logic, in a way a public getter/setter would have to open to everyone, not just the family. Marking year protected instead of package-private, for instance, would let a subclass in a completely different package still adjust it directly — something package-private alone could never do.

The intuition behind all of this: protected is a class designer saying "this isn't part of the public API for everyone, but subclasses may need it to extend or customize what I built." Subclasses get to cooperate with the internals; arbitrary outside code still doesn't. That's what makes protected the modifier of choice for framework and library code specifically — code meant to be extended by callers the author will never meet, without handing those callers free rein over everything inside.

The real payoff is cross-package

protected only becomes truly interesting once the subclass lives in a different package. Inside the same package, it barely adds anything — package-private already grants every class there direct access to non-private members, so a same-package subclass would have reached year anyway, protected or not. The behavior protected uniquely adds is cross-package subclass access: a subclass in another package, another module, even another library entirely, can still reach in and extend the internals, while unrelated classes in that same foreign package still can't. That's precisely the shape of a real framework: its classes and the code extending them are almost never in the same package, so protected's cross-package reach is the part actually doing the work — not the same-package case, which package-private already covered.

A real design that leans on protected: the Template Method pattern

Methods can be protected for exactly the same reason constructors and fields can — and one particular shape of this shows up constantly in real Java code. Vehicle can fix the overall structure of describe() once, while leaving one specific piece for each subclass to fill in:

class Vehicle {
    String make;
    String model;
    int    year;

    public String describe() {
        return year + " " + make + " " + model + extraInfo();
    }

    protected String extraInfo() {
        return ""; // nothing extra, by default
    }
}
class Car extends Vehicle {
    int doors;

    @Override
    protected String extraInfo() {
        return ", " + doors + " doors";
    }
}
Vehicle v = new Car("Toyota", "Corolla", 2022, 4);
System.out.println(v.describe()); // 2022 Toyota Corolla, 4 doors

describe() is the template: the fixed shape every Vehicle's description follows, written once, meant to be called from outside — so it's public. extraInfo() is the hook: the one piece of that shape each subclass is expected to customize — but never meant to be called by outside code on its own; ", 4 doors" in isolation, with no year or make in front of it, isn't a sensible thing for anyone to ask for. protected is what enforces that distinction: Car and any other subclass can override extraInfo() to plug their own piece into the template, while code outside the family can only ever call the template itself, never the hook directly.

Summary: The Hierarchy of Trust

Java's access levels can be viewed as layers of a concentric circle. At each level, we share with more external actors, starting with nobody and ending with everyone:

Modifier Trusts Meaning
private Nobody Trust only myself. No one else. No exceptions.
package-private My trusted neighborhood Trust my coworkers (classes in the same package). If no access modifier is explicitly written, this is the default. Remove private — write nothing — and sharing begins, first with package mates; opening it up further, to classes outside the package too, needs protected or public.
protected Neighborhood + descendants Trust my coworkers (the classes in this package), plus my descendants (subclasses, wherever they live). Key nuance: protected does not mean "subclasses only" — it also lets any class in the same package reach in. Java has no subclass-only modifier.
public Everyone Trust everyone.

Why the "coworker" wins over the "child." A subclass in a different package cannot see a package-private member, but a class in the same package can — even one completely unrelated by inheritance. If a subclass lives in a different package, it's an external entity as far as that member is concerned: it may inherit everything else about the parent, but it lives in a different neighborhood. If Java let subclasses in other packages see package-private members just by extending the class, package-private would stop meaning anything — anyone could gain access to code meant to stay internal simply by writing class Hack extends YourClass in their own project, in their own package. Restricting the default to "same package" is what keeps "internal" actually internal.

The short version: classes in the same package already have a high degree of visibility into each other, reaching anything declared public, protected, or package-private (whether that member was declared directly, or inherited from a superclass). private is the only level that visibility never crosses.

Scope of this chapter

Interfaces, records, and nested classes each add their own wrinkles to access control, and are left for later — this chapter only needed ordinary top-level and member declarations. The general engineering principle behind all of this — expose as little as a class can get away with, called the principle of least privilege — is worth remembering even before every rule above is second nature.
