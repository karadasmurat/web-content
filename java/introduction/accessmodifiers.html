Chapter 11

Packages & Access Modifiers

Marking a field private restricts it to code written inside that field's own class — not other classes, not even a subclass. This chapter covers the full set of access modifiers Java gives a class to make that kind of decision, and the packages that organize classes in the first place.

§11.1

Packages — organizing code into named groups

A package is a named group of related classes — Java's way of organizing a codebase, and of keeping two unrelated classes from colliding if they ever happen to share a name.

Naming a Package

A class declares which package it belongs to with a package statement, as the very first line of the file:

package com.example.shapes;
                
class Rectangle {
    ...
}

Notice the name isn't just shapes — it's com.example.shapes. That's not decoration; it's a convention every real Java codebase follows, built to solve a problem one level up from the collision problem above. shapes alone is exactly the kind of name two completely unrelated projects might both reach for — a graphics library and a geometry teaching tool could both plausibly have a package just called shapes, with no relation to each other at all. If both ever ended up being used in the same program, that's a collision, because the package names themselves are now identical.

The convention is to prefix a package with a reversed domain name the organization actually owns — com.example for a company at example.com, com.google for Google, org.apache for the Apache Software Foundation. Domain names are already globally unique — that uniqueness is the entire reason they cost money and require registration — so reusing one, reversed, as a package prefix gives every organization's code a namespace nobody else can accidentally collide with. com.example.shapes and some unrelated team's com.othercompany.shapes can both exist on the same classpath without incident, because the part before shapes is what actually keeps them apart. This tutorial's own package tree above follows the same idea in miniature: com.store stands in for whatever domain a real store's engineering team would actually own.

Default Package

Every class in this tutorial has omitted package declaration line. Omitting it doesn't mean "no package" — it places the class in Java's single, unnamed default package. Every class shown so far has quietly shared that one package with every other class, which is exactly why none of them ever had to say where any other one lived.

The import statement

A class in a different named package doesn't get that for free. Suppose Rectangle above lives in com.example.shapes, and some billing code, in its own com.example.billing package, wants to use it: 

package com.example.billing;

import com.example.shapes.Rectangle;

class InvoiceLineItem {
    Rectangle area;
}

The import statement names exactly which class, in which package, this file means by the bare word Rectangle everywhere below it. Without it, the compiler has no way to find Rectangle at all — com.example.billing and com.example.shapes are different folders, as far as Java is concerned, and nothing about being "nearby" makes one visible to the other. The alternative to importing is spelling out the fully qualified name every single time instead: com.example.shapes.Rectangle area; works too, with no import at all — import exists purely so the rest of the file can say the short name.

com.example.shapes and com.example.billing hint at something larger than a two-class example: real projects are organized into several packages, one per area of responsibility, all living under one shared root. A small online store might lay its source tree out like this:

  • com/store/
    • model/
      • Customer.java
      • Product.java
      • Order.java
    • billing/
      • Invoice.java
      • Payment.java
    • inventory/
      • Warehouse.java
      • StockItem.java
    • Main.java

The folders aren't just filing cabinets — each one is a package, and the folder path spells the package name directly: Order.java, sitting in com/store/model/, declares package com.store.model; as its first line; Invoice.java, in com/store/billing/, declares package com.store.billing;. Grouping this way keeps each area's classes together and named for what they are — model holds the store's core data, billing the money-handling code, inventory everything about stock — rather than one flat folder of thirty unrelated files. An Invoice that needs an Order reaches across that boundary exactly the way InvoiceLineItem reached Rectangle above: an import com.store.model.Order; line, naming precisely which package the class it needs actually lives in.

How Packages Solve Naming Collisions

Suppose the store integrates with a shipping partner's library, and that library happens to define its own class named Order too — a shipping order, a completely different concept, that just happens to share the name the store already picked for a customer order. Without packages, that's a dead end: Java would have no way to tell the two apart, and whichever one got declared first would be the only Order the compiler could ever recognize. With the help of packages, both Order classes simply live in different places and neither one has to change: 

com.store.model.Order      — a customer's order, placed in the store
com.shipping.Order         — a shipment handed to the carrier

Two classes, same simple name, two completely different types — and Java has no trouble telling them apart, because their fully qualified names, package included, are different. Code that only ever needs one of them just imports that one, same as always:

package com.store.billing;

import com.store.model.Order;

class Invoice {
    Order order; // unambiguous — com.store.model.Order is the only Order this file imported
}

Code that genuinely needs both in the same file has to be explicit about the one that isn't imported by its short name — an import claims the bare word Order for whichever package it names, so the other one falls back to being spelled out in full:

package com.store.fulfillment;

import com.store.model.Order;

class ShipmentTracker {
    Order storeOrder;                 // com.store.model.Order, via the import above
    com.shipping.Order shippingOrder; // com.shipping.Order, written out in full — "Order" was already taken
}

Nothing about this required either Order class to be renamed, and neither library needed to know the other one existed. That's the collision problem packages actually solve: not by preventing two classes from ever sharing a name, but by giving every class a fully qualified name that's guaranteed unique, so a short-name clash can always be resolved by simply being more specific about which one is meant.

§11.2

Why Restrict Access — a class with no way to say no

Before any Java is involved, imagine designing a coffee machine. The person using it should be able to:

  • fill the water tank
  • brew a coffee
  • empty the waste container

But should they be able to:

  • adjust the boiler pressure
  • change the heating algorithm
  • open the electronic components adjusting pressure

Probably not. Those are implementation details.

Every system built for someone else to use ends up with this same split:

  • a public side, meant to be used
  • an internal side, meant to be left alone so the public side can stay simple.
That's not a Java idea, or even a programming one. It's a broader design idea Java's classes happen to run into as well, the moment a class is written for someone else to use.

Translate the coffee machine straight into a class — a water level anyone should be able to change by filling the tank, and a boiler pressure nobody outside the machine should be touching, plus one simple behavior computed from them:

class CoffeeMachine {
    int waterLevel;     // ml remaining in the tank
    int boilerPressure; // bar

    int cupsRemaining() {
        return waterLevel / 200;
    }
}

Every field and method declared this way — no modifier at all, in every chapter up to this one — has, in every example so far, been directly accessible from outside its class, with nothing in between (§11.5 explains exactly why "no modifier" still means something, not "wide open"). For a field, that simply means: outside code can look at the value, and outside code can change it. For a method, it means outside code can call it, same as if it were its own:

CoffeeMachine machine = new CoffeeMachine();
int level = machine.waterLevel;     // read — anyone can see it

machine.waterLevel = -500;          // write — compiles fine
                                    // -500 is a valid int, nothing stops this assignment

machine.boilerPressure = 999;       // exactly the tampering the motivation above warned about

int cups = machine.cupsRemaining(); // call — anyone can invoke it too

Nothing about CoffeeMachine's definition says outside code shouldn't be able to see waterLevel, or that a water level can't be negative, or that boilerPressure is off limits entirely — every field is just an int slot, and any code holding a reference to the object can dereference it, in either direction, and do exactly that. cupsRemaining() is no different in spirit — a class member the same way waterLevel is, just one you call instead of one you read or write, and reachable the same way. The class has no way to say "outsiders shouldn't be allowed to do that" — every field and method has, so far, been left exactly as open as our examples happened to need — visible to read, open to write, free to call.

A well-designed class doesn't leave that gap open. It exposes only what other code actually needs to use — filling the tank, brewing, reading how many cups are left — and keeps everything else, like boilerPressure, hidden from anything outside it. This idea has a name: encapsulation — a class deciding, deliberately, what's exposed and what stays hidden, rather than leaving every field open by default the way CoffeeMachine does above.

Java provides access modifiers to express exactly that design in code — a way for a class to say, field by field and method by method, who's allowed to reach what. private is where that starts.

private — visible only inside the declaring class

Instead of exposing a class member directly, we can make it private. 

class CoffeeMachine {
    private int waterLevel;
    private int boilerPressure;
}

Now direct access is restricted only from within the CoffeeMachine class itself. Even a subclass cannot see private members.

CoffeeMachine machine = new CoffeeMachine();
int level = machine.waterLevel; // compile error — waterLevel has private access in CoffeeMachine
machine.boilerPressure = 999;   // compile error — same reason, and exactly the tampering warned about above
Software Design: What does this class expose?

Notice what actually came first, above: the coffee machine, the question of what a user should and shouldn't touch, the decision to call that encapsulation — and only then, a keyword. That order isn't an accident. This chapter is about class design, not keywords. private, public, protected, and package-private are simply the language's tools for implementing a design decision a class has already made — they don't make the decision for you, and memorizing what each one does is worth little without first asking, for every field and method, the question CoffeeMachine forced above: is this part of what the class exposes, or part of how it works underneath?

§11.3

Getters and Setters — exposing behavior instead of state

private field is there — every CoffeeMachine object still has a waterLevel slot and a boilerPressure slot, laid out exactly as before (§2). private doesn't change the memory layout at all; it changes what the compiler lets others see. Code inside CoffeeMachine's own class body — its constructors, its methods — can still read and write both fields freely. Everyone else is blind to them.

Hiding a field with private is only useful if there's still some way to reach the parts that should be reachable. The standard idiom is a pair of methods:

  • a getter to read the field, and
  • a setter to write it,
enforcing rules the field alone never could. By keeping the field private and exposing carefully designed methods, the object remains responsible for its own state. An object's fields represent its state, and its methods define the safe ways that state can change. 

class CoffeeMachine {
    private int waterLevel;
    private int boilerPressure;

    public CoffeeMachine(int waterLevel) {
        setWaterLevel(waterLevel);
        this.boilerPressure = 15; // bar — fixed internally, never exposed to the caller
    }

    public int getWaterLevel() {
        return waterLevel;
    }

    public void setWaterLevel(int waterLevel) {
        if (waterLevel < 0) {
            throw new IllegalArgumentException("water level cannot be negative");
        }
        this.waterLevel = waterLevel;
    }
}
CoffeeMachine machine = new CoffeeMachine(10);
machine.setWaterLevel(100);  // fine — filling the tank, exactly what the machine should allow
machine.setWaterLevel(-50);   // throws IllegalArgumentException

setWaterLevel is an ordinary method — the same shape as every method since §4 — with one detail worth noticing: this.waterLevel = waterLevel; only runs when the check above it passes. There is no longer any way to give a CoffeeMachine a negative water level, because the only door into waterLevel is a method that refuses to let one through.

Throwing an Exception ← §15

When the program arrives at a situation where its normal logic simply has no next step for, it "throws" an Exception, and execution stops at that line. Java ships many exception types; IllegalArgumentException specifically means "an argument was unacceptable."

Notice what boilerPressure gets: no getter, no setter, nothing. §11.2 already decided adjusting the boiler pressure was never something outside code should do — so unlike waterLevel, it gets no doorway back in at all. Bundling data (waterLevel) together with the code that guards it (setWaterLevel) — so the only way in or out is through methods the class itself controls — is encapsulation in practice, the same idea §11.2 named, now expressed in actual code. It's the reason private fields with public getters and setters are the default shape of a Java class, rather than the plain public fields this tutorial started with.

§11.4

public — visible everywhere

public is the opposite end from private: no restriction at all — any code, anywhere, holding a reference to the object, can call it. It's why getWaterLevel and setWaterLevel are declared public above: they're the class's deliberate, controlled doorway, so they need to be reachable from outside. Constructors are usually public for the same reason — object creation could be limited otherwise.

Notice the constructor can call setWaterLevel instead of assigning waterLevel directly — the validation rule lives in exactly one place, and even CoffeeMachine's own constructor goes through it. boilerPressure isn't even a constructor parameter — there was never a reason to let a caller supply it, so the constructor doesn't offer the option.

Contract, exposed

Restricting access is the mechanism, not the point. What a class actually marks out, member by member, is its public contract: the part it promises to keep working the same way for whoever calls it. Marking getWaterLevel and setWaterLevel public is a promise — other code may depend on this — and once other code does, changing how it behaves means breaking that promise. Marking waterLevel and boilerPressure private is the opposite kind of statement — this is an implementation detail; I reserve the right to change it — CoffeeMachine could switch waterLevel's unit from milliliters to fluid ounces tomorrow, and nothing outside the class would need to know, because nothing outside the class was ever promised a unit in the first place. Every access modifier choice is really this same question, asked one member at a time: is this something callers should be able to depend on, or something the class needs the freedom to change later?

§11.5

Package-Private — the default, when no modifier is written

A field or method can be declared with no access modifier at all. This represents a distinct access level, usually called package-private: visible to every class in the same package, and invisible to everything outside it. 

package com.store.machines;

class CoffeeMachine {
    int waterLevel; // no modifier — package-private
}

This is stricter than public (outside code can't reach it) but looser than private (other classes in the same package still can). Every earlier chapter's field declarations, before this one, were package-private by default — reachable from anywhere in this tutorial's examples because every class happened to share one package (§11.1), which is exactly what made the unrestricted access in §11.2 possible in the first place.

A second class in that same package, com.store.machines, reaches waterLevel exactly the way CoffeeMachine's own code would — no import needed, since same-package classes never need one (§11.1):

package com.store.machines;

class MachineDiagnostics {
    void checkWater(CoffeeMachine machine) {
        System.out.println(machine.waterLevel); // fine — same package
    }
}

A class in a different package doesn't get that, even with an import naming the class directly:

package com.store.billing;

import com.store.machines.CoffeeMachine;

class InvoiceService {
    void checkWater(CoffeeMachine machine) {
        System.out.println(machine.waterLevel);
        // compile error — waterLevel is not public in com.store.machines.CoffeeMachine;
        // cannot be accessed from outside package
    }
}

import only ever does what §11.1 said it does: it lets this file spell CoffeeMachine instead of com.store.machines.CoffeeMachine. It says nothing about which of CoffeeMachine's members InvoiceService is allowed to touch — that question belongs entirely to the access modifier on waterLevel itself, and package-private's answer is the same regardless of which package is asking, except the one it was declared in.

com.store.machines
CoffeeMachine
waterLevelpkg-private
MachineDiagnostics
✓ can reach waterLevel — same package
com.store.billing
InvoiceService
✗ cannot reach waterLevel — different package

The Package as a Trust Boundary

It's worth stepping back from the individual keyword to what a package actually represents. A package isn't just a folder for filing classes away for tidiness — placing several classes in the same package is a statement about them: these classes are designed to work together as one cohesive component. com.store.model's classes (§11.1) all cooperate to represent the store's data; that's exactly why they were grouped together in the first place, not an accident of alphabetizing files.

That statement and the package-private keyword are related, but they're not the same thing. package-private is one specific modifier — the one a member gets when no modifier is written at all, granting access to same-package code and nobody else. But being in the same package as another class grants more than just package-private reach, because nothing about being in the same package makes protected or public members any less visible. A class sitting alongside CoffeeMachine in com.store can reach:

  • its package-private members — by definition, that's exactly what the modifier grants;
  • its protected members — package-private access is part of what protected includes (§11.6);
  • its public members — public holds nothing back from anyone, same package or not.

The only member a same-package class can't reach is a private one. Put the other way around: from inside the package, private is the only wall still standing — every other modifier lets a fellow package member through. That's the sense in which a package functions as a trust boundary: it's not one access level among four, it's the neighborhood inside which three of the four already assume some level of cooperation, and only private still means "just this one class."

§11.6

protected — visible to subclasses too

protected sits between package-private and public: visible within the same package, exactly like package-private, plus visible to subclasses even outside the package.

protected is primarily about allowing subclasses to participate in a design, usually through methods — a subclass overriding a hook, say, or calling a piece of shared logic its superclass exposes just for that purpose. Fields and constructors can be protected too, but methods are where the modifier earns its keep.

Why starting with a protected field is misleading

The classic example looks like this:

class Animal {
    protected String name;
}

class Dog extends Animal {
    void printName() {
        System.out.println(name); // direct access to parent's protected field
    }
}

It works — but what did the student actually learn? They learn: "subclasses need direct access to their parent's variables." That is not the most useful inheritance lesson, and it isn't really what protected is for. A field sitting open for any subclass to read or overwrite, with no rule governing how, is closer to an accident than a design.

A better design leans on methods instead:

class Document {
    private String title;

    public Document(String title) {
        this.title = title;
    }

    // Part of the class's public API:
    public final void generate() {
        System.out.println("Generating: " + title);
        loadData();
        process();
    }

    // NOT part of the class's public API:
    // a subclass can call these directly, but external code in other packages cannot.
    protected void loadData() {
        System.out.println("Document: Loading data");
    }

    protected void process() {
        System.out.println("Document: Processing");
    }
}

// A subclass can override, since loadData() is inherited and visible to it —
// Report has now changed what happens during generate().
class Report extends Document {
    public Report(String title) {
        super(title);
    }

    @Override
    protected void loadData() {
        super.loadData();  // direct access to parent's protected method
        System.out.println("Report: Loading data");
    }
}
Document myReport = new Report("Financials");
myReport.generate();
// Generating: Financials
// Document: Loading data
// Report: Loading data
// Document: Processing

title stays private — no subclass ever touches it directly, and it doesn't need to. generate() is public, and final — here meaning a subclass cannot override it, not the field-reassignment sense §12 covers: it's the fixed shape every Document's generation follows, meant to be called from outside, and no subclass can replace that shape. loadData() and process() are protected: steps of that shape a subclass is allowed to customize, but never meant to be called by outside code on their own — Report overrides loadData() to add its own step, and generate() automatically picks it up, without either the caller or Document itself needing to know Report exists.

Now the lesson is different: "the parent controls what subclasses are allowed to do." Document decides, field by field and method by method, exactly what it exposes to subclasses and exactly what it keeps for itself — protected is one of the tools it uses to draw that line, not an invitation for subclasses to reach in and grab whatever they need.

The real payoff is cross-package

protected only becomes truly interesting once the subclass lives in a different package. Inside the same package, it barely adds anything — package-private already grants every class there direct access to non-private members, so a same-package subclass would have reached loadData() anyway, protected or not. The behavior protected uniquely adds is cross-package subclass access: a subclass in another package, another module, even another library entirely, can still override the hooks and extend the internals, while unrelated classes in that same foreign package still can't. That's precisely the shape of a real framework: its classes and the code extending them are almost never in the same package, so protected's cross-package reach is the part actually doing the work — not the same-package case, which package-private already covered.

The intuition behind all of this: protected is a class designer saying "this isn't part of the public API for everyone, but subclasses may need it to extend or customize what I built." Subclasses get to cooperate with the internals; arbitrary outside code still doesn't. That's what makes protected the modifier of choice for framework and library code specifically — code meant to be extended by callers the author will never meet, without handing those callers free rein over everything inside.

Summary: The Hierarchy of Trust

Java's access levels can be viewed as layers of a concentric circle. At each level, we share with more external actors, starting with nobody and ending with everyone:

Modifier Trusts Meaning
private Nobody Trust only myself. No one else. No exceptions.
package-private My trusted neighborhood Trust my coworkers (classes in the same package). If no access modifier is explicitly written, this is the default. Remove private — write nothing — and sharing begins, first with package mates; opening it up further, to classes outside the package too, needs protected or public.
protected Neighborhood + descendants Trust my coworkers (the classes in this package), plus my descendants (subclasses, wherever they live). Key nuance: protected does not mean "subclasses only" — it also lets any class in the same package reach in. Java has no subclass-only modifier.
public Everyone Trust everyone.

Why the "coworker" wins over the "child." A subclass in a different package cannot see a package-private member, but a class in the same package can — even one completely unrelated by inheritance. If a subclass lives in a different package, it's an external entity as far as that member is concerned: it may inherit everything else about the parent, but it lives in a different neighborhood. If Java let subclasses in other packages see package-private members just by extending the class, package-private would stop meaning anything — anyone could gain access to code meant to stay internal simply by writing class Hack extends YourClass in their own project, in their own package. Restricting the default to "same package" is what keeps "internal" actually internal.

The short version: classes in the same package already have a high degree of visibility into each other, reaching anything declared public, protected, or package-private (whether that member was declared directly, or inherited from a superclass). private is the only level that visibility never crosses.

Scope of this chapter

Interfaces, records, and nested classes each add their own wrinkles to access control, and are left for later — this chapter only needed ordinary top-level and member declarations. The general engineering principle behind all of this — expose as little as a class can get away with, called the principle of least privilege — is worth remembering even before every rule above is second nature.
