Chapter 3

Assignment

Assignment always copies what the variable holds. The value. That's it.

§3.1

The "Value": What Does a Variable Hold?

We already know every variable in Java holds exactly one of two things.

  1. The data itself, stored directly in the slot. This is what every primitive type from §1.3 does.
  2. An address — the location of some data stored elsewhere in memory. This is what reference variables do (§2.3).
A note on terminology

From the language perspective, Java variables of reference type store references, not "addresses" — the specification never promises an actual memory address is what's inside that slot, or that it works the way a raw pointer would in a language like C. This tutorial uses "address" deliberately, as a concrete mental model for what a reference points to and how it gets there. Just don't take "address" as a claim about the JVM's actual internal representation.

Side by side, with the actual values each slot holds:

int age = 20;

Person alice = new Person();
Stack
age = 20
the value itself — nothing to follow
alice = 0x9000
0x9000 is a memory address, not a Person
points to
Heap
Person @ 0x9000

age's slot holds 20 — full stop. There's nowhere else to go, nothing to dereference. alice's slot holds 0x9000 — not a Person, an address of one — and reaching the actual object means following that arrow onto the heap. Same shape of slot, same idea of "a variable holds something," but what that something is is fundamentally different depending on the variable's type. Everything in this chapter follows from that one difference.

§3.2

Primitive Assignment — copies the value

Write a number on a sticky note, then copy it onto a second sticky note. Scribble something new on the second note, and the first one is completely unaffected — they were only ever connected by having matched, once, at the moment of copying. That's exactly what happens when you assign one primitive variable to another: the value in the slot is copied. You end up with two independent slots, each holding the same value.

int a = 10;
int b = a;   // b gets its own slot; the value 10 is copied into it

b = 99;      // modifies b's slot only

System.out.println(a); // 10 — unchanged
System.out.println(b); // 99
Memory model

After int b = a: There are now two distinct slots, a and b, each independently containing 10. These are independent, disconnected values — changing one has no effect on the other.

Stack
a = 10
b = 10 99
a copy — not the same slot as a — then reassigned

No arrows here at all — that's the point. Both slots hold 10 directly, side by side, with nothing connecting them. Change b, and a has no way of finding out.

§3.3

Reference Assignment — copies the address

Hand a friend a copy of your house key, and now there are two keys — but only one house. Either of you can walk in, and anything one of you rearranges inside, the other sees the next time they visit, because you were never dealing with two houses to begin with. That's what happens when you assign one reference variable to another: the address in the slot is copied, not the object it points to. You end up with two variables holding the same address — two names for the same object.

Rectangle r1 = new Rectangle();
r1.width = 10;

Rectangle r2 = r1;  // r2 gets its own slot; the address in r1 is copied into it
Memory model

After Rectangle r2 = r1: There are now two distinct slots, r1 and r2, both containing the same address: 0x4a2f.

One object on the heap. Two variables pointing at it.

Stack
r1 = 0x4a2f
r2 = 0x4a2f
Heap
Rectangle @ 0x4a2f
width10
§3.4

Consequence: Modifications Are Visible Through Both Variables

Because both variables point to the same object, modifying the object through one variable is immediately visible through the other.

r2.width = 99;      // dereferences r2, modifies the object

System.out.println(r1.width); // 99 — r1 sees the change too
System.out.println(r2.width); // 99

This is not a quirk or a bug — it follows directly from the fact that assignment copied the address. There is one object. Both variables hold its address. Whoever writes to it, the other sees the result.

Contrast with primitives

With primitives, assignment copies the value — two independent slots, no connection. With references, assignment copies the address — one object, two names. The behavior after assignment is completely different depending on which kind of variable you're working with.

§3.5

Assignment vs. the Dot

It's worth stating the contrast plainly, because the two operators are easy to blur together — one reaches the object, the other never does. Aliasing (§3.3) sets up the perfect way to see both at once: once r2 and r1 hold the same address, what happens if the program reaches through r1 with the dot — and what happens if it reassigns r1 outright instead? Watch both, one at a time: 

Rectangle r1 = new Rectangle(5, 10);

Rectangle r2 = r1; // r2 gets its own slot; the address in r1 is copied into it

r1.width = 333; // dereference — follows r1's address, mutates the shared object

r1 = new Rectangle(2, 4); // plain assignment — no dereference, r1's slot just gets a new address

System.out.println(r1.width); // 2
System.out.println(r2.width); // 333

r1.width = 333 runs through the dot, so it follows r1's address onto the heap and mutates the object sitting there — the exact same object r2 also holds the address of. r2.width would read 333 immediately afterward too, even though that line never mentioned r2 at all.

The very next line looks similar but does something completely different. r1 = new Rectangle(2, 4) is a plain assignment — the dot never runs at all. It never dereferences anything, never touches the object r2 is still holding the address of. All it does is overwrite what's sitting in r1's own slot — swapping one address for another. r2's slot was never part of that statement, so it goes on holding exactly what it held the line before — including the 333 the dereference just wrote. Watching all four statements run, one at a time, makes this easier to see than the final snapshot alone:

Step 1 — Rectangle r1 = new Rectangle(5, 10);

Stack
r1 = 0xA100
Heap
Rectangle @ 0xA100
width5
height10

Step 2 — Rectangle r2 = r1; — r2 copies r1's address; both now point at the same object:

Stack
r1 = 0xA100
r2 = 0xA100
Heap
Rectangle @ 0xA100
width5
height10

Step 3 — r1.width = 333; — a dereference: it follows r1's address and mutates the shared object itself. Neither slot changes; the object both point to does:

Stack
r1 = 0xA100
r2 = 0xA100
Heap
Rectangle @ 0xA100
width333
height10

Step 4 — r1 = new Rectangle(2, 4); — a plain assignment, no dot involved: r1's slot gets a new address; r2's slot, and the object it points to, are untouched:

Stack
r1 = 0xB200
r2 = 0xA100
Heap
Rectangle @ 0xB200
width2
height4
Rectangle @ 0xA100
width333
height10

Step 4 is where the paths split: r1's arrow now points at a brand-new object; r2's arrow never moved — it's still pointing at the same object r1 mutated back in Step 3, which is exactly why r2.width reads 333 and not 5.

The numbers tell the whole story in order: width starts at 5, becomes 333 the moment the dereference in Step 3 runs — visible through either variable, since there was only ever one object — and then r1 jumps to 2 in Step 4 while r2 stays at 333, because that last line never went anywhere near the object at all.

Connection ← §2.3

§2.3 introduced the dot as a dereference: follow the address, then act on what's there. Assignment (=) never does that — it only ever writes into the variable's own slot. Confusing the two is the most common source of surprise for new Java programmers.

§3.6

The Rule

Assignment always copies what the variable holds:

  • Primitive variable → copies the value. Two independent slots.
  • Reference variable → copies the address. Two names for one object.

There is no third case. Every assignment in Java — whether it appears as a declaration, a standalone statement, or inside a more complex expression — follows one of these two forms. Multiple reference variables can end up holding the same address — aliasing — and through that address they all see the same object.

Looking ahead to §4

When a method is called, the argument is assigned to the parameter. When the method returns, the return value is assigned to the receiving variable. These are ordinary assignments — the same two rules apply. Knowing this now makes method calls in §4 straightforward rather than surprising.
