Chapter 2

Classes & Objects

Beyond Primitive Values: Modeling the Real World

§2.1

Modeling the Real World

So far, we've worked with Java's built-in types: numbers, characters, text, and truth values. 

int age = 25;
double price = 19.99;
boolean active = true;

These types are incredibly useful, but they can only represent simple pieces of information. Real-world concepts are rarely that simple. Most interesting data is made of several things grouped together — the way a real-world kind of thing is described by its attributes. A rectangle has a width and a height. A point has an x and a y. A person has a name and an age.

Classes as Blueprints

Think about how an architect designs a house — not by building one directly, but by drawing up a blueprint first: its dimensions, the number of rooms, where the doors go. The blueprint itself is not a house; you can't live in it. But it's precise enough that a builder can follow it, and many separate houses can be built from that same one blueprint. Each has the same rooms in the same layout — but one is painted blue, another has hardwood floors, another has three bedrooms filled and one empty. Same structure, different values in each house.

Grouping a rectangle's width and height, or a person's name and age, works the same way in code. A class lets you define that grouping: the set of attributes that together describe one kind of thing. A class is a description of a kind of object: it names the fields that every object of that kind will have — the blueprint; individual objects are the houses built from it.

"Blueprint" is a fine way to picture what a class does. But there's a more fundamental sentence underneath it, easy to let slide past unremarked — and it's worth saying directly, once, rather than leaving it implied: 

 Built-in types   ->   User-defined types

  - int                 - Rectangle
  - double              - Person
  - boolean             - Point

Until now, every type in sight has been one Java ships with, built in, already there before a single line of your own code runs. A class is how that list stops being fixed. class Person { ... } doesn't configure an existing type or fill in a template Java already understands — it hands the compiler a type that didn't exist a moment ago.

Defining a class

Our first class uses only primitive fields:

class Rectangle {
    int width;
    int height;
}

This declares that every Rectangle will have two slots: width (an int) and height (an int). The class definition itself is a template — no Rectangle object exists yet, just the description of what one will look like.

The field declarations inside the class look exactly like the variable declarations from §1 — because they are. They're just variables that belong to an object. Once objects exist (§2.2), each one will have its own independent width and height slots. A blueprint on paper and an actual, buildable house are two different things — §2.2 is where that gap closes.

Class as a new type

Not a variable, not a value — a type. A variable's declaration can now have this new type, the same way it could always name primitive int age or double size. Once the class Person is declared, we can now declare a variable with this new type, which will hold the address of an object of this type: 
Rectangle rect;
int age;
double size;

§2.2

Instantiation — what new does

A blueprint on its own is just paper — nobody can live in it. To get an actual house, someone has to build one, using the blueprint as the plan. Java's version of "build one" is the new keyword: the class is the blueprint, and an object is an instance of that blueprint — a concrete chunk of memory laid out according to the class's field declarations. You create one with new: 

Rectangle rect = new Rectangle();

new does three things in sequence:

  1. Allocates a chunk of memory for fields.
  2. Sets defaults for all fields. For our Rectangle: width = 0, height = 0 — §2.5 covers exactly what "default" means for every field type, once constructors are on the table.
  3. Returns the address of the newly allocated chunk.

The object now exists on the heap. The address returned by new has been assigned to a variable named rect, which now holds an address. That's §2.3.

Each object created this way is independent — its own chunk, its own copy of every field.

Memory model: Where do objects live?

A Rectangle object isn't a single value — it is a collection of values. It has a width, and a heigth, bundled together into one object. Java stores them in a special area of memory called the heap. You can think of the heap as the program's storage area for objects.

§2.3

Reference Variables — a variable that holds an address

Imagine a house at 123 XYZ Street, NY — and a piece of paper with that address written on it. The paper is not the house; it's just ink on paper, small enough to fit in your pocket. But it tells you exactly where to go if you want to reach the house: walk to 123 XYZ Street, and there it is.

That's exactly what a variable holding an object looks like in Java. §1.8 said a variable holds either a value or an address — a reference variable is the second kind: it holds an address, the location of an object on the heap, the same way that paper holds an address rather than the house itself. 

Rectangle r = new Rectangle();

Read this as: allocate a Rectangle on the heap, and store its address in the reference variable r. r is that piece of paper; the Rectangle object is the house — sitting on the heap, at that address, never moving. r itself holds none of the house's rooms or furniture (width, height); it holds only the address, exactly the way a piece of paper holds only ink. The variable r does not contain a Rectangle. It contains a number — the memory address where that Rectangle lives.

Stack
r = 0x4a2f
0x4a2f is a memory address
points to
Heap
Rectangle @ 0x4a2f
width0
height0
A variable vs An object
A reference variable and an object are two completely different things that happen to be connected.

  1. The object stores the actual data.
  2. The reference variable gives us a way to refer to the object. Remember, the variable itself does not contain the entire object, just the address of it.

A note on terminology

From the language perspective, a Java variable of reference type stores reference, not address. The specification never promises an actual memory address is what's inside that slot, or that it works the way a raw pointer would in a programming language like C. This tutorial uses address anyway, deliberately, as a concrete mental model for what a reference points to and how it gets there.

The journey: From variable to the actual object it refers

To access the object's fields, you use the dot operator. The dot means: go to the address stored in the variable, then access the member named on my right. 

Rectangle r = new Rectangle();
r.width  = 10;  // go to the object r points to; set its width slot to 10
r.height = 20;  // go to the object r points to; set its height slot to 20

System.out.println(r.width); // dereference r, read its width slot → 10

Every use of the dot operator is a dereference: follow the address, then act on what's there.

§1.7 already named the two things a variable can hold — a value, directly, or an address, pointing elsewhere — without being able to show the second one in action. Now there's enough to show it side by side with the first: 

Primitive variable                Reference variable
          
  int age                         Person user
    │                                  │
    ▼                                  ▼
    20                     The address of a Person object

age's slot holds 20 directly — nothing to follow, nothing standing between the variable and the value. user's slot holds an address of an object of type Person. A primitive variable simply contains the actual data as its value, but a reference variable contains an address, effectively pointing to something else on the memory.

A reference variable that points to nothing

In the code itself, null is a literal — a fixed value spelled directly into the source (§1.4), exactly the way 10 or true are literals for other types. What that literal denotes is a special non-address value a reference variable is allowed to hold. It means exactly what it sounds like — this variable points to nothing at all. 

Rectangle r2 = null;   // r2 holds no address — there is no object to go to

r2 still holds a value here — it isn't somehow empty or uninitialized. That value is null, and null is not an object, nor an address you could follow to find one; it's simply the value that represents the absence of a reference. The same is true of assigning it later, not just at declaration: 

Rectangle r3 = new Rectangle();
r3 = null;   // r3 still holds a value — that value is now null

Dereferencing is impossible without an address to follow. Trying it anyway — r2.width when r2 is null — doesn't read a leftover value or fail silently; Java throws a NullPointerException at runtime.

What "throws an exception" means

An exception is Java's way of reporting that something went wrong while the program was running — not a compile error caught ahead of time, but a problem discovered mid-execution, with no sensible value to produce. When code throws one, normal execution stops right there: the current statement doesn't complete, and control jumps out looking for something to handle the problem. Left unhandled, the program stops and prints a description of what went wrong. A NullPointerException is one specific exception, thrown whenever code tries to dereference null. It won't be the last exception this tutorial mentions — for now, just recognize the shape: a name ending in Exception means the program hit a specific problem at runtime and stopped.

Multiple Objects, Independent State

Each call to new allocates a fresh chunk of heap memory. Two variables created this way hold different addresses, so they point to two separate objects — each with its own fields: 

Rectangle r1 = new Rectangle();
Rectangle r2 = new Rectangle();

r1.width = 10;
r2.width = 50;  // r1.width is still 10 — r1 and r2 are independent objects
Memory model

Two separate heap allocations, two separate Rectangle objects, two separate width/height slot pairs. r1 and r2 hold different addresses, so dereferencing one never touches the other.

Stack
Heap
r1 = 0x7f10
Rectangle @ 0x7f10
width10
height0
r2 = 0x7f88
Rectangle @ 0x7f88
width50
height0

Two different addresses in r1 and r2, two distinct boxes on the heap — that's why writing through one never shows up in the other.

That's the normal case: a new object per new. Assignment is where it gets more interesting — assign one existing reference variable to another, instead of calling new again, and something different happens:

Aliasing

When you assign one reference variable to another, you copy the address — not the object. 

Rectangle r  = new Rectangle();
r.width = 10;

Rectangle r2 = r;   // r2 now holds the same address as r

r2.width = 99;      // dereferences r2, modifies the same object

System.out.println(r.width); // 99 — r and r2 point to the same object

Both r and r2 hold the same address. They are two names for one object. This is called aliasing.

Contrast with primitives ← §1.3

int b = a copies the value — two independent slots. Rectangle r2 = r copies the address — two names for the same object. The difference flows directly from value vs. address (§1.7).

Assignment vs. the dot

Aliasing raises a question worth answering directly: once r2 and r1 hold the same address, does reassigning r1 to something else drag r2 along with it? It's an easy mistake to expect — the two variables were just linked, after all. Watch what happens when r1 is reassigned: 

Rectangle r1 = new Rectangle(5, 10);

Rectangle r2 = r1; // r2 gets its own slot; the address in r1 is copied into it

r1 = new Rectangle(5, 10); // no dereference; only reassignment.
                            // we never touch the heap here —
                            // we just change the value of r1's own slot to a new address.

System.out.println(r1.width); // 5
System.out.println(r2.width); // 5

Both lines print 5 — but don't let the matching numbers suggest r1 and r2 still agree about which object they mean. Follow it step by step:

Step 1 — Rectangle r1 = new Rectangle(5, 10);

Stack
r1 = 0xA100
Heap
Rectangle @ 0xA100
width5
height10

Step 2 — Rectangle r2 = r1; — r2 copies r1's address; both now point at the same object:

Stack
r1 = 0xA100
r2 = 0xA100
Heap
Rectangle @ 0xA100
width5
height10

Step 3 — r1 = new Rectangle(5, 10); — a brand-new object, so r1's address changes; r2's never does:

Stack
r1 = 0xB200
r2 = 0xA100
Heap
Rectangle @ 0xB200
width5
height10
Rectangle @ 0xA100
width5
height10

Step 3 is where the paths split: r1's arrow now points at a brand-new object; r2's arrow never moved — even though both objects happen to hold identical width/height values.

Line 4 (r1 = new Rectangle(5, 10)) is a plain assignment — the same reassignment §1.2 already covered, just with an address as the value instead of a number. It never dereferences anything, never touches the object at 0xA100, and never runs through the dot operator at all. All it does is overwrite what's sitting in r1's own slot — swapping one address for another. r2's slot was never part of that statement, so it goes on holding 0xA100, exactly as it did the line before.

r1.width and r2.width printing the same number is a coincidence of this particular example, not evidence the two variables still point together — they're reading from two different objects that happen to have been built with identical arguments. Change either object's field afterward, and the illusion breaks immediately: r2.width = 99 would leave r1.width untouched at 5, because by that point they were never the same object to begin with.

§2.4

Reference Fields — objects pointing to objects

Up to now, we've only used primitives as fields, like Rectangle's width and height. Now that we've seen how reference variables work, let's move on to fields that hold references too.

A rectangle naturally has two things beyond its dimensions: a position (where its top-left corner sits) and a border (how its edge is drawn). Both are structured values — each made of several primitives — so each becomes its own class: 

class Point {
    int x;
    int y;
}

class Border {
    int     thickness;
    boolean visible;
}

A field can be of any type — including a class, just declared as Point or Border the same way width was declared as int. A field of class type is called a reference field: a slot inside the object that holds an address (or null), exactly the way a reference variable does — the only difference is that this slot lives inside an object instead of inside a method. Now Rectangle can hold references to both: 

class Rectangle {
    int    width;
    int    height;
    Point  origin;  // reference field — holds an address, not a Point itself
    Border border;  // reference field — holds an address, not a Border itself
}

When a Rectangle is created, both reference fields are initialized to null ← §2.3 — the slots exist but point to nothing yet. 

Rectangle r = new Rectangle();
r.width  = 10;
r.height = 20;
// r.origin is null, r.border is null

Point p = new Point();
p.x = 5;
p.y = 8;
r.origin = p;    // copy p's address into r's origin slot

Border b = new Border();
b.thickness = 2;
b.visible   = true;
r.border = b;    // copy b's address into r's border slot
Memory model

Point and Border are separate objects on the heap, each at their own address. Rectangle doesn't contain either one — it just keeps their addresses, in its origin and border reference fields.

origin and border are themselves addresses, so the Rectangle object's own fields point further into the heap, to two more objects:

Stack
r = 0x9000
Heap
Rectangle @ 0x9000
width10
height20
origin0x4a2f
border0x7c10
Point @ 0x4a2f
x5
y8
Border @ 0x7c10
thickness2
visibletrue

The dot operator chains naturally across reference fields. Each dot is one dereference: 

System.out.println(r.origin.x);      // follow r → Rectangle; follow origin → Point;  read x         → 5
System.out.println(r.border.visible); // follow r → Rectangle; follow border → Border; read visible   → true
Connection ← §2.3

origin and border are reference variables that happen to live inside an object rather than in a method. The rules are identical: each holds an address, the dot dereferences it, and aliasing applies. If you write Border b2 = r.border, both b2 and r.border point to the same Border object.

§2.5

Constructors — initialization before the reference is returned

Order a plain, off-the-shelf desk and it shows up ready to use, built to the manufacturer's standard dimensions — nobody customized it for you, it just came assembled. Order a custom one instead, and you specify the height, the wood, the finish, before it's ever built. Either way, something runs to actually assemble the desk before it reaches you — the only difference is whether that assembly step used your specifications or its own defaults.

Creating an object works the same way. We call a class's constructor whenever we instantiate it — that's what's actually running underneath the familiar new syntax: 

Rectangle r = new Rectangle(); // this calls Rectangle's constructor

§2.2 already showed what this particular call does — allocate memory, set every field to its default value, return the address — just without naming it. That behavior is a constructor: the default constructor, supplied automatically because Rectangle hasn't written one of its own.

A constructor, in general, is initialization code that new runs after allocation, before the address is returned to the caller. new always calls one — there's no version of new that skips this step. The only question is which constructor runs: an implicit one Java writes for you, or an explicit one you write yourself. The default constructor above is the implicit case — taking no arguments, its entire job being to leave every field exactly at its default: 

class Rectangle {
    int width;
    int height;
}
Rectangle r = new Rectangle(); // the default constructor runs — no arguments to give it
// r.width  is 0
// r.height is 0

Default Values for Every Field Type

"Every field to its default value" means something different depending on the field's own type. For a primitive (§1.3), the default is always some form of zero or false. For a reference type, there's no zero to speak of — the default is null, an address that points to nothing at all (§2.3 already covered what a null reference actually is):

Type Default value
byte, short, int 0
long 0L
float 0.0f
double 0.0
boolean false
char '\u0000'
Any reference type — Rectangle, or any other class null

This is exactly what §2.2's allocation step already did, without naming it: every field's slot is set to its default (or to null) the moment the object exists, before the constructor body ever runs. The default constructor's entire job, as the code above shows, is to leave that default state untouched.

Fields only — never local variables

This automatic defaulting is specific to fields. A local variable (§1) gets no default at all — using one before it's assigned is a compile error, not a silent 0, because a local variable has no allocation step handing it a default slot the way a field does.

Defaults are rarely what you actually want, though. Most of the time, a new object should start out holding some specific state, not just defaults — a Rectangle with a real width and height, decided at the moment it's created. Writing a constructor of your own replaces the default one entirely: 

class Rectangle {
    int width;
    int height;

    Rectangle(int w, int h) {   // the class's own name, no return type
        width  = w;
        height = h;
    }
}
How a constructor is written

A constructor declaration looks just like a method declaration, except it uses

  1. the class's own name in place of a method name
  2. has no return type at all, not even void
Both differences are non-negotiable: this exact shape is how Java recognizes a constructor as a constructor, rather than as an ordinary method.

Now you pass initial values directly to new:

Rectangle r = new Rectangle(10, 20);
// r.width  is 10
// r.height is 20

The sequence new Rectangle(10, 20) executes as:

  1. Allocate a Rectangle on the heap; set its fields to their defaults.
  2. Run the constructor body with w=10, h=20.
  3. Return the address.

By the time r receives the address, the object is already initialized. The caller never sees the default state.

A field can also be given a value directly at its declaration, without waiting for the constructor to run: 

class Rectangle {
    int width;
    int height;
    String unit  = "px";
    String label = "Rectangle (" + unit + ")";

    Rectangle(int w, int h) {
        width  = w;
        height = h;
    }
}

unit and label never appear inside the constructor's own body, yet every Rectangle object still ends up with both set. That's because a field initializer — the = "px" part of a declaration — isn't really separate from the constructor at all.

Field initializers are syntactic sugar

The compiler lifts every field initializer into the constructor body itself, inserting each one at the top, in declaration order, before any code you actually wrote. The class above compiles as though it had been written: 

Rectangle(int w, int h) {
    this.unit  = "px";                        // ← from the field initializer, declared first
    this.label = "Rectangle (" + unit + ")";  // ← from the field initializer, declared second
    width  = w;                                // your code, unchanged
    height = h;                                // your code, unchanged
}

Declaration order is exactly why label's initializer is allowed to read unit: by the time it runs, unit's own initializer has already executed, one step earlier in that same lifted sequence. Swap the two field declarations, and label would be built from unit's own default value instead of "px" — a real bug, and one field initializers can introduce just as easily as any other ordering mistake.

Scope for now

One constructor, no chaining, no inheritance — just the core idea. The essential point: new = allocate + initialize + return address. And the moment a class defines a constructor of its own, the automatic default one goes away — a class only ever gets it for free when it defines no constructor at all.

§2.6

Static Fields — state that belongs to the class, not the object

§2.1 described a class as an architect's drawing: not a house, but a real, precise thing in its own right — dimensions, room count, door placement, sitting on paper somewhere, distinct from any house built from it. The same is true here. Rectangle the class is itself a real thing, loaded once, the first time it's needed — before any Rectangle object exists. Think of it as building a mothership, once, before any of the individual ships (the objects from §2.2–§2.5) exist. A static field is state that lives on the drawing itself — on the class — instead of on any one house built from it. Every object shares the exact same one. 

class Rectangle {
    static int count = 0;   // lives with the class itself, not repeated on each Rectangle object

    int width;               // one slot per instance
    int height;              // one slot per instance

    Rectangle(int w, int h) {
        width  = w;
        height = h;
        count++;             // every Rectangle updates the one shared slot
    }
}

Here's one Rectangle being created:

Rectangle r1 = new Rectangle(10, 20);
Stack
r1 = 0x9000
Heap
Rectangle @ 0x9000
width10
height20
Class Rectangle
count1

Notice the second arrow is different in kind from the first. r1's arrow is an address: dereference it, and you're standing inside r1's own object, reading width and height directly. The arrow from the Rectangle object to Class Rectangle isn't an address stored anywhere — no field holds it, nothing was assigned. It just shows that every Rectangle is an instance of the class, and so can reach the one shared count that lives there. count is read and written by name (Rectangle.count), not reached through r1 at all — the same one slot, whether zero Rectangles exist or a thousand do.

Create a few more, and watch count track all of them, not just one: 

Rectangle r2 = new Rectangle(5, 5);
Rectangle r3 = new Rectangle(2, 8);

System.out.println(Rectangle.count); // 3

Notice the access: Rectangle.count, through the class name. count doesn't belong to r1, r2, or r3 — it belongs to Rectangle itself. Any object's constructor can change it, but there is exactly one of it, not one per object.

The exception to §2.3 ← §2.3

§2.3 established that every object gets its own independent copy of its fields — that's what "Multiple Objects, Independent State" means. static is the deliberate exception: mark a field static, and every object shares that one slot instead of getting its own. width and height are per-object; count is per-class.

Memory model

count lives with the class itself, not on the heap alongside any Rectangle object. Creating a Rectangle allocates a new chunk for width and height — but there is nowhere in that chunk for count to live, because it was never part of any object's layout. All three Rectangles read and write the exact same slot, the one that's lived on the class since it was loaded.

The same keyword works on methods, for the same reason: belongs to the class, not to any instance. That's §4.2.
