Chapter 18

Collections Framework

An array (§6) is a group of values — but a fixed-size one, decided the moment it's created. This chapter is about a group that can grow and shrink as the program runs, and about the family of interfaces built on top of that one simple idea, each naming a different rule for how the group behaves.

§18.1

The Problem — what an array doesn't give you

What an array actually gives you

§6 already covered arrays; let's be precise about exactly what that gives you, because everything in this section is really about where that precision runs out.

An array is a fixed-size, contiguous block of memory, indexed by position, giving you constant-time access to any element — instant, regardless of how large the array is — as long as you know its index. If it's an array of a reference type, every slot holds a reference to an object (or null); if it's a primitive array, the values themselves sit inline in that contiguous block.

That's the entire contract. Notice what is not part of it:

  • No notion of "how many elements are actually in use" versus how many slots exist.

    array.length tells you the capacity, not the count of meaningful elements. You track that yourself, usually with a separate int size variable.

  • No actual insertion or removal operation.

    There are a fixed number of positions, created once and never revisited — all an array lets you do is set the value already sitting in a slot, never add a new slot or take one away. That's a different act entirely from changing the collection's size: writing guests[2] = "Ana" replaces whatever index 2 already held, it doesn't make the array one element bigger. "Inserting" in the middle, if you want the appearance of one, means manually shifting every subsequent element over by hand — the array itself never grows to make room.
  • No search operation beyond a linear scan you write yourself.

  • No concept of a chosen key.

    The index acts like one, but it's assigned by position, not by meaning — always a contiguous int starting at 0, never something you pick yourself, like an order ID or a person's name.

None of this is a flaw — an array is doing exactly one job, and doing it well: giving you raw, fixed, indexed storage. The friction shows up when your problem asks for something the array's contract was never designed to answer.

Five stories that strain five different properties

Abstract descriptions don't stick the way a concrete story does, so let's walk through five small stories. Each one strains a different property of the array — not the same complaint five times.

Scenario A — fixed number of slots.
You're building the guest list for a meetup. You declare an array sized for 5 people. Four guests arrive and take their seats, one after another. One slot remains, at index 4. Then a fifth guest arrives — but she's brought her daughter along, unannounced. That's two people showing up together, and there's only one slot left. There's no way to fit two people into one slot, so neither of them gets added — they leave without entering. If you tried to force it anyway, writing a second name in once the mother had taken the last slot, you'd hit an out-of-bounds exception — that next index was never created in the first place. The array's size was a decision made the moment it was created, and it simply cannot be revisited — there is no "add one more slot" operation, only "build an entirely new, larger array and copy everything over." 

String[] guests = new String[5];

guests[0] = "Priya";
guests[1] = "Tom";
guests[2] = "Ana";
guests[3] = "Leo";
// one slot left, at index 4

// a guest arrives with her daughter — two people, one slot:
// guests[4] = "mother";    // she could take the last slot...
// guests[5] = "daughter";  // ...but the daughter has nowhere to go

// ArrayIndexOutOfBoundsException: Index 5 out of bounds for length 5

Now picture the opposite guess for a different meetup: the same size is declared, but this time only 3 people actually show up. The remaining two slots hold nothing — allocated the moment the array was created, whether or not they're ever filled. The array doesn't know how many guests actually arrived; it only knows how big it was told to be, and it will report that size forever, regardless of how many slots are genuinely in use. That's the friction, from both directions: the array's size is a one-time guess, made before you know the real count, and it's wrong exactly as often as your guess is — too small, and you overflow; too large, and you waste space that's allocated but never used. 

String[] guests = new String[5];

guests[0] = "Priya";
guests[1] = "Tom";
guests[2] = "Ana";
// only 3 guests show up — guests[3] and guests[4] stay null

System.out.println(guests.length); // 5 — reports the size, not the count

Scenario A, continued — learning the actual size, in use
This raises a natural question: how do you even find out how many guests actually showed up? The array's length will always tell you the same number — the declared capacity, not how many of its slots are genuinely holding a guest. If you want the real count, you have to track it yourself: typically a separate int variable, incremented every time a guest is added, that has to be kept perfectly in sync with the array by hand, for as long as the array is in use. The array holds the data; knowing how much of it is actually meaningful is entirely your responsibility. 

String[] guests = new String[5];
int guestCount = 0;

guests[guestCount++] = "Alice";
guests[guestCount++] = "Bob";

System.out.println(guests.length); // 5 — always the declared capacity
System.out.println(guestCount);    // 2 — the real count, tracked by hand

Scenario B — a club membership list, and Bob leaves.
You're keeping a club membership list, sized for 5.

  • Mike joins first, then Bob,then Alice.
  • Later, Bob wants to quit. So you want him gone from the register.
But there is no remove operation on an array — it doesn't exist, the slots are fixed. The nearest thing you can do is overwrite his slot with null, which doesn't shrink anything: the array's length is exactly the same before and after. All that's changed is what index 1 holds — it went from his name to null. The count of slots is exactly the same; one of them is just empty now. The array itself has no concept of "one fewer member" — only "this slot's value changed." That's the friction: no true removal, only overwrite. 
String[] members = new String[5];

members[0] = "Mike";
members[1] = "Bob";
members[2] = "Alice";

// Bob quits — there's no remove():
members[1] = null;

System.out.println(members.length); // 5, unchanged

Scenario C — the same club, and every member is unique.

  • Mike joins first, inserted at index 0
  • Bob joins next, index 1.
  • Alice joins, index 2.
  • Now Mike tries to join. Again
The actual requirement here is simple to say out loud: you can't be a member of this club twice. The array itself has no idea what "already a member" means — it's not that the check is impossible, it's that the array doesn't do it for you. It will happily let a second copy of his name land at index 3, and now he occupies two different positions, as if he were two different people, because nothing about the array's contract includes enforcing "already a member." The programmer can absolutely know the rule and check for it — but that check has to be written by hand, every time, before every insertion: look through every existing slot, compare it against the new name, and only then decide whether it's allowed in. The requirement is "you can only be a member once"; the array offers no help with it at all — the entire burden of enforcing it sits on the code you write around the array, not on the array itself. 
String[] members = new String[5];

members[0] = "Mike";
members[1] = "Bob";
members[2] = "Alice";

// Mike tries to join again — the array won't stop him:
members[3] = "Mike";  // now "Mike" sits at both index 0 and index 3

// enforcing uniqueness is on you, not the array:
boolean alreadyMember = false;
for (String member : members) {
    if ("Mike".equals(member)) {
        alreadyMember = true;
        break;
    }
}

Scenario D — finding a member by name, not by seat.
An array is perfectly happy answering questions about position: who's sitting in the first slot, whose name is in the third. That's direct and instant, because position is what the array was built to answer. But now someone asks a different kind of question: Is Bob still an active member?

That's not a question about a slot index at all — it's a question about the value. With an array, you have no way to ask that directly — index (position) is the only handle you have on an element, and a name isn't a position. Going from Bob's slot position to his name is immediate. However, if you are asking about the contents, you'd have to scan through every slot yourself, comparing each one against the name you're looking for, just to answer a question that feels like it should be instant. That's the friction: the array only ever answers "What's at this position?" never "Where is this content?" Only if the search key is the index, then comes the answer immediately. 

String[] members = new String[5];
members[0] = "Mike";
members[1] = "Bob";
members[2] = "Alice";

// position -> name is instant:
System.out.println(members[1]); // "Bob"

// name -> position takes a manual scan:
int index = -1;
for (int i = 0; i < members.length; i++) {
    if ("Bob".equals(members[i])) {
        index = i;
        break;
    }
}

Scenario E — a new member has to go in at a specific spot (with room to spare).
Say the club keeps its register in a specific order, and for whatever reason — seniority, a numbered badge, whatever the club's rule is — a new member named Sam has to be inserted at index 2, not just tacked on at the end. The register currently holds four members, and the array was sized for 5, so index 4 is already sitting there, empty and ready.

There is no real "insert at index 2" operation on an array. To make Sam actually land at index 2 and push everyone from there onward one step back, you have to do it by hand, working from the end backward so you don't overwrite anything before you've copied it. It works cleanly here because the slot at the far end — index 4 — already existed. The shifting is manual, but there was always somewhere to shift into. 

String[] members = new String[5];
members[0] = "Mike";
members[1] = "Bob";
members[2] = "Alice";
members[3] = "Priya";
// index 4 is already free

// shift from the end backward, then insert:
members[4] = members[3]; // Priya -> index 4
members[3] = members[2]; // Alice -> index 3
members[2] = "Sam";      // Sam lands at index 2

Scenario E, continued — the same story, but no space to spare.
Now suppose the register had been sized exactly for however many members had joined so far, with no index 4 at all. Insert Sam at index 2 again, and the same shifting logic applies, but it hits a wall immediately: the last member can't move one slot further, because that slot doesn't exist. The whole insertion is impossible until you first build a bigger array and copy everyone into it. That's the extra step this version of the story adds: growing the array is a separate, necessary step before any shifting can happen at all — the manual shift from Scenario E only works when there's already a slot to shift into, and here, there isn't one yet. 

String[] members = new String[4];
members[0] = "Mike";
members[1] = "Bob";
members[2] = "Alice";
members[3] = "Priya";
// full — no room to insert Sam at index 2

// grow into a new, larger array first:
String[] bigger = new String[5];
System.arraycopy(members, 0, bigger, 0, members.length);
members = bigger;

// now shift from the end backward, then insert:
members[4] = members[3]; // Priya -> index 4
members[3] = members[2]; // Alice -> index 3
members[2] = "Sam";      // Sam lands at index 2

Five different stories, five genuinely different gaps — resizing, gap-closing, uniqueness, keyed lookup, insertion at a target position. None of them are the same problem in different clothes.

§18.2

The Big Picture — requirement axes and natural families

Generalizing the pain into requirement axes

Each scenario above was really asking a question about the domain — not about Java. If you strip the Java away entirely, any problem involving "a group of things" forces you to answer some subset of these questions, regardless of what language you're in:

  • Does order matter, or only membership? A shopping cart cares about order (you scroll through items top to bottom); a set of unique tags on a blog post usually doesn't.
  • Are duplicates meaningful, or should they collapse? A list of exam scores keeps every duplicate (two students can both score 87); a collection of "unique visitor IDs today" should never have the same ID twice.
  • Do you need positional access, or only iteration? "Give me the 3rd element" is a different requirement from "let me walk through all of them, I don't care in what order I get access."
  • Do you need to associate a value with a key, rather than a position? "Get me the student with ID 4471" is fundamentally different from "get me the element at index 4471" — one is content-addressed, the other is position-addressed.
  • Do you need automatic ordering — sorted, or insertion-order — or no ordering guarantee at all? Sometimes you want elements to come back out in the order you put them in; sometimes you want them always sorted; sometimes you genuinely don't care, and that indifference is itself useful information because it lets an implementation optimize for something else instead.
  • What operation dominates?
    • Containment checks?
    • Random access by position?
    • Insert/remove at the ends?
    • Insert/remove in the middle?
    This is the question that ultimately decides which performance tradeoffs are acceptable, because no single data structure is fastest at everything simultaneously.

None of these questions mention Java. They're domain questions — the same ones a designer in any language, or even someone designing a physical filing system, has to answer. That's the ground worth standing on before looking at any API: every collection type is really just a settled answer to a particular combination of these questions.

The axes cluster into natural families

Here's the payoff for having done that groundwork: once you have the axes, the standard collection categories aren't a list to memorize — they fall out as natural clusters of answers.

  • Care about order, allow duplicates, want positional access → this is a List. It's the closest conceptual descendant of an array: still ordered and index-addressable, but now with insertion, removal, and automatic resizing handled for you.
  • Care about uniqueness, don't care about position → this is a Set. It answers scenario C directly: Mike can't join the club twice, because the collection itself rejects him the second time — "you can only be a member once" stops being something you enforce by hand on every insertion.
  • Care about processing order — first-in-first-out, or last-in-first-out — rather than positional access → this is a Queue or Deque. You're not asking "give me element 4," you're asking "give me the next one to process."
  • Care about lookup by key rather than by position or identity → this is a Map. It answers scenario D: you ask for "Bob" and get back whether he's a member directly — content-addressed retrieval, keyed rather than indexed.

One important precision point here: a Map is not a Collection, in the strict Java type-hierarchy sense — it doesn't extend the Collection interface at all, because holding key–value pairs is a different contract from holding a simple group of elements. It's conceptually part of the same family — the same "answer one of these domain questions" exercise — but the type hierarchy keeps it structurally separate. Worth being precise about that distinction now, so it doesn't need correcting later.

The Java Collections Framework as a systematic answer

With that groundwork in place, the Java Collections Framework (JCF) itself needs surprisingly little introduction — you can now name what it is rather than list what it contains:

The JCF is an interface hierarchy that codifies these behavioral contracts, paired with a set of implementations that satisfy each contract using different internal data structures, each with its own performance tradeoffs.

That's a familiar shape if you think back to OO design generally: separating what something does (the interface — the contract) from how it does it (the implementation — the internal data structure). List is a contract: ordered, duplicates allowed, positional access. ArrayList and LinkedList are two different implementations of that same contract, making different tradeoffs — one is backed by a resizable array (fast random access, slower middle insertion), the other by a doubly linked list (fast insertion/removal at known positions, slower random access). Same contract, different internal machinery, different performance profile.

That "contract vs. implementation" separation is the single most important architectural idea in this whole framework — everything else is detail layered on top of it.

Why this design pays off

One brief, practical point worth previewing here, even though the full picture comes later: because algorithms and APIs are typically written against the interface rather than a specific implementation, code you write against List<Student> works unchanged whether the underlying object is an ArrayList or a LinkedList. Collections.sort(someList) doesn't care which one it received. This is precisely why "which collection should I use here?" is really shorthand for "which of the requirement axes above dominates in this particular case?" — and that question is exactly what the rest of this chapter answers, implementation by implementation.

Those four intentions — order, processing order, membership, pairing — map onto three interfaces branching directly from Collection, plus Map sitting apart from it entirely:

Everything in the rest of this chapter is one of these three intentions, or a refinement of one.

§18.3

SequencedCollection — a stable, queryable first and last

"Sequence," in everyday usage — and even in some mathematical usage — can just mean "a list of things, one after another, where you could in principle assign positions," without that arrangement needing to be meaningful, stable, or guaranteed. Any iteration produces a sequence in that trivial sense, ordered or not, stable or not: "I got element A, then B, then C" is itself a sequence, even if next time it'd be B, then A, then C. Collection.iterator() is specified to return elements one at a time, but its documentation deliberately says nothing about what order — it doesn't promise the order is stable, doesn't promise it matches insertion order, doesn't even promise it's the same across two separate calls.

SequencedCollection's name isn't about "can this be iterated" — every Collection already can be. It's about a guarantee sitting on top of iteration: that there's a stable, defined first element and last element at any given moment, queryable directly — getFirst(), getLast() — without iterating anything at all, and that stays consistent with whatever the iteration order actually is.

What Problem It Solves

Picture a helpdesk storing its open support tickets in a HashSet<Ticket> (§18.7), chosen for fast contains() checks — "has this ticket already been filed?" Iterating that set absolutely produces some sequence of tickets. But nothing guarantees the first ticket that iteration yields today is the same "first" ticket tomorrow, or after some unrelated ticket gets resolved and removed, or on a different run of the JVM entirely. There's a sequence, in the trivial sense above, but no defined "first" or "last" to ask about independent of walking the whole set. SequencedCollection is precisely the set of types where "first" and "last" are first-class, queryable, stable concepts — not just whatever the iterator happens to emit first.

The Contract: Stable Endpoints

SequencedCollection<E> standardizes exactly the endpoints, nothing more:

There is a well-defined first element and a well-defined last element, and the collection can be traversed in reverse.

That's the entire contract. "First" simply means "at the front of the sequence"; "last" means "at the end." Both are statements about position, not about which value happens to be smallest, largest, or most recently added — position is all the interface promises to define. A helpdesk queue backed by a plain ArrayList already satisfies it, since a List's order is exactly its element positions (§18.4): 

SequencedCollection<String> ticketQueue =
        new ArrayList<>(List.of("T-101", "T-102", "T-103"));

ticketQueue.getFirst(); // "T-101" — the ticket support handles next, no iteration needed
ticketQueue.getLast();  // "T-103" — the most recently filed ticket

ticketQueue.addLast("T-104"); // a new ticket joins the back of the queue

for (String ticket : ticketQueue.reversed()) {
    System.out.println(ticket); // T-104, T-103, T-102, T-101 — newest first, for a status dashboard
}

reversed() hands back a view, not a copy — walking it newest-first for the dashboard doesn't require rebuilding the queue or reversing anything by hand.

Encounter Order

What "first" and "last" actually mean for a given collection depends on that collection's encounter order — the order iteration visits elements in. Different implementations define it completely differently, and SequencedCollection doesn't invent an order of its own; it only guarantees that whatever order already exists has queryable, stable endpoints:

  • For a List (§18.4), encounter order is the elements' index positions — exactly what the ticket queue above relies on.
  • For a LinkedHashSet (§18.7), encounter order is insertion order — useful if the helpdesk wanted to reject duplicate ticket IDs while still processing tickets in the order they arrived.
  • For a TreeSet (§18.7), encounter order is whichever order its sorting defines — natural ordering, or an explicit Comparator — not insertion order at all.
  • A HashSet (§18.7) has no defined encounter order. Nothing in the specification promises any particular iteration order, which is exactly why HashSet doesn't implement SequencedCollection in the first place — there's no stable "first" for getFirst() to return.

If the encounter order changes — for collections that support reordering — the first and last elements change right along with it. §18.7 shows this vividly with a TreeSet: since its encounter order comes entirely from its sorting, not insertion history, adding a single new element can change what getFirst() returns even though nothing was ever explicitly moved to the front.

"First" and "last" are always relative to whatever encounter order the underlying collection already defines. For the ArrayList-backed queue above, that meant arrival order. SequencedCollection just guarantees that, whichever order a given collection defines, its two ends are always a direct query away instead of something only iteration could reveal.

§18.4

List — positional order, open access

Every element in a List has an index, just like an array (§6) — 0, 1, 2, ... — and that index is what defines a List's order. Think of a row of numbered mailboxes: any box is reachable directly, by its number, with no need to check the boxes in front of it first. 

list.get(0); // first box
list.get(4); // fifth box — directly, no traversal

Duplicates are allowed, because nothing about having an index cares whether two elements happen to be equal — two different positions can hold identical values without conflict, the same way two mailboxes could both contain the same letter.

Concrete Implementations

Three classes implement List, and the choice between the two modern ones comes down entirely to what each is actually built from underneath:

Class Built on Fast at Slow at
ArrayList a resizable array reading any index directly — get(i) — and adding or removing at the end inserting or removing anywhere else — everything after has to shift
LinkedList a doubly-linked chain of nodes inserting or removing at either end reading an arbitrary index — has to walk the chain from one end
Vector a resizable array, like ArrayList — everything, due to synchronization overhead most programs never need — legacy; §18.6 covers why

Adding at the end is fast for a subtler reason than reading is: the backing array usually has spare room already, so appending is just writing into the next empty slot — no shifting needed. Occasionally that spare room runs out, and ArrayList has to allocate a bigger array and copy everything into it, an expensive step — but that only happens occasionally, and averaged out over many additions, appending still works out to fast, constant-time behavior overall (what's usually called amortized O(1)). Inserting or removing anywhere else has no such shortcut: every element after the change has to shift, every single time, without exception.

Every List, regardless of which class backs it, shares the same basic vocabulary: 

List<String> fruits = new ArrayList<>();

fruits.add("apple");
fruits.add("banana");
fruits.add("cherry");

fruits.get(1);              // "banana" — read by index
fruits.set(1, "blueberry"); // replaces "banana" at index 1
fruits.add(1, "kiwi");      // inserts at index 1, shifting "blueberry" and "cherry" right

fruits.contains("kiwi");    // true
fruits.indexOf("kiwi");     // 1
fruits.size();              // 4

fruits.remove(0);           // removes by index — "apple" is gone
fruits.remove("cherry");    // removes by value — a different, overloaded method

for (String fruit : fruits) {
    System.out.println(fruit);
}
A real gotcha with List<Integer>

remove(int) and remove(Object) are two different, overloaded methods (§4.2) — one removes by index, the other by value. For a List<String> there's no ambiguity, since a String can never match the int overload: 

List<String> fruits = new ArrayList<>(List.of("apple", "banana", "cherry"));
fruits.remove("banana"); // only remove(Object) could ever match — unambiguous
System.out.println(fruits); // [apple, cherry]

For a List<Integer>, it's a genuine trap — both overloads are now callable, and Java always prefers the more specific one, int, whenever the argument fits: 

List<Integer> numbers = new ArrayList<>(List.of(10, 20, 30));
numbers.remove(1);           // calls remove(int) — removes the element AT index 1
System.out.println(numbers); // [10, 30] — 20 is gone, not "1"

numbers.remove(Integer.valueOf(1)); // forces remove(Object) — removes the value 1, if present
System.out.println(numbers);        // [10, 30] — unchanged, since 1 was never in the list

numbers.remove(1) never removes the value 1 — it removes whatever sits at index 1, every time, because int is the more specific overload and Java always picks it when the argument is a plain int. Removing the integer value specifically requires forcing the other overload with Integer.valueOf(1), wrapping it so it can no longer match remove(int).

§18.5

Defining an Order — Comparable and Comparator

Lining kids up for a school photo by height works because "taller" already has an obvious answer — anyone can look at two kids and settle it, no rule needs to be agreed on first. Now try the same thing with a shelf of books: how do you sort them? By title, alphabetically? By author's last name? By genre, then title within each genre? By the year they were published? None of these is more "correct" than the others — a library could reasonably pick any of them — which is exactly the point: books don't come with a built-in "comes first," the way height does. Someone has to choose a rule before "first" means anything at all, and different reasonable people, wanting different things from the shelf, could choose differently. Every kind of sorting, underneath, keeps asking the same repeatable question about two things at a time — which comes first? — but not every answer is already sitting there waiting to be read off, the way height is.

Java's < and > work directly on numbers the same effortless way height works for kids in a line — no rule to define, the comparison already means something. But an object made up of several fields is exactly like that shelf of books: nothing about it tells Java which of two instances comes first, because nothing has decided what "first" should even mean for that type — by which field, in which direction. That has to be defined, explicitly, the same way a class opts into equality and hashing by overriding equals() and hashCode() (§16.1) — Java doesn't guess an order any more than it guesses equality.

Picture a library's shelf, kept as a plain list of the books on it, that the librarian wants sorted by title — the default way most people expect to find a book on a shelf. Sorting needs exactly the rule just described, and by default, there isn't one: 

record Book(String title, String author, int year) {}

List<Book> shelf = new ArrayList<>(List.of(
        new Book("The Pragmatic Programmer", "Hunt", 1999),
        new Book("Clean Code", "Martin", 2008),
        new Book("Effective Java", "Bloch", 2001)));

Collections.sort(shelf); // compile error: Book doesn't implement Comparable<Book>

This doesn't compile, and deliberately so — Collections.sort(List<E>) is only defined for an E that already knows how to order itself. Book, as written, doesn't: it's just a bundle of a title, an author, and a year, with nothing saying which field, if any, determines the order — the exact question the shelf raised a moment ago. Fixing that means implementing Comparable, the same way a class opts into a default order that equals() and hashCode() (§16.1) opt it into equality and hashing. Its entire contract is a single method: 

public interface Comparable<T> {
    int compareTo(T o);
}

The type parameter T is what keeps this useful — a Book implementing Comparable<Book> gets a compareTo(Book other), not a compareTo(Object other), so there's never a cast to write or an unrelated type to accidentally compare against: 

record Book(String title, String author, int year) implements Comparable<Book> {
    @Override
    public int compareTo(Book other) {
        return title.compareTo(other.title); // alphabetical by title
    }
}

Book cleanCode = new Book("Clean Code", "Martin", 2008);
Book effectiveJava = new Book("Effective Java", "Bloch", 2001);

List<Book> shelf = new ArrayList<>(List.of(
        new Book("The Pragmatic Programmer", "Hunt", 1999), cleanCode, effectiveJava));

Collections.sort(shelf);
for (Book b : shelf) {
    System.out.println(b.title()); // Clean Code, Effective Java, The Pragmatic Programmer
}

title.compareTo(other.title) is just delegating to String's own compareTo(), which already knows how to alphabetize. That's not a coincidence or a special case — plenty of Java's own built-in types already implement Comparable themselves: String implements Comparable<String>, ordering alphabetically, and Integer implements Comparable<Integer>, ordering numerically — the exact same interface Book just implemented. That delegation is why compareTo() reports order as one of three outcomes, the same convention it inherited from String:

  • Negative — this comes before the argument. cleanCode.compareTo(effectiveJava) returns a negative number, since "Clean Code" sorts before "Effective Java".
  • Positive — this comes after the argument. effectiveJava.compareTo(cleanCode) returns a positive number, since "Effective Java" sorts after "Clean Code".
  • Zero — the two are equal for ordering purposes. cleanCode.compareTo(cleanCode) returns zero, since a title can't sort before or after itself.

Book now has exactly one built-in order — by title — and every piece of code that sorts a collection of books without saying otherwise will get that same order, every time.

Comparator: An Order From Outside the Class

Comparable bakes in exactly one order, chosen by whoever wrote the class. But the library's shelf isn't the only reasonable way to look at these books — a reading list for a book club might want them sorted by author instead, and rewriting compareTo() isn't an option, since a class can only implement Comparable one way. A Comparator supplies an order from outside the class instead, as a completely separate object. Its contract is just as small: 

public interface Comparator<T> {
    int compare(T o1, T o2);
}

Same negative/positive/zero convention as compareTo(), just spelled with two arguments instead of one, since a Comparator isn't a method living inside the class being compared — it's handed both objects from the outside. Since it's just an interface, a Comparator can be supplied the same way any interface can be, without a named class built specifically for it — as an anonymous class (§13.6), declared and instantiated in one expression, right where it's needed: 

Comparator<Book> byAuthor = new Comparator<Book>() {
    @Override
    public int compare(Book b1, Book b2) {
        return b1.author().compareTo(b2.author());
    }
};

shelf.sort(byAuthor);
for (Book b : shelf) {
    System.out.println(b.author()); // Bloch, Hunt, Martin — title order ignored entirely
}

new Comparator<Book>() { ... } doesn't instantiate the interface directly — nothing can — it declares an unnamed class implementing Comparator<Book> and creates its one and only instance, all in the same expression, exactly the way an anonymous class works for any other interface. Nothing about Book itself changed; the ordering lives entirely in this one throwaway Comparator object, which is exactly why the same class can have as many of these outside orderings as different call sites need — one per question the shelf could reasonably be asked to answer.

Comparator's single-method shape also makes it a functional interface — Java's later lambda and method-reference syntax (§19) can stand in for exactly this pattern, more tersely than an anonymous class, but the underlying idea is identical: an object that implements compare(), supplied to whatever needs an order.

Preview: the same Comparator, in one line

§19 covers lambdas and method references in full, but it's worth seeing what they do to the anonymous class above, once they're available. A lambda shrinks the whole thing to one expression — the parameter list, then the same comparison, with compare()'s name and signature inferred rather than spelled out: 

Comparator<Book> byAuthor = (b1, b2) -> b1.author().compareTo(b2.author());

And since all that lambda does is pull one value out of each object and let Comparable compare it, a method reference shrinks it further still, through a Comparator.comparing(...) factory method built for exactly this pattern: 

Comparator<Book> byAuthor = Comparator.comparing(Book::author);

Both compile down to an object implementing compare(), exactly like the anonymous class — just with the ceremony gone.

This isn't only for List.sort(). Anywhere Java needs to keep things in order without being told how each time, it reaches for this exact mechanism: TreeSet (§18.7) uses an element's compareTo() — or a Comparator handed to its constructor — to stay continuously sorted, and TreeMap (§18.8) does the same for its keys.

§18.6

Queue — order, not position

A Queue asks a completely different question than a List does.

  • It is not "What is the third element?"
  • It is: "What's next?"

A queue is a collection that holds items temporarily and releases them in an order determined by a policy, built around three concerns:

  • how items enter (which end, and whether the queue reorders on insert),
  • how they wait, and
  • how they leave (who gets out next, and under what condition).

Reduced to what a caller actually does with one, a Queue offers exactly two kinds of operation: submit work (add, offer) and request the next item (remove, poll).

What "next" actually means is answered differently by different implementations:

Structure What does "next" mean?
FIFO queue The first item added (longest waiting item)
Stack (LIFO) The last item added
Priority queue The highest- (or lowest-) priority item present, by a supplied comparator or natural ordering — insertion order plays no role at all

FIFO and LIFO as Abstract Data Types

FIFO (first in, first out) and LIFO (last in, first out) are abstract concepts — pure access discipline, with no implementation detail attached. They're mathematical abstractions: nothing about either one says anything about arrays, linked lists, or any concrete structure at all.

In computer science theory, queue and stack are abstract data types (ADTs): a name, a set of operations, and a contract. An ADT defines what operations are available and what behavior is guaranteed — never how. Nothing about gates, ends, heads, or tails; that's an implementation concern, entirely separate from the contract itself.

ADT Discipline Operations
stack LIFO — strictly push, pop, peek
queue FIFO — strictly enqueue, dequeue, peek

"Strictly" is doing real work in that table: the theoretical stack and queue never allow anything except their one discipline. Java's actual types, as the next table shows, are considerably looser.

Where Java Falls Short of the Theory

Java's named types don't always enforce the discipline their name promises:

Java type Extends Enforces its discipline?
Stack Vector No — index methods leak through
Queue Collection No — FIFO isn't guaranteed (PriorityQueue is a counterexample)
PriorityQueue Queue No — not FIFO, despite being a Queue

The Queue row needs unpacking, because the mismatch there is easy to miss. In CS theory, "queue" on its own always means FIFO — that's not a convention, it's the definition. A priority queue is a genuinely different abstract data type: its defining operations are "insert" and "give me the best one," not "insert" and "give me the oldest one." Theory keeps these as two separate, unrelated ideas that simply happen to share a word in their name.

Java doesn't keep them separate. Queue in Java is one interface covering both disciplines at once, and PriorityQueue is a full, legitimate implementer of it. That means a variable declared as Queue<E> gives no guarantee at all about which discipline is underneath — FIFO, or priority-based — despite "queue" in theory always meaning the former.

Stack is the clearest case of the other kind of mismatch. It was introduced in Java's very first version, and it extends Vector — a synchronized, resizable array, exactly like ArrayList. §9.3 already established what extends actually means: a subclass inherits every public method its superclass has, not just the ones that happen to fit. Stack inherits all of Vector's index-based methods along with it, which means the discipline "only ever touch the top" is not enforced at all — anyone holding a Stack can reach into the middle: 

Stack<Integer> stack = new Stack<>();
stack.push(1);
stack.push(2);

stack.add(0, 99); // legal — Stack IS-A Vector, and Vector allows this
stack.get(1);      // legal — direct access to any index
stack.remove(0);   // legal — remove from any index, not just the top

Stack's name makes a semantic claim — "I enforce LIFO discipline" — that its own extends clause breaks. This is a real, permanent design mistake, not a subtle edge case: reusing Vector's storage through inheritance (§9.3) came bundled with reusing all of its behavior, whether that behavior fit the abstraction being built or not.

A lesson already available from §12 and §13

Stack's mistake was reaching for extends — an is-a claim (§9.1) — to reuse Vector's storage, when what it actually needed was to expose only the methods that belong to "stack," and hide the rest. An interface (§13.3) does exactly that: it names a capability without dragging along an unrelated implementation's full public surface. The next part shows the type Java eventually introduced to fix this properly.

Deque — Symmetric Access, Both Ends

A Queue gives asymmetric access: insert at the tail, inspect and remove at the head. A Deque ("deck," double-ended queue) gives symmetric access instead — two independent ends, each supporting both insertion and removal:

  • addFirst / removeFirst at the head
  • addLast / removeLast at the tail

Deque<E> extends both Queue<E> and SequencedCollection<E> — an interface extending more than one other interface at once, exactly as §13.10 already covered. A Deque is-a Queue, and can do everything a Queue does, but it's a strictly more general structure sitting on top.

That generality means a Deque can serve either role the earlier table described. As a FIFO queue, the idiomatic route is simply Queue's own inherited methods: 

Deque<Integer> queue = new ArrayDeque<>();
queue.add(1);     // enters at the tail
queue.add(2);
queue.remove();   // 1 — leaves from the head, FIFO

As a LIFO stack, push/pop treat the head as the "top": 

Deque<Integer> stack = new ArrayDeque<>();
stack.push(1); // addFirst()
stack.push(2);

stack.pop();   // 2 — removeFirst(), LIFO
stack.peek();  // 1

The exact same discipline works equally well from the tail instead, using Deque's own two-ended vocabulary directly: 

Deque<String> todos = new ArrayDeque<>();

todos.addFirst("First");
todos.addFirst("Second");
System.out.println(todos); // [Second, First]

String todo = todos.removeFirst(); // Second
System.out.println(todos); // [First]

todo = todos.peekFirst(); // First
System.out.println(todos);// [First]
Discipline enforced by usage, not by the type

Deque never claims to be a cs stack or a cs queue specifically — it offers both vocabularies side by side, honestly, as a general double-ended structure. Which discipline actually holds in a given piece of code depends entirely on which methods the caller chooses to use, consistently. This is precisely what fixes Stack's problem, described earlier: nothing is hidden through an inheritance trick that silently leaks unrelated methods; the caller's own consistent method choice is the discipline.

Concrete implementations

ArrayDeque and LinkedList both implement Deque. PriorityQueue is a Queue, but notably not a Deque — its ordering is entirely priority-driven, and a structure that reorders on insert has no fixed "other end" to expose.

§18.7

Set — membership, not position

Scenario C's problem, back in §18.1, was Mike joining the club a second time, with nothing in the array stopping him. A Set is Java's answer: it rejects the second Mike outright, on insertion, rather than leaving the check to be written by hand every single time.

Formally, a Set contains no duplicate elements — no pair of elements e1 and e2 such that e1.equals(e2), and at most one null. It models the mathematical set abstraction directly, and it answers a question neither List nor Queue is built around: "is this present?", answered quickly, without checking every element in turn. Positional access, the whole point of a List (§18.4), simply isn't part of that promise — not forbidden, just never guaranteed, regardless of what a particular Set implementation happens to do internally.

Inside a HashSet: Buckets, hashCode, and equals

"Quickly" deserves a real answer, not just a promise. If contains(...) had to walk every element comparing each one with equals(), a Set would be no faster than a List — the same linear scan, just under a different name. HashSet avoids that scan entirely, and the trick is worth seeing directly instead of taking "fast" on faith.

Picture a mail-sorting facility, letters pouring in by the truckload, with no time to read every address on every envelope before deciding where it goes. What actually happens is much faster: a machine glances at just the zip code on each envelope and drops it straight into the numbered bin for that zip code — no comparing this letter's full address against every other letter's address, just one quick read of a number. A carrier delivering that zip code only ever has to sort through the letters already sitting in that one bin, never the whole truckload.

A hashCode() (§16.1) is exactly that: a number computed quickly from an object's own content, the way a zip code is derived from an address. The same object always produces the same number, and objects with different content usually — though not always, more on that shortly — produce different numbers. It's not a unique ID, and it isn't sorted; it's just a fast, repeatable way to reduce "which of these thousands of objects am I dealing with?" down to "which of these few numbered bins does it belong in?" before ever looking closely at any single object.

Internally, a HashSet keeps an array of buckets — those numbered bins, plain empty slots, more of them than the set currently needs. hashCode() is the machine doing the sorting: it maps each object's number down onto one of those bucket indexes (roughly, hashCode() % number of buckets), which becomes that object's "home" bucket, exactly the way a zip code picks a bin. Adding an element computes its home bucket and drops it there. Checking membership computes that same home bucket for whatever value is being asked about, and looks only inside it:

Each name's hashCode() picks its bucket. Bucket 3 happens to hold two names — a collision, highlighted above.

Looking up "Bob" never touches "Mike", "Alice", or "Zoe" at all — its hash sends it straight to bucket 5, and only bucket 5 gets checked. That's the entire speed advantage: the number of elements contains(...) actually has to compare against is however many happen to share one bucket, not however many are in the Set overall.

Two different objects can still land in the same bucket — "Alice" and "Zoe" above, a collision — since there are far more possible objects than there are buckets. A bucket that already holds something isn't necessarily full; it just means equals() may have to settle it. "May," because a HashSet keeps each occupant's full hashCode() value alongside it, not just its bucket — and before ever calling the (potentially expensive) equals(), it first checks that cached number. Two occupants sharing a bucket can still have different hash codes (the bucket index is only hashCode() % number of buckets, a many-to-one mapping); when they do, equals() never even gets called between them. Only when the numbers already match does equals() get asked to settle it — on insertion, to see if this is a genuine duplicate; on lookup, to decide whether an occupant is actually the one being searched for.

What if equal objects produced a different hash?

This is exactly why hashCode() and equals() have to agree, not just individually work correctly: two objects only ever get compared with equals() if their hash codes already matched. If two objects are equals() but compute different hash codes, that hash check fails before equals() ever runs — regardless of whether they happened to land in the same bucket or not — and a duplicate the Set was supposed to reject slips straight in, undetected, because the two copies never even meet.

Mutating an object already inside a Set

A Set relies on hashCode and equals to manage uniqueness — §16.1 already showed one way this goes wrong, when a class overrides equals without a matching hashCode. There's a second, subtler hazard even when both are written correctly: an object's hash bucket is computed once, at the moment it's inserted. If a field that hashCode or equals depends on changes afterward, the object doesn't move to match — it silently becomes unfindable through contains(...), while still physically sitting in the Set. §11 already gave one reason immutability matters — safe sharing (§11.5) — and this is a second, very concrete one: a Set's correctness depends on its elements never changing in any way that affects equals or hashCode after insertion.

Concrete Implementations: HashSet, LinkedHashSet, and TreeSet

All three implement the same Set contract — no duplicates — the same way ArrayList and LinkedList (§18.4) both implement List: identical behavior from the outside, different internal machinery and different tradeoffs underneath.

  • HashSet is exactly the bucket-array structure just described, with no ordering guarantee at all — iterate over it, and elements come back in whatever order their buckets happen to sit in, which can even change between runs. That's the price paid for its speed: add, remove, and contains are all O(1) on average, since none of them do more work than checking one bucket. Absent a specific reason to want an order, this is the default Set.
  • LinkedHashSet uses the exact same bucket mechanics as HashSet — same hashCode()-driven lookup, same O(1) average cost — plus a linked list threaded through every entry in the order it was inserted. Iteration returns elements in that insertion order, predictably, every time. The extra bookkeeping costs a little more memory per entry; what it buys back is a Set that behaves like a HashSet for lookups but like a List for iteration order.
  • TreeSet isn't bucket-based at all — it's backed by a balanced binary search tree, and it keeps every element in sorted order at all times, not just at iteration time. Sorting needs a way to compare two elements, not just tell if they're equal, so a TreeSet relies on the elements' natural ordering (or an explicit Comparator supplied when the set is created) rather than on hashCode() the way the other two do. That comparison replaces the O(1) bucket lookup with an O(log n) tree search — slower per operation, in exchange for always coming back sorted.

Because a TreeSet is always sorted, it's also a SequencedCollection (§18.3): getFirst() and getLast() return the smallest and largest elements directly, no traversal needed. Picture that same library's catalog, kept as a TreeSet of the same Book from §18.5 — already Comparable by title — where adding a single new book changes what getFirst() returns, even though nothing was ever explicitly moved to the front: 

SequencedCollection<Book> catalog = new TreeSet<>(Set.of(
        new Book("The Pragmatic Programmer", "Hunt", 1999),
        new Book("Clean Code", "Martin", 2008),
        new Book("Effective Java", "Bloch", 2001)));

catalog.getFirst(); // "Clean Code" — alphabetically first, by title

catalog.add(new Book("A Philosophy of Software Design", "Ousterhout", 2018));
catalog.getFirst(); // "A Philosophy of Software Design" — nothing was inserted "at the front"; the set reordered around its comparator

TreeSet's encounter order (§18.3) comes entirely from its comparator, not insertion history — which is exactly why getFirst() can change on an add() that never touched "the front" at all.

§18.8

Map — key-value pairs, outside the Collection family

Scenario D, back in §18.1, asked "is Bob still a member?" — a question about a name, not a position, and one an array had no direct way to answer. A Map answers exactly that kind of question directly: ask for "Bob", get back whatever's associated with him.

A Map<K,V> is an object that maps keys to values — no duplicate keys allowed; each key maps to at most one value. It takes two type parameters (§15.2), not one, because it isn't holding single elements at all — it's holding pairs.

That difference in shape is exactly why Map<K,V> does not extend Collection<E> — a fact worth stating plainly, since it's easy to assume otherwise. Map is still considered part of the Java Collections Framework, just not a Collection itself. What it offers instead are three collection views, each one handing back a genuine Collection-shaped piece of the map's contents:

  • keySet() — a Set of keys (no duplicates possible, matching §18.7 exactly);
  • values() — a plain Collection of values (duplicates allowed — two different keys can map to the same value); and
  • entrySet() — a Set of key-value pairs.

The order a map's views iterate in depends entirely on the implementation, mirroring §18.7's own Set trio exactly: HashMap makes no order guarantee, LinkedHashMap preserves insertion order, and TreeMap guarantees sorted order.

Scope of this chapter

Concurrent collections (BlockingQueue and its relatives), the Collections utility class, writing custom Comparators for sorted order, and the rules around modifying a collection while iterating over it are all left for later. The core shape carries over unchanged: every collection in this framework picks one discipline — ordered, unique, or "whatever's next" — and every type's name is a claim about which one, a claim §18.6 already showed isn't always kept.
