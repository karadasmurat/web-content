Chapter 23

Concurrent Programming

§12.6 promised this chapter the moment threads first came up: sharing mutable state safely between them. Before any of that mechanics, though, comes a more basic question — why bother with more than one thread of execution at all? This chapter starts with a problem every program with the outside world runs into, using nothing but a real-world analogy anyone can picture — then names the vocabulary that problem was already speaking in, before a single line of Java thread code appears.

§23.1

The Problem — one thing at a time, and everything waits

Picture a coffee shop with a single barista and a queue of customers.

Case 1. The barista takes the first customer's order and payment, then presses the button on the espresso machine — and waits for the shot to pull. Once it's ready, they turn on the steamer and wait again while the milk heats. They assemble the drink, call the customer's name — and wait once more for that customer to actually step up and take it. Only once the cup has left the counter does the barista turn to the next person in line and start the entire sequence over. Every other customer in the queue stands there through all three waits, for a drink that isn't even theirs.

Somebody is paying for that barista

The shop owner isn't paying the barista only for the moments spent actually pulling shots and steaming milk — the wage runs the whole shift, waiting included. Every one of Case 1's three waits is time the owner is paying for while nothing productive happens: a resource on the clock, standing still. A barista who spends a third of the shift waiting is a barista the owner is paying full price for one-third less work out of. That's not a people problem or a bad-attitude problem — it's what the sequential arrangement itself guarantees, no matter how diligent the barista is.

Case 2. Now picture the same barista handling the same queue differently. They take the first customer's order and payment, press the button on the espresso machine — and, instead of waiting, immediately take the second customer's order. Since the first machine isn't ready yet, they switch to preparing the second drink: measuring syrup, pouring milk into the shaker, using exactly the dead time the first order's machine is filling. The espresso machine beeps; the barista switches back to the first drink, finishes it, sets it on the counter, and calls the name — then, without waiting for that customer to walk over and collect it, switches straight to whatever's next.

Same barista. Same two customers. Same machine. The only difference is what happens the instant something isn't ready yet:

  1. Wait — do nothing until it is.
  2. Switch — move to something else that's actually ready right now.

Case 1 is a sequential program: each operation completes before the next begins, in a strict, fixed order. The barista here plays the part of the CPU core — the one resource everything in the queue is ultimately waiting on. When the barista is genuinely working — taking an order, assembling a drink — that's pure computation, and it costs nothing: the core is doing exactly what it exists to do, with nothing else it could be doing instead. The waste shows up specifically the moment the barista is waiting rather than working — for the machine to heat, for a customer to arrive to collect their cup — because a resource sitting idle, waiting, is a resource going to waste, whether that resource is a barista or a CPU core. But most real programs aren't pure computation, any more than the barista's job was pure drink-assembly. They read files, query databases, call remote services, wait on a person to click something. Those operations take real, measurable time — not because the CPU is working hard, but because the program is waiting, exactly the way the barista waited on the espresso machine: for a disk to seek, for a network packet to arrive, for some other service that is itself waiting on something further down the chain.

To isolate the actual problem, picture a simplified, theoretical machine running nothing but this one sequential program — no other process for anything else to reach for, nothing else that could possibly use the core in the meantime. In that simplified world, the CPU genuinely sits idle for the entire wait, the same way the barista just stood there instead of touching the second order. A real computer is never actually this empty — dozens of other processes are always around, and something will typically find its way onto that core regardless. But no matter how busy the core stays with other people's work, it does nothing to unfreeze this program: its one sequential thread is still stuck waiting, and everything it was responsible for — including things that had nothing to do with the wait — sits frozen right along with it, exactly like the rest of the queue behind Case 1's barista.

The same shape, in a web browser

Click a link to download a file, and the browser has to actually go fetch it — reading the response from the network as it arrives, which takes real time no disk or CPU speed can shortcut. Written sequentially, that download runs directly on the very same thread that draws the page and listens for clicks. While it runs:

  • The page stops repainting.
  • Clicks are ignored — links, buttons, even scrolling do nothing.
  • The whole browser looks — and behaves — as if it has crashed.

The download might take ten seconds. The browser is frozen for all ten of them — not because the CPU is overloaded (the one thread doing the waiting isn't asking it for anything during that time), but because a single sequential thread of execution was made responsible for two unrelated things, one after the other, and the first one blocks.

The problem, precisely

A single thread of execution cannot make progress on multiple things at once. If one thing it's doing blocks — waiting on a disk, a network, another service — everything else that same thread was responsible for blocks along with it, whether or not that other work had anything to do with the wait.

Case 2's barista already proved this doesn't have to be the outcome — nobody hired a second barista, they just refused to stand still during a wait. §23.3 turns that same move into software.

§23.2

Process, Thread, and the Operating System — vocabulary from a user's chair

Before any of this can be about Java, it helps to have names for a few things every computer user has already seen, usually without needing a name for them. None of what follows is specific to programming — it's just how a modern computer runs more than one thing at a time, described from the outside, the way it looks to whoever is sitting in front of the screen.

Process: one running app, walled off from every other

Open a web browser, a music player, and a word processor, and — on Windows, macOS, or Linux alike — a window like Task Manager or Activity Monitor will list each one separately, each with its own row, its own slice of memory, its own share of CPU time. Each of those rows is a process: one running copy of one program, with its own private memory that no other process can casually reach into. Quit the music player, and the word processor doesn't even notice — as far as it's concerned, the two might as well be running on separate computers.

The operating system: the one thing every process shares

Something has to decide which processes exist, hand each one its own private memory, and let them all share the one CPU, the one disk, the one network connection, without any of them needing to know the others exist. That something is the operating system (the OS) — Windows, macOS, or Linux itself, running underneath every application, the one piece of software every process depends on and none of them can bypass.

Thread: more than one thing happening inside a single process

A process is rarely doing just one single thing internally, either. Open one browser tab on a video-heavy page, and inside that one process, several things are clearly happening together: the page keeps scrolling smoothly, the video keeps playing without stuttering, and a download in another tab keeps ticking upward — all at once, all inside what Task Manager still shows as a single row. Each of those separate, independently progressing activities inside one process is a thread.

Think of a process as one company renting a floor of an office building: it has its own space, its own locked door, its own supplies, none of it accessible to the company on the floor above. A thread is one employee inside that company — several employees can work on different tasks at once, all sharing the same one floor, the same shared filing cabinet, the same shared phone line, precisely because they all belong to the same one company. Different companies (processes) are isolated from each other; different employees of the same company (threads of the same process) share almost everything, on purpose.

Three terms, together

A computer runs many processes at once, each one isolated from the others, all of them managed by the operating system. Inside any one process, multiple threads can be making progress at the same time, sharing that process's own memory. §23.3 picks up from here — with far more threads wanting to run than there are CPU cores to run them on, something has to decide who actually gets one right now.

§23.3

Independent Progress — separate threads for separate activities

§23.1's Case 2 barista pulled off the fix alone, through nothing but discipline: notice when something isn't ready, and switch to other real work instead of standing still. A program could try to replicate that same discipline by hand, everywhere it does anything that might block — but that's exactly the kind of thing worth handing to something more reliable than remembering to do it right, every time, at every blocking call. Give each activity its own thread — its own independent, separately scheduled line of execution — and the operating system enforces the same discipline automatically, without the program having to think about it at every wait. The download runs on one thread; the window-drawing and click-handling run on another. Neither can block the other, because blocking is something that happens to a thread, and now there are two.

This works even on a machine with a single CPU core, because of a part of the operating system §23.2 didn't need yet: the scheduler. With far more threads wanting to run than there are cores to run them on — true even here, with just two — something has to decide, from one moment to the next, which thread actually gets the core, and for how long, before making way for another. That decision-maker is the scheduler, and its entire job is exactly this: pick a thread, let it run for a short slice of time, then pick the next one. Picture a conference with a hundred attendees and one shared microphone: a moderator — the scheduler — decides who speaks next and for how long, cutting each speaker off after their turn and handing the microphone to someone else, over and over, fast enough that everyone who wanted to speak eventually gets heard. No attendee controls when their turn comes; the moderator does.

Handing the microphone to a new speaker isn't free, either. The previous speaker has to be interrupted mid-thought, their place noted so they can resume later exactly where they left off; the next speaker has to get situated before they can usefully begin. The operating system does the equivalent of this every time the scheduler swaps one thread out for another: it saves precisely where the interrupted thread was, then loads in whatever the newly scheduled thread needs to continue. That save-and-restore operation is a context switch — it produces no useful work of its own, purely the overhead of sharing one core among many threads. A handful of switches per second is unnoticeable; §23.4 returns to what happens when there are far more than a handful.

With the scheduler doing this fast enough, the two threads from the download example interleave — switching between them rapidly enough that both appear, to a person watching, to be making continuous progress. The download thread blocks on network I/O; the scheduler immediately switches to the UI thread, which repaints the window and responds to a click; when the network data finally arrives, the scheduler switches back. The core is never left with nothing to do for the whole ten seconds — it's handed to other work the instant the download thread has nothing for it to do.

What concurrency actually means

Concurrency is structuring a program so that independent activities can make progress independently, without blocking each other. That's the whole definition — nothing here requires more than one CPU core. A single core, switching fast enough between threads, is enough to deliver it.

The same principle scales up unchanged. A web server handling thousands of requests at once faces the identical shape of problem: each request involves I/O of its own — reading a socket, querying a database, writing a response. Written sequentially, request B cannot even begin until request A finishes completely, so one slow database query stalls every other client waiting behind it. Give each request its own thread instead, and while thread A waits on its database, thread B computes, thread C writes a response, and thread D reads from its own socket — each advancing on its own schedule. A slow request no longer freezes the server; it only ever blocks the one thread handling it.

The browser, unfrozen

§23.1's Case 2 barista already made the shape of concurrency concrete, with no hardware involved at all — one worker, multiple orders in progress, nothing frozen because nobody stood still during a wait. The same browser that froze solid over a single download in §23.1 runs the identical shape once it's built concurrently, with several activities in flight the moment a page starts loading:

  • The network is fetching HTML, CSS, images, and scripts from the server.
  • The rendering engine is laying out whatever HTML has already arrived.
  • The UI thread keeps the window responsive — scrolling, tab-switching, and clicks all still work.
  • The JavaScript engine may be running a script in the background.

None of these waits for the others to fully finish before starting. The page begins rendering before every image has downloaded; the window can scroll before the page has fully loaded; a slow resource fetching on one tab doesn't stop a click on another. Each activity advances on its own terms, blocking only when it genuinely needs something the others haven't produced yet. A sequential browser — fully download, then fully parse, then fully render, then finally accept input — would be unusable; concurrency is what makes the experience feel alive rather than frozen between steps.

§23.4

Why Concurrency Matters

Concurrency pays off in three distinct ways.

  • Reduced execution time. The payoff looks different depending on what kind of work is actually waiting:

    Scenario What happens
    I/O-bound tasks While one thread waits on a database or a remote service (blocking, not computing), another thread uses the CPU to do real work. The wait stops being wasted time.
    CPU-bound tasks, matched to cores Eight genuinely CPU-bound threads on eight cores each get a core to themselves — the work finishes roughly eight times faster. This is the sweet spot §23.5's parallelism describes.
    CPU-bound tasks, oversubscribed Run a hundred CPU-bound threads on eight cores, and all hundred are fighting over eight execution units. None of them ever blocks on I/O — they're always ready to run — so the scheduler is forced into a constant stream of context switches (§23.3) just to give each one a turn. Those switches have a real cost of their own, and can leave the total work slower than simply running eight threads to begin with.

    The oversubscribed case has a second, less obvious cost sitting underneath the swapping itself: each core keeps the data it's actively working on in a small, extremely fast on-chip cache — the equivalent of a chef keeping every ingredient in reach on the counter, instead of walking back to a distant pantry for each one. A hundred threads competing for the same core constantly evict each other's data from that cache; every time a thread gets its brief turn, it often has to fetch its own working data all over again from much slower main memory, because the previous thread's turn overwrote what was there. More threads than cores doesn't just add switching overhead — it can quietly sabotage the one thing that makes any of this fast in the first place.

  • Better resource utilization. §23.1's underused barista was a cost the shop owner paid regardless of how much got done; a CPU sitting idle is the identical cost, just paid in silicon instead of wages. The CPU, network, and disk are all expensive relative to how often a purely sequential program actually keeps them busy. A CPU is millions of times faster than a disk or a Wi-Fi radio; without concurrency, it sends a request and then sits at effectively 0% utilization until the response arrives. With concurrency, the instant one thread starts waiting on the network, the scheduler hands the CPU to another thread with real work ready — the idle gap gets filled instead of discarded.

  • Improved responsiveness. Of the three, this is the one a person actually feels — it shows up directly as the user experience of the program, not just a number in a performance report. §23.1's frozen download button generalizes to any interface with a main thread responsible for drawing the screen. Run heavy work directly on that thread, and the window freezes for exactly as long as that work takes. Move the same work to a background thread, and the main thread stays free to keep drawing animations, responding to hovers, and updating a progress bar — the same reasoning §23.3 already applied to the download, now framed as a general design habit rather than a one-off fix.

§23.5

Where Parallelism Fits In — a hardware bonus, not the definition

So far, one barista has been doing all the switching — §23.1's Case 2 got two orders moving with exactly one pair of hands, no matter how sharp that barista's switching got, they could still only do one physical thing at any given instant: pulling a shot, or steaming milk, never both. What changes the moment there's a second barista, at a second machine? Two orders can now be worked on at the exact same instant — one barista pulling a shot while the other steams milk for a completely different drink, neither one waiting on the other to free up. That's what a second core buys that no amount of clever switching on the first one ever could.

A core is that second barista. Modern CPUs ship with multiple independent cores — a typical consumer chip has 8 to 16, a server chip 32 or more — each one capable of executing instructions at the same time as every other core. A single-threaded program only ever uses one of them; every other core sits idle as far as that program is concerned, regardless of how busy the one it's using gets.

Running threads on more than one core this way is parallelism, and it adds a second, distinct benefit on top of the responsiveness §23.3 already bought: throughput — genuinely more work completed in the same span of wall-clock time, because more than one core (more than one barista) is doing it at once.

It's worth keeping the two ideas separate, since it's easy to blur them together:

  • Concurrency is a structuring choice: when something causes a wait, switch to something else instead of standing still — exactly the rule §23.1's Case 2 barista followed. It works on a single core.
  • Parallelism is a hardware capability — activities actually executing at the same instant, on separate cores. Two baristas, two machines, two drinks moving forward in the same second, is parallelism. It requires more than one core to exist at all.

Every parallel program is concurrent — running on separate cores at once is one way of making independent progress. But not every concurrent program is parallel — a single barista, interleaving orders through nothing but fast switching, delivers the whole of §23.3's definition without two drinks ever moving forward in the identical instant. The distinction is worth holding onto, because the problems concurrent programs actually have to solve — shared state, coordinating access, one thread seeing another's changes — come from independent progress itself, not from how many cores (or baristas) happen to be available. A single core interleaving two threads has exactly the same coordination problems a sixty-four-core machine running the same two threads in true parallel does — the shared filing cabinet from §23.2's office-building analogy is just as much in dispute either way.

A hybrid in practice

Real systems usually land in between. Picture four baristas handling eight orders between them: four orders move forward at the exact same instant — one per barista, genuine parallelism — while each barista additionally switches between their own two orders to keep all eight progressing. Swap in a four-core machine running eight threads, and it's the identical picture. Parallelism explains why four things are moving at the exact same time; concurrency explains why all eight are moving at all.

§23.6

The Main Thread — where the first thread comes from

§23.8 is about to show how a program hands a task off to a thread — but creating a thread takes an already-running thread to do the creating. That raises an obvious question: if every thread is created by another thread that's already running, where did the very first one come from?

The genesis, at the OS level

The answer is that it isn't created inside the Java world at all — it's handed to Java by the operating system, before the JVM has done anything of its own. Running java app.Main from a terminal sets off a short handoff, each step passing control to the next:

  1. The operating system creates a new process (§23.2) for the command.
  2. The OS starts exactly one native execution thread inside that process, and hands it to the newly started JVM.
  3. The JVM uses that one thread to initialize itself, then searches the named class for one specific signature: public static void main(String[] args).

That thread — the one the OS handed over, now running the method the JVM found — is the main thread. Every Java application starts with exactly one of these, and every other thread the program ever creates (§23.8 shows how) traces back to it, directly or indirectly.

Not actually alone

It's tempting to picture the main thread starting out by itself, with the whole process to itself until it deliberately creates a second thread. That's not quite what happens. By the time the JVM hands control to main(), it has already quietly started several of its own background threads just to keep itself running:

  • A garbage collector thread, reclaiming heap memory that's no longer reachable.
  • A finalizer thread, running cleanup code for objects on their way out.
  • A signal dispatcher thread, handling signals from the OS — what actually notices, for instance, when a user presses Ctrl+C in the terminal running the program.

None of this code was written by the program itself, and none of it appears anywhere in main()'s own source. By the time your first line of code runs, the main thread is already sharing the process with a handful of threads the JVM created for its own purposes — a concurrent program before a single line you wrote has asked for one.

§23.7

Step 1: Creating a Task — defining work, independent of how it will run

Getting concurrent work done in Java breaks cleanly into two separate steps, worth keeping apart from the start: first, define the work itself — a task; only then decide what will actually run it. This section is entirely about the first step. §23.8 is entirely about the second.

Java has supported threads since its very first version, and over time two distinct ways of defining a task emerged. Understanding both — and why the second replaced the first as the default — is the foundation everything else in this section is built on.

The original model: a task is a thread

The earliest approach fuses the work itself and the machinery that runs it into one object, by extending Thread directly:

class DownloadTask extends Thread {
    @Override
    public void run() {
        System.out.println("Downloading...");
    }
}
DownloadTask task = new DownloadTask();
task.start(); // Downloading...

DownloadTask is a Thread (§10.3) — not something that merely uses one. That inheritance relationship is exactly the problem. The download logic and the thread that runs it are permanently one object; there's no way to hand just the logic somewhere else. Want to run this same work through a thread pool instead of a raw thread later on? There's no "just the logic" to hand over — run()'s body only exists bolted onto a full Thread, and Java classes only get to extend one thing (§10.3). A class that's already spent its one extends on Thread can't also extend anything else, and the task can't be run any way except by starting this particular kind of Thread.

The modern model: a task is handed to a thread

The cleaner approach keeps the two ideas separate from the start:

  • A task is a unit of work — what should happen.
  • A thread (or, later, an executor) is the execution mechanism — what actually runs it.

A task defined this way is an ordinary object implementing an interface (§14) — Runnable, in the simplest case, whose single method is exactly the run() body from before, with the Thread machinery entirely removed:

class DownloadTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Downloading...");
    }
}

DownloadTask no longer is a Thread — it has no relationship with one at all yet. That relationship only forms later, at the exact moment something decides to actually run it, which is what §23.8 is entirely about. The task object itself stays exactly this simple no matter what ends up running it — a plain Thread, a configured one, or a managed pool — with not one line of DownloadTask ever needing to change. Runnable is the one plug every execution mechanism in Java is built to accept.

Aspect extends Thread implements Runnable
Relationship IS-A — the class is a specialized kind of thread HAS-A — the class has work; a thread only carries it out
What it says "I am a worker" — the task and the execution mechanism are the same object "I have a task" — the work is defined once, independent of whatever ends up running it
Inheritance cost Spends the class's one extends slot (§10.3) — it can never extend anything else Uses one implemented interface among as many as the class likes (§14.5) — extends stays free for something else entirely
Generally preferred? Rarely — only when the class-per-thread coupling is genuinely wanted Yes — the default choice for defining work in Java

The inheritance row is worth restating plainly: Java permits a class to implement as many interfaces as it likes, but only ever extend one class (§10.3). A class that spends that one slot on Thread has no extends left for anything else it might legitimately need to be. A class implementing Runnable spends nothing from that same budget — it's free to extend some other class entirely, on top of carrying the task. Preferring HAS-A here is the identical instinct §27.5 gives a formal name to: reach for composition before inheritance, unless "is a more specific kind of" is genuinely the relationship being modeled — and a task is not a more specific kind of thread, any more than a BookEdition was a more specific kind of Book.

Runnable and Callable, formally

Runnable is declared in the standard library, in its entirety, as one method:

public interface Runnable {
    void run();
}

No parameters in, nothing returned, no checked exception declared. That shape fits a task run purely for its side effect — print something, write a file, update the screen — where the caller never needs anything handed back. It's a functional interface (§20.5), exactly one abstract method, which is exactly why DownloadTask above could just as easily have been written as a lambda instead of a named class:

Runnable task = () -> System.out.println("Downloading...");

Some work doesn't fit that shape, though — a task that computes something the caller actually needs back, or one that might fail in a way worth reporting as a checked exception. Callable<V> is the counterpart built for exactly that case, generic (§16) over whatever type its result will be:

public interface Callable<V> {
    V call() throws Exception;
}
Callable<Integer> sum = () -> 2 + 3;
System.out.println(sum.call()); // 5

call() returns a value of type V — Integer here — where run() always returns nothing at all. call()'s signature also declares throws Exception, meaning it's allowed to throw a checked exception (§15.5) — one the caller is required to acknowledge, either by catching it or declaring the same throws itself. run() has no such room in its signature; a Runnable can only ever throw the kind of exception nothing forces a caller to plan for. Both interfaces are functional (§20.5), and both get handed to the identical execution mechanisms §23.8 covers — the choice between them is entirely about whether the task needs to hand a result back.

Why this is the foundation, not just a style preference

Every more advanced execution mechanism Java offers — thread pools, scheduled executors, virtual threads — exists to run something. What they run is a Runnable or a Callable, never a raw Thread subclass. Defining work as a task, independent of how it will run, is what lets the exact same task move between execution mechanisms without ever being rewritten. The IS-A model forecloses that; the HAS-A model is what makes the rest of the concurrency API possible at all.

A task is an object

Every task, however it's written, is still just an object (§2.2) implementing Runnable or Callable — created with new, or, since both are functional interfaces, written as a lambda; sitting on the heap, reachable through a reference. §20.6 already made the point once, and it applies here without modification: a lambda is never a free-floating block of code, it's an instance of some class implementing the target interface. A task is that same idea, aimed at concurrency specifically.

Because a task is an object, it can capture references from the scope it was created in — the same closure behavior §20's lambdas already have. What it captures is what decides whether running it is safe:

Runnable greeting = () -> System.out.println("Hello!");

greeting captures nothing — no field, no reference to anything outside itself. It's stateless: handing this exact same object to a hundred different threads, all running it at once, is perfectly safe, because there's no shared data anywhere for any of them to disagree about.

class Counter {
    int value;
}
Counter counter = new Counter();
Runnable increment = () -> counter.value++;

increment is a different shape entirely. The lambda captures counter — a reference to a mutable object outside itself — making increment a closure over shared, mutable state. Hand this same Runnable to two threads, and both are now reading and writing the identical counter.value field, with none of the coordination described so far having said anything about yet. That's not a flaw in this particular example — it's the general shape of the hazard multi-threaded programs run into the moment a task captures something mutable, and later sections in this chapter come back to exactly what it takes to make that safe.

The one question worth asking about any task

Defining a task is defining an object, and every object has fields, captured or otherwise. Before handing a Runnable or Callable to more than one thread, the question that matters is simply: what does this object actually capture? Nothing — and it's stateless, safe to share without a second thought. Something mutable — and every hazard shared, mutable state has ever caused (§12.6 named the category first) is back on the table, this time with more than one thread able to trigger it at once.

Expressing a task: four equivalent forms

§20 built one progression in full, using DiscountStrategy as its running example: the same behavior, expressed as a named class, then an anonymous class, then a lambda, then a method reference — each one shrinking the ceremony without changing what runs. Runnable and Callable are functional interfaces exactly like DiscountStrategy was, so every one of those four forms applies to a task without a single new idea — only a new domain to apply them to.

A named class.

class FileProcessor implements Runnable {
    private final Path path;

    FileProcessor(Path path) {
        this.path = path;
    }

    @Override
    public void run() {
        // process file at path
    }
}
Runnable task = new FileProcessor(somePath);

The task is a first-class citizen here — a name, a constructor, explicit fields. Its state is declared openly, not captured implicitly from wherever the object happens to be built. This form earns its keep when the task carries meaningful state deserving real fields, when it needs to be tested on its own, when it's complex enough that a name adds real clarity, or when the same task type gets instantiated repeatedly with different parameters — exactly the reasoning §20.1 already gave for when a named class pays for itself.

An anonymous class.

Runnable task = new Runnable() {
    @Override
    public void run() { }
};

A one-off implementation with no name (§20.4), capable of capturing variables from its enclosing scope but noticeably more verbose than a lambda doing the identical thing. It still fits when a task is simple, used exactly once, and a name would add nothing — but as §20.6 already showed, a lambda almost always says the same thing more directly, which is why this form has been largely superseded since Java 8 introduced the alternative.

A lambda.

Runnable task = () -> {};

Runnable task2 = () -> Utils.process();

Callable<String> task3 = () -> Files.readString(path);

The most common form in modern Java for exactly the reason §20.7 already gave: concise, readable, capturing whatever it needs from the enclosing scope without ceremony. It's the right choice when the task is short, has no name worth giving it, and captures at most a handful of variables.

Captured variables and shared state

This is where this section's earlier warning becomes concrete, in code:

List<Integer> nums = new ArrayList<>();
Runnable task = () -> nums.add(42); // captures nums

A lambda that captures a variable creates a closure (§20.6), and the captured reference becomes a field of the object the compiler generates — this lambda holds the exact same address nums does, pointing at the exact same ArrayList on the heap (§2.3's aliasing, again). Run this one task on several threads, and every one of them reaches that identical list: shared mutable state, with every race condition that shape has ever caused (§12.6). A stateless task is safe by construction — there's nothing to disagree about. A task that captures something mutable needs exactly the same care any other shared mutable state needs in a concurrent program, no more and no less.

Not always a new object

Every lambda expression is an object (§20.6), but not necessarily a fresh one on every evaluation. A lambda that captures nothing is stateless, and the JVM is free to cache and reuse one instance across every place it's evaluated. A lambda that captures a variable can't be shared this way — the captured value is part of that object now, so capturing a different value means building a genuinely new instance, one per capture.

A method reference.

class Utils {
    public static void process() { }
}
Runnable task = Utils::process;

Runnable staticRef   = MyClass::doWork;   // static method
Runnable instanceRef = instance::doWork;  // instance method
Runnable ctorRef      = MyClass::new;     // constructor

A method reference (§20.7) is a lambda whose entire body is already just a call to one existing method — () -> Utils.process() and Utils::process run identically, the second just says it with less noise. It's the right call whenever a task's whole body is exactly one existing method, cleaner than wrapping that same call in a lambda for no reason.

Four forms, one underlying object every time — a named class, an anonymous class, a lambda, or a method reference, chosen the same way §20 already taught, now aimed specifically at defining the tasks §23.8 hands off to run.

§23.8

Step 2: Handing a Task to an Executor — three ways to actually run one

A task from §23.7 doesn't run itself — something has to actually execute it. Java offers three ways to make that handoff, in increasing order of how much control they give over the thread doing the running: a raw Thread, a configured one built through Thread.Builder, or a managed pool through an ExecutorService.

2.1 — new Thread: the simplest handoff

Runnable task = () -> System.out.println("Downloading...");

Thread thread = new Thread(task);
thread.start(); // Downloading...

new Thread(task) builds a thread configured with nothing beyond the task itself — no name worth mentioning, no daemon status deliberately chosen, no priority set — and start() immediately hands it to the OS scheduler (§23.2) to begin running. It's the most direct way to get a task onto its own thread, and also the least configurable one; the next two approaches exist for exactly the cases where that matters.

2.2 — The Thread Builder API: configuring before handoff

new Thread(task) constructs a thread directly — there's no room in that expression to configure anything about the thread itself before it exists. Thread.Builder is a fluent, nested interface for exactly that: constructing and customizing a thread — its name, whether it's a daemon, which kind of thread it even is — before deciding whether to start it immediately or hold onto it for later.

Getting a builder. Two static factory methods on Thread produce one, depending on which kind of thread is wanted:

Thread.Builder platform = Thread.ofPlatform(); // a platform thread — backed directly by an OS thread (§23.2)
Thread.Builder virtual  = Thread.ofVirtual();   // a virtual thread — lightweight, managed by the JVM itself

Every thread this chapter has discussed so far — the ones §23.2 defined, the one §23.6 traced back to the OS — has been a platform thread: one genuinely backed by a native OS thread, exactly as heavy as that implies. A virtual thread is a newer, much lighter alternative the JVM manages internally rather than handing off to the OS one-for-one; a later section is where that distinction actually earns its keep. For now, the builder API itself works identically either way — everything below applies to both.

An active build step. A typical builder is passive: configure it property by property, then call a build() that hands back a finished, but otherwise inert, object. Thread.Builder's equivalent step does something more:

Thread t = Thread.ofPlatform().start(task);

start(Runnable) is Thread.Builder's version of build() — except it doesn't just construct the Thread, it also immediately tells the operating system to begin running it. Building and starting happen in the same call. When that's one step too many — the thread needs to exist first, and run later — unstarted(Runnable) is the genuinely passive version:

Thread t = Thread.ofPlatform().unstarted(task); // built, not running yet

// ...do other setup...

t.start(); // now it runs

Either way, the object handed back is an ordinary Thread (§23.2) — the builder's job ends the moment one exists; start() and join() from here on are plain Thread methods, not builder methods. t.join(), in particular, is worth naming now: it blocks whichever thread calls it until t finishes running, the ordinary way to wait for a thread's work to actually be done before moving on.

Configuring before building. Every configuration method returns the same builder, so calls chain naturally before the final start() or unstarted():

Thread thread = Thread.ofPlatform()
        .name("worker")
        .daemon()
        .unstarted(() -> System.out.println("Working..."));

thread.start();

name(...) is self-explanatory; daemon() is not, without a callback. §23.6's garbage collector, finalizer, and signal dispatcher threads were all daemon threads: background helpers the JVM never expects to be the reason the program keeps running. A non-daemon thread keeps the whole program alive until it finishes; a daemon thread doesn't — once every non-daemon thread has finished, the JVM exits, daemon threads included, mid-execution if need be. Marking a thread a daemon is saying, explicitly, "don't wait on this one to decide the program is done."

One builder, many threads. A builder isn't consumed by building — the same instance can produce several threads, one per call:

List<Thread> threads = new ArrayList<>();
var builder = Thread.ofPlatform().name("worker-", 0);

for (int i = 0; i < 3; i++) {
    threads.add(builder.start(task)); // start() returns the built Thread
}

name("worker-", 0) takes a prefix and a starting count; each thread this builder goes on to create is numbered automatically — worker-0, worker-1, worker-2 — without the loop having to track that count itself.

Builders as a ThreadFactory. A thread pool needs to build its own threads internally, on its own schedule, using whatever naming or configuration a caller wants — which is exactly what a builder is for. ThreadFactory is the interface that connects the two: a functional interface (§20.5) with one method, Thread newThread(Runnable r), and a builder is a natural way to implement it:

try (var executor = Executors.newFixedThreadPool(4,
        r -> Thread.ofPlatform().name("worker-", 0).unstarted(r))) {

    for (int i = 0; i < 12; i++) {
        executor.submit(task);
    }
}

The lambda r -> Thread.ofPlatform().name("worker-", 0).unstarted(r) is the ThreadFactory — a task expressed exactly the way §23.7 catalogs (a lambda implementing a functional interface), handed to newFixedThreadPool so that every thread the pool ever creates for itself comes out named and configured the same deliberate way, instead of whatever a plain new Thread(r) would have produced by default. That call is also this section's first real look at ExecutorService, which the next part covers properly.

2.3 — ExecutorService: handing off to a managed pool

The first two approaches still create one thread per task, by hand, every time. That habit generalizes badly the moment the number of tasks stops being small and fixed:

// naive web server — one thread per request
while (true) {
    Socket connection = serverSocket.accept();
    new Thread(() -> handleRequest(connection)).start();
}

This runs perfectly well under light load — a handful of requests, a handful of threads, nothing under any real pressure. Under heavy load, the exact same code degrades in ways that have nothing to do with how much actual work is being done:

  • Memory fills up with reserved stack space, one reservation per thread, whether that thread is doing anything yet or not.
  • The OS scheduler (§23.2) struggles to fairly divide time across thousands of native threads at once.
  • Context switches (§23.3) climb in direct proportion to the thread count, not the amount of real work.
  • Throughput plateaus and latency climbs — not because the CPU has too much work to do, but because it's spending an increasing share of its time managing threads instead of running them.

The fundamental problem sits one level up from any of these symptoms: the number of OS threads is being driven directly by the number of concurrent requests — a quantity the program doesn't control at all. A traffic spike produces a proportional thread spike, and nothing about new Thread(...).start() puts any ceiling on how far that's allowed to go.

Decoupling task count from thread count. The fix doesn't touch the tasks themselves — a task defined the HAS-A way §23.7 already built doesn't change at all — it changes what's allowed to create threads and when:

  • Create a fixed pool of threads once, upfront, sized deliberately rather than reactively.
  • Submit tasks to a queue instead of starting a thread per task.
  • Pool threads pull tasks from that queue and run them, one at a time, each.
  • When a thread finishes a task, it goes back to the pool and picks up whatever's next in the queue.
tasks submitted → [ queue ] → thread 1, thread 2, ... thread N → results

The number of threads is now a controlled variable, sized to the hardware and the workload instead of to however many requests happen to be arriving this second. A pool of a hundred threads can work through thousands of submitted tasks, just in batches, one queue's worth at a time — memory use, OS overhead, and context-switching are all bounded by the pool size, not by however large a traffic spike turns out to be.

The executor framework

This pattern — a fixed pool, a queue, workers pulling from it — is common enough that Java standardizes it directly, rather than leaving every program to build its own version by hand. That standardized implementation is the executor framework, built around the ExecutorService interface glimpsed above — the subject the rest of this chapter turns to next.

§23.9

ExecutorService in Practice — predefined pools, shutdown, and Future<V>

Predefined implementations: the Executors utility class

§23.8's last example called Executors.newFixedThreadPool(4, ...) without pausing on it. Executors is a utility class — never instantiated, existing purely to hold a set of static factory methods — each one building and configuring a specific ExecutorService strategy so that no program has to assemble a fixed pool, a queue, and a worker loop by hand, the way §23.8's "Decoupling task count from thread count" walked through conceptually.

The motivation behind supplying these ready-made strategies rather than leaving every program to build its own is worth stating plainly:

  • Reusable, well-tested thread pool strategies — the same handful of pooling strategies almost every concurrent program actually needs, implemented once, correctly, instead of reinvented per project.
  • Safe task queuing and execution — the queue-and-worker coordination §23.8 described is exactly the kind of code that's easy to get subtly wrong; Executors gets it right so individual programs don't have to.
  • Better performance than constantly creating and destroying threads — the create-per-task cost §23.8 opened with, avoided by construction rather than by discipline.
  • Simplified concurrency management — one factory call in place of manually wiring a pool, a queue, and a rejection policy together.
A factory, in the pattern sense

Executors is a factory: its methods hide the concrete class actually being built and hand back only the ExecutorService interface (§14) it implements. Calling code never names, or needs to name, whichever class is doing the pooling underneath — it programs to the interface (§27.6 gives this idea its formal name), and could swap which factory method built that ExecutorService without a single other line changing. The same HAS-A instinct §23.7 already built into Runnable shows up again here, one level up: a caller depends on what an executor can do, never on which concrete class is doing it.

Shutting down an executor: why it's not optional

Why shutdown matters.

An ExecutorService holds live threads, and — unlike §23.6's JVM-internal helper threads, or a thread explicitly marked with .daemon() back in §23.8 — those pool threads are not daemon threads by default. §23.8 already established what a non-daemon thread does: it keeps the whole program running for as long as it's alive, and a pool thread is exactly that kind of thread. A program that creates an executor and never shuts it down will hang indefinitely after every line of its own application code has finished, waiting on threads that will never be asked to do anything again.

Shutdown isn't an optional cleanup step, the way forgetting to close a file eventually gets cleaned up anyway. It's a required part of an executor's lifecycle.

Why the executor stays reachable. Say a method creates an executor, submits some tasks, and returns — the local variable holding the executor reference goes out of scope. §12.7 already covered what normally happens next: an object with no reachable reference becomes eligible for garbage collection. An executor doesn't follow that rule on its own, and the reason is worth tracing all the way through. Each pool thread's entire job, for its whole life, is running a loop shaped roughly like this:

Runnable workerLoop = () -> {
    while (true) {
        Runnable task = queue.take();
        task.run();
    }
};

new Thread(workerLoop).start();

There are two levels of Runnable stacked on top of each other here, easy to conflate:

  • The worker Runnable — workerLoop itself, started once when the pool thread is created, running for that thread's entire lifetime. This loop is the thread's own task: sit here forever, picking up and running whatever arrives.
  • The task Runnable — the actual work submitted to the executor (§23.7's tasks), picked up and executed one at a time, inside that outer loop.

queue.take() blocks indefinitely whenever the queue is empty — the thread is parked, burning no CPU, but still fully alive. That queue is a field of the executor object itself, the same internal BlockingQueue every submitted task passes through on its way to a pool thread. A live thread blocked inside queue.take() holds a reference back to that field, which means the executor object it belongs to is still reachable — not because any of the caller's own code still points to it, but because one of its own threads does.

GC root

§12.7 introduced unreachable as the condition that makes an object eligible for collection. A GC root is one of a small set of places the garbage collector always treats as reachable by definition — a running thread's own stack is one of them. A live thread, parked in queue.take() with the executor's queue reachable from its stack, is exactly such a root: everything reachable from it — the queue, and the executor object that owns it — gets traced as alive, no matter how many other references to that same executor have long since gone out of scope.

So, once the last external reference is gone and the queue has drained: no new tasks can arrive, since nothing is left to submit one; no queued tasks remain, since the queue is empty; every pool thread is still alive, parked in queue.take(), waiting for a task that can now never come; and the executor object stays on the heap, reachable only through those very threads. The JVM can't exit either — it's still waiting on live, non-daemon threads, and those threads are waiting on it.

shutdown() — graceful. The standard, polite way to stop an executor: it lets work already in flight finish rather than cutting it off.

executor.shutdown();

shutdown() signals the executor to stop accepting new tasks, and returns immediately — it doesn't wait for anything to actually finish. After the call: submit() throws RejectedExecutionException for any new task offered; tasks already running keep running to completion; tasks already queued still execute, in order; and once every one of those finishes, the executor moves to a TERMINATED state — the point where its pool threads finally exit their loops for good, and the executor object becomes reachable from nowhere again.

awaitTermination() — waiting for completion. shutdown() only signals; it doesn't wait. Code that needs to know the work is actually finished — before, say, reading results those tasks produced — calls awaitTermination(...) instead, which blocks the calling thread until either every task completes or a given timeout elapses, whichever comes first. It returns true if the executor reached TERMINATED inside that window, false if the timeout ran out first, with some of the executor's tasks possibly still unfinished.

Try-with-resources on ExecutorService. Since Java 19, ExecutorService extends AutoCloseable — the interface (§14) any resource needing guaranteed cleanup can implement, by declaring exactly one method: void close() throws Exception. A class implementing it can be declared as the resource in a try-with-resources statement, and Java guarantees close() runs automatically the moment the block exits, on success or on failure, without a separate finally block around to remember it.

For ExecutorService, close() performs exactly the graceful shutdown just described, in one call: it invokes shutdown(), then blocks indefinitely on awaitTermination() until every task completes. The code immediately following the block is guaranteed to see every task's finished results — a coordination guarantee formally called happens-before, which a later section in this chapter returns to in full.

try (ExecutorService executor = Executors.newFixedThreadPool(4)) {
    executor.submit(task);
    // close() called automatically, no checked exception to handle
}

This is the cleanest pattern for a bounded, known set of tasks — declare the executor as the resource, submit whatever it needs to do, and let the block's exit guarantee both shutdown and completion without an explicit shutdown()/awaitTermination() pair. Code written before Java 19 has no close() to rely on; an explicit shutdown() call, paired with awaitTermination() wherever completion genuinely needs to be waited on, is what appears in its place.

A narrowed throws clause

AutoCloseable.close() declares throws Exception — any implementor is allowed to throw a checked exception (§15.5) from close(), which is exactly why a try-with-resources block (§15.8 covers the general mechanism) normally has to plan for one. ExecutorService.close() overrides that same method (§10.5 covered overriding) and declares no checked exception at all. This is legal Java: an overriding method may narrow or eliminate the checked exceptions its parent declared, but it can never add ones the parent didn't. The practical effect: a try-with-resources block using an ExecutorService never needs a catch for a checked exception that close() is now guaranteed not to throw.

Future<V>: a handle to a result that isn't ready yet

What Future<V> represents.

Submitting a Callable to an ExecutorService doesn't run it on the calling thread — it runs on a pool thread (§23.8), asynchronously. The calling thread gets something back immediately, before the task has necessarily even started: a Future<V>, a handle to a computation that may still be running, or may not have begun at all.

Future<V> represents exactly one thing: a result that will be available at some point. It decouples the moment a task is submitted from the moment its result is actually needed — and in the gap between those two moments, the calling thread is free to do other work.

The Future<V> interface. Future<V>, in java.util.concurrent, is a placeholder for a result that may not yet exist, with methods to check on it and eventually retrieve it:

  • boolean isDone() — has the computation finished, one way or another?
  • boolean isCancelled() — was it cancelled before it finished?
  • V get() — blocks until the result is ready, then returns it.
  • V get(long timeout, TimeUnit unit) — waits at most the given time for the result, then returns it if it's available.
  • boolean cancel(boolean mayInterruptIfRunning) — attempts to cancel the computation.

isDone(): a non-blocking check.

Future<String> future = executor.submit(task);

if (future.isDone()) {
    String result = future.get();
}
// if isDone() returns false, this skips get() entirely —
// the result is never retrieved!

isDone() returns true once the task has completed — successfully, with an exception, or through cancellation — without ever blocking to find out. That makes it useful for checking progress without committing to a wait, but the snippet above shows the trap: if isDone() happens to return false, the if body simply doesn't run, and the result — once it does arrive — is never retrieved at all. Nothing crashes; the value is just silently lost.

isDone() alone is not a complete retrieval strategy. It only earns its keep when there's genuinely something else to do if the task isn't finished yet, and a real plan to come back to it:

if (future.isDone()) {
    String result = future.get(); // already done — no block
} else {
    useDefault(); // task not done yet — fall back, or retry later
}
while (!future.isDone()) {
    doOtherWork(); // genuinely useful work while waiting
}

String result = future.get(); // now guaranteed not to block

Optional<T> vs Future<V>. Optional (§21) and Future can look alike at a glance — both wrap a value that might not be sitting right there waiting — but they answer two entirely different questions.

Optional<T> Future<V>
Question answered Is the value there at all? Is the result ready yet?
Problem it solves A null return (§2.3) nobody remembered to check for Slow work that shouldn't freeze the calling thread
State check isPresent() — a fixed, binary state isDone() — a state that changes over time on its own
Blocking? Never — accessing it is instantaneous Potentially — get() waits if the work isn't done
Empty/unready access get() throws NoSuchElementException immediately get() simply blocks until the result exists
Best for Replacing a null return so absence must be handled Offloading work to a background pool (§23.8)

Optional's box is either full or empty right now, permanently, the instant it's created. A Future's box starts empty and may fill in on its own, at any point after submission — and even once it's full, it still carries the history of how it got there, since get() can surface an exception the task threw instead of a result.

The submit-all-then-collect pattern. Future<V> doesn't mean a result is available now — only that it will be, eventually — and get() blocks until it is. That raises a fair question: if waiting is still required, what's actually been gained?

Future<String> future = executor.submit(() -> fetchFromDatabase());
String result = future.get(); // blocks until result is available

Calling get() immediately after submit(), with no other work done in between, gains nothing — the calling thread blocks right away, and the whole arrangement is effectively sequential. The value in Future comes specifically from the gap between submission and retrieval; whether anything actually happens in that gap is entirely up to the caller.

The pattern that exploits it: submit every task first, then collect every result — never submit and immediately get() one at a time.

// Wrong — defeats the purpose
Future<String> f1 = executor.submit(task1);
String r1 = f1.get(); // blocks here — task2 hasn't even started yet

Future<String> f2 = executor.submit(task2);
String r2 = f2.get(); // blocks here — effectively sequential

Future<String> f3 = executor.submit(task3);
String r3 = f3.get(); // blocks here — effectively sequential
// Right — all three tasks may run concurrently
Future<String> f1 = executor.submit(task1);
Future<String> f2 = executor.submit(task2);
Future<String> f3 = executor.submit(task3);
// all three are now either running or queued to run

String r1 = f1.get(); // wait for the first
String r2 = f2.get(); // probably already done
String r3 = f3.get(); // probably already done

submit() places a task in the executor's queue and returns immediately; a pool thread picks it up as soon as one is free (§23.8). Submitting all three before calling get() on any of them leaves all three running in parallel during the gap — the calling thread is free the whole time, and the total wait ends up close to the length of the single longest task, not the sum of all three the way calling get() after each individual submit() would force. invokeAll() exists specifically because this submit-all-then-collect pattern is common enough to deserve one method that does it directly.
