Chapter 7

Control Flow

Every program so far has run top to bottom, once, every line. Control flow lets you decide whether a line runs at all, or how many times it repeats.

§7.1

The Flow of Execution — one statement at a time

When a program starts, it doesn't execute everything at once. Instead, it works its way through the code, one statement at a time. Consider this simple example:

System.out.println("Preparing coffee...");
System.out.println("Steaming milk...");
System.out.println("Coffee is ready!");

The output is:

Preparing coffee...
Steaming milk...
Coffee is ready!

The second statement isn't executed until the first one finishes. The third waits for the second. The program follows a path through the code, moving from one statement to the next.

This movement is called the control flow.

Changing the Flow

So far, the flow has been completely predictable: one statement after another. But real programs need to make decisions.

  • Sometimes a statement should execute only if a condition is true.
  • Sometimes a group of statements should be executed many times.
  • Sometimes execution jumps into another method before returning.

These are all ways of changing the program's normal flow. Java controls it through a handful of distinct mechanisms: conditional statements (if, switch), loops (for, while), method calls, and exceptions. This chapter covers the first two — the conditionals and loops that make up the most common, everyday decisions a program makes.

§7.2

Variable Scope — a variable doesn't exist everywhere

So far, every variable declared has stayed visible for the rest of the method. But that's not the actual rule; it just looks that way so far. Whenever Java encounters a pair of braces { }, it creates a new block — and a block doesn't just group statements together, it also defines the scope of any variable declared inside it: exactly how much of the program is allowed to see that variable at all. 

{
    int age = 25;
    System.out.println(age);
}

Inside the block, age exists exactly as before. But once execution leaves the block, the variable disappears — not just becomes hard to reach, but stops existing entirely:

{
    int age = 25;
}
System.out.println(age); // compile error — cannot find symbol

The compiler rejects this — age no longer exists by the time that println runs. Its scope ended at the block's closing brace, one line above.

A clean desk for the next task

Picture a variable as a sticky note placed on your desk while you work on one task. When the task is done, you throw the note away — the next task starts with a clean desk, not a pile of old notes from tasks that no longer matter. A block works the same way: whatever gets declared inside it exists only while that block is running, and is thrown away the moment execution leaves.

The smallest scope wins

Scope exists to stop a variable from outliving its usefulness. A variable should be visible exactly where it's needed, and nowhere else — the moment it could be misread or reused somewhere it doesn't belong, its scope should already have ended. Two separate calculations can each keep their own subtotal, confined to the block that needs it, with no risk of one accidentally picking up the other's leftover value: 

{
    int subtotal = 12 * 3;
    System.out.println(subtotal); // 36
}

{
    int subtotal = 8 * 2;
    System.out.println(subtotal); // 16
}

By the time the second block runs, the first subtotal doesn't exist anymore — there's nothing left to confuse it with. If both had instead been declared once, further up, in a scope wide enough to reach both blocks, the second calculation could silently start from whatever the first one left behind, and a bug like that only gets harder to spot the further apart the two blocks end up living. Keeping every variable's visible lifetime as short as possible is what makes a large program readable at all: anyone reading a line of code only has to think about what's in scope right there, not everything ever declared anywhere above it.

One thing to remember

Braces create a new scope. That's the entire rule, and it applies no matter what put the braces there — a bare block on its own, like the one above, or a block that belongs to some other statement entirely. The next few sections introduce if, while, and for, and every block any of them opens follows this exact same rule.

§7.3

Conditional Branching — if, else, and else if

if — running a block only when a condition holds

Picture a bouncer checking IDs at a door: only if you're old enough does anything happen — the door opens, you walk through. Otherwise nothing happens at all; you just keep standing where you were. Every program needs the same kind of gate sooner or later: run this step, but only under some condition, and skip it entirely otherwise.

Java's version of that gate is if. An if runs a block only when its condition is true — and simply skips it, moving straight on to whatever comes after, when the condition is false. The condition is any expression that produces a boolean — exactly the comparison and logical operators from §5: 

int width = 10;

if (width > 5) {
    System.out.println("wide");
}
System.out.println("done");

With width at 10, the condition is true, so both lines print. Change width to 3, and the condition becomes false — "wide" never prints, but "done" still does. Nothing about the rest of the program depends on which way the condition went; if only ever decides whether to run its own block, one time, on the way past.

Only a boolean

Java requires the condition to be a genuine boolean — not an int standing in for one, the way some languages allow. if (width) is a compile error unless width is itself declared boolean. If you mean "not zero," you write if (width != 0).

else — the alternative branch

if, on its own, only ever decides whether to run one block — it has nothing to say about what happens when the condition is false, beyond simply skipping. Wanting something to happen in that case too, without else, means writing a second, separately-checked condition: 

int width = 3;

if (width > 5) {
    System.out.println("wide");
}
if (width <= 5) {
    System.out.println("narrow");
}

This works, but it says the same thing twice — width <= 5 is just width > 5, negated, written out by hand. Nothing enforces that the two conditions actually stay opposite of each other if either one is edited later. else states the relationship directly instead of re-deriving it:

int width = 3;

if (width > 5) {
    System.out.println("wide");
} else {
    System.out.println("narrow");
}

Exactly one of the two blocks runs, every time — never both, never neither. else doesn't have its own condition to check; it simply means "every other case," whatever that turns out to be.

else if — chaining more than two cases

Plain if/else only covers exactly two outcomes. A third category — say, a "medium" width sitting between "wide" and "narrow" — doesn't fit either branch. else if chains another condition onto the else, checked only if every condition above it was false: 

int width = 10;

if (width > 20) {
    System.out.println("wide");
} else if (width > 5) {
    System.out.println("medium");
} else {
    System.out.println("narrow");
}
// medium

The conditions are checked in order, top to bottom, and the first one that's true runs — every condition after it is skipped entirely, even if it would also have been true. With width = 10: width > 20 is false, so Java moves to width > 5, which is true — "medium" runs, and the final else never even gets checked. Order matters here in a way a single if/else never had to worry about: swap the two conditions, and a width of 25 would incorrectly print "medium", since 25 > 5 is already true before 25 > 20 ever gets a turn.

Each Branch Is Its Own Scope

Every block above — the if's, the else's, each else if's — creates its own scope, exactly like the bare blocks §7.2 introduced. A variable declared inside one of these blocks doesn't survive past its own closing brace:

if (loggedIn) {
    String message = "Welcome!";
    System.out.println(message);
}

System.out.println(message); // compile error — cannot find symbol

message was created for the if block alone. Reaching for it afterward is exactly the error §7.2 already showed — this block just happens to belong to an if instead of standing on its own.

Because if and else are separate blocks, they can even declare a variable with the same name without any conflict:

if (loggedIn) {
    String greeting = "Welcome!";
} else {
    String greeting = "Please sign in.";
}

These are two completely different variables that happen to share a name. Each one's scope is its own block, and the two blocks never overlap — exactly one of them ever runs, so at most one greeting ever exists at a time. Nothing here conflicts.

Nested ifs Share the Outer Scope

An if can also sit inside another if, to check a second condition that only matters once the first one already held — say, checking whether an admin is logged in, which only makes sense to ask at all once someone is logged in in the first place:

if (loggedIn) {
    String welcomeMessage = "Welcome back!";

    if (isAdmin) {
        System.out.println(welcomeMessage); // "Welcome back!" — still visible in here
    }
}

This is a genuinely different situation from if/else sitting side by side. The inner if's block isn't next to the outer one — it's nested inside it, so it doesn't start with a clean slate the way else's block does. Everything the outer if already declared, like welcomeMessage, is still visible in there too, on top of anything the inner block declares for itself.

That also means the inner if can't redeclare a variable the outer one already has — the name is already taken, for as long as the outer block is still running:

if (loggedIn) {
    String welcomeMessage = "Welcome back!";

    if (isAdmin) {
        String welcomeMessage = "Welcome, admin!"; // compile error — variable welcomeMessage is already defined
    }
}

Compare that to greeting above: if and else could each declare their own because those two blocks never coexist — exactly one runs. Here, the outer block is still very much running for the entire time the inner block runs, so a second welcomeMessage inside it would mean two things with the same name, in scope at the same instant. That's exactly the ambiguity scope exists to rule out.

Free Shipping Threshold

Part 1. An online store gives free shipping on orders of $50 or more. Using a plain if, print "Free shipping!" when orderTotal meets that threshold.

double orderTotal = 62.00;
Free shipping!

Part 2. Orders under $50 still need to know what shipping costs — a flat $5.99. Add an else so every order prints exactly one line, whichever case it falls into.

Part 3. The store adds a middle tier: orders from $25 up to (but not including) $50 get discounted shipping at $2.99, instead of the full $5.99. Chain in an else if to handle all three cases.

Show solution

Part 1 — just the threshold, nothing printed for orders below it:

if (orderTotal >= 50) {
    System.out.println("Free shipping!");
}

Part 2 — else guarantees every order prints something:

if (orderTotal >= 50) {
    System.out.println("Free shipping!");
} else {
    System.out.println("Shipping: $5.99");
}

Part 3 — the new middle tier slots in as an else if, checked only once the $50 case has already failed:

if (orderTotal >= 50) {
    System.out.println("Free shipping!");
} else if (orderTotal >= 25) {
    System.out.println("Shipping: $2.99");
} else {
    System.out.println("Shipping: $5.99");
}

Order matters here exactly as the else if explanation above described: checking orderTotal >= 25 first would incorrectly catch a $62 order too, since 62 >= 25 is also true. Checking the highest threshold first, and only falling through when it fails, is what keeps the three tiers from stepping on each other.

§7.4

while — repeat while a condition holds

Imagine pouring water into a glass, checking after every pour whether there's still room left, and stopping the moment there isn't. You don't decide up front exactly how many pours it'll take — you just keep repeating "pour, then check" for as long as the condition "is there still room?" stays true.

A while loop is that same pattern in code: it re-checks its condition before every run of its block, and stops the moment that condition is false:

int i = 0;
while (i < 5) {
    System.out.println("In the loop: " + i);
    i++;
}
System.out.println("After the loop: " + i);
In the loop: 0
In the loop: 1
In the loop: 2
In the loop: 3
In the loop: 4
After the loop: 5

Three moving parts make this work, and all three are things you already know: i starts at a known value, the condition (§5's <) is checked before each iteration, and i++ (§5) changes the value the condition depends on. Leave that last part out, and the condition never becomes false — the loop runs forever.

Why the last line says 5, not 4

i is declared before the loop, not inside it — a while loop's braces are a block like any other (§7.2), but they don't reach backward to claim a variable declared ahead of them, so i is still very much alive once the loop is done with it. And it holds 5, not 4, because of the exact order things happen in: the last iteration runs with i at 4, prints it, then i++ makes it 5 — then the condition is checked again, finds 5 < 5 false, and only then does the loop exit. The increment that ends the loop always happens before the check that notices it did.

§7.5

for — the loop with its bookkeeping up front

A for loop repeats a block of code a known number of times, with its bookkeeping — start, check, update — gathered into one line instead of scattered around a while loop:

for (initialize; check; update) {
    // loop body
}
for (int i = 0; i < 5; i++) {
    System.out.println(i);
}
// 0 1 2 3 4

The loop works like this:

  1. initialize: create i and give it the value 0 — runs once, before anything else.
  2. check: is i < 5?
    • Yes → run the loop body.
    • No → exit the loop.
  3. update: increase i by one.
  4. Repeat from step 2.

This is precisely the previous example, rearranged — nothing new is executing, it's just harder to forget the update when it lives next to the condition.

Unlike the while loop's i, this one doesn't survive

§7.4's i was declared before the while, so it was still around once that loop finished. This i is declared inside the for's own parentheses, and that's exactly where its scope ends too (§7.2) — i stops existing the moment the loop is done. Try System.out.println(i) right after this loop, and it won't print 5; it won't compile at all — cannot find symbol.

Now the array chapter's opening example can finally be written the way it was always heading:

int[] temperatures = {68, 71, 69, 73, 75, 70, 66};

int total = 0;
for (int i = 0; i < temperatures.length; i++) {
    total += temperatures[i];
}
double average = total / (double) temperatures.length;
System.out.println(average); // 70.28571428571429
Same idea as mon + tue + ... ← §6.1

§6.1 opened by hand-writing mon + tue + wed + thu + fri + sat + sun — one variable per value, and a sum that has to name every one of them. temperatures[i], with i counting 0 through temperatures.length - 1, reaches every element the same way that hand-written sum did, without a fixed number of terms baked into the code. Ask for thirty temperatures instead of seven, and this loop doesn't change at all.

Print a row of stars

Using a single for loop, write a program that prints a row of n asterisks on one line. For n = 3, the output should be:

***

Hint: System.out.println(...) always starts a new line right after it prints. System.out.print(...) — without the ln — doesn't, which is exactly what keeps every star on the same line here.

Show solution 
int n = 3;
for (int i = 0; i < n; i++) {
    System.out.print("*");
}

The loop runs n times, printing one * per iteration with no line break in between — exactly the *** the exercise asked for, nothing more.

§7.6

for-each — when you don't need the index

Most of the time, a loop over an array wants every element, in order, and never actually uses i for anything except temperatures[i]. The for-each loop drops the index entirely:

int total = 0;
for (int temp : temperatures) {
    total += temp;
}

Read for (int temp : temperatures) as: "for each int in temperatures, call it temp." Every iteration, temp holds the next element's value — copied out, exactly like any other primitive assignment. There's no i to initialize, check, or increment, because there's no index at all.

When you still need the classic for

For-each can't skip elements, go backwards, or change which element it's looking at — it always visits every element, front to back, once. It also can't tell you the index of the element it's holding. Need any of that — or need to modify the array itself through the loop variable — and the classic for (int i = 0; ...) is still the right tool.

Find the Hottest Day

Using a for-each loop over the temperatures array from earlier in this chapter, find and print the single highest temperature of the week:

75

Hint: keep a "highest so far" variable. Start it at the first temperature, then update it whenever a hotter one comes along — you never need to know which day that was, only the number itself, which is exactly what for-each is good at.

Show solution 
int hottest = temperatures[0];
for (int temp : temperatures) {
    if (temp > hottest) {
        hottest = temp;
    }
}
System.out.println(hottest);

hottest starts as the first temperature so there's always something to compare against. Each iteration asks one question — is this one hotter than the best I've seen so far? — and only updates hottest when the answer is yes. By the time the loop has visited all seven temperatures, hottest can only be the largest one.

§7.7

The Ternary Operator — folding a small if/else into a value

A shopping cart needs a short description of what's inside it. Most of the time that means showing how many items are there — but a cart can also be empty, and "0 item(s)" is a strange thing to show a shopper. What's wanted is a plain "empty cart" for that one case, and the count for every other case.

That's exactly the same kind of decision §7.3's if/else already handles: one condition, two possible outcomes. Written that way, though, both branches exist purely to assign the same variable:

int quantity = 0;

String label;
if (quantity == 0) {
    label = "empty cart";
} else {
    label = quantity + " item(s)";
}

Four lines of scaffolding to make one decision, and the actual choice — which string label ends up holding — is buried in the middle of it. The ternary operator asks the same question and produces the answer directly, as a value, with no branches to declare and no variable to assign twice:

int quantity = 0;
String label = (quantity == 0) ? "empty cart" : quantity + " item(s)";

Read condition ? valueIfTrue : valueIfFalse left to right: check the condition; if it's true, the whole expression evaluates to valueIfTrue; if it's false, to valueIfFalse. Exactly one side ever runs — same rule as else, just written as an expression instead of two blocks.

That's the entire difference from if/else: if is a statement — it runs code, and has nothing to hand back. A ternary is an expression — it evaluates to a value, the same way quantity + 1 or a method call does (§4.3). It never needs a variable of its own at all: 

System.out.println(quantity == 0 ? "empty cart" : quantity + " item(s)");
A value, not a block ← §4.3

§4.3 drew the line between methods that return a value and ones that don't — and showed that a returned value doesn't have to be stored, it can be printed, combined, or used right where it's produced. A ternary draws the exact same line inside a single expression: if/else is all action, no value; a ternary is all value, evaluated inline, wherever an expression is allowed — an assignment's right-hand side, a method argument, another expression's operand.

One decision, not a chain

A ternary can be nested — a ? b : (c ? d : e) compiles — but nesting more than one quickly turns unreadable, since neither branch lines up visually the way else if does. If choosing the value takes more than one condition, reach for if/else if/else (§7.3) instead and assign the result there. Save the ternary for exactly what §7.3's else is doing: one condition, two values, nothing more.

Scope of this chapter

switch, do-while, and the break/continue statements that fine-tune a loop mid-run all build on exactly the ideas here — a boolean condition, and a block that runs some number of times — and are left for later.
