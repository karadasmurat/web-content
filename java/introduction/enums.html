Chapter 17

Enums

Compass directions, seasons, chess pieces — some categories are closed by nature, complete the moment you can name every member. Before Java had a dedicated way to model that, the usual trick was a handful of int constants, and every guarantee a closed set promises quietly evaporated. This chapter builds the real thing: a class (§2) with the one property an ordinary class can't offer on its own — a compiler-enforced, permanently fixed set of instances.

§17.1

The Problem — closed sets, and the constants that faked them

Days of the week. Seasons. Compass directions. Chess pieces. Programs constantly need a type that can only ever hold one of a small, known handful of values — and before Java had a dedicated feature for this, the common pattern was a handful of int constants:

public class Navigation {
    static final int NORTH = 0;
    static final int SOUTH = 1;
    static final int EAST = 2;
    static final int WEST = 3;

    public static void move(int direction) {
        if (direction == NORTH) {
            System.out.println("Moving north");
        } else if (direction == SOUTH) {
            System.out.println("Moving south");
        } else if (direction == EAST) {
            System.out.println("Moving east");
        } else if (direction == WEST) {
            System.out.println("Moving west");
        } else {
            System.out.println("Unknown direction");
        }
    }
}
Navigation.move(Navigation.NORTH); // works fine

The idea seems reasonable enough: each direction is represented by a distinct int — NORTH is 0, SOUTH is 1, and so on — and move compares whatever it's handed against each constant in turn, printing the matching message. And it works, exactly as shown above.

But look again at the parameter itself: move(int direction). The method doesn't accept "one of NORTH, SOUTH, EAST, or WEST" — it accepts any int at all. NORTH through WEST are just four particular values the programmer intended callers to use; nothing about the method's signature actually limits a caller to them. That gap — between what the constants were meant for and what the compiler will actually let through — is where every one of the following three problems comes from.

  1. The type is real, but it's the wrong one. The compiler does check move(int)'s argument — pass a String or a boolean and it won't compile. But int is the type of every whole number in the program, not just 0 through 3. The check that runs is "is this an int," never "is this one of the four directions" — so every one of these passes it:

    Navigation.move(Navigation.NORTH); // intended
Navigation.move(0);                // same as NORTH, but unclear at the call site
Navigation.move(42);               // compiles fine, but completely invalid
Navigation.move(-1);               // compiles fine, but completely invalid

  2. Semantic meaning is lost at runtime. NORTH is a compile-time alias for 0 — nothing more. Once the code runs, only 0 remains; the fact that this number represents a compass direction is gone the moment you'd actually want it, while debugging:

    int d = Navigation.NORTH;
System.out.println(d); // prints: 0

  3. The constants are not related. Four separate ints happen to live in the same class, but the compiler does not know they form a group. Nothing stops this:

    static final int MAX_SPEED = 2; // coincidentally the same value as EAST
Navigation.move(MAX_SPEED);      // compiles fine, silently wrong

    Two completely unrelated constants share the same int value, and the compiler has no way to catch the mistake — as far as it's concerned, MAX_SPEED and EAST are simply the number 2, twice.

What's actually missing

All three problems trace back to one thing: int is int. The compiler tracks types (§1.2), and every one of these constants has the same type as every other integer in the program — 2, 42, and EAST are indistinguishable to it. What's needed is a genuinely distinct type for "a compass direction," one the compiler can actually check against. That's exactly what an enum is.

Named constants are already one step up from something worse: a magic number, a literal value hardcoded directly into logic with no name explaining what it means at all:

// what is 2? what is 1?
if (user.getStatus() == 2) {
    shipOrder();
} else if (user.getStatus() == 1) {
    promptPayment();
}

Nothing here is wrong syntactically — it's just unreadable without cross-referencing whatever else in the codebase defines what 2 and 1 mean, and nothing stops a third, unrelated piece of code from reusing 2 for something else entirely. Giving the numbers names, the way Navigation does, fixes exactly that one problem — readability at the call site — and none of the other three.

A real example still in the JDK

java.awt.Font exposes exactly this pattern today: public static final int PLAIN = 0;, BOLD = 1;, ITALIC = 2;, alongside a protected int style field they're meant to populate. It works, and it ships in the standard library — but it predates enums, kept for backward compatibility rather than as a model to follow now.

Before reaching for that fix, though, it's worth checking that NORTH/SOUTH/EAST/WEST actually has the property that made it a good fit in the first place: it's a closed set, complete and sealed by nature, every member known upfront, membership never changing. Not every set that feels fixed actually is:

Genuinely closed:

  • Cardinal directions — NORTH, SOUTH, EAST, WEST. There is no fifth direction to discover.
  • Seasons — SPRING, SUMMER, AUTUMN, WINTER. Fixed by nature.
  • Chess pieces — KING, QUEEN, ROOK, BISHOP, KNIGHT, PAWN. The rules of chess define exactly six.

Apparently closed, but actually open:

  • Countries — feel finite, but countries split, merge, and are created over time.
  • Job titles at a company — change as the organization evolves.
  • HTTP status codes — new ones get added by standards bodies.

The distinction matters because of how the fix this chapter builds actually works: its whole set of values gets baked into the compiled class file. A closed set is a good candidate for it. An open one is not — every time it grew, you'd have to recompile and redeploy just to add a value; open sets are better stored in a database or a config file, loaded at runtime instead. That isn't a flaw in the fix — it's the direct cost of the guarantee it's built to provide, and different parts of the very same program are free to answer the question differently for different data. The question worth asking before reaching for it, every time: is this set closed by nature, or could it grow? (§17.8 returns to this question with concrete alternatives.)

§17.2

Declaring an Enum — a fixed set of instances, guaranteed by the compiler

That's exactly the gap §17.1 just opened up: instead of a handful of int constants nothing stops you from misusing, Java gives "a season" — or a compass direction, a chess piece — its own fixed, compiler-checked set of values.

An enum declares a class whose instances are fixed and finite, fully determined at compile time, and created by the JVM once, when the class is loaded:

enum Season {
    SPRING, SUMMER, AUTUMN, WINTER;
}

Each name is not a number with a label attached — it's a reference to an actual object on the heap, exactly the kind of reference §2.3 already covered, just declared for you by the compiler instead of written by hand with new.

"Fully determined at compile time" and "created at runtime" aren't in tension — they describe two different moments. The compiler knows everything about the instances ahead of time: their names, how many there are, what order they're declared in. But the objects themselves don't exist until the class is actually loaded, and Java loads classes lazily — not every class up front at startup, but only once something in the running program first refers to it. The first time anything touches Season, the JVM loads the class and runs its static initialization, in the exact order it's written in the source:

  1. The enum constants are created first — SPRING, SUMMER, AUTUMN, WINTER, assigned to static final fields the compiler generates for you. This step is implicit; you never write it.
  2. Any explicit static fields you declared, in order.
  3. Any static initializer blocks, in order.
The constants always come first

Java requires the enum constants to be the first thing declared in the enum body, ahead of any fields or methods. That ordering isn't a style preference — it's what guarantees that by the time any of your static fields or static blocks run, SPRING through WINTER already exist on the heap and can safely be referenced from them.

What the compiler actually builds

enum is syntactic sugar: a more convenient syntax that the compiler expands into something more verbose before the JVM ever sees it. You write the short form; the compiler generates the long form; the JVM only ever runs the long form. Understanding the expansion is understanding what actually executes. The Season declaration above compiles to roughly this:

public final class Season extends Enum<Season> {
    public static final Season SPRING = new Season("SPRING", 0);
    public static final Season SUMMER = new Season("SUMMER", 1);
    public static final Season AUTUMN = new Season("AUTUMN", 2);
    public static final Season WINTER = new Season("WINTER", 3);

    private static final Season[] $VALUES = { SPRING, SUMMER, AUTUMN, WINTER };

    private Season(String name, int ordinal) { // private — no one else can instantiate it
        super(name, ordinal);
    }

    public static Season[] values() {
        return $VALUES.clone();
    }

    public static Season valueOf(String name) {
        return Enum.valueOf(Season.class, name);
    }
}

The compiler-generated constructor takes two implicit parameters — the constant's name as a String and its position as an int, called the ordinal — and passes them straight to Enum's own constructor. The JVM sees exactly this: a regular class, four static final fields, a private constructor. Every bit of "enum behavior" was handled by the compiler before this class file ever existed.

§17.3

Objects, Not Constants — why "named constant" is the wrong mental model

Enum values are sometimes called "named constants," which suggests they're just integers with labels attached — the old Navigation pattern from §17.1, dressed up. That's exactly backward. An enum value is an object: an instance of a real class, capable of fields, methods, and behavior, same as any class from §2 onward. It's also not a compile-time constant the way a final int or a string literal is — a compile-time constant has to be a primitive or a String, fixed entirely by the source text, where SPRING and the rest are object references, built by running a constructor when the class loads (§17.2). The two keywords on their declaration are each pulling their own weight:

  • final guarantees the reference can never be reassigned — Direction.NORTH can never be made to point at a different object (§3).
  • static only says there's one shared field on the class, instead of one per instance.

Together they guarantee a single, permanent reference — not that the value was already known before the program ran, and not that the object on the other end of it can never change internally. A more accurate mental model: enum values are constant references to singleton objects, and it's the full set of those objects that's sealed at compile time, not necessarily their internal state.

enum Direction {
    NORTH, SOUTH, EAST, WEST;

    private int travelCount = 0;

    public void travel() {
        travelCount++;
    }

    public int getTravelCount() {
        return travelCount;
    }
}
Direction.NORTH.travel();
Direction.NORTH.travel();

System.out.println(Direction.NORTH.getTravelCount()); // 2

The NORTH object itself changed — its travelCount field went from 0 to 2 — while Direction.NORTH still refers to the exact same object on the heap the whole time. Nothing about the singleton guarantee was violated: there's still exactly one NORTH, and there always will be. It just isn't frozen.

The reference is constant; the object doesn't have to be

Since enums are classes, giving one a non-final field makes that field just as mutable as it would be on any other class (§11.3). Most enums never need this — Season and Direction as bare constant sets have nothing to mutate — but it's worth knowing the restriction lives in your own field declarations, not in some special enum-wide immutability rule. By convention, enum constant names are written in UPPERCASE — NORTH, SPRING, MONDAY — a widely followed style, not a compiler requirement.

§17.4

Comparing Enum Values — why == is not just allowed, but better

§8.6 warned against using == to compare Strings, because two equal-looking strings can be different objects. Enums are the opposite case. Because there is only ever one Season.SUMMER instance in the entire running program — the singleton guarantee from §17.2 — comparing two enum references for identity (§8.6's ==) and comparing them for logical equality (.equals()) can never disagree. There's no second, look-alike SUMMER object anywhere for .equals() to be needed against.

Season a = Season.SUMMER;
Season b = Season.SUMMER;
System.out.println(a == b);              // true
System.out.println(a == Season.SUMMER);  // true

a and b are two independent local variables (§1), yet both hold the identical address, 0x7200 — not because one points to the other, but because each assignment independently copied whatever address Season.SUMMER held at that moment (§3's aliasing). There's exactly one place that address could have come from — the one SUMMER slot living on Season.class, set once when the class loaded (§17.2) and never reassigned since, and pointing at exactly one Season object on the heap. Every reference to Season.SUMMER, anywhere in the program, resolves to a copy of that same one address — which is why both a == b and a == Season.SUMMER are guaranteed true. Whether it's two variables, or a variable against the field itself, == is always comparing copies of that one number, all naming the same one object, never two different objects that merely look alike.

== is not just permitted here — it's the better choice. It's null-safe (an == comparison against null simply evaluates to false, where a.equals(...) would throw a NullPointerException if a itself were null), and it's faster, since it compares two addresses directly instead of running equals()'s logic.

What == cannot do is cross between an enum and a primitive. An enum value is an object, never an int, no matter what its ordinal happens to be:

enum Season { WINTER, SPRING, SUMMER, FALL }

if (Season.SUMMER == 2) { ... } // compile error: incompatible types Season and int

To compare against a numeric position at all, go through ordinal() explicitly — the position §17.2's desugaring assigned each constant, in declaration order, starting at 0:

boolean res = Season.WINTER.ordinal() == 0; // true
§17.5

Type Characteristics — what an enum can and can't do

Every enum implicitly extends java.lang.Enum<E> — itself a generic type (§15), parameterized by the very enum type extending it, which is how values() and ordinal() can be declared once, in Enum, and still work correctly for every enum that will ever exist. Because Java classes can only extend one thing (§9.3), and that one thing is already Enum, two consequences follow immediately:

  • No further class inheritance. An enum cannot extend anything else — the slot is taken.
  • Interfaces are still fine. An enum can implement as many interfaces as it likes (§13.5), used polymorphically exactly like any other implementing class. The standard library does this routinely — java.nio.file.StandardCopyOption, for instance, is declared enum StandardCopyOption implements CopyOption.

A related question: can you extend an enum to add more constants later? No — and not as an incidental restriction, but as the direct consequence of the single-instance guarantee §17.2 built. Imagine, hypothetically, that it were allowed:

enum Direction { NORTH, SOUTH, EAST, WEST }

// not allowed — but imagine if it were
class ExtendedDirection extends Direction {
    NORTHEAST, NORTHWEST; // new instances of type Direction
}

Direction would now have six possible instances instead of four, depending on what's been loaded — and every guarantee this chapter has built so far would be gone: the "closed set" from §17.1 wouldn't be closed, and any code that assumed it had already seen every Direction could be wrong. To prevent exactly this, every enum is implicitly final as well — a modifier the compiler adds automatically, and one you're not allowed to write yourself:

final enum Size { S, M, L; } // compile error: illegal modifier for the enum

Writing final explicitly is redundant, since the compiler already guarantees it — and the language specification simply disallows redundant modifiers here rather than silently ignoring them.

Feature Ordinary class Enum
Instantiated with new? Yes No — constructor is always private
Fields and methods? Yes Yes
Implements interfaces? Yes Yes
Extends another class? Yes No — implicitly extends Enum<E> already
Usable in a switch? No (except via a wrapper type or String) Yes — and checked for exhaustiveness

That last row is worth a brief, concrete look, even though switch's full generality is left for a future chapter. In its simplest form, a switch compares one value against a list of possibilities:

Direction d = Direction.NORTH;

switch (d) {
    case NORTH -> System.out.println("Moving north");
    case SOUTH -> System.out.println("Moving south");
    case EAST  -> System.out.println("Moving east");
    case WEST  -> System.out.println("Moving west");
}

This is the direct replacement for §17.1's if/else chain on raw ints — same idea, but now the compiler knows the full set of Direction values ahead of time, and can check that every one of them is handled (an exhaustive switch), something an if/else chain on an int could never offer. The JVM also compiles a switch over an enum into a jump table — a direct lookup by ordinal, rather than a sequence of comparisons — which is part of why the row above says "Yes" so plainly.

When to use an enum

Reach for one when the set of values is fixed and known at compile time (statuses, directions, days, piece types), when raw numbers or strings would otherwise stand in for a meaningful category, or when values need real behavior attached (§17.6). Avoid one when the set can grow from outside the code — from a database row or a config file — or when the values are genuinely numeric and need arithmetic done on them; an enum is an object, never a number to compute with.

§17.6

Fields, Constructors, and Methods — an enum is still a class

An enum constant carries two pieces of information for free — its name and its ordinal (§17.2). Every example so far has stopped there, treating each constant as little more than a label. But recall §17.1's Navigation.move: every direction needed the same one piece of behavior — print a message — and the only place that behavior could live was outside the constants entirely, in an if/else chain that had to check identity before it could act. Scale that up — a coin needing its face value, an HTTP status needing a numeric code and a human-readable message — and the external-lookup approach gets worse with every new piece of data: a parallel array here, a switch there, each one a second place that has to stay in sync with the constants it describes.

An enum constant doesn't have to settle for that, because it's a real object (§17.3), and a real object can hold its own state and answer its own questions. A coin has a face value; an HTTP status has a numeric code and a message. Fields let that data live directly on each constant, instead of being looked up separately in a map or a parallel switch every time it's needed:

enum Coin {
    PENNY(1), NICKEL(5), DIME(10), QUARTER(25); // passing a value to the enum's own constructor

    private final int cents;

    Coin(int cents) {
        this.cents = cents;
    }

    public int getCents() {
        return cents;
    }
}
Coin.QUARTER.getCents(); // 25
Coin.DIME.getCents();    // 10

Writing PENNY(1) passes 1 to Coin's constructor, run once for that constant when the class loads. A slightly richer example carries two fields and a derived method:

enum HttpStatus {
    OK(200, "OK"),
    NOT_FOUND(404, "Not Found"),
    INTERNAL_SERVER_ERROR(500, "Internal Server Error"),
    UNAUTHORIZED(401, "Unauthorized"),
    FORBIDDEN(403, "Forbidden");

    private final int code;
    private final String message;

    HttpStatus(int code, String message) {
        this.code = code;
        this.message = message;
    }

    public int getCode()       { return code; }
    public String getMessage() { return message; }
    public boolean isError()   { return code >= 400; }
}

isError() is the detail worth noticing: it's not a stored field, but a small computation over one that is. An enum constant isn't limited to handing back the data it was built with — like any object, it can derive new facts from its own state, exactly the way an ordinary class method would. HttpStatus.NOT_FOUND.isError() is true without ever storing a boolean anywhere.

The constructor is always private

Enums don't support new at all — creating an instance is illegal from outside the class, and, as §17.2's desugaring showed, the compiler-generated constructor is private even inside it. This isn't optional: every enum constructor is implicitly private, the modifier may be omitted but never changed, and writing public or protected is a compile error:

enum Flavor {
    VANILLA, CHOCOLATE, STRAWBERRY;

    public Flavor() {} // compile error: illegal modifier for the enum constructor; only private is permitted
}

Each constructor call happens exactly once, at class-loading time (§17.2), not whenever a constant is first mentioned in your own code:

enum Season {
    WINTER, SPRING, SUMMER, FALL;

    Season() {
        System.out.print("construct ");
    }
}

public class Main {
    public static void main(String[] args) {
        System.out.println("main");
        System.out.print(Season.WINTER + " ");
        System.out.print(Season.SPRING);
    }
}
main
construct construct construct construct WINTER SPRING

"main" prints first — the constructor calls only run once something finally references Season, which happens on the very next line. At that point, all four constructors run together, in declaration order, before WINTER is available to be printed at all. Nothing was constructed lazily per-constant; the whole enum loads as one unit.

Instance and static methods

Fields solve half the original problem — the data. The other half is the same question §17.1's move answered externally for every direction: given a constant, what does it actually do, or what can be derived from it? A method attaches that logic to the constant itself, the same way isError() just did above, so the answer travels with the value instead of living in a separate function that has to be handed the value and asked. Like any class, an enum can declare both instance methods and static ones:

enum DayOfWeek {
    MON, TUE, WED, THU, FRI, SAT, SUN;

    public boolean isWeekend() {
        return this == SAT || this == SUN;
    }
}
DayOfWeek day = DayOfWeek.FRI;

if (day.isWeekend()) {
    System.out.println("Have a good weekend!");
} else {
    System.out.println("Work, work, work!");
}

isWeekend() reads exactly like a method on any other class, because that's all it is — this refers to whichever constant it was called on, compared against the two constants that count as a weekend. Nothing about it being an enum constant changes how instance methods work; the fixed set of instances from §17.2 is the only thing that's special here.

A static method can search across every constant using values() — the method §17.2's desugaring showed returning a defensive copy of the constant array:

enum ResponseGroup {
    INFORMATIONAL(1), SUCCESSFUL(2), REDIRECTION(3), CLIENT_ERROR(4), SERVER_ERROR(5);

    private final int group;

    ResponseGroup(int group) {
        this.group = group;
    }

    public static Optional<ResponseGroup> resolve(int statusCode) {
        int group = statusCode / 100;
        for (ResponseGroup r : ResponseGroup.values()) {
            if (r.group == group) {
                return Optional.of(r);
            }
        }
        return Optional.empty();
    }
}
var group = ResponseGroup.resolve(200);

if (group.isPresent()) {
    System.out.println(group.get()); // SUCCESSFUL
}

resolve is a static method precisely because looking up a constant doesn't belong to any one constant — it's a question about the whole set, so it's attached to the enum type itself rather than to any particular ResponseGroup. This is the general shape for turning an outside value (a status code fetched from somewhere) into the matching enum constant: loop over values(), compare, return the match.

Optional<T>, briefly

resolve might reasonably have nothing to return — a status code outside every known range. Rather than returning null (§2.3) and leaving every caller to remember to check for it, Optional<T> is a small standard-library wrapper that makes "might legitimately be absent" part of the return type itself: Optional.of(r) wraps a present value, Optional.empty() represents no value at all, and isPresent()/get() ask and unwrap explicitly, so absence has to be handled rather than forgotten.

§17.7

Enums vs. Magic Numbers

§17.1 named the two bad patterns this fixes: a bare magic number with no name at all, and Navigation's named int constants, which fix readability but nothing else. The comparison below lays out exactly what an enum buys back over both:

Feature Magic numbers Enums
Readability Poor — "what does 2 mean?" High — Status.CANCELLED reads on sight
Type safety Weak — checked as an int, not as a status High — checked as a Status, one of its declared values
Maintainability Difficult — every 2 has to be found and checked Easy — change the one definition
Namespace Global, easily polluted Encapsulated inside the enum type
Validation Manual — if (val < 4), easy to forget Automatic — an invalid value can't be constructed

Navigation's named int constants (§17.1) sit partway between the two columns above — readability fixed, every other row still failing exactly as it did there: the type checked is still int, never "a valid direction"; the constants share one global namespace regardless of domain; and printing one for debugging shows a bare number.

Accessing enum constants

Enum constants are implicitly public static final (§17.2's desugaring made this explicit) and are read with ordinary dot notation:

Season sn = Season.SUMMER;
System.out.println(sn); // SUMMER

That output comes from toString() — every enum inherits an implementation from Enum that prints the constant's own name, with no extra work required. A static import drops the type prefix when a single enum is used heavily throughout one file:

import static com.myapp.Color.*;

Color favorite = RED; // instead of Color.RED

And this closes the loop back to §17.1's opening problem. An enum used as a parameter type gets the same compile-time protection §17.1 showed missing from move(int):

public void setDirection(Direction direction) { ... }

setDirection(Direction.NORTH); // correct
setDirection(0);               // compile error — even though NORTH's own ordinal is 0

Even though Direction.NORTH happens to have ordinal 0 internally, 0 itself is never accepted where a Direction is expected. The type system enforces exactly the boundary §17.1's int-based version couldn't.

§17.8

Closed Sets vs. Evolving Reality — choosing safety or flexibility, per dimension

§17.1 asked the question this section answers concretely: closed by nature, or could it grow? The right answer isn't always the same one, even within a single program — here's what each side of that trade actually looks like in code.

Keep the enum when the domain is genuinely stable and you want the compiler to force every call site to keep up:

enum BodyType { SOLID, ACOUSTIC, SEMI_HOLLOW }
enum PickupType { NONE, MAGNETIC, PIEZO }

This buys exhaustive switches (§17.5) and a clear, closed domain, at the cost of needing a code change for every new value — worth it when you own the domain and it rarely changes.

Reach for an open value type when new values can legitimately appear without a code change — from a database row or an external API, for instance. A record (§16) wrapping a single value is the simplest version:

record BodyType(String value) { }
record PickupType(String value) { }

record Guitar(
        String name,
        int numberOfStrings,
        BodyType bodyType,
        PickupType pickupType) { }

This accepts a brand-new value like new BodyType("CARBON_FIBER_HYBRID") with no redeploy at all — at the cost of losing exhaustiveness checking entirely, and needing some validation layer of your own, since the compiler no longer limits what a BodyType can hold.

A middle ground keeps a registry of known values while still allowing new ones to be discovered at runtime:

class BodyTypeRegistry {
    private static final Map<String, BodyType> CACHE = new HashMap<>();

    static {
        register("SOLID");
        register("ACOUSTIC");
        register("SEMI_HOLLOW");
    }

    static BodyType register(String value) {
        BodyType type = new BodyType(value);
        CACHE.put(type.value(), type);
        return type;
    }

    static BodyType getOrCreate(String value) {
        // computeIfAbsent: return the cached BodyType if value is already a key,
        // otherwise build one with BodyType::new (a method reference — §19.7 covers these) and cache it first
        return CACHE.computeIfAbsent(value.toUpperCase(), BodyType::new);
    }

    static Collection<BodyType> knownValues() {
        return Collections.unmodifiableCollection(CACHE.values());
    }
}

The known values are pre-registered in a static block (§17.2's ordering rules apply here too), exactly the way an enum's constants would be — but getOrCreate leaves the door open for a value nobody anticipated at compile time, backed by a Map (§18.6 covers the Collections Framework in full) and a constructor reference (§19.7 covers method references). Nothing here is a fundamentally new idea, just tools this tutorial hasn't reached yet — Records, the Collections Framework, and method references, combined to sit deliberately between an enum's total safety and a bare value type's total flexibility. Worth a second look once those chapters are behind you, if this registry pattern doesn't fully click on a first read.

Choose per dimension, not per program

A single program can use enums for some fixed categories (say, DayOfWeek) and open value types for others (say, product categories fetched from a catalog service) — the choice is made once per concept being modeled, not once for the whole codebase. The question from §17.1 is worth asking every time: is this particular set closed by nature, or could it grow?

§17.9

Key Takeaways

  • An enum models a set of values that's closed by nature — every member known upfront, membership never changing. If the set can grow after the code ships, an enum is the wrong tool; consider a value type (a record) or a registry instead (§17.8).
  • Enum constants are objects, not labeled integers — real instances of a real class, created once by the JVM when the class loads, and held in static final fields the compiler generates for you.
  • static final fixes the reference; it doesn't make the object immutable. An enum constant can carry mutable fields, though most never need to.
  • Every enum implicitly extends java.lang.Enum<E> and is implicitly final — consequences of the same single-instance guarantee that makes the set closed in the first place. Interfaces remain fully available.
  • Because there's only ever one instance per constant, == is not just legal but the better comparison — null-safe and faster than .equals().
  • Enums can have fields, constructors (always private), and both instance and static methods, exactly like an ordinary class — the fixed set of instances is the only thing that's special.
  • Compared to magic numbers or public static final int constants, enums add real type safety, a real namespace, and a real string representation, at the cost of needing a source change whenever the set of values grows.
