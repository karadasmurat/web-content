Chapter 14

Exceptions

Some failures have no honest value to return at all — dividing by zero, a file that isn't there, an index past the end of an array. §2.3 already used the word informally for exactly this shape of problem. This chapter builds it out in full: what an exception actually is, the hierarchy Java organizes them into, the difference between one the compiler forces a caller to confront and one it doesn't, and the syntax — try, catch, finally, throw, throws — for raising and handling them.

§14.1

The Problem — when there's no sensible value to return

A program, most of the time, does exactly what it's meant to: each piece of work finishes, and whatever depends on its result picks up from there, assuming everything before it went fine. Picture a simple reporting tool, built around three pieces of work in this order — read a file, process what's in it, produce a report from the result:

Start
  │
  ▼
Read the file
  │
  ▼
Process what's in it
  │
  ▼
Produce the report
  │
  ▼
Finish

Call this the happy path: everything completes, in this order, and there's nothing more interesting to say about it than that. This diagram deliberately says nothing about which method calls which — §14.2 fills that in, and the order shown here turns out to survive it completely unchanged.

Unexpected situations

Real programs interact with a world outside their own control, and every one of those steps is a place that world can fail to cooperate:

Open a file       → file missing
Connect to a database → database unavailable
Parse a number    → invalid input
Read network data → connection lost

Each of these is the program arriving at a situation its normal logic simply has no next step for — not a wrong answer to compute around, but no honest answer to give at all.

What are the options?

Take the very first step from the happy path above — reading the raw file, before anything has been processed or reported on:

String contents = readFile("sales.txt");

What should happen if sales.txt doesn't exist? A few genuinely different answers are possible.

Return a special value.

String contents = readFile("sales.txt");

if (contents == null) {
    // ...handle the missing file...
}

Return a special value, which has a special meaning. Pick some particular value, reserve it, and give it a name so a reader isn't left decoding something unexplained — for a method returning an index or a count, that might be an int like -1:

static final int ERROR_FILE_NOT_FOUND = -1;

int lineCount = countLines("sales.txt");

if (lineCount == ERROR_FILE_NOT_FOUND) {
    // ...handle the missing file...
}

The trouble shows up the moment every possible value is already a legitimate one. Suppose the method were this instead:

int readTemperature()

-1 can no longer be reserved for "reading failed" — it might be an entirely real temperature. There's no int left over to sacrifice as a special value, because the method's whole range is already spoken for. This is exactly why special values don't scale: they only work in domains that happen to have a value nothing legitimate would ever produce, and plenty of domains don't.

Why exceptions exist

The first two options work, but they share a cost that only shows up once a single method has more than one place where things can go wrong. Consider what generateReport() itself has to do internally: read the file, then process what it read. Each of those is a call site needing its own check, mixed directly into the logic that's actually trying to get something done:

if (readFileFails) {
    // handle error
} else {
    if (processDataFails) {
        // handle error
    } else {
        // ...build and return the report...
    }
}

Repeated at every step, that shape buries the actual point of generateReport() — read a file, process it, produce a report — under a pyramid of failure handling that has nothing to do with what the method is for. Exceptions exist to separate those two concerns entirely: normal program logic in one place, error handling in another, connected by a mechanism instead of by nested conditionals at every step:

String contents = readFile(path);
List<Entry> entries = processData(contents);
return buildReport(entries);

The happy path stays exactly as readable as §14.1 opened with — one statement after another — and none of it needs to mention failure at all. Handling failure doesn't disappear; it just isn't here anymore. Which raises the obvious next question: who, exactly, is now responsible for handling it?

Not every language answers that question the same way. Some — Rust's Result, Go's multiple-return-value convention — keep success and failure as two ordinary values riding along in the same return, and hand the responsibility straight to whoever called the method: inspect what came back, right there, before doing anything else with it. Nothing but the caller's own discipline enforces that inspection.

Java was designed around a different answer. A method's normal return type describes only success — Report, never "a Report, or a failure." Failure doesn't travel through that return type at all; it takes an entirely separate path, the one the rest of this chapter is about. Responsibility for handling it isn't assigned to the immediate caller by default — it belongs to whichever piece of code, anywhere up the chain of calls, actually catches it. §14.2 turns that idea into the precise mechanics: what that path up the chain of calls is actually called, and what happens if it runs out of road entirely.

§14.2

The Exception Lifecycle — unwinding the call stack until something agrees to handle it

This is the core mental model for the whole chapter, worth having firmly in place before any of the syntax: an exception unwinds the call stack until something agrees to handle it — or the program terminates if nothing does. Everything from here to the end of the chapter is detail on top of that one sentence.

Creation and throwing

When something goes wrong inside a method, that method creates an exception object and hands it off to the runtime — the JVM itself. The object carries information about the failure: its type, and typically a description of what happened. Creating that object and handing it to the runtime is called throwing an exception, and it's always done with the throw statement (§14.6), no matter what code is doing the throwing — your own code, a third-party library, or the JVM acting on its own.

There are two different ways the exception object actually gets created, though both end the same way — thrown:

  • Explicit creation. Nothing implicit at all: the code calls new itself, and throws the result itself.

    void withdraw(double amount) {
    if (amount > balance) {
        throw new InsufficientFundsException("Not enough balance");
    }
}

  • Implicit creation. Many exceptions are constructed automatically, by the JVM, the instant it detects a violation — no new anywhere in sight in the code that triggers it:

    int[] arr = new int[3];
int x = arr[5]; // the JVM constructs and throws ArrayIndexOutOfBoundsException on your behalf

Either way, the moment an exception is thrown, the method it was thrown from stops executing immediately, right there — no further line in that method runs, and no value is ever returned from it.

The search for a handler

§14.1's happy path completed three pieces of work in order. Here's the piece §14.1 left out: main() calls generateReport(), and generateReport() is the one actually doing the work — it calls readFile() first to get the raw data, then calls processData() to turn that into something reportable, then assembles the report itself from the result:

void main() {
    Report report = generateReport();
}

Report generateReport() {
    String contents = readFile(path);      // called first
    List<Entry> entries = processData(contents); // called next
    return buildReport(entries);
}

The chain of calls actually in progress at any one moment — main() waiting on generateReport(), which is waiting on whichever of readFile() or processData() it's currently running — is the call stack.

Say readFile() is the one running, and the file it was asked to open is missing. At that exact moment, the call stack is just three frames deep: main(), waiting on generateReport(), waiting on readFile() — processData() was never even called yet, since generateReport() was still waiting on readFile()'s result before it could call anything else.

Exception propagation

If readFile() throws an IOException and doesn't catch it, the exception moves up to generateReport(). If generateReport() doesn't catch it either, it keeps moving up, to main(). This upward movement, one frame of the call stack at a time, is exception propagation — and it stops the instant something along the way actually catches it.

Underneath that upward movement, the JVM is asking one question at each frame it reaches: does this one know how to handle an exception of this type (§14.5 covers the syntax for that)? If not, that frame is abandoned entirely — this is the "unwinding" the mental model opened with, and it's why nothing after the original throw, in any of those methods, ever runs — including processData(), which was waiting to be called and now never will be.

Program termination

If that search reaches the very bottom of the call stack — every method unwound, nothing anywhere willing to catch it — the program, or at minimum the one thread (§22.2) that was running it, terminates.

"I give up. You deal with it."

An exception is Java's way of saying exactly that: I don't know what to do about this right now, so I'm handing the problem to whoever called me. Writing a method that might fail always comes down to the same choice, for every exception it might let escape: deal with it right there, with a catch, or explicitly make it the calling code's problem, with a throws declaration (§14.6) — passing the same choice one level further up the exact call stack this section just described.

§14.3

The Exception Hierarchy — every exception is an object, in a family tree

An exception in Java isn't a special language construct sitting outside the type system — it's an ordinary object (§2.2), an instance of an ordinary class, related to every other exception class the same way any classes are related: through inheritance (§9).

Every exception class ultimately extends Throwable, which splits into two branches immediately below it:

Throwable
├── Error                     (JVM-level trouble — not meant to be caught)
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception
    ├── RuntimeException      (unchecked — §14.4)
    │   ├── NullPointerException
    │   ├── ArrayIndexOutOfBoundsException
    │   ├── ArithmeticException
    │   └── NoSuchElementException
    └── IOException, and other checked exceptions (§14.4)

Error represents problems below the level a normal program is expected to recover from — the JVM running out of memory, or a stack overflowing from runaway recursion (§4). Catching an Error is legal but rarely useful: by the time one occurs, the JVM itself is often in no state to keep going reliably. Exception is the branch this chapter is actually about — problems a program can reasonably anticipate and respond to, from a missing file to a malformed number to an index that ran off the end of an array.

An exception is just an object, thrown

Every box in that tree is a class, constructed with new exactly like any other object — new ArithmeticException("/ by zero") is a perfectly ordinary constructor call. What makes it an exception isn't anything about the object itself; it's the throw keyword (§14.6) applied to it, which hands that object to the runtime specifically to interrupt normal execution and go looking for a handler.

§14.4

Checked vs. Unchecked — what the compiler actually enforces

Some failures are ones a well-written program should genuinely expect and plan for — a file that might not be there, a network request that might fail. Others are just bugs — an index that should never have gone out of range, a value that should never have been null. Java draws a hard line between these two categories, and backs it with the compiler:

  • A checked exception is one the compiler forces every caller to actually confront — either catch it (§14.5), or declare it in the caller's own throws clause (§14.6) — before the code is even allowed to compile.
  • An unchecked exception gets no such enforcement. Code is free to ignore the possibility entirely, and it will still compile.
// Bad — does not compile
void loadConfig(String path) {
    FileReader reader = new FileReader(path); // constructor declares throws FileNotFoundException — unhandled here
}

This is exactly what checked-exception enforcement exists to prevent. FileReader's constructor is declared throws FileNotFoundException in the standard library — not "might fail, who knows," but an explicit, compiler-visible declaration that this specific checked exception is possible. Because the declaration is there, Java refuses to even compile a caller that ignores it. Catching it, or declaring it in turn, is not optional:

// Good — compiles, because the possibility is acknowledged
void loadConfig(String path) throws FileNotFoundException {
    FileReader reader = new FileReader(path);
}

An unchecked exception gets none of this ceremony, deliberately. It's worth seeing exactly what Java spared every program from by making that choice, using a genuinely ordinary piece of business logic: a teacher's grade book, computing a class average from a completed set of exam scores:

class GradeBook {
    private final int[] scores;

    GradeBook(int[] scores) {
        this.scores = scores;
    }

    int scoreAt(int index) {
        return scores[index]; // index comes from the caller — genuinely might be out of range
    }

    double average() {
        int total = 0;
        for (int i = 0; i < scores.length; i++) {
            total += scores[i];
        }
        return (double) total / scores.length;
    }
}
public class ReportCard {
    public static void main(String[] args) {
        GradeBook gradeBook = new GradeBook(new int[] { 90, 85, 78, 92 });
        System.out.println("First score: " + gradeBook.scoreAt(0));
        System.out.println("Class average: " + gradeBook.average());
    }
}

scoreAt(int index) is the honest case: index comes from whoever calls it, and nothing about GradeBook can guarantee it's in range the way average()'s own loop bound guarantees safety for itself. Calling gradeBook.scoreAt(10) against a four-score array genuinely does throw. If Java had made ArrayIndexOutOfBoundsException checked, this is exactly the kind of method that rule was supposedly protecting against — and the ceremony it would demand:

// Bad — imagine if array access worked like a checked exception
int scoreAt(int index) throws ArrayIndexOutOfBoundsException {
    return scores[index];
}
// ...and now every caller needs one, whether or not the index is ever actually wrong
try {
    System.out.println("First score: " + gradeBook.scoreAt(0));
} catch (ArrayIndexOutOfBoundsException e) {
    // ...what would this even do? "0" is obviously fine — but the compiler can't tell that
}

A human reading scoreAt(0) can see the index is fine; the compiler enforcing a checked exception can't — it has no way to evaluate whether a given call site is actually safe, so it would demand the identical catch at every single call, correct index or not. Multiply that across every array access in every program ever written, and checked-exception ceremony stops being a safeguard and starts being noise nobody reads. That's the actual dividing line: because ArrayIndexOutOfBoundsException is unchecked, scoreAt() compiles exactly as first written, with none of this —

// Good — no ceremony, because this is what unchecked buys you
int scoreAt(int index) {
    return scores[index]; // still throws ArrayIndexOutOfBoundsException if index is bad — just not enforced
}

— and on the occasion a caller genuinely does pass a bad index (a real bug: a typo, an off-by-one, an index meant for a different GradeBook entirely), the exception still happens, still propagates, and still crashes exactly where the bug actually is — which is a far more useful outcome for whoever has to fix it than a catch block that exists at every call site just in case, quietly hiding a caller's mistake instead.

Which category a given exception falls into isn't a special marker stuck on the class — it's simply where that class sits in §14.3's hierarchy. Every subclass of RuntimeException is unchecked; every other subclass of Exception is checked. Error and its subclasses are technically unchecked too, but since they're not meant to be handled in the first place, that distinction rarely comes up in practice.

Checked Unchecked
Hierarchy Exception, excluding RuntimeException RuntimeException and its subclasses
Compiler enforcement Must be caught or declared — code won't compile otherwise None — catching it is always optional
Typical cause A condition outside the program's control — a missing file, a network failure A programming mistake — a bad index, a null dereference, invalid arguments
Example IOException NullPointerException, ArrayIndexOutOfBoundsException

That "typical cause" row is the practical rule of thumb worth internalizing: reach for a checked exception when a caller genuinely might encounter the failure through no fault of their own, and needs to be reminded to plan for it — a file that might not exist, a connection that might drop. Reach for unchecked when the failure signals a bug in the calling code itself — an index that should never have been out of range, an argument that violated a documented precondition. Forcing every caller to catch a NullPointerException around every dereference would make checked exceptions a burden instead of a safeguard; leaving a missing file uncaught would let a program silently limp forward on a false assumption. §14.7 returns to this same question when deciding which to extend for a custom exception.

§14.5

Handling Exceptions — try, catch, and finally

A try block marks a region of code whose exceptions should be caught rather than left to propagate; each catch that follows handles one specific exception type:

try {
    int result = 10 / 0; // throws ArithmeticException
    System.out.println(result); // never reached
} catch (ArithmeticException e) {
    System.out.println("Can't divide by zero: " + e.getMessage());
}

The instant 10 / 0 throws, everything else remaining in the try block is skipped — result is never assigned, the println never runs — and control jumps directly to the first catch whose declared type matches the exception that was thrown. e.getMessage() retrieves whatever description the exception was constructed with (§14.3's new ArithmeticException("/ by zero"), in this case).

More than one catch can follow the same try, each handling a different possible failure:

try {
    int[] values = { 1, 2, 3 };
    System.out.println(values[5] / 0);
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Index out of range.");
} catch (ArithmeticException e) {
    System.out.println("Division by zero.");
}

Java checks each catch in order, top to bottom, and runs the first one whose type matches — only one catch block ever runs per exception.

The Rule of Specificity

When handling a hierarchy of related exceptions, the order of catch blocks is critically important — get it wrong, and the code fails to compile at all, not just at runtime. The Golden Rule: catch more specific (subclass) exceptions before more general (superclass) ones. A catch for a supertype placed before a catch for its subtype would catch the subtype's exceptions too (polymorphism, §9.5, applies here exactly as it does anywhere else) — put Exception itself at the top, and it silently swallows every single error, leaving every specific handler beneath it unreachable. Java actually refuses to compile a catch block it can prove can never run this way. The rule is simple: order every catch from most specific to most general.

When two or more exception types genuinely deserve identical handling, multi-catch combines them into a single block with a |:

try {
    riskyOperation();
} catch (ArrayIndexOutOfBoundsException | ArithmeticException e) {
    System.out.println("Something went wrong: " + e.getMessage());
}

This is exactly equivalent to writing the same catch body twice, once per type — multi-catch exists purely to avoid that duplication, not to change what gets caught.

Catching the wrong exception

For a checked exception (§14.4), the compiler doesn't just check that some catch exists — it checks that the catch actually covers everything the declared type could mean. A method declared throws IOException is free to throw any subclass of it at runtime — FileNotFoundException, SocketException, or others §14.6's polymorphic throwing already named — and catching only one narrower subclass doesn't account for the rest:

void riskyBusiness() throws IOException { } // declares the checked exception

void process() {
    try {
        riskyBusiness(); // compile error
    } catch (FileNotFoundException e) { // doesn't cover every IOException riskyBusiness() could throw
        System.out.println("We have an exception!");
    }
}

This doesn't compile. riskyBusiness() is declared to throw IOException — any IOException, not specifically a FileNotFoundException — and the only catch present handles just that one narrow subtype. The compiler can see that a SocketException, or any other IOException that isn't a FileNotFoundException, would leave this try block with nowhere to go, so it refuses to compile process() at all.

Fixing this means the catch clauses actually have to cover everything the try block might throw — catching the exact declared type, a broader supertype of it, or the narrower subtype alongside the exact declared type:

void process() {
    try {
        riskyBusiness();
    } catch (FileNotFoundException e) {
        System.out.println("Specifically missing a file.");
    } catch (IOException e) { // catches every other IOException riskyBusiness() might throw
        System.out.println("Some other I/O problem.");
    }
}

finally: the block that always runs

A finally block, attached after a try (and any catches), runs no matter what happens inside the try — whether it completed normally, threw an exception that got caught, or threw one that didn't:

try {
    System.out.println("Trying...");     // prints: Trying...
    int result = 10 / 0;                 // throws ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("Caught: " + e.getMessage()); // prints: Caught: / by zero
} finally {
    System.out.println("This always runs."); // prints: This always runs.
}

"Always" is meant precisely here: finally runs even if the try block returns a value, even if the catch block itself throws a new exception, even if neither block finishes normally at all.

Here's exactly why that guarantee matters. Say a connection is opened, and something throws before it's closed:

// Bad — the connection leaks if sendData() throws
Connection conn = openConnection();
sendData(conn);       // throws IOException
conn.close();         // never reached

conn.close() was right there on the next line, and it still never runs — the exception from sendData(conn) skips straight past it, exactly as §14.2 already described. The connection stays open, forever, unless something further up the call stack happens to close it, which nothing here promises. Putting the cleanup in a finally block fixes this directly:

// Good — close() always runs, even if sendData() throws
Connection conn = openConnection();
try {
    sendData(conn);
} finally {
    conn.close();     // runs whether sendData() succeeded, threw, or was caught elsewhere
}

This is exactly the cleanup finally exists for: closing a connection, releasing a resource, anything that would otherwise need to be duplicated at the end of both the success path and every failure path. §14.9 covers the modern alternative for exactly this cleanup case, which most new code reaches for instead of a manual finally.

Double trouble: an exception inside catch or finally

Nothing stops a catch or finally block from throwing an exception of its own — and a real one, saving an order to a database, is exactly the kind of place both can fail at once:

// Bad — the real problem disappears
try {
    saveOrder(order);       // throws OrderException: "Invalid order data"
} finally {
    connection.close();     // ALSO throws IOException: "Connection already closed"
}

saveOrder(order) throws first, with the actual, useful problem: the order data was invalid. But finally still runs before that exception is allowed to propagate (§14.5 already established this), and connection.close() throws too. The second exception doesn't join the first — it replaces it. The IOException about a connection is what reaches the caller; the OrderException that would have actually explained what went wrong is simply gone, discarded the moment the second exception was thrown on top of it.

The fix is to keep cleanup failures from ever being allowed to overwrite whatever the try block was already reporting:

// Good — the real problem still gets through
try {
    saveOrder(order);       // throws OrderException: "Invalid order data"
} finally {
    try {
        connection.close();
    } catch (IOException closeException) {
        System.err.println("Also failed to close connection: " + closeException.getMessage());
        // the OrderException from saveOrder() is still the one that propagates
    }
}

Wrapping the cleanup itself in its own try/catch means a failure while closing the connection gets logged rather than thrown — it's real, and worth knowing about, but it's not allowed to bury the original failure that actually caused the problem. §14.9's try-with-resources handles this exact situation automatically, keeping both exceptions instead of losing one, which is one more reason it's usually preferred over a hand-written finally.

Nested try/catch

The close() call above wasn't the only place a fallback action could fail — it's a general problem. Picture loading a user's preferences: if the primary file is corrupted, falling back to a bundled set of defaults is reasonable. But that fallback is itself a file read, and file reads fail:

// Bad — if loadDefaultPreferences() also fails, nothing here says why
try {
    loadPrimaryPreferences();
} catch (IOException e) {
    loadDefaultPreferences(); // itself throws IOException — propagates uncaught
}

If loadDefaultPreferences() throws, that new IOException propagates straight out of this method entirely uncaught — indistinguishable from the original failure, with no indication that this was actually the fallback breaking down, not the primary attempt. The recovery action needs handling of its own, distinct from the failure it was recovering from:

// Good — the fallback's own failure gets its own, more specific response
try {
    loadPrimaryPreferences();
} catch (IOException primaryError) {
    try {
        loadDefaultPreferences();
    } catch (IOException fallbackError) {
        throw new PreferencesUnavailableException(
            "Neither the primary nor the default preferences could be loaded", fallbackError);
    }
}

A try block nested inside a catch (or, just as legitimately, inside another try) is handled exactly like any other — its own catch only sees exceptions thrown within its own try, entirely independent of the outer one. Here, that means a failure in the fallback gets converted into a single, honest exception (§14.8's chaining preserves fallbackError as the cause) that says plainly what actually happened: not one preferences source failed, but both.

Summary

  1. You place code that might throw an exception inside a try block. The try block wraps the code segment that might throw an exception — the "protected" block. If everything inside succeeds, every catch block is skipped entirely.

  2. You define one or more catch blocks to intercept specific types of exceptions. A catch block declares the exception type it's willing to handle; inside it, you write the code that responds to that particular failure — recovering from it, or at least managing it gracefully. That might involve:

    • Logging: logger.error("Error processing file", e);
    • A user-facing message: System.err.println("An error occurred: " + e.getMessage());
    • Partial recovery: re-prompting the user for input, or retrying an operation after a delay.
    • Setting a flag: operationSuccessful = false;
    • Converting exception types: throwing a different, perhaps more abstract or custom (§14.7), exception instead.

    A try can be followed by several catch blocks, each handling a different exception type. Order matters — see The Rule of Specificity above.

  3. The finally block contains code that must run regardless of whether an exception occurred in the try block or was caught by a catch block. It's primarily used for cleanup — critically, for closing resources like file streams, database connections, and network sockets, preventing exactly the kind of leak this section opened with.

§14.6

throw and throws — raising an exception, and declaring one

Two keywords, easy to conflate by how similar they look, doing entirely different jobs. throw is a statement — it actually raises one specific exception instance, right now:

static int divide(int a, int b) {
    if (b == 0) {
        throw new ArithmeticException("Cannot divide by zero");
    }
    return a / b;
}

throw takes a Throwable object — built with new, exactly as §14.3 described — and hands it to the runtime to interrupt execution immediately; nothing after a throw statement in the same block ever runs.

throws, by contrast, is part of a method's signature, not something that executes — it declares which checked exceptions (§14.4) a method might let escape, so that every caller is compiler-enforced to deal with them:

void readConfig(String path) throws IOException {
    // ...code that might throw IOException...
}

A method that calls readConfig now has exactly two options for its own checked-exception obligation: catch IOException itself, or declare throws IOException in its own signature and pass the responsibility one level further up the call chain. An unchecked exception (§14.4) never needs a throws declaration — nothing forces one, though a method is still free to add one for documentation's sake even when it isn't required.

Calling a method that declares a checked exception

The moment a method's signature declares a checked exception, calling it hands that exception straight to the caller as a decision they now have to make — there's no third option, and no way to simply ignore it. Every checked exception a method can throw is effectively part of its public interface, exactly as visible to a caller as its parameter list or return type: whoever calls the method needs to know what it might throw, precisely so they can decide what to do about it — catch it there (§14.5), or declare it themselves and pass the decision one level further up.

One consequence of this rule is easy to miss: a method can declare a checked exception it never actually throws, anywhere in its own body:

void eatCarrot() throws NutritionException {
    System.out.println("Munch.");
    // NutritionException is never actually constructed or thrown here
}

The compiler never checks whether a method's body actually produces every exception it declares — only that whoever calls a method handles or declares every checked exception that method's signature lists. So eatCarrot()'s declaration alone is enough to bind every one of its callers:

void feedRabbit() throws NutritionException { // required, even though eatCarrot() never actually throws it
    eatCarrot();
}

This is genuinely useful when a method is expected to throw something in a future version but doesn't yet: declaring the exception now means every caller already handles or declares it, so adding the real throw later never breaks a single caller that compiles against this method today.

Polymorphic throwing

A method that saves a file can fail in several distinct ways at once — the network drops mid-write, the file itself has vanished, the destination directory no longer exists. Each of those is a real, separate exception type in the standard library, and declaring all of them individually gets unwieldy fast:

// Bad — every new failure type means editing this signature again
public void saveFile() throws SocketException, FileNotFoundException, NoSuchFileException {
    // ...
}

A caller now has to know about, and handle, three specific types — and a fourth I/O failure the method starts throwing next year would mean updating this signature and every caller's catch blocks all over again. But SocketException, FileNotFoundException, and NoSuchFileException all extend the same class: IOException (§14.3's hierarchy). Declaring the parent instead of the children covers the whole family at once, exactly the way a supertype reference can hold any of its subtypes (polymorphism, §9.5):

// Good — one declaration, whole family of I/O failures covered
public void saveFile() throws IOException {
    // ...
}

The method itself is free to throw a SocketException today and a brand-new IOException subclass next year, without its signature ever needing to change. A caller only needs to know that some kind of I/O problem is possible — catch (IOException e) handles every one of them, named or not, the same way a variable declared as a supertype can be handed any subtype without the caller needing to know which one specifically arrived (§9.5 covered this same substitutability for ordinary method parameters; a throws clause gets it for free, too).

§14.7

Custom Exceptions — modeling a domain-specific failure

The standard library's exception classes cover generic problems — a bad index, a missing file — but a program's own domain often has failures with no natural standard-library name. Without one, a method is left borrowing something generic and stuffing the real problem into a string:

// Bad — no exception type actually means "insufficient funds"
void withdraw(double amount, double balance) throws Exception {
    if (amount > balance) {
        throw new Exception("Cannot withdraw " + amount + " from balance of " + balance);
    }
}
try {
    withdraw(500, 100);
} catch (Exception e) {
    // catches insufficient funds — and every other possible failure this
    // method (or anything it calls) could ever throw, with no way to tell them apart
}

A caller here can't catch "insufficient funds" specifically — only Exception, the same catch-all that would also swallow a bug, a null dereference three calls deep, or anything else. Extending Exception or RuntimeException (§9.3) instead creates a new exception type that names the problem precisely, catchable on its own:

// Good — a real type for this one specific failure
class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String message) {
        super(message);
    }
}
void withdraw(double amount, double balance) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Cannot withdraw " + amount + " from balance of " + balance);
    }
}

The constructor does nothing but forward its message to Exception's own constructor via super(...) (§9.4) — everything getMessage(), the exception hierarchy, and every catch block already know how to do with an exception, InsufficientFundsException inherits for free. What's new is purely the name and the type: catch (InsufficientFundsException e) now catches exactly this domain failure and nothing else, distinct from an ArithmeticException, an IOException, or the generic Exception the bad version above was forced to use.

The choice of what to extend is exactly §14.4's checked-vs-unchecked question, applied to this one new type: extend Exception when callers genuinely need to be compiler-reminded to handle the failure (as above — a caller forgetting to plan for insufficient funds is a real, likely mistake); extend RuntimeException when the failure represents a caller's own programming error, the same way IllegalArgumentException does for a bad argument.

§14.8

Exception Chaining — preserving the original problem

Catching one exception only to throw a different, more domain-appropriate one is common — a low-level IOException caught and re-thrown as a ConfigLoadException, say. Done carelessly, that step throws away exactly the information that would explain what actually went wrong underneath:

try {
    loadFile(path);
} catch (IOException e) {
    throw new ConfigLoadException("Could not load configuration"); // original cause lost
}

Throwable has a constructor overload built specifically to avoid this — one that takes a cause alongside the message:

try {
    loadFile(path);
} catch (IOException e) {
    throw new ConfigLoadException("Could not load configuration", e); // e preserved as the cause
}

The new exception now carries the original one inside it, retrievable later with getCause(). A stack trace printed from the outer exception shows both — the higher-level failure a caller actually asked about, and, chained beneath it, the lower-level one that actually caused it. Whenever wrapping one exception in another, passing the original along as the cause costs nothing and keeps the real explanation from disappearing.

§14.9

Try-with-Resources — cleanup that can't be forgotten

A resource that must be closed — a file, a network connection, a thread pool — traditionally needed a manual finally block (§14.5) to guarantee that cleanup, even when an exception interrupted whatever was using it. But FileReader.close() is itself declared throws IOException — the exact "cleanup that can itself fail" problem §14.5 just walked through — so doing this properly meant a nested try/catch just to handle close() potentially throwing too:

// Bad — correct, but this much ceremony for one resource
void readFile(String path) throws IOException { // still required — FileReader(...) and reader.close() both declare it
    FileReader reader = new FileReader(path);
    try {
        // ...use reader...
    } finally {
        try {
            reader.close(); // must remember this, every time — and it can throw too
        } catch (IOException closeError) {
            System.err.println("Failed to close reader: " + closeError.getMessage());
        }
    }
}

Every class fit to be cleaned up this way implements AutoCloseable — an interface (§13) with exactly one method, void close() throws Exception. Try-with-resources is syntax built directly around that interface, specifically to make this ceremony unnecessary: declare the resource inside the parentheses of a try, and Java calls close() automatically the moment the block ends, success or failure, handling a failure in close() itself without a nested try/catch ever being written by hand:

void readFile(String path) throws IOException { // still required — try-with-resources doesn't remove this
    try (FileReader reader = new FileReader(path)) {
        // ...use reader...
    } // reader.close() is called automatically here
}

More than one resource can be declared the same way, separated by semicolons, and they're closed in the reverse of the order they were opened — the last one declared is the first one closed. §22.10 puts this exact mechanism to work for ExecutorService specifically; its close() is simply one particular AutoCloseable implementation, doing the graceful-shutdown work appropriate to a thread pool instead of releasing a file handle.

AutoCloseable and Closeable

Any resource declared inside a try's parentheses has to implement one of exactly two interfaces: java.lang.AutoCloseable, or java.io.Closeable.

public interface AutoCloseable {
    void close() throws Exception;
}
public interface Closeable extends AutoCloseable {
    void close() throws IOException;
}

Closeable actually extends AutoCloseable — it isn't a separate, competing interface, just a more specific one, overriding close() to narrow its declared throws Exception down to throws IOException. An overriding method is always free to narrow or drop the checked exceptions its parent declared, just never add ones the parent didn't (§22.9 shows this same narrowing again, for ExecutorService.close()). Reader — and through it, FileReader — implements Closeable, along with BufferedReader, Socket, Scanner, and InputStream: all specifically I/O resources.

AutoCloseable is the broader of the two, and it exists precisely because try-with-resources wasn't meant to stay limited to I/O. When it was introduced, declaring close() against the base Exception type — rather than the narrower IOException — made the interface general enough for any resource needing cleanup, I/O or not: a database connection, or a thread pool.

public interface ExecutorService extends Executor, AutoCloseable

ExecutorService (§22.9) implements AutoCloseable directly, not Closeable — shutting down a thread pool has nothing to do with I/O, so the broader interface is the correct fit. Either way, notice that both interfaces' close() declares a checked exception — Exception for AutoCloseable, IOException for Closeable — which is exactly why the "what try-with-resources does not do" callout below still applies to every resource declared this way, regardless of which of the two it implements.

Automatic execution of close()

In a try-with-resources statement, close() runs automatically as part of exiting the try block — whether that exit is normal or via an exception — and it runs before any catch or finally attached to that same try. That ordering matters: it means a catch block here covers exceptions thrown by close() itself, not just ones thrown by the body of the try.

There's one important nuance to that, though. If the try body throws and close() also throws while cleaning up after it, only one of those two exceptions can actually propagate to the catch block — and it's the body's, not close()'s:

try (FileReader reader = new FileReader(path)) {
    throw new RuntimeException("Something went wrong reading");
    // reader.close() also throws IOException here, while unwinding
} catch (Exception e) {
    System.out.println(e.getMessage());          // "Something went wrong reading"
    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println("Also: " + suppressed.getMessage()); // the IOException from close()
    }
}

The exception from close() isn't discarded the way §14.5's "double trouble" showed a hand-written finally discarding one — it's attached to the primary exception as a suppressed exception, retrievable through getSuppressed(), without ever propagating on its own. This is exactly the improvement try-with-resources makes over a manual finally: both failures survive, instead of the second one silently overwriting the first.

What try-with-resources does not do

Try-with-resources automates exactly one thing: calling close() on the resource, guaranteed, regardless of how the block exits. That's the whole feature. It says nothing about exceptions themselves — a checked exception thrown from inside the try body still propagates exactly as it would without it, and still needs, per §14.4, either a catch block to handle it locally or a throws declaration to push the responsibility to the caller. Without one of those two, the compiler rejects the method — the same rule as always, unaffected by whether the resource happens to be declared in the try's parentheses.

§14.10

Exceptions Are Not Control Flow — reserved for what's actually exceptional

Everything so far has been about failures a program's normal logic has no next step for at all — a missing file, a bad index, a connection that dropped. Not every unsuccessful-looking outcome belongs in that category, and treating one that doesn't is one of the easiest ways to misuse everything else this chapter builds.

Consider a method that looks a user up by ID. Not finding one is often a routine, expected outcome — the ID simply doesn't exist yet, the same way a search coming up empty is still a completed search, not a failed one. Modeling that as an exception is a common mistake:

// Bad — "not found" isn't exceptional here
User user = findUser(id); // throws UserNotFoundException if missing
try {
    User user = findUser(id);
    // ...
} catch (UserNotFoundException e) {
    // ...
}

A try/catch used this way isn't handling a failure at all — it's an ordinary if/else wearing exception syntax, for a branch of logic that's a completely normal, everyday outcome of calling findUser. §20 covers Optional<T> in full, but the shape of the better fix is already visible here: a return value that represents "might not be there" directly and honestly, without treating a routine miss as if it were a breakdown.

The dividing line

An exception should represent something exceptional actually happening — a genuine failure, a violated assumption, a job that could not be completed. It should not represent one ordinary branch of a method's normal logic, reached for instead of an if purely because throwing felt like the more deliberate way to write it. If a caller is expected to see a particular outcome regularly, in the ordinary course of using a method correctly, that outcome is normal — not exceptional — no matter which syntax reports it.

§14.11

Best Practices & Anti-Patterns — habits worth building, and ones worth breaking

Everything in this chapter comes with real ways to misuse it. A few habits account for most of the trouble in practice — each one easy to fall into, and each with a straightforward fix.

Anti-pattern: swallowing an exception

// Bad — the failure vanishes without a trace
try {
    riskyOperation();
} catch (IOException e) {
    // ignored
}

An empty catch block doesn't make the problem go away — it just deletes every trace that it happened. Whatever riskyOperation() was trying to do silently didn't happen, and nothing here says so; a caller three layers up, or a user staring at a blank screen, has no way to know why. At an absolute minimum, a caught exception should be logged, even when there's genuinely nothing else to do about it:

// Good — the failure is invisible to the program's logic, not to whoever reads the logs
try {
    riskyOperation();
} catch (IOException e) {
    logger.error("riskyOperation failed", e);
}

Anti-pattern: catching Throwable or Error

// Bad — catches far more than was ever intended
try {
    doWork();
} catch (Throwable t) {
    // ...
}

Throwable sits above both branches of §14.3's hierarchy — this catches every Exception, but also every Error: an OutOfMemoryError, a StackOverflowError, conditions the JVM itself is in no state to recover from. Catching one of these doesn't fix anything; it just delays a crash that was already coming, often while masking the fact that something in the program is fundamentally broken. Catch Exception, not Throwable, unless there's a genuinely deliberate reason — logging a fatal condition right before the program exits, say — to see an Error at all:

// Good — catches real, recoverable problems, and nothing below Error
try {
    doWork();
} catch (Exception e) {
    logger.error("doWork failed", e);
}

Anti-pattern: logging the same failure at every level

// Bad — one failure, logged three times on its way up
void processData() throws IOException {
    try {
        readFile();
    } catch (IOException e) {
        logger.error("processData failed", e);
        throw e; // still propagates — and gets logged again by whoever catches it next
    }
}

Propagation (§14.2) means the same exception can pass through several methods on its way to a handler — and logging it at every one of those methods turns one real failure into a wall of duplicate log entries, each describing the identical problem. Log once, at the point that actually has enough context to act on it or report it meaningfully — typically much further up the call stack than where the exception was first thrown — and let it propagate silently everywhere beneath that:

// Good — propagates quietly; logged exactly once, where it's actually handled
void processData() throws IOException {
    readFile(); // no catch here — nothing at this level can act on a failure anyway
}

Anti-pattern: losing the original exception

§14.8 already covered exception chaining, but it's common enough to misuse that it belongs here again, stated plainly as what not to do:

// Bad — the real cause is discarded
} catch (SQLException e) {
    throw new UserNotFoundException("User not found"); // cause dropped
}

e is right there, holding the actual database problem that led here, and the new exception simply doesn't mention it. Whoever eventually handles or logs the UserNotFoundException sees a plausible-looking message and nothing else — the SQLException underneath, which might have said exactly why the lookup really failed, is gone the moment this line runs. Passing it as the cause costs nothing and keeps it recoverable:

// Good — the cause is preserved
} catch (SQLException e) {
    throw new UserNotFoundException("User not found", e); // cause preserved
}

Recap: rules from earlier in this chapter

  • Order catch blocks from most specific to most general (§14.5's Rule of Specificity) — a broad catch placed first silently swallows every subtype beneath it.
  • Don't use exceptions for routine, expected outcomes (§14.10) — a normal "not found" belongs in a return value, not a try/catch standing in for an if.
  • Prefer try-with-resources over a hand-written finally (§14.9) — it closes resources automatically and keeps a cleanup failure from silently overwriting the original one.
§14.12

Key Takeaways

Every mechanic in this chapter is detail on top of one shape, and it's worth ending exactly where §14.2 began — with that shape drawn out in full:

Normal execution
       │
       ▼
    Success

          OR

Problem occurs
       │
       ▼
Exception thrown
       │
       ▼
Search upward for handler
       │
       ▼
Handled — or program terminates
  • An exception is an ordinary object, an instance of a class descending from Throwable — throw is what turns constructing one into interrupting normal execution.
  • Thrown, an exception unwinds the call stack, method by method, until something agrees to handle it; reaching the bottom with nothing willing to catch it terminates the program (or the thread).
  • Error represents JVM-level trouble not meant to be caught; Exception represents problems a program can reasonably anticipate and recover from.
  • A checked exception (any Exception subclass outside RuntimeException) must be caught or declared, enforced by the compiler; an unchecked exception (RuntimeException and its subclasses) carries no such enforcement.
  • An exception should represent something exceptional — a genuine failure. A routine, expected outcome like "not found" belongs in an ordinary return value, not a try/catch standing in for an if/else.
  • try/catch handles an exception where it's actually manageable; multiple catch blocks are checked top to bottom, and multi-catch (A | B) combines identical handling for more than one type.
  • finally runs no matter how the try block exits — normally, by a caught exception, or by one that wasn't caught at all.
  • throw raises one specific exception instance; throws, part of a method's signature, declares which checked exceptions might escape it.
  • A custom exception, extending Exception or RuntimeException, names a domain-specific failure precisely — checked when callers genuinely need reminding to handle it, unchecked when it signals the caller's own programming error.
  • Passing the original exception as a cause when wrapping it in a new one preserves the real underlying problem instead of discarding it.
  • Try-with-resources calls close() on any AutoCloseable automatically once its block ends, removing the need for a hand-written finally just to guarantee cleanup.
