Chapter 14

Exceptions

Programs don't execute in a perfect world. Files disappear, networks fail, users enter unexpected input, and assumptions sometimes turn out to be wrong. Exceptions are Java's mechanism for reporting that a method cannot complete the work it promised, and for transferring responsibility to code that's in a better position to decide what should happen next.

§14.1

The Problem — when there's no sensible value to return

A program, most of the time, does exactly what it's supposed to do. One piece of work finishes, hands its result to the next, and execution continues under the assumption that everything has gone according to plan.

Picture a simple reporting tool, built around three pieces of work in this order — read a file, process what's in it, produce a report from the result:

Start
Read the file
Process what's in it
Produce the report
Finish

Call this the happy path: everything completes, in this order, and there's nothing more interesting to say about it than that.

Unexpected situations

Real programs constantly interact with a world they do not control. Every interaction introduces the possibility that something won't go as planned. 

Open a file           → file missing
Connect to a database → database unavailable
Parse a number        → invalid input
Read network data     → connection lost

When one of these situations occurs, the method currently executing may no longer be able to fulfill its contract. A method that promised to return a Report cannot honestly produce one if the file doesn't exist.

Things can go wrong in different ways

  • Sometimes the caller violates the method's contract by providing invalid input—for example, a negative value where only positive values are allowed.
  • Other times, the environment prevents the method from succeeding, even though it was called correctly. A file may not exist, a network connection may be lost, or a database may be unavailable.
  • Finally, the failure may be caused by a bug in the program itself. The method may make an incorrect assumption, dereference a null reference, or access an array beyond its bounds.

What are the options?

Suppose loadReport() discovers that the requested file is missing. Should it:

  • ask the user to choose another file?
  • create an empty report?
  • retry after a few seconds?
  • download the file from another location?
  • terminate the program?

There isn't a universally correct answer.

  • A desktop application might prompt the user for another file.
  • A web server might return an HTTP 404 response.
  • A batch-processing job might simply log the problem and continue with the next file.

The method that detected the failure knows what happened, but it usually doesn't know how the application should respond. Instead of guessing, it reports the problem by throwing an exception. The exception then travels up the call stack until it reaches code with enough context to decide what should happen next. That's the central idea behind Java's exception mechanism: the code that detects a failure isn't always the code that's best equipped to recover from it.

When the method cannot return the value it promised, throwing an exception is Java's way of saying:
"I couldn't complete my side of the contract."

That naturally raises an important question:

If the method doesn't handle the failure itself, who does?

What about returning an error object?

Different programming languages handles exceptions in different ways. Some languages, such as Rust with its Result type or Go with its multiple return values, treat failure as just another value. A method returns either a successful result or an error, and the caller is expected to inspect that result immediately before continuing. Responsibility stays with the immediate caller, who decides what to do next.

Java takes a different approach.

A method's return type describes only its successful outcome. If a method returns a Report, then a successful call produces a Report—nothing more. Failures don't travel through the return value at all. Instead, they follow a completely separate path. When an exception is thrown, normal execution stops immediately. Java creates an exception object and begins propagating it up the call stack. Each method on the stack is given an opportunity to handle the exception. If a method doesn't catch it, that method also terminates, and the exception continues to the next caller. This process repeats until a method handles the exception. If no method handles it and the exception reaches the top of the call stack, the program terminates.

The rest of this chapter explores that second path: how exceptions travel through a program, how they are caught, and what happens when nobody catches them at all.

§14.2

The Exception Lifecycle — unwinding the call stack until something agrees to handle it

What Exceptions Actually Are

An exception is just an object — an instance of a class — that represents "something abnormal happened here."

An exception is an event, which occurs during the execution of a program, that disrupts the normal flow of the program's instructions.

In other words: your program is executing line by line, following a plan — and then something happens that the plan didn't account for. A file isn't there. A number gets divided by zero. An array index goes out of bounds. The normal, expected flow can't continue.

Java's answer to this is to represent that disruptive event as an object. This object — an instance of a class like IOException or ArithmeticException — carries everything you need to understand what happened:

  • Its type, which itself communicates meaning (e.g., FileNotFoundException vs NullPointerException)
  • A message with human-readable detail
  • A stack trace — exactly where and how execution got there
  • Optionally, a cause — another exception that triggered this one

1. Creation and throwing

When something goes wrong, an exception object is created to represent the failure. It records information about what happened—its type and usually a descriptive message. The object is then given to the JVM, which begins searching for code that can handle it. This process is called throwing an exception. We have three actors separately:

  • Exception object → represents the failure.
  • JVM → transports the exception.
  • Handler (catch) → decides what to do.

This is the central mental model for the rest of the chapter:

an exception travels up the call stack until something handles it—or, if nothing does, the program terminates.

There are two different ways the exception object actually gets created, though both end the same way — thrown:

  1. Explicit creation.

    Nothing implicit at all: the code calls new itself, and throws the result itself. 

    void withdraw(double amount) {
    if (amount > balance) {
        throw new InsufficientFundsException("Not enough balance");
    }
}

  2. Implicit creation.

    Many exceptions are constructed automatically, by the JVM, the instant it detects a violation — no new anywhere in sight in the code that triggers it: 

    int[] arr = new int[3];
int x = arr[5]; // the JVM constructs and throws ArrayIndexOutOfBoundsException on your behalf

Either way, the moment an exception is thrown, the method it was thrown from stops executing immediately, right there — no further line in that method runs, and no value is ever returned from it.

Two keywords, easy to conflate by how similar they look, doing entirely different jobs. throw is a statement — it actually raises one specific exception instance, right now: 

static int divide(int a, int b) {
    if (b == 0) {
        throw new ArithmeticException("Cannot divide by zero");
    }
    return a / b;
}

throw takes a Throwable object — built with new, exactly as §14.4 described — and hands it to the runtime to interrupt execution immediately; nothing after a throw statement in the same block ever runs.

throws, by contrast, is part of a method's signature, not something that executes — it declares which checked exceptions (§14.5) a method might let escape, so that every caller is compiler-enforced to deal with them: 

void readConfig(String path) throws IOException {
    // ...code that might throw IOException...
}

A method that calls readConfig now has exactly two options for its own checked-exception obligation: catch IOException itself, or declare throws IOException in its own signature and pass the responsibility one level further up the call chain. An unchecked exception (§14.5) never needs a throws declaration — nothing forces one, though a method is still free to add one for documentation's sake even when it isn't required.

Polymorphic throwing

A method that saves a file can fail in several distinct ways at once — the network drops mid-write, the file itself has vanished, the destination directory no longer exists. Each of those is a real, separate exception type in the standard library, and declaring all of them individually gets unwieldy fast: 

// Bad — every new failure type means editing this signature again
public void saveFile() throws SocketException, FileNotFoundException, NoSuchFileException {
    // ...
}

A caller now has to know about, and handle, three specific types — and a fourth I/O failure the method starts throwing next year would mean updating this signature and every caller's catch blocks all over again. But SocketException, FileNotFoundException, and NoSuchFileException all extend the same class: IOException (§14.4's hierarchy). Declaring the parent instead of the children covers the whole family at once, exactly the way a supertype reference can hold any of its subtypes (polymorphism, §9.5): 

// Good — one declaration, whole family of I/O failures covered
public void saveFile() throws IOException {
    // ...
}

The method itself is free to throw a SocketException today and a brand-new IOException subclass next year, without its signature ever needing to change. A caller only needs to know that some kind of I/O problem is possible — catch (IOException e) handles every one of them, named or not, the same way a variable declared as a supertype can be handed any subtype without the caller needing to know which one specifically arrived (§9.5 covered this same substitutability for ordinary method parameters; a throws clause gets it for free, too).

2. Exception Propagation: The search for a handler

Lets consider a workflow for report generation:
  1. main() calls generateReport()
  2. generateReport() calls readFile() first to get the raw data,
  3. and then calls processData() to turn that into something reportable,
  4. then assembles the report itself from the result:
void main() {
    Report report = generateReport();
}

Report generateReport() {
    String contents = readFile(path);      // called first
    String entries = processData(contents); // called next
    return buildReport(entries);
}

The chain of calls actually in progress at any one moment. Execution begins in main(), which calls generateReport(). Before generateReport() can continue, it must first wait for readFile() to finish. At that moment, the active chain of method calls looks like this: 

main()
  ↓   waiting for
generateReport()
  ↓   waiting for
readFile()

This chain of active method calls is called the call stack.

Notice that processData() isn't on the stack yet. It hasn't been called because generateReport() is still waiting for readFile() to return.

Now suppose the file doesn't exist. Instead of returning normally, readFile() throws an IOException.

What happens next?

Assume readFile() doesn't handle the exception itself, the JVM removes its stack frame and passes the exception to the method that called it: 
main()
  ↓
generateReport()
  ↓
readFile()  // does not handle failure, terminates
Now generateReport() has an opportunity to handle the exception. If it doesn't, its own stack frame is removed as well, and the exception continues upward: 
main()
  ↓
generateReport() // does not handle failure, terminates
Finally, main() gets the same opportunity.
  • If main() catches the exception, execution continues from there.
  • If it doesn't, the exception has nowhere left to go, and the program terminates.
This journey from one method to its caller is called exception propagation. As the exception moves upward, each method on the call stack gets exactly one chance to handle it. The propagation stops as soon as a matching catch block is found.

Underneath that upward movement, the JVM is asking one question at each frame it reaches:

does this method know how to handle an exception of this type?
If not, that frame is abandoned entirely — this is the "unwinding" the mental model opened with, and it's why nothing after the original throw, in any of those methods, ever runs — including processData(), which was waiting to be called and now never will be.
"I give up. You deal with it."

An exception is Java's way of saying exactly that: I don't know what to do about this right now, so I'm handing the problem to whoever called me.

3. Program termination

If that search reaches the very bottom of the call stack — every method unwound, nothing anywhere willing to catch it — the program, or at minimum the one thread (§22.2) that was running it, terminates.

§14.3

Handling Exceptions — try, catch, and finally

Accepting Responsibility

We've seen that an exception propagates up the call stack until some method decides to handle it. A try/catch block is how a method says:

"If this operation fails, I'll take responsibility for deciding what happens next."

A try block marks code that may fail.

try {
    // code that might throw an exception. 
}

There are two possible outcomes:

  • The operation completes successfully, or
  • an exception is thrown

As we have seen, if an exception is thrown, normal execution stops immediately.

Catching an Exception

The moment an exception is thrown, the JVM abandons the rest of the try block. Any remaining statements inside it are skipped, and control jumps directly to the first catch block capable of handling that exception. 

try {
    String contents = readFile(path);    
    processData(contents); 
} catch (IOException e) {
    System.out.println("Unable to load the report.");
}

A method containing a matching catch block is the first one that says:

"I know what this means. I'm prepared for it. I'll take it from here."

Responsibility has been accepted, and execution continues after the entire try/catch statement.

Handling Several Exceptions the Same Way

Sometimes different exceptions deserve exactly the same response. Instead of duplicating the same code, Java allows multiple exception types to share a single catch block. 

try {
    riskyOperation();
} catch (ArrayIndexOutOfBoundsException | ArithmeticException e) {
    System.out.println("Something went wrong: " + e.getMessage());
}

This is known as multi-catch. It behaves exactly as though two separate catch blocks contained the same body. Its only purpose is to eliminate duplicated code.

Catching the wrong exception

For a checked exception (§14.5), the compiler doesn't just check that some catch exists — it checks that the catch actually covers everything the declared type could mean. A method declared throws IOException is free to throw any subclass of it at runtime — FileNotFoundException, SocketException, or others §14.2's polymorphic throwing already named — and catching only one narrower subclass doesn't account for the rest: 

void riskyBusiness() throws IOException { } // declares the checked exception

void process() {
    try {
        riskyBusiness(); // compile error
    } catch (FileNotFoundException e) { // doesn't cover every IOException riskyBusiness() could throw
        System.out.println("We have an exception!");
    }
}

This doesn't compile. riskyBusiness() is declared to throw IOException — any IOException, not specifically a FileNotFoundException — and the only catch present handles just that one narrow subtype. The compiler can see that a SocketException, or any other IOException that isn't a FileNotFoundException, would leave this try block with nowhere to go, so it refuses to compile process() at all.

Fixing this means the catch clauses actually have to cover everything the try block might throw — catching the exact declared type, a broader supertype of it, or the narrower subtype alongside the exact declared type: 

void process() {
    try {
        riskyBusiness();
    } catch (FileNotFoundException e) {
        System.out.println("Specifically missing a file.");
    } catch (IOException e) { // catches every other IOException riskyBusiness() might throw
        System.out.println("Some other I/O problem.");
    }
}

finally: the block that always runs

A finally block, attached after a try (and any catches), runs no matter what happens inside the try — whether it completed normally, threw an exception that got caught, or threw one that didn't: 

try {
    System.out.println("Trying...");     // prints: Trying...
    int result = 10 / 0;                 // throws ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("Caught: " + e.getMessage()); // prints: Caught: / by zero
} finally {
    System.out.println("This always runs."); // prints: This always runs.
}

"Always" is meant precisely here: finally runs even if the try block returns a value, even if the catch block itself throws a new exception, even if neither block finishes normally at all.

Here's exactly why that guarantee matters. Say a connection is opened, and something throws before it's closed: 

// Bad — the connection leaks if sendData() throws
Connection conn = openConnection();
sendData(conn);       // throws IOException
conn.close();         // never reached

conn.close() was right there on the next line, and it still never runs — the exception from sendData(conn) skips straight past it, exactly as §14.2 already described. The connection stays open, forever, unless something further up the call stack happens to close it, which nothing here promises. Putting the cleanup in a finally block fixes this directly: 

// Good — close() always runs, even if sendData() throws
Connection conn = openConnection();
try {
    sendData(conn);
} finally {
    conn.close();     // runs whether sendData() succeeded, threw, or was caught elsewhere
}

This is exactly the cleanup finally exists for: closing a connection, releasing a resource, anything that would otherwise need to be duplicated at the end of both the success path and every failure path. §14.8 covers the modern alternative for exactly this cleanup case, which most new code reaches for instead of a manual finally.

Double trouble: an exception inside catch or finally

Nothing stops a catch or finally block from throwing an exception of its own — and a real one, saving an order to a database, is exactly the kind of place both can fail at once: 

Connection conn = openConnection();
try {
    saveOrder(order);       // throws OrderException: "Invalid order data"
} finally {
    connection.close();     // ALSO throws IOException: "Connection already closed"
}
  1. saveOrder(order) throws first, with the actual, useful problem: The order data is invalid.
  2. But finally still runs before that exception is allowed to propagate (§14.3 already established this), and connection.close() throws too.

In this scenario, the second exception doesn't join the first — but replaces it:

  • The caller receives an IOException about the connection.
  • The OrderException that would have actually explained what went wrong is simply gone, discarded the moment the second exception was thrown on top of it.

The fix is to keep cleanup failures from ever being allowed to overwrite whatever the try block was already reporting: 

// Good — the real problem still gets through
try {
    saveOrder(order);       // throws OrderException: "Invalid order data"
} finally {
    try {
        connection.close();
    } catch (IOException closeException) {
        System.err.println("Also failed to close connection: " + closeException.getMessage());
        // the OrderException from saveOrder() is still the one that propagates
    }
}

Wrapping the cleanup itself in its own try/catch means a failure while closing the connection gets logged rather than thrown — it's real, and worth knowing about, but it's not allowed to bury the original failure that actually caused the problem. §14.8's try-with-resources handles this exact situation automatically, keeping both exceptions instead of losing one, which is one more reason it's usually preferred over a hand-written finally.

Nested try/catch

Let's generalize the problem. Picture loading a user's preferences: if the primary file is corrupted, falling back to a bundled set of defaults is reasonable. But that fallback is itself a file read, and file reads fail: 

// Bad — if loadDefaultPreferences() also fails, nothing here says why
try {
    loadPreferences();
} catch (IOException e) {
    loadDefaultPreferences(); // itself throws IOException — propagates uncaught
}

If loadDefaultPreferences() throws, that new IOException propagates straight out of this method entirely uncaught — indistinguishable from the original failure, with no indication that this was actually the fallback breaking down, not the primary attempt. The recovery action needs handling of its own, distinct from the failure it was recovering from: 

// Good — the fallback's own failure gets its own, more specific response
try {
    loadPrimaryPreferences();
} catch (IOException primaryError) {
    try {
        loadDefaultPreferences();
    } catch (IOException fallbackError) {
        throw new PreferencesUnavailableException(
            "Neither the primary nor the default preferences could be loaded", fallbackError);
    }
}

A try block nested inside a catch (or, just as legitimately, inside another try) is handled exactly like any other — its own catch only sees exceptions thrown within its own try, entirely independent of the outer one. Here, that means a failure in the fallback gets converted into a single, honest exception (§14.7's chaining preserves fallbackError as the cause) that says plainly what actually happened: not one preferences source failed, but both.

Summary

  1. You place code that might throw an exception inside a try block. The try block wraps the code segment that might throw an exception — the "protected" block. If everything inside succeeds, every catch block is skipped entirely.

  2. You define one or more catch blocks to intercept specific types of exceptions. A catch block declares the exception type it's willing to handle; inside it, you write the code that responds to that particular failure — recovering from it, or at least managing it gracefully. That might involve:

    • Logging: logger.error("Error processing file", e);
    • A user-facing message: System.err.println("An error occurred: " + e.getMessage());
    • Partial recovery: re-prompting the user for input, or retrying an operation after a delay.
    • Setting a flag: operationSuccessful = false;
    • Converting exception types: throwing a different, perhaps more abstract or custom (§14.6), exception instead.

    A try can be followed by several catch blocks, each handling a different exception type. Order matters — see The Rule of Specificity, §14.4.

  3. The finally block contains code that must run regardless of whether an exception occurred in the try block or was caught by a catch block. It's primarily used for cleanup — critically, for closing resources like file streams, database connections, and network sockets, preventing exactly the kind of leak this section opened with.

§14.4

The Exception Hierarchy — every exception is an object, in a family tree

An exception in Java isn't a special language construct sitting outside the type system — it's an ordinary object, an instance of an ordinary class, related to every other exception class the same way any classes are related: through inheritance (§9).

Every exception class ultimately extends Throwable, which splits into two branches immediately below it: 

Throwable
├── Error                     (JVM-level trouble — not meant to be caught)
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception
    ├── RuntimeException      (unchecked — §14.5)
    │   ├── NullPointerException
    │   ├── ArrayIndexOutOfBoundsException
    │   ├── ArithmeticException
    │   └── NoSuchElementException
    └── IOException, and other checked exceptions (§14.5)

Error represents problems below the level a normal program is expected to recover from — the JVM running out of memory, or a stack overflowing from runaway recursion (§4). Catching an Error is legal but rarely useful: by the time one occurs, the JVM itself is often in no state to keep going reliably. Exception is the branch this chapter is actually about — problems a program can reasonably anticipate and respond to, from a missing file to a malformed number to an index that ran off the end of an array.

An exception is just an object, thrown

Every box in that tree is a class, constructed with new exactly like any other object — new ArithmeticException("/ by zero") is a perfectly ordinary constructor call. What makes it an exception isn't anything about the object itself; it's the throw keyword (§14.2) applied to it, which hands that object to the runtime specifically to interrupt normal execution and go looking for a handler.

The Rule of Specificity: Ordering Catch Blocks

Picture a fruit packing house grading a crate of oranges by size — small, medium, large, jumbo — using a stack of trays, each one a mesh of holes, one size per tray. Pour the oranges in at the top, and they work their way down through tray after tray. If the top tray has the biggest holes, nearly every orange falls straight through it on the first pass, small and large alike, and the finer trays underneath never get a chance to do their job, because there's nothing left for them to catch. Flip the stack so the smallest holes are on top instead, and it works: the small oranges get caught right away, the mediums fall through to the next tray and get caught there, then the larges, until only the jumbos are left to reach the bottom. Small holes first, biggest holes last.

catch blocks work through that same family tree, top to bottom, stopping at the first one the exception fits through — but Java is far less forgiving about it. As long as the exceptions are unrelated, the order between them doesn't matter. The moment two catch blocks belong to the same branch of the hierarchy — one catching a broad category, another catching a more specific case within it — putting the broad one first doesn't just make the specific one pointless, it makes the code refuse to compile. Consider transfer(), which throws InsufficientFundsException when an account doesn't have enough funds. Like every exception, InsufficientFundsException is also an Exception — so the two belong to the same chain: 

void transfer(Account from, Account to, double amount) throws InsufficientFundsException {
    if (from.getBalance() < amount) {
        throw new InsufficientFundsException(from.getId(), amount);
    }
    // ... move the money
}

void processTransfer() {
    try {
        transfer(checking, savings, 500.0);
    } catch (Exception e) { // catches EVERYTHING below it too
        System.out.println("Transfer failed: " + e.getMessage());
    } catch (InsufficientFundsException e) { // compile error: unreachable
        System.out.println("Please add funds and try again.");
    }
}

This doesn't compile. Exception is the supertype of InsufficientFundsException (polymorphism, §9.5, applies here exactly as it does anywhere else), so the first catch already matches every InsufficientFundsException that could arrive. The second catch can never run, and Java refuses to compile a catch block it can prove is unreachable this way. Swap the order — most specific first — and both blocks become reachable, each handling exactly the case it was written for: 

void processTransfer() {
    try {
        transfer(checking, savings, 500.0);
    } catch (InsufficientFundsException e) { // narrowest subtype first
        System.out.println("Please add funds and try again.");
    } catch (Exception e) { // catches every other Exception
        System.out.println("Transfer failed: " + e.getMessage());
    }
}

The rule is simple: order every catch from most specific (subclass) to most general (superclass). A broad handler placed first doesn't just steal cases from the specific ones below it — it makes them permanently unreachable code, which is exactly what the compiler is refusing to let through.

§14.5

Checked vs. Unchecked — what the compiler actually enforces

Java divides exceptions into two categories: checked and unchecked.

The distinction isn't how exceptions are thrown or propagated. Once a program is running, both behave identically. An exception is thrown, propagates up the call stack, and either finds a matching catch block or terminates the program. The difference exists entirely at compile time. It lies in what the compiler requires of the code calling the method.

Can "check before act" guarantee success?

Every line of code can fail. Some failures are bugs — preventable through careful logic, validation, and correct assumptions.

Before dividing:

if (divisor != 0) { ... }

Before accessing an array:

if (index >= 0 && index < scores.length) { ... }

If these checks are written correctly, the corresponding failures simply cannot occur.

However, not every situation works like this. Consider these scenarios:

  • Opening a file. A caller can check whether the file exists, but that check can never become a guarantee. Between the check and the moment the file is actually opened, the file might be deleted by another process.
  • An account's balance — a caller can check before spending, but the check never becomes a guarantee, because another thread may be acting concurrently and modifying shared state.
The caller can reduce the likelihood of failure, but it cannot eliminate it through careful programming alone.

These are the kinds of situations Java models with checked exceptions.

Checked Exceptions

When an operation can fail for reasons the caller cannot completely prevent, as these failures are outside the caller's control, Java requires the caller to explicitly deal with that possibility. In effect, the compiler is saying:

You can't assume this operation will always succeed. No amount of your checks can guarantee succees. You must decide what should happen if it fails.

Before the code can compile, Java requires a small amount of ceremony to make that decision explicit. The caller must either handle the failure itself or or explicitly pass that responsibility to its own caller:

  • handle the exception with a try/catch block (§14.3), or 
    try{ 
    // code that may throw a checked exception
}catch(Type e){ 
    // handling logic
}
  • Propagate the exception by declaring it in a throws clause (§14.2): 
    void myMethod throws Type{
    ... 
}

    A throws clause doesn't handle the failure. Instead, it shifts responsibility one level up the call stack.

    It forces the caller to stop and make a conscious decision:

    • Will I handle this failure here?
    • Or will I pass the responsibility to my own caller?
    Every throws clause is essentially saying:
    "Hey, caller—it's your problem now! Better decide what happens next."

Unchecked Exceptions

Unchecked exceptions are different. They generally indicate that one of the assumptions required for the method to operate has already been violated. That can happen in two ways.

  • Sometimes, the failure originates inside the method itself. An unexpected null reference, an impossible internal state, or an invalid calculation may reveal a defect in the program's implementation. 
    String name = null; 
String up = name.toUpperCase();

int age = 25/0;
  • Other times the caller violates the method's contract by supplying an argument that isn't valid. For example, a method may require a positive amount, a non-null reference, or an index within a valid range. If those preconditions are not met, the method is free to reject the call immediately.

    Some unchecked exceptions are deliberately thrown by methods to enforce their contracts: 

    if (amount < 0) {
    throw new IllegalArgumentException("Amount must be positive.");
}

    Here, the method is working exactly as intended. It has detected that its precondition was violated and refuses to continue.

In both cases, Java assumes these situations are not part of the method's normal operation. They represent violated assumptions rather than expected outcomes that every caller should plan for. For that reason, the compiler places no requirements on the caller. A method may throw unchecked exceptions without declaring them, and callers are not required to catch or propagate them.

Defensive boilerplate everywhere?

Every line of code can fail. Some failures are bugs — preventable through careful logic, validation, and correct assumptions. Others come from external events — a full disk, a dropped connection, a deleted file — which no amount of checking can rule out in advance, because the world can change between the check and the action. The conclusion isn't that every line needs a try/catch: handling every conceivable failure would drown the actual logic in defensive boilerplate, making code unreadable and unmaintainable. Instead, exception handling is a design decision — decide deliberately where failure is expected and recoverable (catch it, translate it, or return Optional), where it's a contract violation (let it propagate, or fail fast), and accept that "no method call is safe" is a fact to design around, not a mandate to defend against everywhere.

No method call is inherently safe.

  • Checked exceptions are the compiler-enforced subset — the compiler forces you to catch or declare them.
  • Unchecked exceptions (RuntimeException and its subclasses, ie, ArrayIndexOutOfBoundsException, NullPointerException) get no such enforcement — nothing stops you from calling a method that can throw one without any try-catch in sight.

The absence of a throws clause — or the absence of a compiler-enforced try/catch requirement — is not a guarantee of safety. It only means the exception, if one occurs, is unchecked. The method can still fail at runtime:

  • Every method call involving object references can trigger NullPointerException if invoked on or passed a null value.
  • Any method touching arrays, collections, arithmetic, casting, or parsing can throw the corresponding unchecked exception — ArrayIndexOutOfBoundsException, ArithmeticException, NumberFormatException, etc — with zero declaration required.

The compiler has simply chosen not to force you to acknowledge that possibility in advance. Compiler's silence is not a promise of success. It just tells you nothing is required explicitly, not that nothing can go wrong.

Example: Where no decleration required

As production code here is a bug waiting to happen: no precondition check, no validation, nothing. Zero defense against b == 0 

public class Main {

    // redundant throws decleration
    // zero compiler effect. It doesn't force callers
    static int divide(int a, int b) throws ArithmeticException {
        return a / b;
    }

    public static void main(String[] args) {
        divide(10, 0); // compiles fine. throws ArithmeticException
    }
}

Note that ArithmeticException is unchecked.

  • The throws clause adds no compile-time requirement for the caller.
  • If you removed the throws clause entirely, this code behaves identically.
  • This declaration is mostly documentation.

Note: Many style guides (and linters) discourage it because it can mislead readers into thinking it behaves like a checked declaration. @throws in javadoc is purely documentation by design. It can't mislead a reader into thinking it's compiler-enforced, unlike a throws clause on the signature, which looks identical whether the exception is checked or unchecked. 

/**
 * Divides a by b.
 *
 * @throws ArithmeticException if b is zero
 */
static int divide(int a, int b) {
    return a / b;
}

Compile-Time vs Run-Time

A method may always throw unchecked exceptions without declaring them. Otherwise, every method would need a throws clause simply because it might contain an internal programming error.

For the same reason, callers are not required to catch or propagate them. If the compiler enforced unchecked exceptions as well, virtually every method call would need to be surrounded by a try/catch block, making ordinary code unnecessarily verbose without improving its correctness.

For unchecked exceptions, the compiler does not place any requirements on the caller. The calling code is free to ignore that a method may throw an unchecked exception. How can the caller be sure the method it has called has not made required checks? Opposite approach woud cause all method calls to be surrounded with try/catch blocks.

Compiles without any additional ceremony at all:

  • divide() doesn't declare throws ArithmeticException.
  • The caller isn't forced to wrap the call in a try/catch block. 
public class Main {
    static int divide(int a, int b) {
        return a / b;
    }

    public static void main(String[] args) {
        int divisor = 0;
        divide(10, divisor); // compiles fine. throws ArithmeticException
    }
}
Compile-Time only
This is a compiler level requirement - therefore it exists only at compile time. Once the program is running, checked and unchecked exceptions behave identically. Both are thrown, both propagate up the call stack, and both terminate the program if nobody catches them. The compiler is the only component that treats them differently.
Example: A teacher's grade book

Every method defines a contract:

  • Its parameters describe what the caller must provide.
  • Its return type describes what the caller can expect in return.

For example: int scoreAt(int index) The contract is simple: "Give me a valid index, and I'll return the score stored at that position." As long as the caller fulfills that contract, the method can fulfill its own. 

class GradeBook {
    private final int[] scores;

    GradeBook(int[] scores) {
        this.scores = scores;
    }

    int scoreAt(int index) {
        return scores[index]; // index comes from the caller — genuinely might be out of range
                              // throws ArrayIndexOutOfBoundsException if index is bad
                              // not enforced by the compiler
    }

    double average() {
        int total = 0;
        for (int i = 0; i < scores.length; i++) {
            total += scores[i];
        }
        return (double) total / scores.length;
    }
}
public class ReportCard {
    public static void main(String[] args) {
        GradeBook gradeBook = new GradeBook(new int[] { 90, 85, 78, 92 });
        System.out.println("First score: " + gradeBook.scoreAt(0));
        System.out.println("Class average: " + gradeBook.average());
    }
}

Case 1: Caller breaking the contract - Implicit throwing

index comes from whoever calls it, and nothing about GradeBook can guarantee it's in valid index range. What if the caller calls: scoreAt(-1); There is no score at index -1. The method can't return an integer because no correct answer exists.

At this point, it has only two options:

  1. invent a value

    Returning a made-up value such as 0 would be misleading. Zero might be a perfectly valid score

  2. report that the contract has been violated.

    The method signals that it cannot complete its contract by throwing an exception. Rather than pretending everything is fine, the method says: "I can't produce the result you asked for because the request itself is invalid."

Throwing an exception is saying:

"Your request doesn't make sense according to this method's contract."
Ask the right question

Don't ask "Can I recover?" Ask "Does this input satisfy the method's contract?" Exceptions are about contract violations, not simply "something went wrong."

  • If yes: return a meaningful result. If the situation is part of the normal business problem ("customer not found", "product unavailable", "search returned no results"): model that possibility in the return type
    • return null
    • returning Optional is preferred in newer APIs (§20)
  • If no: fail fast, usually with an IllegalArgumentException, IndexOutOfBoundsException, or another appropriate exception.

The core question: is "nothing to return" a normal outcome, or a failure? Prefer returning a null-like signal (or better, Optional) when absence is a legitimate, expected outcome — not an error. For example, "find a user by email," where not finding one is a completely normal result of a normal search: 

public Optional<User> findByEmail(String email) {
    // not found is normal — searches miss all the time
    return repository.query(email);
}

Prefer throwing when the missing value represents a contract violation — the caller did something wrong, or a precondition wasn't met, and proceeding further would be meaningless or dangerous. You want the failure to be impossible to silently ignore.

That's why List.get(index) throws, while methods like Map.get(key) return null (or, in newer APIs, Optional). At first glance these methods seem similar — list.get(index) and map.get(key) are both asking for a value — yet one throws an exception and the other returns null. Why? The answer lies in what each method promises. The contract of List.get() is essentially:

  • If the index is valid, I'll return the element.
  • If it isn't, you've violated the contract.

So it throws IndexOutOfBoundsException. A map is different — a map isn't organized by position, it's a lookup table: 

"Alice" → 42
"Bob"   → 37

When you ask map.get("Charlie"), you're really asking: "Is there a value associated with this key?"

  • Sometimes there is.
  • Sometimes there isn't. That's not an error.

Historically, Java chose null to mean "no mapping."

Case 2: Validation - Explicit throwing

Now you're enforcing the contract yourself. Why bother? Because you can:

  • produce a clearer error message,
  • validate before accessing the array,
  • decouple your API from the internal implementation.
Notice we used IndexOutOfBoundsException, not ArrayIndexOutOfBoundsException. That's what the JDK itself usually does. 
int scoreAt(int index) {
    if (index < 0 || index>= scores.length) {
        throw new IndexOutOfBoundsException("Index: " + index + ", size: " + scores.length);
    }
    
    return scores[index];
}

Which category a given exception falls into isn't a special marker stuck on the class — it's simply where that class sits in §14.4's hierarchy. Every subclass of RuntimeException is unchecked; every other subclass of Exception is checked. Error and its subclasses are technically unchecked too, but since they're not meant to be handled in the first place, that distinction rarely comes up in practice.

Checked Unchecked
Hierarchy Exception, excluding RuntimeException RuntimeException and its subclasses
Compiler enforcement Must be caught or declared — code won't compile otherwise None — catching it is always optional
Typical cause A condition outside the program's control — a missing file, a network failure A programming mistake — a bad index, a null dereference, invalid arguments
Example IOException NullPointerException, ArrayIndexOutOfBoundsException

That "typical cause" row is the practical rule of thumb worth internalizing: reach for a checked exception when a caller genuinely might encounter the failure through no fault of their own, and needs to be reminded to plan for it — a file that might not exist, a connection that might drop. Reach for unchecked when the failure signals a bug in the calling code itself — an index that should never have been out of range, an argument that violated a documented precondition. Forcing every caller to catch a NullPointerException around every dereference would make checked exceptions a burden instead of a safeguard; leaving a missing file uncaught would let a program silently limp forward on a false assumption. §14.6 returns to this same question when deciding which to extend for a custom exception.

Checked or Not: A Compiler Experiment

Let's see what the compiler actually does and doesn't demand, across four small programs.

A method can throw an exception when it cannot complete its contract. But does the compiler always require the caller to handle that? Consider this program: 

public class Main {
    public static void main(String[] args) {
        int result = 10 / 0;
    }
}

Our program fails because it attempts an operation without first verifying that the operation is valid. .The problem isn't that division is dangerous. The problem is that we attempted an operation without first checking whether it was valid.

The code compiles without complaint. If you run it, when the program reaches the division, the JVM throws an ArithmeticException. Notice what the compiler didn't require: it didn't ask for a try/catch block, and it didn't ask for anything declared in a throws clause. The program compiled exactly as written.

Java treats ArithmeticException as a programming error. Could we eliminate failure? A safer approach is to verify the condition first. This idea appears throughout programming and is often called check before act .

Be careful: check before act may not always prevents exceptions. For arithmetic, it does, but for a filesystem operation, it does not. 

public class Main {

    public static void main(String[] args) {
        int dividend = 10;
        int divisor = 0;
        if (divisor != 0) {
            int result = dividend / divisor;
        }
    }
}

Now extract the division into its own method:

void main() {
    calculate();
}

void calculate() {
    int result = 10 / 0;
}

Again, the code compiles perfectly. The exception still propagates from calculate() to main(), but the compiler remains completely silent. Apparently, a method is allowed to throw some exceptions without declaring them at all.

Now try something different:

void main() {
    FileReader reader = new FileReader("report.txt"); // compile error — unreported exception FileNotFoundException; must be caught or declared to be thrown
}

This time, the compiler immediately reports an error. It refuses to compile the program until it either catches the exception, or declares that main() throws it. Fixing it means picking one of those two: 

// Option 1 — catch it
void main() {
    try {
        FileReader reader = new FileReader("report.txt");
    } catch (FileNotFoundException e) {
        System.out.println("Could not open report.txt");
    }
}
// Option 2 — declare it, and pass the obligation to whoever calls main()
void main() throws FileNotFoundException {
    FileReader reader = new FileReader("report.txt");
}

Suppose the code is extracted into another method, the same way calculate() was above: 

void main() {
    openReport();
}

void openReport() {
    FileReader reader = new FileReader("report.txt"); // compile error — unreported exception FileNotFoundException; must be caught or declared to be thrown
}

The compiler still isn't satisfied — the error has simply moved to openReport(). Unlike the previous example, this exception cannot be ignored, and openReport() faces the exact same two choices main() faced above: catch it right there, or declare it and pass it up further still.

Catching it right there resolves the problem locally:

void main() {
    openReport();
}

void openReport() {
    try {
        FileReader reader = new FileReader("report.txt");
    } catch (FileNotFoundException e) {
        System.out.println("Could not open report.txt");
    }
}

main() never even needs to know a FileNotFoundException was ever possible — the whole thing begins and ends inside openReport().

Declaring it instead doesn't make the possibility go away — it just relocates it again, this time to whoever calls openReport(): 

void main() throws FileNotFoundException { // still required — openReport() passes the obligation straight through
    openReport();
}

void openReport() throws FileNotFoundException {
    FileReader reader = new FileReader("report.txt");
}

Now main() faces the exact same choice openReport() just made. It could declare it and pass it up further still, as above — but main() is the program's entry point; there's nothing further up the call stack to pass it to. Catching it here instead is the more natural fit: 

void main() { // catches here — no throws clause needed on main() itself
    try {
        openReport();
    } catch (FileNotFoundException e) {
        System.out.println("Could not open report.txt");
    }
}

void openReport() throws FileNotFoundException {
    FileReader reader = new FileReader("report.txt");
}

Whichever level does it, the obligation only stops traveling once some method actually catches it.

Something changed

These two pairs of examples behave very differently. In both cases a method may fail, and in both cases an exception can propagate up the call stack, all the way to main() — yet only one pair causes a compilation error. Why?

The answer is that Java places these exceptions into different categories.

  • ArithmeticException is an unchecked exception — the compiler allows it to propagate without requiring any caller to acknowledge it.
  • IOException — the family FileReader's constructor belongs to (§14.4) — is a checked exception: the compiler insists that every method either handle it or explicitly declare that it may propagate further.
The important observation

Nothing changed at runtime. Both exceptions are thrown; both travel up the call stack in exactly the same way. The only difference is what happened before the program even ran — the compiler enforced one contract but not the other. That distinction — what the compiler enforces, not what happens once the program is running — is the entire difference between checked and unchecked exceptions — precisely the distinction that made catching (§14.3) and declaring (§14.2) worth having as two separate, deliberate choices, rather than one default behavior imposed on every method.

Calling a method that declares a checked exception

The interesting question is: "When should the caller be forced to acknowledge that a method might fail?"

That leads naturally to the two categories.

Let's frame the distinction around the method's contract:

  • A checked exception communicates: "A correct caller must be prepared for this possibility."
  • An unchecked exception communicates: "A correct caller should never violate this part of the contract."

The moment a method's signature declares a checked exception, calling it hands that exception straight to the caller as a decision they now have to make — there's no third option, and no way to simply ignore it. Every checked exception a method can throw is effectively part of its public interface, exactly as visible to a caller as its parameter list or return type: whoever calls the method needs to know what it might throw, precisely so they can decide what to do about it — catch it there (§14.3), or declare it themselves and pass the decision one level further up.

One consequence of this rule is easy to miss: a method can declare a checked exception it never actually throws, anywhere in its own body: 

void eatCarrot() throws NutritionException {
    System.out.println("Munch.");
    // NutritionException is never actually constructed or thrown here
}

The compiler never checks whether a method's body actually produces every exception it declares — only that whoever calls a method handles or declares every checked exception that method's signature lists. So eatCarrot()'s declaration alone is enough to bind every one of its callers: 

void feedRabbit() throws NutritionException { // required, even though eatCarrot() never actually throws it
    eatCarrot();
}

This is genuinely useful when a method is expected to throw something in a future version but doesn't yet: declaring the exception now means every caller already handles or declares it, so adding the real throw later never breaks a single caller that compiles against this method today.

§14.6

Custom Exceptions — modeling a domain-specific failure

The standard library's exception classes cover generic problems — a bad index, a missing file — but a program's own domain often has failures with no natural standard-library name. Without one, a method is left borrowing something generic and stuffing the real problem into a string: 

// Bad — no exception type actually means "insufficient funds"
void withdraw(double amount, double balance) throws Exception {
    if (amount > balance) {
        throw new Exception("Cannot withdraw " + amount + " from balance of " + balance);
    }
}
try {
    withdraw(500, 100);
} catch (Exception e) {
    // catches insufficient funds — and every other possible failure this
    // method (or anything it calls) could ever throw, with no way to tell them apart
}

A caller here can't catch "insufficient funds" specifically — only Exception, the same catch-all that would also swallow a bug, a null dereference three calls deep, or anything else. Extending Exception or RuntimeException (§9.3) instead creates a new exception type that names the problem precisely, catchable on its own: 

// Good — a real type for this one specific failure
class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String message) {
        super(message);
    }
}
void withdraw(double amount, double balance) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Cannot withdraw " + amount + " from balance of " + balance);
    }
}

The constructor does nothing but forward its message to Exception's own constructor via super(...) (§9.4) — everything getMessage(), the exception hierarchy, and every catch block already know how to do with an exception, InsufficientFundsException inherits for free. What's new is purely the name and the type: catch (InsufficientFundsException e) now catches exactly this domain failure and nothing else, distinct from an ArithmeticException, an IOException, or the generic Exception the bad version above was forced to use.

The choice of what to extend is exactly §14.5's checked-vs-unchecked question, applied to this one new type: extend Exception when callers genuinely need to be compiler-reminded to handle the failure (as above — a caller forgetting to plan for insufficient funds is a real, likely mistake); extend RuntimeException when the failure represents a caller's own programming error, the same way IllegalArgumentException does for a bad argument.

§14.7

Exception Chaining — preserving the original problem

Catching one exception only to throw a different, more domain-appropriate one is common — a low-level IOException caught and re-thrown as a ConfigLoadException, say. Done carelessly, that step throws away exactly the information that would explain what actually went wrong underneath: 

try {
    loadFile(path);
} catch (IOException e) {
    throw new ConfigLoadException("Could not load configuration"); // original cause lost
}

Throwable has a constructor overload built specifically to avoid this — one that takes a cause alongside the message: 

try {
    loadFile(path);
} catch (IOException e) {
    throw new ConfigLoadException("Could not load configuration", e); // e preserved as the cause
}

The new exception now carries the original one inside it, retrievable later with getCause(). A stack trace printed from the outer exception shows both — the higher-level failure a caller actually asked about, and, chained beneath it, the lower-level one that actually caused it. Whenever wrapping one exception in another, passing the original along as the cause costs nothing and keeps the real explanation from disappearing.

§14.8

Try-with-Resources — cleanup that can't be forgotten

A resource that must be closed — a file, a network connection, a thread pool — traditionally needed a manual finally block (§14.3) to guarantee that cleanup, even when an exception interrupted whatever was using it. But FileReader.close() is itself declared throws IOException — the exact "cleanup that can itself fail" problem §14.3 just walked through — so doing this properly meant a nested try/catch just to handle close() potentially throwing too:

Step 1: Summary: What's actually wrong with this pattern 
try {
    // use the resource
} catch (Type e) {
    e.printStackTrace();
} finally {
    // try closing the resource
}
Step 2: Let's make the problem obvious: 
FileReader reader = null;
try {
    reader = new FileReader("file.txt"); // 1. instantiation: constructor throws

    // 2. using the reader also throws

} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (reader != null) {
        try {
            reader.close(); // 3. closing the resource: close() throws
        } catch (IOException e) {
            e.printStackTrace(); // now nested try/catch just to close
        }
    }
}

Try-with-resources is syntax built specifically to remove this ceremony 

try (FileReader reader = new FileReader(path)) {
    // ...use reader...
} // reader.close() is called automatically here

More than one resource can be declared the same way, separated by semicolons, and they're closed in the reverse of the order they were opened — the last one declared is the first one closed. §22.10 puts this exact mechanism to work for ExecutorService specifically; its close() is simply one particular AutoCloseable implementation, doing the graceful-shutdown work appropriate to a thread pool instead of releasing a file handle.

AutoCloseable and Closeable

Any resource declared inside a try's parentheses has to implement one of exactly two interfaces: java.lang.AutoCloseable, or java.io.Closeable. 

public interface AutoCloseable {
    void close() throws Exception;
}
public interface Closeable extends AutoCloseable {
    void close() throws IOException;
}

Closeable actually extends AutoCloseable — it isn't a separate, competing interface, just a more specific one, overriding close() to narrow its declared throws Exception down to throws IOException. An overriding method is always free to narrow or drop the checked exceptions its parent declared, just never add ones the parent didn't (§22.9 shows this same narrowing again, for ExecutorService.close()). Reader — and through it, FileReader — implements Closeable, along with BufferedReader, Socket, Scanner, and InputStream: all specifically I/O resources.

AutoCloseable is the broader of the two, and it exists precisely because try-with-resources wasn't meant to stay limited to I/O. When it was introduced, declaring close() against the base Exception type — rather than the narrower IOException — made the interface general enough for any resource needing cleanup, I/O or not: a database connection, or a thread pool. 

public interface ExecutorService extends Executor, AutoCloseable

ExecutorService (§22.9) implements AutoCloseable directly, not Closeable — shutting down a thread pool has nothing to do with I/O, so the broader interface is the correct fit. Either way, notice that both interfaces' close() declares a checked exception — Exception for AutoCloseable, IOException for Closeable — which is exactly why the "what try-with-resources does not do" callout below still applies to every resource declared this way, regardless of which of the two it implements.

Automatic execution of close()

In a try-with-resources statement, close() runs automatically as part of exiting the try block — whether that exit is normal or via an exception — and it runs before any catch or finally attached to that same try. That ordering matters: it means a catch block here covers exceptions thrown by close() itself, not just ones thrown by the body of the try.

Suppressed Exceptions: When Cleanup Also Fails

§14.3's "double trouble" showed the failure mode a hand-written finally is exposed to: if the try body throws, and then cleanup in finally throws too, the second exception silently replaces the first — the real cause, the invalid order or the corrupted file, is gone, and all the caller ever sees is a connection error that says nothing about what actually went wrong. Try-with-resources runs its cleanup — the implicit call to close() — in exactly that same risky spot, after the try body, while an exception may already be in flight. Left unhandled, it would have the same problem to solve: 

try (FileReader reader = new FileReader(path)) {
    throw new RuntimeException("Something went wrong reading");
    // reader.close() also throws IOException here, while unwinding
} catch (Exception e) {
    System.out.println(e.getMessage());          // "Something went wrong reading"
    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println("Also: " + suppressed.getMessage()); // the IOException from close()
    }
}

Both exceptions happened here — the RuntimeException from the try body, and an IOException from reader.close() while unwinding because of it — but unlike the hand-written finally case, neither one is lost. The body's exception is still the one that propagates to catch, exactly as before. But the close() failure isn't discarded either: the JVM attaches it to the propagating exception as a suppressed exception, retrievable through getSuppressed(), which returns an array because more than one resource can fail to close on the way out. It never gets its own catch block and never propagates independently — it just rides along, available if anyone asks for it. This is exactly the improvement try-with-resources makes over a manual finally: both failures survive, instead of the second one silently overwriting the first.

What try-with-resources does not do

Try-with-resources automates exactly one thing: calling close() on the resource, guaranteed, regardless of how the block exits. That's the whole feature. It says nothing about exceptions themselves — a checked exception thrown from inside the try body still propagates exactly as it would without it, and still needs, per §14.5, either a catch block to handle it locally or a throws declaration to push the responsibility to the caller. Without one of those two, the compiler rejects the method — the same rule as always, unaffected by whether the resource happens to be declared in the try's parentheses.

§14.9

Best Practices & Anti-Patterns — habits worth building, and ones worth breaking

Everything in this chapter comes with real ways to misuse it. A few habits account for most of the trouble in practice — each one easy to fall into, and each with a straightforward fix.

Anti-pattern: swallowing an exception

// Bad — the failure vanishes without a trace
try {
    riskyOperation();
} catch (IOException e) {
    // ignored
}

An empty catch block doesn't make the problem go away — it just deletes every trace that it happened. Whatever riskyOperation() was trying to do silently didn't happen, and nothing here says so; a caller three layers up, or a user staring at a blank screen, has no way to know why. At an absolute minimum, a caught exception should be logged, even when there's genuinely nothing else to do about it: 

// Good — the failure is invisible to the program's logic, not to whoever reads the logs
try {
    riskyOperation();
} catch (IOException e) {
    logger.error("riskyOperation failed", e);
}

Anti-pattern: catching Throwable or Error 

// Bad — catches far more than was ever intended
try {
    doWork();
} catch (Throwable t) {
    // ...
}

Throwable sits above both branches of §14.4's hierarchy — this catches every Exception, but also every Error: an OutOfMemoryError, a StackOverflowError, conditions the JVM itself is in no state to recover from. Catching one of these doesn't fix anything; it just delays a crash that was already coming, often while masking the fact that something in the program is fundamentally broken. Catch Exception, not Throwable, unless there's a genuinely deliberate reason — logging a fatal condition right before the program exits, say — to see an Error at all: 

// Good — catches real, recoverable problems, and nothing below Error
try {
    doWork();
} catch (Exception e) {
    logger.error("doWork failed", e);
}

Anti-pattern: logging the same failure at every level

// Bad — one failure, logged three times on its way up
void processData() throws IOException {
    try {
        readFile();
    } catch (IOException e) {
        logger.error("processData failed", e);
        throw e; // still propagates — and gets logged again by whoever catches it next
    }
}

Propagation (§14.2) means the same exception can pass through several methods on its way to a handler — and logging it at every one of those methods turns one real failure into a wall of duplicate log entries, each describing the identical problem. Log once, at the point that actually has enough context to act on it or report it meaningfully — typically much further up the call stack than where the exception was first thrown — and let it propagate silently everywhere beneath that: 

// Good — propagates quietly; logged exactly once, where it's actually handled
void processData() throws IOException {
    readFile(); // no catch here — nothing at this level can act on a failure anyway
}

Anti-pattern: losing the original exception

§14.7 already covered exception chaining, but it's common enough to misuse that it belongs here again, stated plainly as what not to do: 

// Bad — the real cause is discarded
} catch (SQLException e) {
    throw new UserNotFoundException("User not found"); // cause dropped
}

e is right there, holding the actual database problem that led here, and the new exception simply doesn't mention it. Whoever eventually handles or logs the UserNotFoundException sees a plausible-looking message and nothing else — the SQLException underneath, which might have said exactly why the lookup really failed, is gone the moment this line runs. Passing it as the cause costs nothing and keeps it recoverable: 

// Good — the cause is preserved
} catch (SQLException e) {
    throw new UserNotFoundException("User not found", e); // cause preserved
}

Anti-pattern: using exceptions for routine, expected outcomes

Everything so far has been about failures a program's normal logic has no next step for at all — a missing file, a bad index, a connection that dropped. Not every unsuccessful-looking outcome belongs in that category, and treating one that doesn't is one of the easiest ways to misuse everything else this chapter builds.

Consider a method that looks a user up by ID. Not finding one is often a routine, expected outcome — the ID simply doesn't exist yet, the same way a search coming up empty is still a completed search, not a failed one. Modeling that as an exception is a common mistake: 

// Bad — "not found" isn't exceptional here
User user = findUser(id); // throws UserNotFoundException if missing
try {
    User user = findUser(id);
    // ...
} catch (UserNotFoundException e) {
    // ...
}

A try/catch used this way isn't handling a failure at all — it's an ordinary if/else wearing exception syntax, for a branch of logic that's a completely normal, everyday outcome of calling findUser. §20 covers Optional<T> in full, but the shape of the better fix is already visible here: a return value that represents "might not be there" directly and honestly, without treating a routine miss as if it were a breakdown.

The dividing line

An exception should represent something exceptional actually happening — a genuine failure, a violated assumption, a job that could not be completed. It should not represent one ordinary branch of a method's normal logic, reached for instead of an if purely because throwing felt like the more deliberate way to write it. If a caller is expected to see a particular outcome regularly, in the ordinary course of using a method correctly, that outcome is normal — not exceptional — no matter which syntax reports it.

Recap: rules from earlier in this chapter

  • Order catch blocks from most specific to most general (§14.4's Rule of Specificity) — a broad catch placed first silently swallows every subtype beneath it.
  • Don't use exceptions for routine, expected outcomes (above) — a normal "not found" belongs in a return value, not a try/catch standing in for an if.
  • Prefer try-with-resources over a hand-written finally (§14.8) — it closes resources automatically and keeps a cleanup failure from silently overwriting the original one.
§14.10

Key Takeaways

Every mechanic in this chapter is detail on top of one shape, and it's worth ending exactly where §14.2 began — with that shape drawn out in full: 

Normal execution
       │
       ▼
    Success

          OR

Problem occurs
       │
       ▼
Exception thrown
       │
       ▼
Search upward for handler
       │
       ▼
Handled — or program terminates
  • An exception is an ordinary object, an instance of a class descending from Throwable — throw is what turns constructing one into interrupting normal execution.
  • Thrown, an exception unwinds the call stack, method by method, until something agrees to handle it; reaching the bottom with nothing willing to catch it terminates the program (or the thread).
  • Error represents JVM-level trouble not meant to be caught; Exception represents problems a program can reasonably anticipate and recover from.
  • A checked exception (any Exception subclass outside RuntimeException) must be caught or declared, enforced by the compiler; an unchecked exception (RuntimeException and its subclasses) carries no such enforcement.
  • An exception should represent something exceptional — a genuine failure. A routine, expected outcome like "not found" belongs in an ordinary return value, not a try/catch standing in for an if/else.
  • try/catch handles an exception where it's actually manageable; multiple catch blocks are checked top to bottom, so order them from most specific to most general — a broad catch placed first silently swallows every subtype beneath it. Multi-catch (A | B) combines identical handling for more than one type.
  • finally runs no matter how the try block exits — normally, by a caught exception, or by one that wasn't caught at all.
  • Never leave a catch block empty — at minimum, log what was caught, since silence deletes the only evidence that something went wrong. Catch Exception, not Throwable, so a JVM-level Error doesn't get mistaken for something worth handling.
  • Log a failure once, at the point that can actually act on it or report it meaningfully — not at every method it happens to pass through on the way there, which turns one real failure into a wall of duplicate log entries.
  • throw raises one specific exception instance; throws, part of a method's signature, declares which checked exceptions might escape it.
  • A custom exception, extending Exception or RuntimeException, names a domain-specific failure precisely — checked when callers genuinely need reminding to handle it, unchecked when it signals the caller's own programming error.
  • Passing the original exception as a cause when wrapping it in a new one preserves the real underlying problem instead of discarding it.
  • Try-with-resources calls close() on any AutoCloseable automatically once its block ends, removing the need for a hand-written finally just to guarantee cleanup.
