Chapter 16

Generics

A container built around Object can hold anything — which is exactly the problem. This chapter is about telling the compiler what a container actually holds, so it can catch the mistake you'd otherwise only discover at runtime.

§16.1

The Problem — a container that forgets what it holds

A class meant to hold "anything" reaches, naturally enough, for Object — every type is some kind of Object (§11.3's unwritten extends Object), so a field of that type accepts anything at all:

class Box {
    private Object content;

    public void set(Object content) { this.content = content; }
    public Object get()              { return content; }
}
Box box = new Box();
box.set("hello");
String s = (String) box.get(); // must cast — get() only promises an Object

get() only ever promises an Object — the compiler has no way to know a String is what's actually inside, so reaching for a String specifically requires an explicit downcast (§11.7), every single time. And nothing stops the mistake this whole class exists to prevent:

box.set(42);                     // compiles fine — 42 is an Object too
String oops = (String) box.get(); // compiles fine — throws ClassCastException at runtime

The cast on the last line is a promise to the compiler that turns out to be false — and like every unchecked downcast (§11.7), the compiler has no way to catch it. The mistake was made the moment box.set(42) ran; it just doesn't announce itself until some much later line tries to use the result as a String.

§16.2

Type Parameters — <T>

A generic class declares a placeholder for a type, filled in later, at the point it's actually used:

class Box {
    private T content;

    public void set(T content) { this.content = content; }
    public T get()              { return content; }
}

T is a type parameter — read Box<T> as "a Box of some type T, decided by whoever creates one." Creating a Box now names that type explicitly:

Box box = new Box<>();
box.set("hello");
String s = box.get(); // no cast — box was declared to hold String, specifically

Every T in the class body becomes String for this particular box — set only accepts a String, and get returns one directly, no cast required. The mistake from §16.1 is no longer a runtime surprise; it's a compile error, at the exact line that caused it:

box.set(42); // compile error — box is a Box<String>, 42 is not a String
The empty <> is inference, not omission

new Box<>() doesn't mean "no type parameter" — the compiler already knows the type from the left side of the assignment, Box<String>, and fills in <String> on the right automatically. This is the diamond operator: shorthand for new Box<String>(), not a different, looser kind of Box.

§16.3

Generic Methods — a type parameter on a single method

A type parameter doesn't require a generic class around it — a single method can declare its own, scoped only to that method, written just before the return type:

static  T firstNonNull(T a, T b) {
    return a != null ? a : b;
}
String name = firstNonNull(null, "default");     // T is inferred as String
Integer count = firstNonNull(null, 5);             // T is inferred as Integer

firstNonNull isn't written for String, or for Integer, or for any one type — T is decided fresh at each call, inferred from whatever arguments actually show up. One method body, used at as many types as callers need, each call fully type-checked on its own terms.

§16.4

Bounded Types — <T extends ...>

Plain T promises nothing except "some type" — not even Object's methods are assumed beyond what every type already has. Calling anything more specific on a value of type T requires telling the compiler what T is guaranteed to be. A bounded type parameter does exactly that:

static  void renewAll(T[] items) {
    for (T item : items) {
        item.renew();
    }
}
Subscription[] subs = { new Subscription(), new Subscription() };
renewAll(subs); // fine — Subscription implements Renewable

T extends Renewable restricts what T is allowed to be to types that implement Renewable (§15.2) — despite the keyword, this works identically whether Renewable is a class being extended or, as here, an interface being implemented. That restriction is precisely what makes item.renew() legal inside the method: the compiler knows, for any type that could ever fill in T here, that renew() is guaranteed to exist.

§16.5

Compile Time Only — what generics don't do

Every guarantee generics make is a compile-time one. Once the code compiles, the type parameter itself is gone — Java's generics work by checking everything up front, then discarding the type information, a design called type erasure. A Box<String> and a Box<Integer> are, underneath, running the exact same class at runtime; nothing about T survives to be inspected later.

This is not a loophole to work around — it's the actual scope of what generics promise. They don't add a new runtime check; they move a whole category of mistake (§16.1's ClassCastException) to a place it can be caught immediately, at compile time, instead of a much later, harder-to-trace line where the bad value finally gets used.

Scope of this chapter

Wildcards (List<? extends T>), multiple bounds, generic interfaces, and the finer rules around type erasure and raw types are all left for later. The core idea carries over unchanged: a type parameter lets a class or method make a promise about what type it's working with, checked once, at compile time, everywhere it's used.
