Chapter 16

Generics

A container built around Object can hold anything — which is exactly the problem. This chapter is about telling the compiler what a container actually holds, so it can catch the mistake you'd otherwise only discover at runtime.

§16.1

The Problem — a container that forgets what it holds

Let's start with code you might have actually written — or would have written — before generics existed.

Picture a storage box at home labeled "misc." It holds anything — a screwdriver, a cable, a photo — which is exactly what makes it convenient, right up until you reach in expecting the screwdriver and pull out the photo instead. A box that accepts everything can't promise what comes back out.

A class meant to hold "anything" reaches, naturally enough, for the same idea: every type is some kind of Object (§10.3's unwritten extends Object), so a field of that type accepts anything at all.

The Pre-Generics World: Raw Types and Object Casts

Before Java 5, the only way to write a reusable container was to hold everything as Object: 

class Box {
    // Since Object is the superclass of all classes, you can put any type into content.
    // The compiler cannot check whether you're putting in the "right" type.
    private Object content;

    public Box(Object content) {
        this.content = content;
    }

    public void setValue(Object content) {
        this.content = content;
    }

    // Type safety concern: content can be any type at all.
    public Object getValue() {
        return content;
    }
}

Using it looked like this:

// the box is initialized with a String
Box strBox = new Box("Hello");

String s = (String) strBox.getValue(); // cast required — compiler has no idea what's inside

This compiles. It runs. It seems fine. But now consider a slightly larger codebase where the box gets passed around: 

// ...200 lines later, a problematic value gets set silently, with no warning —
// the compiler sees nothing wrong

strBox.setValue(2014); // someone puts an Integer in

String value = (String) strBox.getValue(); // compiles perfectly — throws ClassCastException at runtime

The compiler had no complaint. The problem only surfaced at runtime — potentially in production, potentially in a code path that's rarely exercised.

What this means

You don't commit to a specific type upfront, while writing the class — but now everything is Object, and type safety goes with it. The compiler can't help you; every mistake it would normally catch moves to runtime instead.

Why Runtime Failures Are Worse Than Compile-Time Errors

This is worth sitting with, because it shapes every design decision in generics.

A compile-time error is:

  • caught immediately, before any code runs
  • localized — the compiler points at the exact line
  • free — it costs nothing in production

A runtime error is:

  • discovered only when that code path actually executes
  • potentially rare — a bug can live in a codebase for months
  • potentially dangerous — it crashes a running program, possibly losing state

The ClassCastException above is particularly insidious because the cast — (String) — looks completely reasonable to anyone reading that line in isolation. The real mistake was made somewhere else, at strBox.setValue(2014) — but there's no error there either. The compiler accepted both lines without a word.

The fundamental failure

The type contract is invisible to the compiler. There's nothing in the type of Box that says "this box holds Strings." That knowledge lives only in the programmer's head.

The Core Tension: Type Safety vs. Reusability

Before generics, there were exactly two ways to write this class, and neither one was enough on its own:

Reusable? Type-safe?
Use Object (one Box) Yes No — casts everywhere, runtime risk
Write StringBox, IntegerBox, … No — a new class per type Yes — but only by giving up on reuse

The second option removes every cast, at the cost of writing the identical class over and over, once per type: 

class StringBox {
    private String value;

    public StringBox(String value) {
        this.value = value;
    }

    public void setValue(String value) {
        this.value = value;
    }

    public String getValue() {
        return value;
    }
}

class IntegerBox {
    private Integer value;

    public IntegerBox(Integer value) {
        this.value = value;
    }

    public void setValue(Integer value) {
        this.value = value;
    }

    public Integer getValue() {
        return value;
    }
}
StringBox strBox = new StringBox("Hello");
String s = strBox.getValue(); // no cast — but a whole class was written just for String

IntegerBox intBox = new IntegerBox(100);
Integer n = intBox.getValue(); // no cast here either — and a second whole class, just for Integer

StringBox and IntegerBox are word-for-word identical except for one type name, repeated in four places each. Every new kind of box needed at some point means writing, testing, and maintaining an entire new class — for logic that never actually changes.

A cast is a promise

A cast is a promise to the compiler that you — not it — are responsible for type correctness. §16.2 picks up exactly here: a way to keep Box as the single, reusable class it was, while handing that promise back to the compiler, where it belongs.

§16.2

Type Parameters — <T>

The fix is to let Box say up front what it holds, instead of defaulting to "anything" the way the misc. box did. A generic class declares a placeholder for a type, filled in later, at the point it's actually used: 

class Box<T> {
    private T content;

    public void set(T content) { this.content = content; }
    public T get()              { return content; }
}

T is a type parameter — read Box<T> as "a Box of some type T, decided by whoever creates one." Creating a Box now names that type explicitly: 

Box<String> box = new Box<>();
box.set("hello");
String s = box.get(); // no cast — box was declared to hold String, specifically

Every T in the class body becomes String for this particular box — set only accepts a String, and get returns one directly, no cast required. The mistake from §16.1 is no longer a runtime surprise; it's a compile error, at the exact line that caused it: 

box.set(42); // compile error — box is a Box<String>, 42 is not a String
The empty <> is inference, not omission

new Box<>() doesn't mean "no type parameter" — the compiler already knows the type from the left side of the assignment, Box<String>, and fills in <String> on the right automatically. This is the diamond operator: shorthand for new Box<String>(), not a different, looser kind of Box.

§16.3

Class-Level Generics — one type parameter, shared by the whole class

§16.2's T was declared on the class itself: class Box<T>. That placement matters — it means every field, constructor, and instance method in the class body shares the exact same T, and that T is fixed the moment a particular Box is created. A generic class, in other words, is a reusable template that adapts to a specific type per instance: Box<String> is a box that only ever holds Strings, Box<Integer> only ever holds Integers — same logic, different types, both fully type-safe.

Reach for a class-level type parameter whenever the class is a container or a service built around one piece of data, and every field and method needs to agree on what that data's type actually is.

Scope: Which Members Can Actually Use T

A class-level type parameter's scope is the whole class — but "whole class" specifically means:

  • non-static fields
  • constructors
  • instance methods

Static members are left out on purpose.

Why Static Methods Can't Use Class-Level Generics

The confusion here usually stems from thinking that Box<Integer> and Box<String> are two different classes. In reality, they're the same class, producing different instances — and that's exactly why a static member can't get involved.

A static member belongs to the class itself, not to any one instance — every instance shares the same one. Suppose two Boxes exist at once: 

Box<Integer> intBox = new Box<>();
Box<String> strBox = new Box<>();

If Box had a static field typed T, both of these would be reaching for the exact same field — intBox setting it to hold an Integer, strBox setting it to hold a String. Since there's only one copy of a static field, shared by every instance, it can't be an Integer and a String at the same time. Worse, a static member exists even before any instance does — so there isn't even an instance yet to ask which T it should mean.

Memory area Content
Static area Class Box (only one copy exists)
Heap — instance 1 Box object (thinks T is Integer)
Heap — instance 2 Box object (thinks T is String)

The static area is a single shelf, shared by the whole class; each instance on the heap is its own separate item, free to disagree with every other instance about what T actually is. A static member has no instance to ask, so the compiler refuses outright to let it reference the class's own T: 

public class Box<T> {
    private T content;

    public static void printContent(T content) { // compile error — cannot make a static reference to the non-static type T
        System.out.println(content);
    }
}

If a static method genuinely needs to be generic, it declares its own type parameter, entirely independent of the class's — exactly the mechanism §16.6 covers next.

Type Parameter vs. Type Argument

This distinction is worth being precise about, because the two terms name two different moments:

  • Type parameter — the placeholder declared on the class: T in class Box<T>.
  • Type argument — the concrete type supplied by the caller: String in Box<String>.
A contractual placeholder

Think of T as a promise the class makes to itself: "I don't know the exact type yet, but whoever creates me will pick one, and every field and method here will honor that choice." The moment a caller writes Box<String>, that promise is filled — T is locked to String for that object, for good.

Binding Time: One Template, Many Instances

Box<T> itself is never bound to any type — only each instance is, and the binding happens at the moment of construction: 

Box<Integer> intBox = new Box<>(); // T is locked to Integer for this object, and this object alone
intBox.set(2);

Box<String> strBox = new Box<>(); // a completely separate binding — T is String here
strBox.set("two");

intBox and strBox are both Boxes, but as far as the compiler is concerned they're different species from this point on — nothing that compiles against one says anything about the other. And yet, at runtime, both are secretly running the exact same class: 

System.out.println(intBox.getClass().getName()); // Box
System.out.println(strBox.getClass().getName()); // Box — identical, despite Integer vs. String

That's type erasure — the compiler checks every use of T against the type argument, then discards T entirely once compilation is done, leaving one ordinary Box class behind. §16.10 covers exactly what that means and doesn't mean; for now, the point is narrower: Box<Integer> and Box<String> are two different contracts enforced by the compiler, on top of one shared class underneath.

Object get() T get()
Flexibility High — can hold anything High — can hold anything, per instance
Safety Low — cast, and hope High — the compiler guarantees the type
Responsibility The caller must remember the type The class tracks the type, for every caller
The soda fountain

Box<T> is one machine — one nozzle, one button, one drain — the same way a soda fountain is one machine regardless of what's in the cup. T is the flavor selected. Pick "Cola," and the machine's physical structure never changes — but for that one cup, it behaves strictly as a Cola machine. Nobody needs to taste it to rule out orange juice; the selection already guaranteed it.

One restriction applies no matter what: a type argument can never be a primitive. There's no Box<int> — reach for the wrapper type instead, Box<Integer>.

§16.4

Generic Interfaces — a shared contract, still parameterized by type

Box<T> is one class. Suppose the codebase grows a second, genuinely different way of holding a single value — a Crate<T>, say, that logs every time its value changes. Both classes make the same promise: put a value of some type in, get a value of that same type back out. Nothing today says so: 

class Box<T> {
    private T content;

    public void set(T content) { this.content = content; }
    public T get()              { return content; }
}

class Crate<T> {
    private T content;

    public void set(T content) {
        System.out.println("Storing: " + content);
        this.content = content;
    }
    public T get() { return content; }
}

Box<T> and Crate<T> share an identical shape, by coincidence — nothing in either declaration says they're interchangeable. A method that wants to accept "anything that holds a single value of type T" has no type to ask for. Writing it against Box<T> specifically excludes every Crate<T>, and there's no shared supertype linking the two: 

static <T> void printContents(Box<T> box) { // locked to Box specifically
    System.out.println(box.get());
}

printContents(new Crate<>()); // compile error — Crate is not a Box

This is the exact shape §14's Renewable problem: a capability shared by unrelated classes, with no lineage connecting them. The fix is the same one — an interface — except this time the capability itself is generic: 

interface Container<T> {
    void set(T value);
    T get();
}

Container<T> declares the same two method signatures Box and Crate already had, parameterized the same way a generic class is — T here is a placeholder filled in by whatever implements the interface, exactly as it's filled in by whatever creates a Box<T>.

Choice 1: Implement It Generically, Passing T Straight Through

The most common choice: the class stays exactly as generic as the interface, and its own type parameter simply is the interface's: 

class Box<T> implements Container<T> {
    private T content;

    @Override
    public void set(T content) { this.content = content; }

    @Override
    public T get() { return content; }
}

class Crate<T> implements Container<T> {
    private T content;

    @Override
    public void set(T content) {
        System.out.println("Storing: " + content);
        this.content = content;
    }

    @Override
    public T get() { return content; }
}

Box<T> and Crate<T> lose nothing — either can still hold any type at all, decided per instance exactly as before (§16.3). What they gain is a shared type to be referred to by: 

static <T> void printContents(Container<T> container) { // accepts anything implementing Container
    System.out.println(container.get());
}

Container<String> c1 = new Box<>();
c1.set("hello");

Container<String> c2 = new Crate<>();
c2.set("world"); // Storing: world

printContents(c1); // hello
printContents(c2); // world

c1 and c2 are a Box and a Crate underneath — unrelated classes, no lineage in common beyond Object — yet both are fully substitutable wherever a Container<String> is expected. That's §14's interface-based polymorphism, with a type parameter riding along on top of it.

Choice 2: Implement It for One Fixed Type

A class doesn't have to stay generic to implement a generic interface — it can supply a concrete type argument instead, the same way a variable declares one: 

class StringContainer implements Container<String> {
    private String content;

    @Override
    public void set(String content) { this.content = content; }

    @Override
    public String get() { return content; }
}

StringContainer isn't generic at all — it's an entirely ordinary class, with no <T> of its own. It just happens to satisfy Container<String>, permanently, because it wrote String everywhere the interface wrote T. This is the right choice when a class is only ever going to hold one specific type, and being generic itself would just be unused flexibility: 

Container<String> c3 = new StringContainer();
c3.set("!");
printContents(c3); // !

Choice 3: Implement It Raw — legal, not advisable

A class can also implement Container with no type argument at all — a raw type: 

class RawContainer implements Container { // raw type — no <T> at all
    private Object content;

    @Override
    public void set(Object content) { this.content = content; }

    @Override
    public Object get() { return content; }
}
Why this compiles, and why it's still §16.1's problem

Raw types exist purely so pre-generics code (anything written before Java 5) still compiles unmodified. Every occurrence of T is erased to its bound — here, plain Object — and every one of the type-safety guarantees this chapter builds toward is gone: set accepts anything, get returns only Object, and the compiler is back to warning, not enforcing. Reach for a raw type only when working with legacy code that requires it; a new class implementing Container should always supply a type argument, whether its own T (Choice 1) or a concrete type (Choice 2).

§16.5

Diamond Operator, Type Inference, and var — when the compiler fills in the type, and when it can't

§16.2 introduced the diamond operator, <>, as shorthand — new Box<>() instead of writing new Box<String>() in full. It's worth being precise about exactly where that shorthand gets its answer from, because there's exactly one place it looks — and one common way of writing the line removes that place entirely.

What the Diamond Actually Infers From

The diamond operator doesn't guess. It copies the type argument from the target type — the declared type standing right there on the left of the assignment: 

Box<String> box = new Box<>(); // <> copies String from the declared type on the left

Read the line right to left: the compiler sees Box<String> being declared, so when it reaches new Box<>(), there's exactly one type that could possibly go in the empty brackets. This is type inference in its simplest form — the compiler isn't inventing information, it's copying information that's already sitting in plain sight, one token to the left.

The var Trap

var (§1.5) also infers a type — but from the right side of the assignment, working out what type a variable should be from whatever expression is being assigned to it. Combine var with the diamond operator, and each side is left waiting on the other: 

var box = new Box<>(); // what is box's type? what does <> mean? neither side says.

var has no declared type of its own for the diamond to copy — that's the entire point of writing var. And the diamond operator has nothing on its own to offer var either, since Box's constructor here takes no argument to infer a type from. Left with no information anywhere, the compiler falls back to the one type every reference can hold: Object. The line compiles — but silently, box is a Box<Object>, not the Box<String> it probably looked like: 

var box = new Box<>(); // box is actually a Box<Object>

box.set("hello"); // compiles — Object accepts a String
box.set(42);      // ALSO compiles — box's T silently became Object, not String
Var and the diamond don't infer from each other

var and <> each depend on information the other one was supposed to provide, and neither will invent a type out of nothing — the fallback is always Object, quietly reopening §16.1's exact problem. Combining them safely means giving the compiler a real source to infer from: either drop var and declare the type on the left, or drop the diamond and name the type argument explicitly on the right. 

Box<String> box = new Box<>(); // declared type on the left — <> has something to copy

var box = new Box<String>();  // type argument named explicitly — var has something to infer

Either line is fully type-safe; only pairing var with an empty diamond, and nothing else pinning the type down, loses the guarantee this whole chapter is about.

§16.6

Generic Methods — a type parameter on a single method

A type parameter doesn't require a generic class around it — a single method can declare its own, scoped only to that method, written just before the return type: 

static <T> T firstNonNull(T a, T b) {
    return a != null ? a : b;
}
String name = firstNonNull(null, "default");     // T is inferred as String
Integer count = firstNonNull(null, 5);             // T is inferred as Integer

firstNonNull isn't written for String, or for Integer, or for any one type — T is decided fresh at each call, inferred from whatever arguments actually show up. One method body, used at as many types as callers need, each call fully type-checked on its own terms.

§16.7

Bounded Types — <T extends ...>

Plain T promises nothing except "some type" — not even Object's methods are assumed beyond what every type already has. Calling anything more specific on a value of type T requires telling the compiler what T is guaranteed to be. A bounded type parameter does exactly that: 

static <T extends Renewable> void renewAll(T[] items) {
    for (T item : items) {
        item.renew();
    }
}
Subscription[] subs = { new Subscription(), new Subscription() };
renewAll(subs); // fine — Subscription implements Renewable

T extends Renewable restricts what T is allowed to be to types that implement Renewable (§14.2) — despite the keyword, this works identically whether Renewable is a class being extended or, as here, an interface being implemented. That restriction is precisely what makes item.renew() legal inside the method: the compiler knows, for any type that could ever fill in T here, that renew() is guaranteed to exist.

§16.8

Invariance — the illusion of subtype polymorphism

The most common way to build a heterogeneous collection — one that holds several different concrete types at once — is to declare it in terms of a common supertype or interface, and let it hold instances of anything that is one: 

List<Vehicle> vehicles = new ArrayList<>();

vehicles.add(new Car()); // ok — a Car is a Vehicle
vehicles.add(new Bus()); // ok — a Bus is a Vehicle

This works, but it's really two distinct ideas working together, and it's worth pulling them apart:

  • polymorphism, which is about what a single Vehicle-typed slot — a variable, a field, an element, a method parameter like add(Vehicle), or a method's declared return type — can actually hold, accept, or hand back, and
  • invariance, which is about whether one whole parameterized type can stand in for another.

Polymorphism: What a Single Slot Can Hold or Accept

vehicles holds Vehicle references — and thanks to ordinary subtype polymorphism (§10.5), a Vehicle reference can point at an actual Car, a Bus, or any other subtype, the exact same way a plain variable of type Vehicle could. Every one of those really is a Vehicle: 

Car car1 = new Car();
Car car2 = new Car();

List<Car> cars = List.of(car1, car2);
List<Vehicle> vcars = List.of(car1, car2); // the exact same two objects, viewed as Vehicles

Both cars and vcars can end up holding addresses of the identical two objects — only the declared element type differs. Adding to a List<Vehicle> follows the same upcasting rule (§10.5) as passing an argument: anything that is a Vehicle is accepted, automatically: 

vehicles.add(new Car()); // upcast: Car → Vehicle, exactly like passing a Car to a Vehicle parameter
vehicles.add(new Bus()); // upcast: Bus → Vehicle

Retrieval works in reverse: whatever comes back out of a List<Vehicle> is typed as Vehicle, full stop — the list doesn't remember that a particular element started life as a Car. Reaching for anything more specific than Vehicle needs an explicit instanceof check and a cast, the same as any other downcast (§10.7).

How Far Can This Go? List<Object>

Every class in Java extends Object (§10.3), so a List<Object> is the most permissive version of this idea — it accepts anything at all: 

List<Object> list = new ArrayList<>();

list.add("hello");      // String — upcast to Object
list.add(42);            // Integer — upcast to Object
list.add(new Vehicle()); // Vehicle — upcast to Object

Every one of those is valid for the same reason: String, Integer, and Vehicle all are Objects, so they all satisfy at least what Object guarantees (toString(), equals(), and so on). This buys flexibility of storage, not behavior — without a cast, nothing more specific than Object's own methods is callable on what comes back out.

The real restriction

The restriction generics enforce was never about what a list can hold — it's about what reference it can be assigned to: 

List<Object> list = new ArrayList<Object>(); // fine

List<Object> list2 = new ArrayList<String>(); // compile error — incompatible types

That fails even though every String genuinely is an Object — the reason why is exactly what the rest of this section is about.

Reading Back Out: instanceof and Pattern Matching 

List<Number> numbers = new ArrayList<>(); // holds any subtype of Number

numbers.add(10);   // Integer, upcast automatically
numbers.add(3.14); // Double
numbers.add(100L); // Long

var e = numbers.get(0); // declared type: Number

if (e instanceof Integer i) { // pattern-matching instanceof (§10.7)
    System.out.println("Integer: " + i);
}

Storing several subtypes of Number costs nothing — but reading one back out means the specific type is gone unless something checks for it. That's the trade-off a heterogeneous list always makes: type safety on the way out is lower than on the way in, and a cast to the wrong type fails only at runtime, with a ClassCastException (§16.1) — never at compile time. 

Vehicle v = vehicles.get(0);      // always safe — every element genuinely is a Vehicle
Car c = (Car) vehicles.get(0);    // compiles, but risky — fails at runtime if that element isn't actually a Car

Invariance: Why List<Car> Isn't a List<Vehicle>

Given cars and vcars from earlier — built from the exact same two objects — it's tempting to assume one can substitute for the other, the same way a Car substitutes for a Vehicle. It can't: 

vcars = cars; // compile error — incompatible types, List<Car> is not a List<Vehicle>

This is invariance: even though Car is a Vehicle, List<Car> is not a List<Vehicle> — the subtyping relationship between the element types doesn't carry over to the parameterized types built from them. Two lists can hold identical objects and still be incompatible types, because what each one allows differs.

Why Invariance Matters in Practice

A method call is an implicit assignment — the compiler checks an argument against a parameter exactly the way it checks the right side of = against the left. So invariance shows up here too, not just at an explicit =: 

void process(List<Vehicle> list) {
    list.add(new Bus());     // fine — the compiler knows list is a List<Vehicle>
    list.add(new Vehicle());  // fine
}

ArrayList<Car> cars = new ArrayList<>();
process(cars); // compile error — List<Car> is not a List<Vehicle>

This isn't an arbitrary restriction. Inside process, the compiler is certain list is a List<Vehicle> — which means it's certain list.add(new Bus()) is legal. If process(cars) were allowed, that same add call would just have slipped a Bus into what the caller still believes is a List<Car>. Invariance is exactly what closes that gap — it exists specifically to protect a list that might be written to.

Making this flexible again means being explicit about which direction the list is actually used in: 

// Only reading: accepts List<Car>, List<Bus>, List<Vehicle>, ... — cannot safely add
void readVehicles(List<? extends Vehicle> list) { }

// Only adding: accepts List<Vehicle>, List<Object>, ... — cannot safely read as anything specific
void addVehicles(List<? super Vehicle> list) { }
Producer extends, consumer super

A list a method only ever reads from is producing values for it — ? extends Vehicle fits, because every element, whatever its exact subtype, can be read as a Vehicle. A list a method only ever adds to is consuming values from it — ? super Vehicle fits, because anything that can legally hold a Vehicle can accept one. Trying to add through ? extends, or read anything specific out of ? super, is exactly the case each wildcard was built to rule out.

The Unbounded Wildcard: List<?>

Sometimes the element type genuinely doesn't matter at all. List<?> — a list of some unknown type — accepts a reference to any List, of any element type, but the price is steep: 

List<?> list = List.of(new Car(), new Bus(), new Vehicle()); // ok

list.add(null);         // ok — null belongs to every type
list.add(new Bus());    // compile error — the compiler has no idea what "?" actually is here

Object element = list.get(0); // safe, but only as Object — the specific type is gone

Once a list is declared List<?>, nothing can be added to it except null — the compiler has thrown away the one piece of information (what ? actually stands for) that would let it check anything else. Reading still works, but only as far as Object. List<? super Vehicle> is the useful middle ground: unknown exactly what supertype of Vehicle it holds, but definitely a supertype, so adding a Vehicle — or any of its subtypes — is still safe: 

List<? super Vehicle> list = new ArrayList<>();

list.add(new Car());     // ok
list.add(new Bus());     // ok
list.add(new Vehicle());  // ok
§16.9

Wildcards — accepting a broader promise than the one asked for

A wildcard, ?, stands for a type argument that isn't known — or doesn't need to be known — at the point something is declared. It never appears where T would, as a type parameter on a class or method; a wildcard only ever fills in the <> on something already generic, as a type argument. Java has three forms:

Form Reads as Allows
List<?> a list of some unknown type reading elements as Object; writing nothing but null
List<? extends Vehicle> a list of some unknown subtype of Vehicle reading elements as Vehicle; writing nothing — the exact subtype is unknown
List<? super Vehicle> a list of some unknown supertype of Vehicle writing a Vehicle or any of its subtypes; reading elements only as Object

§16.8 already put each of these three forms to work, one at a time — readVehicles, addVehicles, and the fully unbounded List<?>. What follows is a single worked example, named for what it actually solves, showing why a method from the standard library itself needs exactly this.

Use Case: Reusing a Broader Predicate

Predicate<Vehicle> represents a function that can test any Vehicle — its test(Vehicle) method takes a Vehicle parameter, and thanks to ordinary polymorphism (§10.5), a Car argument is accepted without complaint: 

Predicate<Vehicle> canHandle = vehicle -> vehicle != null; // stands in for some real business rule

Car myCar = new Car();
canHandle.test(myCar); // fine — test(Vehicle) accepts a Car, same as any Vehicle-typed parameter would

canHandle genuinely can test a Car. It doesn't know or care that it's being handed a Car specifically — a Predicate<Vehicle> is built to test anything that's a Vehicle, which a Car always is.

Where This Breaks: Stream<T>.filter()

Stream<T> declares a filter method that takes a predicate to decide which elements survive. If it were declared the obvious way — filter(Predicate<T> predicate) — then for a Stream<Car>, that would demand a Predicate<Car>, exactly: 

Stream<Car> carStream = cars.stream();

carStream.filter(canHandle); // if filter took Predicate<T> exactly: compile error
                              // Predicate<Vehicle> is not a Predicate<Car> — invariance (§16.8)

This is invariance again, showing up one level down — inside a method parameter instead of a variable assignment. And it's rejecting something that obviously ought to work: canHandle can test a Car perfectly well, since every Car is a Vehicle. The restriction isn't protecting anything here — filter only ever calls test, handing it elements; it never asks the predicate to hand anything back that needs to be exactly a Car.

The Real Declaration: Predicate<? super T>

Which is exactly why the actual signature isn't the obvious one: 

Stream<T> filter(Predicate<? super T> predicate)

The wildcard opts in to a kind of polymorphism at the parameter level — the same way a plain Vehicle parameter accepts a Car argument, Predicate<? super T> accepts a predicate built for T, or for anything broader than T: 

carStream.filter(canHandle); // compiles — Predicate<Vehicle> satisfies Predicate<? super Car>

This is contravariance: as the element type narrows from Vehicle to Car, the set of predicates filter will accept actually grows wider, not narrower — every Predicate<Car> still qualifies, and so does every Predicate built for any supertype of Car. That's the same producer/consumer reasoning §16.8 already named: filter only ever consumes elements by handing them to the predicate, so ? super is precisely the direction that stays safe.

§16.10

Compile Time Only — what generics don't do

Every guarantee generics make is a compile-time one. Once the code compiles, the type parameter itself is gone — Java's generics work by checking everything up front, then discarding the type information, a design called type erasure. A Box<String> and a Box<Integer> are, underneath, running the exact same class at runtime; nothing about T survives to be inspected later.

This is not a loophole to work around — it's the actual scope of what generics promise. They don't add a new runtime check; they move a whole category of mistake (§16.1's ClassCastException) to a place it can be caught immediately, at compile time, instead of a much later, harder-to-trace line where the bad value finally gets used.

Scope of this chapter

Multiple bounds, generic interfaces, and the finer rules around raw types are all left for later. The core idea carries over unchanged: a type parameter lets a class or method make a promise about what type it's working with, checked once, at compile time, everywhere it's used.
