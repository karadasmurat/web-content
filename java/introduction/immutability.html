Chapter 11

Immutability

String's refusal to change (§8.4) wasn't a special privilege only the standard library gets. It's a design any class can choose — and once you see why it's worth choosing, it changes how you look at every class before this one.

§11.1

Two Real-World Analogies — things that change, and things that don't

Open a bank account with a starting balance of $100. Deposit $50, and the balance becomes $150. Withdraw $30, and it becomes $120. At every point in this little story, it's the exact same account — same account number, same owner, the very account you opened at the start. What changed is its balance, moving through a sequence of different values as time passed. The account has an identity that never moves, and a state — the balance — that does. Ask "what's the balance of this account?" and the honest answer is "it depends - at what point in time?"

A car works the same way. Turn the key, and the engine starts. It's still the same car — same VIN, the same car parked in the driveway last night — but a moment ago it was off, and now it's running. Same car, different current condition.

Both stories have the same shape: one fixed identity, and a state that moves through a timeline of different values while that identity stays put. Most of the objects built so far in this tutorial work exactly this way — §11.2 makes that precise, and §11.3 puts it in code. But not everything is built to allow this. Some things are deliberately constructed so that, once they exist, nothing about them is ever allowed to move again.

Look back at the account above — not its balance this time, but its history. Every deposit and every withdrawal was recorded as a transaction: a line in the ledger saying what happened and when. Once posted, a transaction entry is never edited — a mistake gets corrected by recording a new, offsetting transaction, not by rewriting the original line. The record of what happened at that moment stays exactly as it was, forever; correcting it means adding a new entry, never touching the old one. The balance moves; the trail of entries describing how it got there does not.

A printed book works the same way. Once a copy comes off the press, its pages don't change. So what happens when the author makes changes? The author doesn't reach into libraries and living rooms to overwrite pages in copies that already exist — that's not even possible. Instead, the publisher prints a second edition: a new, separate set of pages, sitting on the shelf right alongside the first. Every first-edition copy still says exactly what it always said; nothing about it ever changed. The author's revision produced a new object, not a modification of the old one.

The transaction ledger and the book are both frozen snapshots of a moment in time — the opposite of the account balance and the running engine, whose entire nature was to keep moving. A Java object can be built to behave either way, and this chapter is about making that a deliberate choice instead of an accident.

§11.2

Identity and State — what an object actually is

An object's fields, taken together, are a representation: a snapshot, held in memory, of some condition at a moment in time. A BankAccount with balance = 150 represents an account currently holding $150 — that number is, as far as the program is concerned, what the account's current state means, exactly like the balance from §11.1's story.

Every class considered so far has quietly assumed that a representation can change. Reassign account.balance, and the object's fields hold a different number than they did a moment ago — the object's state changed. Nothing about its identity changed: it's still the same object, still the same address (§2.3), still the account the reference has pointed to the entire time. Only what it represents is different now. This is worth naming precisely, because it's a choice, not a law of nature: identity — which object this is — and state — what that object currently represents — are two separate things, and a class gets to decide whether the second one is allowed to move while the first one stays fixed.

§11.3

A Question of Time — what mutation actually costs

Every object considered so far has been mutable: its state can change after construction, while its identity stays fixed, and the object is meant to represent one thing, continuously evolving through the kind of timeline §11.1 described. A BankAccount's balance this morning might not be its balance this afternoon. The identity never moves — same object, same address. What changes is what you find when you dereference it, depending on when you ask:

class BankAccount {
    double balance;

    BankAccount(double balance) {
        this.balance = balance;
    }
}
BankAccount account = new BankAccount(100);
System.out.println(account.balance); // 100.0

account.balance += 50;  // a deposit
System.out.println(account.balance); // 150.0 — same object, different answer

Two calls to the exact same expression, account.balance, gave two different answers — not because the expression changed, but because time passed, and the object underneath it changed with it, exactly the deposit from §11.1's story, now in code. Call this temporal instability: for a mutable object, "what does this reference mean" doesn't have one fixed answer. It has an answer for right now — one that a line of code five minutes from now, written by someone who never touches this variable directly, might already have replaced.

An immutable object refuses to have that problem. Once constructed, its state is a frozen snapshot — forever. A String's characters, at the moment it's created, are its characters for the rest of its existence; no method call, no elapsed time, ever changes that. Representing a new state doesn't mean changing the object — it means constructing an entirely new one, the way toUpperCase() builds a new String rather than editing the old one (§8.4).

What can still change: the reference, never the object

Immutability doesn't freeze variables — only objects. A variable holding an immutable object's address can still be reassigned to a different object's address later; that's ordinary reference assignment (§3), unrelated to whether either object can change. What immutability guarantees is narrower, and more powerful: once you have a specific object's address, dereferencing it gives the same answer every single time, no matter when you ask. The object itself never has a "later" that contradicts its "now."

This is the entire payoff. Mutation makes an object's meaning depend on when you observe it — the same reference can answer a question two different ways, five minutes apart, with no code in front of you explaining why, only elapsed time and some other code, somewhere else, that reached in. Immutability removes that variable from the equation entirely: a reference to an immutable object means exactly one thing, and it means that thing permanently.

§11.4

Building an Immutable Class — private, final, no setters

Four ingredients turn an ordinary class immutable. The first is already familiar: every field private (§10.3), so nothing outside the class can reach in directly. The second is new: final on a field, which restricts it to being assigned exactly once — inside the constructor — and never again, anywhere, for the rest of the object's life. The third is simply a discipline: write no setters, so there's no method left that could reassign a field even from inside the class's own code. The fourth locks the class itself: final on the class, so no subclass (§9) can extend it and add a mutable field of its own, or override a method in a way that breaks the guarantee. 

final class Transaction {
    private final String type;    // "DEPOSIT" or "WITHDRAWAL"
    private final double amount;

    Transaction(String type, double amount) {
        this.type = type;
        this.amount = amount;
    }

    String getType()   { return type; }
    double getAmount() { return amount; }
}

this.amount = amount; runs exactly once, inside the constructor. Try to write this.amount = 0; anywhere else in the class — another method, say — and the compiler refuses: final isn't a request, the way a comment or a naming convention would be. It's enforced.

Notice there's no setType, no setAmount — only getters. Without a setter, and with final blocking direct reassignment, there is no code path, anywhere, that changes a Transaction after it's created — from outside the class, or from any method inside it.

That covers every line already inside Transaction. The one door left is a subclass, written somewhere else entirely, extending Transaction and adding a mutable field of its own, or overriding getAmount() to return something that changes over time. final on the class itself closes that door: final class Transaction makes class Subclass extends Transaction a compile error, so there is no subclass to worry about, because there can't be one at all.

Notice, too, what's missing compared to a point that might need to move, or a balance that might need to grow: there's no method here returning a new Transaction representing some updated version. A posted transaction doesn't need one — it's written once, at the moment it happens, and that's its entire life story. Each new entry just sits beside the last one, forever accurate to the exact moment it was created:

Transaction deposit    = new Transaction("DEPOSIT", 50);
Transaction withdrawal = new Transaction("WITHDRAWAL", 30);

System.out.println(deposit.getAmount());    // 50.0 — its own entry, permanently
System.out.println(withdrawal.getAmount()); // 30.0 — a separate entry, entirely unaffected by the first

This is exactly the ledger from §11.1, now in code: the account's balance field keeps moving (§11.3), while each Transaction describing how it moved, once created, never does.

The Core Pillars — a five-point checklist

Everything this chapter is building toward reduces to five checkable rules. A class satisfying all five has no code path, anywhere, that changes its state after construction:

Rule Why
fields private Prevents outside code from reaching in and reading or writing state directly, instead of only through the class's own controlled methods (§10.3).
fields final Prevents the field from being reassigned anywhere after the constructor runs, above — even by the class's own methods.
no setters Removes the one other path a well-encapsulated class would normally offer for changing state (§10.4) — nothing to call means nothing to change.
class final Prevents a subclass (§9) from adding mutable fields of its own, or overriding a method in a way that breaks the guarantee. final class Transaction makes class Subclass extends Transaction a compile error — there is no subclass to worry about, because there can't be one at all.
no exposed mutable state Closes the one gap the first four don't: a final reference field can still point to something mutable, and that state stays reachable through any reference into or out of the object (§11.5, next). Defensive copying — cloning on the way in and the way out — is the usual fix, though the exact technique depends on the type involved.

Transaction, above, already satisfies the first four by construction; it never needed the fifth, because it never held a reference to anything mutable in the first place. ImmutableBox, next, needs all five.

§11.5

final Isn't Deep — the reference-field gotcha

final only locks the field's own slot — the address it holds. If that address points to something mutable, final says nothing at all about what happens on the other end of it:

Copying a reference is copying a key

Assigning a reference is like handing someone a copy of your house key, not a photo of the house. Whoever holds a copy can let themselves in and rearrange the furniture whenever they like — the house doesn't know or care which key was used to open the door, only that it opened. Two variables holding the same address are two keys to the same house: mutate the object through either one, and both variables see the change, because there was never a second house to begin with.

final class ImmutableBox {
    private final int[] values;

    public ImmutableBox(int[] values) {
        this.values = values;
    }

    public int[] getValues() {
        return values;
    }
}
int[] data = {1, 2, 3};
ImmutableBox box = new ImmutableBox(data);

data[0] = 999;
System.out.println(box.getValues()[0]); // 999 — the "immutable" box just changed

values itself is never reassigned — final is satisfied, technically. But values and data hold the exact same address (§3's aliasing, again), so writing through data reaches the identical array box was holding onto the whole time. Worse: getValues() hands out that same live address to any caller, so box.getValues()[0] = 42; would reach in and mutate it just as directly, from entirely outside the class.

Stack
data = 0x8000
box.values = 0x8000
Heap
int[3] @ 0x8000
[0]999
[1]2
[2]3

Two reference variables, one array, no copy anywhere — exactly the aliasing picture from §3, just now undermining the one guarantee this class exists to make. The real problem is broader than this one constructor: any time a final field's address leads to something mutable, that mutable state is still reachable from outside the object, no matter how the address got there or who's holding a copy of it. The fix here is defensive copying: copy the array on the way in, and copy it again on the way out, so no outside reference ever touches the object's real internal state:

public ImmutableBox(int[] values) {
    this.values = values.clone();  // a copy — not the caller's array
}

public int[] getValues() {
    return values.clone();  // a copy — not the internal array
}

Now data[0] = 999; only reaches the caller's own array; box's internal copy is untouched. And box.getValues()[0] = 42; mutates a throwaway copy handed out just for that call — the object's real state, still only reachable through values, never moves.

Cloning is just this array's answer. The actual rule to carry forward is more general: an immutable class can't let any reference — incoming or outgoing — stay pointed at mutable state it doesn't fully own by itself. A different mutable type might call for a copy constructor instead of clone(), or for wrapping the value in an immutable view rather than duplicating it — the mechanism varies, but the requirement doesn't: no outside key still opens a door into this object's real state.

§11.6

Why It Matters — safe sharing

§3 introduced aliasing as a fact of reference types, with a quiet warning attached: modifying an object through one variable is visible through every other variable that happens to hold the same address. With mutable objects, that's a real hazard — hand a BankAccount's address to some other code, and that code can change it out from under you, with no warning.

BankAccount shared = new BankAccount(500);
logForAudit(shared);      // hands the address to some other code
shared.balance -= 500;    // and now, whatever that code kept, it sees a balance of 0 too

With an immutable object, this entire category of surprise is impossible — not mitigated, impossible. Handing out a Transaction's address gives another reference to something that categorically cannot change. Every piece of code holding that address is guaranteed to see the exact same state, forever, no matter what any of the others do. Aliasing stops being a hazard to manage and becomes simply free — sharing an immutable object's address costs nothing, because there's nothing a second reference could do to betray the first.

Everything above already holds even if "some other code" isn't just another method running a moment later, but another thread — a separate, independently running line of execution, active at the exact same instant as the one holding shared. Two threads reading and writing the same mutable object at once can each observe the other's change mid-update, or overwrite it entirely, producing a race condition: a bug that depends on timing so precise it may not even reproduce the same way twice. An immutable object sidesteps that too, for the identical reason it sidesteps ordinary aliasing surprises — if nothing about it can ever change, there is nothing for two threads to race over. This is why immutable types are the standard first choice for data shared across threads, and it's the main reason this chapter has been worth the trouble: the full mechanics of threads, and what it actually takes to share mutable state safely between them, are the subject of a later chapter on concurrent programming.

§11.7

The Cost of Immutability — StringBuilder

Everything so far has argued for immutability. It isn't free, though — and String itself shows where the bill comes due:

String result = "";
for (int i = 0; i < 5; i++) {
    result = result + i;
}

result + i doesn't grow the existing String (§8.4) — it builds an entirely new one, copying every character seen so far, then discards the previous object. Five iterations, five String objects, four of them thrown away the instant they're made. For a loop this small, it's invisible. For a loop over thousands of appends, it means allocating and immediately discarding thousands of objects, each copy a little longer — and a little more expensive — than the last.

Trace it iteration by iteration. result is a single stack slot — it only ever holds one address at a time. Each pass through the loop builds a new String, points result at it, and in that same instant leaves the previous String with no reference to it anywhere. Nothing deletes that orphaned object — it simply becomes unreachable, sitting on the heap until the garbage collector eventually reclaims its memory:

The stack side is a single slot, result — never five different variables, just one, reassigned once per iteration. Each reassignment points it at a new String and, in that same instant, drops the previous String to zero incoming references. Nothing else in the program was ever holding those earlier addresses, so the moment result moves on, they become unreachable — not deleted, just no longer reachable from anywhere a running program could look. One live reference, four orphans: by the time the loop exits, four of the five String objects built along the way are unreachable, and only "01234", the last one built, is still reachable, through the one result slot that's held it since the loop ended.

Unreachable isn't the same as gone

The four dashed objects above haven't been erased — nothing in Java reaches in and destroys an object the instant it loses its last reference. They're sitting on the heap, fully intact, exactly as they were built. "Unreachable" only means the program itself has no path left to find them. That's what makes an object eligible for garbage collection — whether and when it's actually reclaimed is a separate question, decided later by the JVM's garbage collector on its own schedule, not by this line of code. A running program could easily have several of these lying around at once, still technically present in memory, simply waiting to be swept up.

StringBuilder is the standard library's answer to exactly that cost:

StringBuilder sb = new StringBuilder();
for (int i = 0; i < 5; i++) {
    sb.append(i);
}
String result = sb.toString();

sb.append(i) modifies the same object, in place, every time — no new object, no copying, no discarding. One buffer, growing.

StringBuilder is mutable on purpose

This isn't a loophole in everything this chapter just argued — it's Java making the identical design choice from §11.3/§11.4, deliberately, in the other direction, because building something up piece by piece is exactly the case where mutability is cheaper than immutability. And notice where it ends: .toString() hands back an ordinary, immutable String. The mutable tool's entire job is to do its work and then get out of the way, leaving you with something safe again. Immutable by default; mutable when the cost specifically demands it — String and StringBuilder, side by side, are that whole principle in miniature.

§11.8

Putting It Together — mutable vs. immutable, side by side

One class, deliberately built with all three kinds of field this tutorial has covered, makes every idea in this chapter concrete at once:

class Data {
    int version;          // primitive field — holds a value directly
    String name;           // reference field — points to an immutable object
    StringBuilder notes;    // reference field — points to a mutable object

    Data(int version, String name, String notes) {
        this.version = version;
        this.name = name;
        this.notes = new StringBuilder(notes);
    }
}
Data d1 = new Data(1, "Bob", "init");

Look at what actually happened to each object, not just each field. version changed the ordinary way primitives always have — no reference involved at all. name now reads 0x7100 instead of 0x7000 — but the String at 0x7000, "Bob", is still sitting on the heap, completely unchanged; d1.name simply doesn't point to it anymore. That's the only one of the three that's real mutation: the same object, changed underneath a reference that never moved.

An immutable counterpart to Data applies every pillar from §11.4 to close off that third case entirely:

final class ImmutableData {
    private final int version;
    private final String name;
    private final String notes;   // String, not StringBuilder — nothing left to defend

    public ImmutableData(int version, String name, String notes) {
        this.version = version;
        this.name = name;
        this.notes = notes;
    }

    public int getVersion() { return version; }
    public String getName()  { return name; }
    public String getNotes() { return notes; }

    public ImmutableData withNotes(String newNotes) {
        return new ImmutableData(version, name, newNotes);
    }
}

Swapping StringBuilder for String is what actually closes the gap — with every field now pointing only to immutable types (or holding a primitive directly), final on the fields is enough on its own; there's no mutable object left on the other end of any reference for a defensive copy to guard. withNotes(...) replaces notes.append(...): instead of reaching into existing state, it returns a whole new ImmutableData, exactly the way String's own methods have all along (§8.4).

Scope of this chapter

record — a dedicated, more concise syntax for exactly the shape built by hand in §11.4 and §11.8 — is left for later, along with the deeper rules for keeping equals and hashCode consistent on immutable value types (§11.6 already previewed why threads make immutability worth the trouble in the first place). The core idea carries over unchanged: a class earns the right to call itself immutable the moment nothing, anywhere, can change its state after construction.
