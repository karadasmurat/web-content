Chapter 13

Immutability

String's refusal to change (§8.4) wasn't a special privilege only the standard library gets. It's a design any class can choose — and once you see why it's worth choosing, it changes how you look at every class before this one.

§13.1

Identity and State — what an object actually is

An object's fields, taken together, are a representation: a snapshot, held in memory, of some condition at a moment in time. A Rectangle with width = 10 and height = 20 represents a 10-by-20 rectangle — that pair of numbers is, as far as the program is concerned, what a rectangle of that shape means.

Every class considered so far has quietly assumed that a representation can change. Reassign r.width, and the object's fields hold different numbers than they did a moment ago — the object's state changed. Nothing about its identity changed: it's still the same object, still the same address (§4.2), still the thing r has pointed to the entire time. Only what it represents is different now. This is worth naming precisely, because it's a choice, not a law of nature: identity — which object this is — and state — what that object currently represents — are two separate things, and a class gets to decide whether the second one is allowed to move while the first one stays fixed.

§13.2

A Question of Time — what mutation actually costs

Every object considered so far — Rectangle, Car, Vehicle — has been mutable: its state can change after construction, while its identity stays fixed, and the object is meant to represent one thing, continuously evolving. A Rectangle's width today might not be its width tomorrow. It's still the same object, the same address — but what that address means depends on when you ask:

Rectangle r = new Rectangle(10, 20);
System.out.println(r.width); // 10

r.width = 999;
System.out.println(r.width); // 999 — same object, different answer

Two calls to the exact same expression, r.width, gave two different answers — not because the expression changed, but because time passed, and the object underneath it changed with it. Call this temporal instability: for a mutable object, "what does this reference mean" doesn't have one fixed answer. It has an answer for right now — one that a line of code five minutes from now, written by someone who never touches this variable directly, might already have replaced.

An immutable object refuses to have that problem. Once constructed, its state is a frozen snapshot — forever. A String's characters, at the moment it's created, are its characters for the rest of its existence; no method call, no elapsed time, ever changes that. Representing a new state doesn't mean changing the object — it means constructing an entirely new one, the way toUpperCase() builds a new String rather than editing the old one (§8.4).

What can still change: the reference, never the object

Immutability doesn't freeze variables — only objects. A variable holding an immutable object's address can still be reassigned to a different object's address later; that's ordinary reference assignment (§5), unrelated to whether either object can change. What immutability guarantees is narrower, and more powerful: once you have a specific object's address, dereferencing it gives the same answer every single time, no matter when you ask. The object itself never has a "later" that contradicts its "now."

This is the entire payoff. Mutation makes an object's meaning depend on when you observe it — the same reference can answer a question two different ways, five minutes apart, with no code in front of you explaining why, only elapsed time and some other code, somewhere else, that reached in. Immutability removes that variable from the equation entirely: a reference to an immutable object means exactly one thing, and it means that thing permanently.

§13.3

Building One — private, final, no setters

Two ingredients turn an ordinary class immutable. The first is already familiar: every field private (§12.3), so nothing outside the class can reach in directly. The second is new: final on a field, which restricts it to being assigned exactly once — inside the constructor — and never again, anywhere, for the rest of the object's life.

class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }
    public int getY() { return y; }
}

this.x = x; runs exactly once, inside the constructor. Try to write this.x = 5; anywhere else in the class — another method, say — and the compiler refuses: final isn't a request, the way a comment or a naming convention would be. It's enforced.

Notice there's no setX, no setY — only getters. Without a setter, and with final blocking direct reassignment, there is no code path, anywhere, that changes x or y after construction. A method that needs to represent a moved point returns a new one, exactly like String's methods do:

public ImmutablePoint translate(int dx, int dy) {
    return new ImmutablePoint(x + dx, y + dy);
}
ImmutablePoint p1 = new ImmutablePoint(0, 0);
ImmutablePoint p2 = p1.translate(5, 5);

System.out.println(p1.getX()); // 0 — unchanged
System.out.println(p2.getX()); // 5 — a new object entirely
§13.4

final Isn't Deep — the reference-field gotcha

final only locks the field's own slot — the address it holds. If that address points to something mutable, final says nothing at all about what happens on the other end of it:

class ImmutableBox {
    private final int[] values;

    public ImmutableBox(int[] values) {
        this.values = values;
    }

    public int[] getValues() {
        return values;
    }
}
int[] data = {1, 2, 3};
ImmutableBox box = new ImmutableBox(data);

data[0] = 999;
System.out.println(box.getValues()[0]); // 999 — the "immutable" box just changed

values itself is never reassigned — final is satisfied, technically. But values and data hold the exact same address (§5's aliasing, again), so writing through data reaches the identical array box was holding onto the whole time. Worse: getValues() hands out that same live address to any caller, so box.getValues()[0] = 42; would reach in and mutate it just as directly, from entirely outside the class.

Stack
data = 0x8000
box.values = 0x8000
Heap
int[3] @ 0x8000
[0]999
[1]2
[2]3

Two reference variables, one array, no copy anywhere — exactly the aliasing picture from §5, just now undermining the one guarantee this class exists to make. The fix is defensive copying: copy the array on the way in, and copy it again on the way out, so no outside reference ever touches the object's real internal state:

public ImmutableBox(int[] values) {
    this.values = values.clone();  // a copy — not the caller's array
}

public int[] getValues() {
    return values.clone();  // a copy — not the internal array
}

Now data[0] = 999; only reaches the caller's own array; box's internal copy is untouched. And box.getValues()[0] = 42; mutates a throwaway copy handed out just for that call — the object's real state, still only reachable through values, never moves.

§13.5

Why It Matters — safe sharing

§5 introduced aliasing as a fact of reference types, with a quiet warning attached: modifying an object through one variable is visible through every other variable that happens to hold the same address. With mutable objects, that's a real hazard — hand a Rectangle's address to some other code, and that code can change it out from under you, with no warning.

Rectangle shared = new Rectangle(10, 20);
logForAudit(shared);  // hands the address to some other code
shared.width = 999;   // and now, whatever that code kept, it sees 999 too

With an immutable object, this entire category of surprise is impossible — not mitigated, impossible. Handing out an ImmutablePoint's address gives another reference to something that categorically cannot change. Every piece of code holding that address is guaranteed to see the exact same state, forever, no matter what any of the others do. Aliasing stops being a hazard to manage and becomes simply free — sharing an immutable object's address costs nothing, because there's nothing a second reference could do to betray the first.

§13.6

The Cost of Immutability — StringBuilder

Everything so far has argued for immutability. It isn't free, though — and String itself shows where the bill comes due:

String result = "";
for (int i = 0; i < 5; i++) {
    result = result + i;
}

result + i doesn't grow the existing String (§8.4) — it builds an entirely new one, copying every character seen so far, then discards the previous object. Five iterations, five String objects, four of them thrown away the instant they're made. For a loop this small, it's invisible. For a loop over thousands of appends, it means allocating and immediately discarding thousands of objects, each copy a little longer — and a little more expensive — than the last.

StringBuilder is the standard library's answer to exactly that cost:

StringBuilder sb = new StringBuilder();
for (int i = 0; i < 5; i++) {
    sb.append(i);
}
String result = sb.toString();

sb.append(i) modifies the same object, in place, every time — no new object, no copying, no discarding. One buffer, growing.

StringBuilder is mutable on purpose

This isn't a loophole in everything this chapter just argued — it's Java making the identical design choice from §13.2/§13.3, deliberately, in the other direction, because building something up piece by piece is exactly the case where mutability is cheaper than immutability. And notice where it ends: .toString() hands back an ordinary, immutable String. The mutable tool's entire job is to do its work and then get out of the way, leaving you with something safe again. Immutable by default; mutable when the cost specifically demands it — String and StringBuilder, side by side, are that whole principle in miniature.

§13.7

The Core Pillars — a five-point checklist

Everything in this chapter reduces to five checkable rules. A class satisfying all five has no code path, anywhere, that changes its state after construction:

Rule Why
fields private Prevents outside code from reaching in and reading or writing state directly, instead of only through the class's own controlled methods (§12.3).
fields final Prevents the field from being reassigned anywhere after the constructor runs (§13.3) — even by the class's own methods.
no setters Removes the one other path a well-encapsulated class would normally offer for changing state (§12.4) — nothing to call means nothing to change.
class final Prevents a subclass (§11) from adding mutable fields of its own, or overriding a method in a way that breaks the guarantee. final class ImmutablePoint makes class Subclass extends ImmutablePoint a compile error — there is no subclass to worry about, because there can't be one at all.
defensive copies Closes the one gap the first four don't: a final reference field can still point to something mutable (§13.4). Copying on the way in and the way out severs that connection entirely.

ImmutablePoint (§13.3) already satisfied the first four by construction; it never needed the fifth, because it never held a reference to anything mutable in the first place. ImmutableBox (§13.4) needed all five.

§13.8

Putting It Together — mutable vs. immutable, side by side

One class, deliberately built with all three kinds of field this tutorial has covered, makes every idea in this chapter concrete at once:

class Data {
    int version;          // primitive field — holds a value directly
    String name;           // reference field — points to an immutable object
    StringBuilder notes;    // reference field — points to a mutable object

    Data(int version, String name, String notes) {
        this.version = version;
        this.name = name;
        this.notes = new StringBuilder(notes);
    }
}
Data d1 = new Data(1, "Bob", "init");

Three fields, three different relationships to change:

d1.version = 2;              // 1. the value itself, overwritten directly
d1.name = "Alice";            // 2. the field now points to a brand-new String
d1.notes.append("-update");   // 3. the SAME StringBuilder, mutated through the field
Stack
d1 = 0x8000
Heap
Data @ 0x8000
version2
name0x7100
notes0x7500

«immutable»"Bob"
String @ 0x7100
«immutable»"Alice"
StringBuilder @ 0x7500
«mutable»"init-update"

Look at what actually happened to each object, not just each field. version changed the ordinary way primitives always have — no reference involved at all. name now reads 0x7100 instead of 0x7000 — but the String at 0x7000, "Bob", is still sitting on the heap, completely unchanged; d1.name simply doesn't point to it anymore. Only notes stayed at the same address and changed — 0x7500 held "init", and now, still at 0x7500, it holds "init-update". That's the only one of the three that's real mutation: the same object, changed underneath a reference that never moved.

An immutable counterpart to Data applies every pillar from §13.7 to close off that third case entirely:

final class ImmutableData {
    private final int version;
    private final String name;
    private final String notes;   // String, not StringBuilder — nothing left to defend

    public ImmutableData(int version, String name, String notes) {
        this.version = version;
        this.name = name;
        this.notes = notes;
    }

    public int getVersion() { return version; }
    public String getName()  { return name; }
    public String getNotes() { return notes; }

    public ImmutableData withNotes(String newNotes) {
        return new ImmutableData(version, name, newNotes);
    }
}

Swapping StringBuilder for String is what actually closes the gap — with every field now pointing only to immutable types (or holding a primitive directly), final on the fields is enough on its own; there's no mutable object left on the other end of any reference for a defensive copy to guard. withNotes(...) replaces notes.append(...): instead of reaching into existing state, it returns a whole new ImmutableData, exactly the way String's own methods have all along (§8.4).

Scope of this chapter

record — a dedicated, more concise syntax for exactly the shape built by hand in §13.3 and §13.8 — is left for later, along with the deeper rules for keeping equals and hashCode consistent on immutable value types, and why immutability matters specifically for safely sharing data across threads. The core idea carries over unchanged: a class earns the right to call itself immutable the moment nothing, anywhere, can change its state after construction.
