Chapter 0

Java at a Glance

What Java is, how it works, and the two or three mechanical facts that everything else in this tutorial depends on.

What is Java?

Java is a compiled, statically typed, object-oriented programming language designed to run on any machine without modification. You write source code in .java files. The Java compiler (javac) translates that source into bytecode — a compact, platform-neutral instruction set stored in .class files. The Java Virtual Machine (JVM) then loads and executes the bytecode.

The JVM is available on Windows, macOS, Linux, and more, and the same .class file runs on all of them. This is Java's central guarantee — write once, run anywhere — and it's worth unpacking exactly what makes it possible.

Write Once, Run Anywhere

The Java Workflow, End to End

That word "compiled" is worth pausing on first. A CPU only understands machine instructions — numbers meaningful to that specific processor — and nobody writes programs that way. You write source code: text in a language designed for humans to read and reason about. Compilation is the development-time step that bridges the two — a compiler reads your source, rejects anything that doesn't parse or type-check, and translates whatever survives into an executable form. That step is the first of three distinct stages a Java application moves through, and they don't have to happen on the same machine:

  1. Development. You write Java source code and compile it with a Java compiler (javac) on your development machine. This produces .class files containing bytecode.
  2. Distribution. You package the .class files, along with any resources they need, into a distributable format — typically a .jar file.
  3. Execution. The user on the target platform needs a compatible JVM installed. They run your application, and their JVM executes the bytecode — no recompilation, regardless of what platform built it.

Compilation happens once, on the developer's machine; execution happens anywhere a JVM is installed, unmodified. That separation — compile once, distribute the bytecode, run on any JVM — is what "write once, run anywhere" actually refers to. The next section looks at the mechanics that make the separation possible.

"A compatible JVM installed" means installing a JDK — the Java Development Kit. It bundles javac, the java launcher (which starts the JVM and runs your bytecode), the standard library, and the rest of the developer tooling in one package. You'll sometimes see the term JRE (Java Runtime Environment) in older material — it referred to a separate, smaller download containing just the JVM and libraries needed to run a compiled program, with none of the development tools. Oracle stopped distributing a standalone JRE starting with Java 11: today there's just the JDK, and both the developer compiling the code and the end user running it install the same thing.

Compiling to Native Code vs. Compiling to Bytecode

To see why that guarantee holds, it helps to compare Java's pipeline to a language like C, which compiles straight to native machine code.

Java inserts a layer of indirection: javac never targets a CPU at all. It targets the JVM's instruction set, which is the same everywhere. Portability moves from the compiler to the runtime — a JVM is built once for each platform, and it knows how to translate that one bytecode format into whatever native instructions the local CPU needs. Java's output stays identical no matter where it was compiled; it's the JVM that varies from platform to platform, not the bytecode.

The trade-off

C's approach produces a binary that talks to the CPU directly — no intermediate translation at runtime, and no extra program has to be installed to run it. Java's approach gives up that directness in exchange for portability: the same compiled artifact runs anywhere a JVM exists, at the cost of needing that JVM present and doing a bit of translation work as the program runs.

Your First Program

Here is the simplest complete Java program:

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, world.");
    }
}

Don't worry about public, static, or String[] args yet — those will make sense by the end of this tutorial. The key observation: even this trivial program is organized as a class. In Java, everything lives inside a class. There are no freestanding functions.

Save it as Hello.java — the file name must match the public class name exactly, including case. With a JDK installed, here's how to compile and run it, step by step:

  1. Open a terminal. Also called the command line, or a shell — a text window where you type the name of a program and it runs, rather than clicking an icon. On macOS that's Terminal; on Windows, Command Prompt or PowerShell; on Linux, whichever shell your distribution ships.
  2. Navigate to the folder containing Hello.java.
  3. Compile it: 
    javac Hello.java
    Console Output

    (nothing printed — a silent, empty terminal means it compiled successfully)

    javac reads the text in Hello.java and, if it compiles cleanly, produces no console output at all. Instead it writes a new file, Hello.class, sitting right next to it on disk — the bytecode.
  4. Run it: 
    java Hello
    Console Output
    Hello, world.
    java starts the JVM and tells it to run the class named Hello. Notice it's given the class name, not a file name — no .class extension. The JVM finds Hello.class on disk, loads its bytecode, and executes it, which is what finally prints the line above to the terminal.

That's the Development stage from earlier, compressed onto a single machine: javac compiles, java is the JVM executing the result. Ship that same Hello.class file to any other machine with a JVM installed, and java Hello runs it there too, unmodified.

Key Features

Statically Typed

Every variable has a type declared at compile time. The compiler rejects programs where types don't line up. Many errors are caught before the program ever runs.

Object-Oriented

Programs are organized around objects — bundles of data (fields) and behavior (methods). This tutorial builds that model from the ground up.

Garbage Collected

You allocate objects with new; the JVM frees them automatically when they're no longer reachable. There is no free() or delete.

Strongly Typed

There are no silent conversions between unrelated types. An int is not a boolean. A String is not a Rectangle. The type system is not optional.

Why this matters for the tutorial

Because Java is statically typed, every variable declaration includes a type. That type determines whether the variable holds a value directly (primitive) or an address (reference). That distinction is the central thread of everything that follows.
