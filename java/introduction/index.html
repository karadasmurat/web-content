Chapter 0

Java at a Glance

What Java is, why it's still worth learning, how it works, and the two or three mechanical facts that everything else in this tutorial depends on.

§0.1

What Is Java?

Java is a general-purpose programming language, used to build everything from small command-line tools to large-scale systems serving millions of users.

Language and platform

Java is both a programming language and a platform. The language is what you write. The platform — the JVM, the standard library, the tooling — is what provides everything needed to run what you wrote.

§0.2

Where Is Java Used?

Java powers software across a wide range of industries and environments — often invisibly, from the perspective of the people using it:

Area Examples
Backend services Web APIs, business applications, cloud services
Banking & finance Transaction processing systems, trading platforms
E-commerce Online stores, payment processing systems
Big data Apache Hadoop, Apache Kafka, and their ecosystems
Android development Historically one of Android's primary languages
Desktop applications Cross-platform desktop software

Even when users never realize it, many of the applications and services they interact with every day are running on Java behind the scenes.

§0.3

Why Java?

Why invest time learning a language whose first version shipped in 1996? A few reasons hold up well:

Portable

The same compiled bytecode runs unmodified on any platform with a JVM installed — the mechanics are §0.9's subject in full.

Reliable

A strong type system, automatic memory management, and a mature standard library help developers write software that's easier to maintain and less prone to common errors.

High Performance

Early Java was criticized for interpreting bytecode slowly. Modern JVMs use Just-In-Time (JIT) compilation and runtime profiling to turn frequently executed code into highly optimized machine code while the program runs — fast enough for large-scale backend and financial systems today.

Rich Ecosystem

An extensive standard library, backed by thousands of open-source libraries, frameworks, and development tools.

Built for Concurrency

Comprehensive support for multithreading and concurrent programming, suited to today's multi-core processors.

Mature & Backward-Compatible

Nearly thirty years of continuous development, with a strong emphasis on compatibility — applications written years ago commonly still run on modern Java releases with little or no modification.

Throughout this tutorial

You'll see how these design principles influence the language itself. Features that seem restrictive at first often exist because they help software stay understandable and maintainable as a project grows from a few hundred lines to millions.

§0.4

The Java Ecosystem

When people say "Java," they usually mean Java SE (Standard Edition) — the core platform: the language itself, the compiler, the JVM, and the standard libraries every Java developer uses. This tutorial focuses entirely on Java SE; everything covered here applies whether you later build desktop applications, backend services, or command-line tools.

Most real-world Java applications are also built on a framework — Spring Boot is the most popular, simplifying web applications and microservices; Quarkus and Micronaut are two others. These frameworks are built on top of Java, not instead of it — learning the language first gives you a solid foundation for picking up any framework afterward.

§0.5

Java Versions

Java is actively developed, with a new feature release every six months. Some releases are designated Long-Term Support (LTS) — these receive updates for years and are the usual choice for production systems. Other releases receive support for a much shorter window, aimed at developers who want early access to new features.

Type Typical use
LTS release Production systems and long-term projects
Feature release Trying new language features, early adoption

This tutorial targets Java 25. Most examples also work on earlier versions, but occasionally something introduced more recently gets used deliberately, favoring modern, idiomatic style over maximum backward reach.

§0.6

Your First Program

Here is the simplest complete Java program:

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, world.");
    }
}

Don't worry about public, static, or String[] args yet — those will make sense by the end of this tutorial. The key observation: even this trivial program is organized as a class. In Java, everything lives inside a class. There are no freestanding functions.

Save it as Hello.java — the file name must match the public class name exactly, including case. With a JDK installed, here's how to compile and run it, step by step:

  1. Open a terminal. Also called the command line, or a shell — a text window where you type the name of a program and it runs, rather than clicking an icon. On macOS that's Terminal; on Windows, Command Prompt or PowerShell; on Linux, whichever shell your distribution ships.
  2. Navigate to the folder containing Hello.java.
  3. Compile it: 
    javac Hello.java
    Console Output

    (nothing printed — a silent, empty terminal means it compiled successfully)

    javac reads the text in Hello.java and, if it compiles cleanly, produces no console output at all. Instead it writes a new file, Hello.class, sitting right next to it on disk — the bytecode.
  4. Run it: 
    java Hello
    Console Output
    Hello, world.
    java starts the JVM and tells it to run the class named Hello. Notice it's given the class name, not a file name — no .class extension. The JVM finds Hello.class on disk, loads its bytecode, and executes it, which is what finally prints the line above to the terminal.

That's the Development stage from earlier, compressed onto a single machine: javac compiles, java is the JVM executing the result. Ship that same Hello.class file to any other machine with a JVM installed, and java Hello runs it there too, unmodified.

A note about modern Java

Recent Java versions also allow small examples to be written with less ceremony:

void main() {
    System.out.println("Hello, world.");
}

This simplified form is handy for quick experiments, but this tutorial uses the traditional class-based style throughout — it's the shape almost every real Java codebase actually takes.

§0.7

Key Features

Statically Typed

Every variable has a type declared at compile time. The compiler rejects programs where types don't line up. Many errors are caught before the program ever runs.

Object-Oriented

Programs are organized around objects — bundles of data (fields) and behavior (methods). This tutorial builds that model from the ground up.

Garbage Collected

You allocate objects with new; the JVM frees them automatically when they're no longer reachable. There is no free() or delete.

Strongly Typed

There are no silent conversions between unrelated types. An int is not a boolean. A String is not a Rectangle. The type system is not optional.

§0.8

Key Takeaways

  • Java is a general-purpose language used across many industries, usually through the Java SE platform plus a framework built on top of it.
  • A Java program is compiled once with javac into platform-neutral bytecode, then executed by a JVM — the same bytecode runs unmodified on any platform with a compatible JVM installed.
  • Releases are either LTS (long-term support, for production) or feature releases (faster-moving, for early access); this tutorial targets Java 25.
  • This tutorial teaches the traditional class-based style, since that's the shape most real Java code takes, alongside the core object-oriented concepts the rest of the tutorial builds from the ground up.
§0.9

Appendix: Write Once, Run Anywhere — the mechanics in full

§0.1 and §0.3 already named the portability guarantee. This appendix is the mechanical detail behind it — optional on a first read, worth returning to once the rest of the tutorial has given you more Java to anchor it to.

The Java Workflow, End to End

That word "compiled" is worth pausing on first. A CPU only understands machine instructions — numbers meaningful to that specific processor — and nobody writes programs that way. You write source code: text in a language designed for humans to read and reason about. Compilation is the development-time step that bridges the two — a compiler reads your source, rejects anything that doesn't parse or type-check, and translates whatever survives into an executable form. That step is the first of three distinct stages a Java application moves through, and they don't have to happen on the same machine:

  1. Development. You write Java source code and compile it with a Java compiler (javac) on your development machine. This produces .class files containing bytecode.
  2. Distribution. You package the .class files, along with any resources they need, into a distributable format — typically a .jar file.
  3. Execution. The user on the target platform needs a compatible JVM installed. They run your application, and their JVM executes the bytecode — no recompilation, regardless of what platform built it.

Compilation happens once, on the developer's machine; execution happens anywhere a JVM is installed, unmodified. That separation — compile once, distribute the bytecode, run on any JVM — is what "write once, run anywhere" actually refers to. The next section looks at the mechanics that make the separation possible.

"A compatible JVM installed" means installing a JDK — the Java Development Kit. It bundles javac, the java launcher (which starts the JVM and runs your bytecode), the standard library, and the rest of the developer tooling in one package. You'll sometimes see the term JRE (Java Runtime Environment) in older material — it referred to a separate, smaller download containing just the JVM and libraries needed to run a compiled program, with none of the development tools. Oracle stopped distributing a standalone JRE starting with Java 11: today there's just the JDK, and both the developer compiling the code and the end user running it install the same thing.

Compiling to Native Code vs. Compiling to Bytecode

To see why that guarantee holds, it helps to compare Java's pipeline to a language like C, which compiles straight to native machine code.

A C compiler (gcc, clang, MSVC) translates hello.c directly into instructions for one specific CPU and operating system. Target five different machines, and you compile the same source five separate times — once per machine, on that machine — and get five incompatible binaries out:

Notice that "Windows" alone isn't even one target: the two Windows rows are both Windows, but a different CPU architecture still means a different binary, the same way it does between the two Macs.

Java inserts a layer of indirection: javac never targets a CPU at all. It targets the JVM's instruction set, which is the same everywhere. Portability moves from the compiler to the runtime — a JVM is built once for each platform, and it knows how to translate that one bytecode format into whatever native instructions the local CPU needs. One compile is all it takes, and that one output runs everywhere, because translating bytecode to native instructions is the JVM's job, not javac's:

Line up the two pictures: the C compiler and the Java compiler occupy the same slot, but the C compiler's output is tied to one machine — recompiling is the only way to reach another. Java's output stays identical no matter where it was compiled; it's the JVM that varies from platform to platform, not the bytecode.

The trade-off

C's approach produces a binary that talks to the CPU directly — no intermediate translation at runtime, and no extra program has to be installed to run it. Java's approach gives up that directness in exchange for portability: the same compiled artifact runs anywhere a JVM exists, at the cost of needing that JVM present and doing a bit of translation work as the program runs.
