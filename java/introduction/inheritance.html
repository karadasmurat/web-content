Chapter 11

Inheritance

Some classes are more specific versions of other classes. They have all of that other class's fields and methods, plus more.

§11.1

Taxonomy — general at the top, specific below

A poodle is a dog. A dog is a mammal. A mammal is an animal — but a mammal is just as much a cat, or a human, and an animal is just as much a bird or a reptile. Every level branches into several more specific categories below it, and every one of those still has everything the broader category requires, plus something of its own:

  • Animal
    • Mammal
      • Dog
        • Poodle
      • Cat
      • Human
    • Bird
    • Reptile

Nothing is lost moving down a branch — only added. Every animal needs energy from somewhere; every mammal does too, plus it has hair and feeds its young milk; every dog does all of that, plus it barks and has a breed. This kind of hierarchy is a taxonomy: categories arranged by what they have in common. The most general traits — the ones true of everything — sit at the top, stated exactly once. Move down any branch, and it narrows while still satisfying everything above it. Move up, and you're asking "what do all of these actually share?"

Classes can be organized the same way. A car is a vehicle. A truck is a vehicle. Both have everything a vehicle has — plus something of their own.

§11.2

The Problem — repeated structure

Suppose the program needs both cars and trucks. Both have a make, a model, and a year — and both need a way to describe themselves:

class Car {
    String make;
    String model;
    int    year;

    String describe() {
        return year + " " + make + " " + model;
    }
}

class Truck {
    String make;
    String model;
    int    year;

    String describe() {
        return year + " " + make + " " + model;
    }
}

Every field and every line of describe() is duplicated, word for word. Add a third kind of vehicle, and it happens again. Fix a bug in describe(), and it needs fixing in every copy. Inheritance is Java's way of writing that shared structure exactly once.

§11.3

extends — inheriting fields and methods

Pull the shared part into its own class, and have the specific ones declare that they're a kind of it:

class Vehicle {
    String make;
    String model;
    int    year;

    String describe() {
        return year + " " + make + " " + model;
    }
}

class Car extends Vehicle {
    int doors;
}

Car extends Vehicle means exactly what it says: a Car is a Vehicle, plus whatever Car adds on top. Vehicle is the superclass; Car is the subclass. Nothing in Car's own body re-declares make, model, year, or describe() — but every Car object has all four anyway, inherited from Vehicle:

Car c = new Car();
c.make  = "Toyota";
c.model = "Corolla";
c.year  = 2022;
c.doors = 4;

System.out.println(c.describe()); // 2022 Toyota Corolla
Stack
c = 0x9000
Heap
Car @ 0x9000
make"Toyota"
model"Corolla"
year2022
own to Car
doors4

There is exactly one Car object on the heap. It doesn't contain a separate Vehicle object inside it — make, model, and year are slots in the same object as doors, laid out there because Car extends Vehicle. Inheritance adds fields to the layout; it doesn't nest one object inside another the way a reference field does (§4.3).

You've been extending a class all along ← §4.1

Every class in this tutorial — Rectangle, Point, Border, Vehicle — has had an unwritten extends Object at the top the whole time. Object is the one class every other class inherits from, directly or indirectly, whether it says so or not. That's where the default, address-comparing equals(Object) from §8 actually comes from: it's a method defined on Object, and every class inherits it unless it overrides it.

§11.4

Constructors — the superclass runs first, via super(...)

A Car's constructor needs to initialize the Vehicle part of itself too — and it does that by calling the Vehicle constructor explicitly, as the very first line:

class Vehicle {
    String make;
    String model;
    int    year;

    Vehicle(String make, String model, int year) {
        this.make  = make;
        this.model = model;
        this.year  = year;
    }

    String describe() {
        return year + " " + make + " " + model;
    }
}

class Car extends Vehicle {
    int doors;

    Car(String make, String model, int year, int doors) {
        super(make, model, year);  // runs Vehicle's constructor first
        this.doors = doors;
    }
}
Car c = new Car("Toyota", "Corolla", 2022, 4);
System.out.println(c.describe()); // 2022 Toyota Corolla

super(make, model, year) is a call to Vehicle's constructor, using Car's own parameters as its arguments. It must be the first statement in Car's constructor — the inherited part of the object is built before the part Car adds. This is the exact same allocate-then-initialize sequence from §4.4, just with the initializing done in two stages: the superclass's constructor first, then the subclass's own.

Left out, not left off

If a subclass constructor doesn't write super(...) itself, Java inserts a call to the superclass's no-argument constructor automatically, as the first line. That only works if the superclass has a no-argument constructor. Vehicle above doesn't — its only constructor takes three arguments — so Car is required to call super(...) explicitly, with matching arguments.

§11.5

Polymorphism — one supertype, many behaviors underneath

Polymorphism — "many forms" — is the name for what the last two sections set up: a single supertype reference can refer to objects of many different subtypes, and the behavior that actually runs depends on which one is really there. Three mechanics make this work together, in the order they matter: a subtype reference can substitute for a supertype one, code can ask, at runtime, which subtype it actually received, and a subclass can replace inherited behavior — the practical payoff of substituting in the first place.

Substitutability — a subclass reference fits where the superclass is expected

Because a Car is a Vehicle, a variable declared as Vehicle can hold the address of a Car object — no cast, no conversion, nothing special:

Vehicle v = new Car("Toyota", "Corolla", 2022, 4);
System.out.println(v.describe()); // 2022 Toyota Corolla

v holds the exact same kind of thing every reference variable holds — an address (§4.2) — and a Car's address is a perfectly good value for a Vehicle-typed variable to hold, because everything a Vehicle reference is allowed to do (read make, call describe()) is something the Car object can actually do. What v cannot do is reach doors — as far as v's declared type is concerned, it's a Vehicle, and doors isn't part of Vehicle. The object is still fully a Car underneath; the variable's type just limits what you can reach through it.

Nothing stops the same address from living in two variables of two different declared types at once — it's just an ordinary reference assignment, the same rule from §5:

Car myCar = new Car("Kia", "Sportage", 2007, 4);
Vehicle vc = myCar;   // reference assignment — vc's slot gets a copy of myCar's address
Stack
myCar = 0x9500
vc = 0x9500
Heap
Car @ 0x9500
make"Kia"
model"Sportage"
year2007
doors4

One Car object on the heap. Two variables holding its address — myCar, declared Car, and vc, declared Vehicle. Through myCar, every field is reachable. Through vc, only what Vehicle declares is — vc.doors won't compile, myCar.doors will. Same object, same address, two different views of what's allowed, depending entirely on which variable you go through.

Two types, one object

Every object now has two types in play. Its declared type is whatever type the variable holding it was declared as — a compile-time fact, fixed in the source code, and the only thing the compiler consults when deciding what you're allowed to write. Its runtime type is the type it was actually constructed with — fixed the moment new ran, and never changing afterward. vc's declared type is Vehicle; its runtime type is Car. Both are true at once, and the next part is about telling them apart.

instanceof — checking the runtime type

Give Car a method Vehicle doesn't have:

class Car extends Vehicle {
    int doors;

    Car(String make, String model, int year, int doors) {
        super(make, model, year);
        this.doors = doors;
    }

    void openTrunk() {
        System.out.println("Trunk open.");
    }
}

Substitutability showed that a Car's address fits in a Vehicle-typed variable. But that variable's declared type still limits what you can write:

Vehicle v = new Car("Toyota", "Corolla", 2022, 4);
v.openTrunk(); // compile error — Vehicle has no openTrunk()

This doesn't compile, even though the object v points to is, underneath, fully capable of it. The compiler never looks at what's actually on the heap — it only ever checks the declared type, and as far as it's concerned, v is a Vehicle, full stop.

The instanceof operator asks the question the compiler won't: is the object actually a Car? Its syntax puts the reference on the left, the type on the right:

System.out.println(v instanceof Vehicle); // true — every Car is a Vehicle
System.out.println(v instanceof Car);     // true — its runtime type really is Car
System.out.println(v instanceof Truck);   // false — it isn't one
Why it has to dereference

== on references (§9) never looks past the address — it only compares what's sitting in the two variables' own slots. instanceof can't get its answer that way: the variable only knows its declared type, and that's not what's being asked. So instanceof follows the address to the object on the heap and checks the type it was actually constructed with — its runtime type — against the type on the right.

Knowing v's object is a Car still doesn't let you call openTrunk() through v — its declared type hasn't changed. Reaching Car-specific members requires a downcast: telling the compiler, explicitly, to treat the reference as the more specific type.

Car c = (Car) v;   // explicit downcast
c.openTrunk();     // Trunk open.

An unchecked downcast is a promise to the compiler — and the compiler can't verify it. If the object underneath isn't actually a Car, the promise is broken at run time:

Vehicle v2 = new Truck("Ford", "F-150", 2023, 6.5);
Car bad = (Car) v2; // compiles fine — throws ClassCastException at runtime

Checking with instanceof first makes the cast safe:

if (v instanceof Car) {
    Car c = (Car) v;
    c.openTrunk();
}

Java lets you fold the check and the cast into one step — the variable c is only declared, and only in scope, inside the branch where the check already succeeded:

if (v instanceof Car c) {
    c.openTrunk(); // c is already a Car here — no separate cast needed
}

This is the most common shape the operator takes in real code: a method can declare a parameter of some broad type, like Vehicle, and be called with a subtype's object instead — Car, Truck, or any other Vehicle — by the exact substitutability described above. instanceof is how the method checks, once inside, whether the object it actually received is more specific than its parameter's type lets on.

Overriding — replacing an inherited method

Sometimes the inherited behavior isn't quite right for the subclass. A Truck's description should probably mention its bed length. A subclass can override an inherited method — provide its own body with the same name and parameters — and that new body replaces the inherited one, for objects of that subclass:

class Truck extends Vehicle {
    double bedLength;

    Truck(String make, String model, int year, double bedLength) {
        super(make, model, year);
        this.bedLength = bedLength;
    }

    @Override
    String describe() {
        return super.describe() + ", " + bedLength + "ft bed";
    }
}
Vehicle v3 = new Truck("Ford", "F-150", 2023, 6.5);
System.out.println(v3.describe()); // 2023 Ford F-150, 6.5ft bed

@Override tells the compiler "this is meant to replace an inherited method" — it's not required, but it catches typos (a misspelled method name would otherwise silently define a brand new method instead of overriding anything). super.describe() inside the override calls Vehicle's original version explicitly, so Truck's version can build on it rather than duplicate it.

Notice v3 is declared Vehicle — exactly the substitutability from earlier in this section — but v3.describe() runs Truck's version, not Vehicle's. Java always runs the override that belongs to the object's actual type on the heap, regardless of the variable's declared type. This is dynamic dispatch, and it's the payoff of everything else in this section: substitutability lets any subtype's object arrive through a supertype reference, and overriding is what lets each subtype answer the same call, like describe(), in its own way — automatically, with no instanceof check needed, because Java picks the right version by itself.

Scope of this chapter

This is single-level, single-parent inheritance — the essential mechanics. Multiple levels of subclassing, abstract classes and methods, interfaces, and the protected access level all build on exactly this foundation, and are left for later. So is writing an actual equals(Object) override — §8 deferred it, and this chapter now has all the pieces (inheritance, Object, overriding) to explain how one works, but the method itself has a few extra rules of its own that deserve their own space. instanceof's pattern-matching form, combined with switch, is a much larger feature in modern Java than shown here — this chapter only needed the single-branch case.
