Chapter 9

Inheritance

Some classes are specialized versions of other classes.

§9.1

Taxonomy — general at the top, specific below

Humans naturally organize knowledge by grouping similar things together. We classify animals, plants, vehicles, musical instruments, and countless other objects into categories. Each category shares common characteristics, while more specific categories introduce additional traits.

  • Animal
    • Mammal
      • Dog
      • Cat
      • ...
    • Bird
    • ...
The essence of the IS-A relationship:
  • Every dog is a mammal.
  • Every dog is an animal.
  • Every cat is a mammal.
  • Every cat is an animal.
  • Every bird is an animal.

Transitivity of Inheritance

"Every dog is a mammal" needed nothing but the tree above to justify it — Dog sits directly under Mammal. "Every dog is an animal" is a different kind of claim: Dog isn't directly under Animal at all, it's two levels down. It's true anyway, because inheritance is transitive: if Dog extends Mammal, and Mammal extends Animal, then Dog is a subclass of Animal too — automatically, with nothing extra written anywhere. The is-a relationship doesn't stop propagating after a single step; it carries all the way up the chain, however many classes long that chain is. §9.5 puts this to work directly, once there's a reference variable to substitute into.

This kind of hierarchy is a taxonomy: categories arranged by what they have in common. Every level branches into several more specific categories below it. Specific categories still have everything the broader parent category requires, plus something of its own:

Nothing is lost moving down a branch — only added.

  • Every animal needs energy from somewhere;
  • every mammal does too, plus it has hair and feeds its young milk;
  • every dog does all of that, plus it barks and has a breed.
The most general traits sit at the top, stated exactly once. Move down any branch, and it narrows while still satisfying everything above it. Move up, and you're asking "what do all of these actually share?"

Classes can be organized the same way. An invoice is a document. A contract is a document. Both have everything a document has — plus something of their own.

  • Document
    • Invoice
    • Contract
    • Report
    • Resume
§9.2

The Problem — repeated structure

Suppose the program needs both invoices and contracts. Both have a title, an author, a creation date, and a version — and both need a way to save themselves:

class Invoice {
    String title;
    String author;
    String createdDate;
    int    version;

    String save() {
        return "Saved: " + title + " by " + author + " (v" + version + ")";
    }
}

class Contract {
    String title;
    String author;
    String createdDate;
    int    version;

    String save() {
        return "Saved: " + title + " by " + author + " (v" + version + ")";
    }
}

Every field and every line of save() is duplicated, word for word. Add a third kind of document, and it happens again. Fix a bug in save(), and it needs fixing in every copy. Inheritance is Java's way of writing that shared structure exactly once.

§9.3

extends — inheriting fields and methods

What do they have in common?

Pull the shared part into its own class, and have the specific ones declare that they're a kind of it:

class Document {
    String title;
    String author;
    String createdDate;
    int    version;

    String save() {
        return "Saved: " + title + " by " + author + " (v" + version + ")";
    }
}

class Invoice extends Document {
    String customer;
    double amount;
}

Invoice extends Document means exactly what it says: an Invoice is a Document, plus whatever Invoice adds on top. Document is the superclass; Invoice is the subclass. Nothing in Invoice's own body re-declares title, author, createdDate, version, or save() — but every Invoice object has all four anyway, inherited from Document. §9.4 shows this concretely, once Invoice has a proper constructor to create one with.

You've been extending a class all along ← §2.2

Every class in this tutorial — Rectangle, Point, Border, Document — has had an unwritten extends Object at the top the whole time. Object is the one class every other class inherits from, directly or indirectly, whether it says so or not. That's where the default, address-comparing equals(Object) from §8 actually comes from: it's a method defined on Object, and every class inherits it unless it overrides it.

§9.4

Constructors — the superclass runs first, via super(...)

An Invoice's constructor needs to initialize the Document part of itself too — and it does that by calling the Document constructor explicitly, as the very first line:

class Document {
    String title;
    String author;
    String createdDate;
    int    version;

    Document(String title, String author, String createdDate, int version) {
        this.title       = title;
        this.author      = author;
        this.createdDate = createdDate;
        this.version     = version;
    }

    String save() {
        return "Saved: " + title + " by " + author + " (v" + version + ")";
    }
}

class Invoice extends Document {
    String customer;
    double amount;

    Invoice(String title, String author, String createdDate, int version, String customer, double amount) {
        super(title, author, createdDate, version);  // runs Document's constructor first
        this.customer = customer;
        this.amount   = amount;
    }
}
Invoice i = new Invoice("Q3 Services", "Priya Shah", "2024-09-30", 1, "Acme Corp", 4200.00);
System.out.println(i.save()); // Saved: Q3 Services by Priya Shah (v1)
Stack
i = 0x9000
Heap
Invoice @ 0x9000
title"Q3 Services"
author"Priya Shah"
createdDate"2024-09-30"
version1
own to Invoice
customer"Acme Corp"
amount4200.00

There is exactly one Invoice object on the heap. It doesn't contain a separate Document object inside it — title, author, createdDate, and version are slots in the same object as customer and amount, laid out there because Invoice extends Document. Inheritance adds fields to the layout; it doesn't nest one object inside another the way a reference field does (§2.4). super(...) is what filled in the first four slots — the constructor call below explains how.

super(title, author, createdDate, version) is a call to Document's constructor, using Invoice's own parameters as its arguments. It must be the first statement in Invoice's constructor — the inherited part of the object is built before the part Invoice adds. This is the exact same allocate-then-initialize sequence from §2.5, just with the initializing done in two stages: the superclass's constructor first, then the subclass's own.

Left out, not left off

If a subclass constructor doesn't write super(...) itself, Java inserts a call to the superclass's no-argument constructor automatically, as the first line. That only works if the superclass has a no-argument constructor. Document above doesn't — its only constructor takes four arguments — so Invoice is required to call super(...) explicitly, with matching arguments.

The "Onion" Model

The diagram above only has two layers, Document and Invoice, but the same layering continues for as long as the hierarchy does — every class extends something (§9.3), all the way up to Object at the root. Picture the whole hierarchy as an onion: Object's fields form the innermost layer, each class up the chain adds a layer around it, and the class you actually write new for is the outermost layer. new Invoice(...) doesn't build four nested objects, one per layer — it calculates the size of the whole onion at once and allocates a single contiguous block on the heap big enough to hold every layer, in one motion (§2.2).

Layout is one thing; filling each layer in is another, and that's the job the constructor chain does. Building the outer layer of an onion before the inner ones exist wouldn't make sense — so super(...) runs first, completing Document's layer, before a single line of Invoice's own constructor body executes on the layer it owns. Two constructors run, back to back, but there is only ever one object underneath them, one address, one identity — never a separate "Document object" and "Invoice object" glued together.

super(...) doesn't create a second object

It's tempting to picture super(...) as instantiating a hidden "Document part" on its own — it doesn't. new already allocated the entire onion, Document's layer included, as one reserved block, before any constructor body has run at all. new is the architect and the builder; a constructor, superclass's or subclass's, is only the interior decorator for the layer it's responsible for — it initializes memory that already exists, it doesn't create more of it. Skip super(...) where no implicit no-argument call is available, and the inner layer isn't missing — it's just sitting there unfurnished: space for title, author, createdDate, and version exists in the object the moment new runs, with no value assigned to any of it yet.

§9.5

Polymorphism — one supertype, many behaviors underneath

Polymorphism — "many forms" — is the name for what the last two sections set up: a single supertype reference can refer to objects of many different subtypes, and the behavior that actually runs depends on which one is really there. Two mechanics make the "refer to" half of that work: a subtype reference can substitute for a supertype one, and code can ask, at runtime, which subtype it actually received. What actually makes that worth doing — a subclass replacing inherited behavior — is substantial enough to earn its own section right after this one.

Substitutability — a subclass reference fits where the superclass is expected

Because an Invoice is a Document, a variable declared as Document can hold the address of an Invoice object — no cast, no conversion, nothing special:

Document doc = new Invoice("Q3 Services", "P Shah", "2024-09-30", 1, "Acme Corp", 4200.00);
System.out.println(doc.save()); // Saved: Q3 Services by Priya Shah (v1)

d holds the exact same kind of thing every reference variable holds — an address (§2.3) — and an Invoice's address is a perfectly good value for a Document-typed variable to hold, because everything a Document reference is allowed to do (read title, call save()) is something the Invoice object can actually do. What d cannot do is reach customer — as far as d's declared type is concerned, it's a Document, and customer isn't part of Document. The object is still fully an Invoice underneath; the variable's type just limits what you can reach through it.

Nothing stops the same address from living in two variables of two different declared types at once — it's just an ordinary reference assignment, the same rule from §3:

Invoice myInvoice = new Invoice("Annual Support", "J Reyes", "2023-01-15", 2, "Globex", 9800.00);
Document doc = myInvoice;   // reference assignment — doc's slot gets a copy of myInvoice's address
Stack
myInvoice = 0x9500
doc = 0x9500
Heap
Invoice @ 0x9500
title"Annual Support"
author"Jon Reyes"
createdDate"2023-01-15"
version2
customer"Globex"
amount9800.00

One Invoice object on the heap. Two variables holding its address — myInvoice, declared Invoice, and doc, declared Document. Through myInvoice, every field is reachable. Through doc, only what Document declares is — doc.customer won't compile, myInvoice.customer will. Same object, same address, two different views of what's allowed, depending entirely on which variable you go through.

Two types, one object

Every object now has two types in play. Its declared type is whatever type the variable holding it was declared as — a compile-time fact, fixed in the source code, and the only thing the compiler consults when deciding what you're allowed to write. Its runtime type is the type it was actually constructed with — fixed the moment new ran, and never changing afterward. doc's declared type is Document; its runtime type is Invoice. Both are true at once, and the next part is about telling them apart.

Substitutability Is Transitive Too

§9.1 introduced transitivity through a taxonomy three levels deep — Dog under Mammal under Animal — and promised it would matter once there was a reference variable to substitute into. Now there is. Translated into classes, that taxonomy is a two-step extends chain:

class Animal { }
class Mammal extends Animal { }
class Dog    extends Mammal { }

Dog's own declaration mentions only Mammal — extends Animal appears nowhere in it. Substitutability doesn't stop at one step either, though: a Dog reference fits anywhere a Mammal is expected, exactly as shown above, but it also fits anywhere an Animal is expected, despite Dog never mentioning Animal directly: 

Animal a = new Dog();  // no cast, no conversion — Dog is an Animal, two steps removed

Dog inherits everything Mammal has, and Mammal in turn inherits everything Animal has, so a Dog object satisfies everything an Animal-typed variable is allowed to expect, the same way doc above was satisfied by an Invoice. Substitutability follows the extends chain exactly as far as it goes — one level or several, it works the same way at every step.

instanceof — checking the runtime type

Give Invoice a method Document doesn't have:

class Invoice extends Document {
    String customer;
    double amount;

    Invoice(String title, String author, String createdDate, int version, String customer, double amount) {
        super(title, author, createdDate, version);
        this.customer = customer;
        this.amount   = amount;
    }

    double calculateTotal() {
        return amount;
    }
}

Contract is another Document subclass, shaped the same way — its own fields, plus everything Document already declares:

class Contract extends Document {
    String parties;
    String expirationDate;

    Contract(String title, String author, String createdDate, int version, String parties, String expirationDate) {
        super(title, author, createdDate, version);
        this.parties        = parties;
        this.expirationDate = expirationDate;
    }
}

Substitutability showed that an Invoice's address fits in a Document-typed variable. But that variable's declared type still limits what you can write:

Document d = new Invoice("Q3 Services", "Priya Shah", "2024-09-30", 1, "Acme Corp", 4200.00);
d.calculateTotal(); // compile error — Document has no calculateTotal()

This doesn't compile, even though the object d points to is, underneath, fully capable of it. The compiler never looks at what's actually on the heap — it only ever checks the declared type, and as far as it's concerned, d is a Document, full stop.

The instanceof operator asks the question the compiler won't: is the object actually an Invoice? Its syntax puts the reference on the left, the type on the right:

System.out.println(d instanceof Document); // true — every Invoice is a Document
System.out.println(d instanceof Invoice);  // true — its runtime type really is Invoice
System.out.println(d instanceof Contract); // false — it isn't one
Why it has to dereference

== on references (§5) never looks past the address — it only compares what's sitting in the two variables' own slots. instanceof can't get its answer that way: the variable only knows its declared type, and that's not what's being asked. So instanceof follows the address to the object on the heap and checks the type it was actually constructed with — its runtime type — against the type on the right.

Knowing d's object is an Invoice still doesn't let you call calculateTotal() through d — its declared type hasn't changed. Reaching Invoice-specific members requires a downcast: telling the compiler, explicitly, to treat the reference as the more specific type. 

Invoice inv = (Invoice) d;   // explicit downcast
inv.calculateTotal();        // 4200.00

An unchecked downcast is a promise to the compiler — and the compiler can't verify it. If the object underneath isn't actually an Invoice, the promise is broken at run time:

Document d2 = new Contract("Vendor Agreement", "Priya Shah", "2024-02-01", 1, "Acme Corp & Globex", "2025-02-01");
Invoice bad = (Invoice) d2; // compiles fine — throws ClassCastException at runtime

Checking with instanceof first makes the cast safe:

if (d instanceof Invoice) {
    Invoice inv = (Invoice) d;
    inv.calculateTotal();
}

Java lets you fold the check and the cast into one step — the variable inv is only declared, and only in scope, inside the branch where the check already succeeded:

if (d instanceof Invoice inv) {
    inv.calculateTotal(); // inv is already an Invoice here — no separate cast needed
}

This is the most common shape the operator takes in real code: a method can declare a parameter of some broad type, like Document, and be called with a subtype's object instead — Invoice, Contract, or any other Document — by the exact substitutability described above. instanceof is how the method checks, once inside, whether the object it actually received is more specific than its parameter's type lets on.

Substitutability and instanceof are both about the reference — what it can point at, and how to ask what's really there. Neither one, on its own, explains why any of this is worth having. That payoff is the next section's entire subject.

§9.6

Overriding — replacing an inherited method

§9.5 covered two of polymorphism's three mechanics — a subtype reference substituting for a supertype one, and instanceof asking what's really there. Neither one changes what actually runs when a method is called. Overriding is the third mechanic, and the one that does: a subclass can replace an inherited method's body outright, and that's what makes substituting in a subtype's object actually pay off.

Step away from Document for a moment. Suppose an application can send different kinds of notifications:

  • Notification
    • EmailNotification
    • SmsNotification
    • PushNotification

Ask the same question §9.1's taxonomy asked: if an object is an EmailNotification, is it also a Notification? The answer is obviously yes — the same is-a relationship as Dog and Animal, just with a different pair of names.

Now declare a variable using the broad type:

Notification notification;

This variable doesn't say which notification it refers to — it can't, yet; it doesn't even hold an address yet (§1.2). All the declaration says is: "I can refer to any object whose type is Notification" — an EmailNotification qualifies, an SmsNotification qualifies, anything else that is-a Notification qualifies too, by the exact transitivity and substitutability already covered in this chapter. Assign it an object, and that's exactly what shows up: 

notification = new EmailNotification("alice@example.com", "Your order has shipped.");

Later, in the same variable:

notification = new SmsNotification("+1-555-0100", "Your OTP is 4821.");

Nothing changed about notification itself — it's still declared Notification, same as always, and reassignment works exactly as it always has for a reference variable (§3): the slot's old address is replaced with a new one. Only the object on the other end of that address changed, from an EmailNotification to an SmsNotification. The variable's type is fixed at declaration and never moves; what varies, call to call, is which object it happens to be pointing at.

A notification system sends the same kind of thing — a recipient, a message — through very different channels. Every notification shares those two fields; what differs, completely, is how sending one actually works:

class Notification {
    String recipient;
    String message;

    Notification(String recipient, String message) {
        this.recipient = recipient;
        this.message   = message;
    }

    void send() {
        System.out.println("Notification for " + recipient + ": " + message);
    }
}

Notification's own send() is deliberately generic — it doesn't know about email, SMS, or anything else, because at this level in the hierarchy, it can't. An EmailNotification needs to talk to an SMTP server; an SmsNotification needs an SMS gateway. Same method name, same job in spirit, completely different mechanics underneath — exactly the case for overriding: a subclass provides its own body for an inherited method, with the same name and parameters, and that new body replaces the inherited one for objects of that subclass: 

class EmailNotification extends Notification {
    EmailNotification(String recipient, String message) {
        super(recipient, message);
    }

    @Override
    void send() {
        super.send();
        System.out.println("  -> delivered through the SMTP server");
    }
}

class SmsNotification extends Notification {
    SmsNotification(String recipient, String message) {
        super(recipient, message);
    }

    @Override
    void send() {
        super.send();
        System.out.println("  -> delivered through the SMS gateway");
    }
}
Notification n1 = new EmailNotification("alice@example.com", "Your order has shipped.");
Notification n2 = new SmsNotification("+1-555-0100", "Your OTP is 4821.");

n1.send();
// Notification for alice@example.com: Your order has shipped.
//   -> delivered through the SMTP server

n2.send();
// Notification for +1-555-0100: Your OTP is 4821.
//   -> delivered through the SMS gateway

Declared Type vs Actual Type

Look closely at what n1 and n2 actually are. Each variable has a declared type — Notification, written right there in the declaration — and that's the only type the compiler will ever check against; as far as the source code is concerned, n1 is a Notification, nothing more specific. But what n1 holds is an address (§2.3), and the object that address points to on the heap isn't a plain Notification at all — it's a complete EmailNotification, built by new EmailNotification(...), with an EmailNotification-shaped body sitting at the other end of that address. A broader, declared reference type doesn't stop the object it points to from being a more specialized subtype underneath — that gap between what a variable is declared as and what object it actually addresses is exactly what makes the next question worth asking.

Dynamic Dispatch

n1 is declared Notification. But the object n1 actually addresses is an EmailNotification.

Should n1.send() run

  • Just prints the generic line, through Notification's own send()? or
  • Send an email, through EmailNotification's override of send()?

Sitting with that tension is exactly what makes the motivation for dynamic dispatch obvious: a call written in terms of the declared type still has to somehow reach the more specific behavior that lives on the actual object.

Java's rule resolves it directly: when calling an overridden instance method, Java chooses the implementation based on the object's actual type , not the variable's declared type. That's the entire idea. n1.send() runs EmailNotification's version, n2.send() runs SmsNotification's, and neither one runs Notification's own — regardless of the fact that both variables are declared Notification, and regardless of what the compiler saw when it checked the call.

@Override tells the compiler "this is meant to replace an inherited method" — it's not required, but it catches typos (a misspelled method name would otherwise silently define a brand new method instead of overriding anything). super.send() inside each override calls Notification's original version explicitly, so each channel's version builds on the shared logging line rather than duplicating it — the same relationship super(...) has with a constructor, just for an ordinary method instead.

This is the payoff of everything else in this section: substitutability lets any subtype's object arrive through a supertype reference, and overriding is what lets each subtype answer the same call, like send(), in its own way — automatically, with no instanceof check needed, because Java picks the right version by itself. A PushNotification would slot in the same way, talking to a push service instead — the pattern doesn't change no matter how many channels get added.

Scope of this chapter

This is single-level, single-parent inheritance — the essential mechanics. Multiple levels of subclassing, abstract classes and methods, interfaces, and the protected access level all build on exactly this foundation, and are left for later. So is writing an actual equals(Object) override — §8 deferred it, and §9.7 covers the simpler toString(), but equals and its partner hashCode() have a few extra rules of their own that deserve their own space. instanceof's pattern-matching form, combined with switch, is a much larger feature in modern Java than shown here — this chapter only needed the single-branch case.

§9.7

Overriding Object's Methods — toString()

If every object in Java is different, is there anything they all have in common? The answer is yes. Every class in Java ultimately inherits from a single class called Object. That means these two declarations are equivalent: 

class Book { }
          
class Book extends Object { }
Because of this, every object in Java is also an Object. Object defines behavior that makes sense for every object, regardless of what it represents. For example, every object can:
  • produce a textual representation (toString())
  • be compared with another object (equals())
  • produce a hash code (hashCode())
A Book and a BankAccount are very different, but asking "Can you describe yourself as text?" makes sense for both. That's why they're defined once, in Object, and automatically inherited by every class. 

Notification n = new EmailNotification("alice@example.com", "Your order has shipped.");
System.out.println(n); // EmailNotification@1b6d3586 — not exactly useful

System.out.println calls toString() on whatever it's handed and prints the result. Object's own toString() reports the class name and the object's hash code.

Overriding it works exactly like overriding send() did:

class Notification {
    String recipient;
    String message;

    Notification(String recipient, String message) {
        this.recipient = recipient;
        this.message   = message;
    }

    void send() {
        System.out.println("Notification for " + recipient + ": " + message);
    }

    @Override
    public String toString() {
        return "Notification[recipient=" + recipient + ", message=" + message + "]";
    }
}
Notification n = new EmailNotification("alice@example.com", "Your order has shipped.");
System.out.println(n); // Notification[recipient=alice@example.com, message=Your order has shipped.]

EmailNotification never wrote its own toString() at all — it inherits Notification's override the same ordinary way it inherits send()'s original body whenever a subclass doesn't replace it (§9.6). Overriding isn't limited to methods a class itself declared; Object's methods are exactly as overridable as any other inherited method, because they're inherited the exact same way.

public is not optional here

Object.toString() is declared public, and an override can never narrow the access an inherited method already has — only keep it the same or widen it. Writing String toString() with no modifier, package-private by default (§10.6), wouldn't override anything at all; it would fail to compile, since a package-private method can't stand in for a public one. This is a general rule about overriding, not something specific to toString() — it just happens to be the first override in this chapter where forgetting it would actually break something.

Almost every debugging session, log line, and IDE inspector leans on toString() being overridden well — it's one of the most quietly useful overrides a class can have, precisely because so much of the rest of the platform calls it automatically, without ever being asked to.
