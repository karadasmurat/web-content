Chapter 15

Interfaces

Not everything two classes share is an "is-a" relationship — and Java only lets a class extend one thing anyway. This chapter covers the other kind of shared shape, and modern Java's biggest revision to it: an interface can carry real behavior, not just a promise.

§15.1

The Problem — a shared capability, no shared ancestor

A plane flies. So does a bird. So does an insect, for that matter. All three share the exact same capability, described by the exact same word — fly — but how each one manages it has nothing in common at all: fixed wings and jet engines, feathers and flapping, an entirely different mechanism built on an entirely different body plan. And a Plane, a Bird, and an Insect share no real ancestry either — a Plane is not a kind of Bird, and inventing a shared parent class just so all three can have a fly() method would invent a relationship that isn't really there. Same capability — the what — expressed identically; completely different implementation — the how — underneath; no "is-a" relationship forcing them together at all.

Software runs into this constantly, usually over something more mundane than flight. A Subscription can be renewed. So can a Passport. Neither fact has anything to do with the other — a Subscription isn't a kind of Passport, and forcing one to extend the other, or inventing a shared superclass just to hold a renew() method, would be the exact same mistake as forcing Plane and Bird under one parent just for fly(): inventing a lineage a class doesn't actually have, purely to satisfy the compiler.

class Subscription {
    void renew() {
        System.out.println("Subscription renewed for another year.");
    }
}

class Passport {
    void renew() {
        System.out.println("Passport renewed for another decade.");
    }
}

Both classes happen to have a method with the same name and meaning — renewable is a real, shared idea — but nothing in the code says so. Worse, Java only allows extends on one class at a time (§11.3): even if a shared superclass made sense here, a class already extending something else — a Passport that's also a kind of GovernmentDocument, say — couldn't extend a second thing just to pick up renew() too. Inheritance alone can't express "this class can do X," separately from "this class is a kind of Y."

§15.2

Interface vs. Abstract Class — which fits

Both let a class promise a method without yet supplying it. The difference is in what kind of relationship each one is built to express:

Abstract class (§14) Interface
Relationship "Is a kind of" — a true, single lineage (§11.1) "Can do" — a capability, unrelated to lineage
How many, per class One (extends allows only one) As many as needed (implements takes a list)
Instance fields Yes — Vehicle's make, model, year No instance state at all
Constructors Yes — subclasses call it via super(...) None — nothing to initialize

Vehicle stayed an abstract class because Car and Truck genuinely are vehicles, sharing real fields and a real constructor. Renewable is an interface because being renewable has nothing to do with what else a class is — a Subscription and a Passport share the capability without sharing an ancestor, a field, or a constructor.

§15.3

Declaring and Implementing — interface, implements

An interface declares a capability as a set of method signatures, with no bodies — a contract, not an implementation. Any class, regardless of what it extends, can promise to fulfill that contract:

interface Renewable {
    void renew();
}
class Subscription implements Renewable {
    @Override
    public void renew() {
        System.out.println("Subscription renewed for another year.");
    }
}

class Passport implements Renewable {
    @Override
    public void renew() {
        System.out.println("Passport renewed for another decade.");
    }
}

implements is the relationship an interface creates — distinct from extends (§11.3), and not mutually exclusive with it: a class can extend exactly one class and implement as many interfaces as it needs. Notice, too, that renew() is public in both classes: every method declared in an interface is implicitly public, and every class implementing it must match that — there's no way to implement an interface method with anything more restrictive (§12.5).

Declaring interface Renewable doesn't just declare a contract — it declares a new type, exactly as much a type as Subscription or Passport themselves. A variable can be declared with that type directly, and hold the address of any object whose class implements it:

Renewable r1 = new Subscription();
Renewable r2 = new Passport();
r1.renew(); // Subscription renewed for another year.
r2.renew(); // Passport renewed for another decade.

Nothing is ever instantiated as an interface — new Renewable() isn't legal, any more than new Vehicle() was once Vehicle became abstract (§14.2); what actually gets built by new is always a concrete class, Subscription or Passport. What changes is what the caller sees afterward: r1's declared type is Renewable, so that's the entire view r1 offers — the fact that a Subscription specifically is sitting at that address is invisible through this reference, the same declared-type limit §11.6 already established for any reference type. A reference variable, in other words, is always three things layered together: an interface (or class) type it's declared as, a concrete class that actually implements or extends it, and one specific object of that class, instantiated on the heap. This is the exact same substitutability from §11.6, just with an interface standing in for a superclass, and with no shared ancestor required at all.

Introducing an interface purely for a shared capability — never for lineage — earns two things at once, from two different angles. From the type system's side: unrelated classes become interchangeable through that interface — polymorphism without lineage. r1 and r2 above are a Subscription and a Passport underneath, with no common ancestor beyond Object, yet both are fully substitutable wherever a Renewable is expected. From the dependency's side: any code written against Renewable — a method taking a Renewable parameter, say — depends on the interface alone, never on Subscription or Passport specifically. That's loose coupling: the call site doesn't know, and doesn't need to know, which concrete class it actually received.

§15.4

Overriding vs. Overloading — same name, two different things

Subscription supplying a real renew() for the one Renewable declared is a form of overriding — the same relationship a subclass has with an inherited method it redefines (§11.5), just against an interface's promise instead of a superclass's default. Nothing stops Subscription from also declaring more methods sharing that name, as long as each one's parameter list is different:

class Subscription implements Renewable {
    @Override
    public void renew() {
        System.out.println("Subscription renewed for another year.");
    }

    public void renew(int years) { // an overload — same name, different parameters
        System.out.println("Subscription renewed for " + years + " years.");
    }
}
Subscription sub = new Subscription();
sub.renew();  // Subscription renewed for another year.
sub.renew(3); // Subscription renewed for 3 years.

This is overloading: multiple methods sharing one name, distinguished entirely by parameter list — different number of parameters, different types, or both. Which one runs is decided at compile time, from the arguments actually written at the call site: sub.renew() matches the no-argument version, sub.renew(3) matches the one taking an int. Nothing about the object's runtime type is involved in that choice at all.

Do not confuse these

Overriding replaces an inherited or interface-promised method with a new body for the same signature — same name, same parameter list — and which version runs is chosen at runtime, from the object's actual type (§11.5's dynamic dispatch). Overloading defines multiple, genuinely separate methods that happen to share a name, distinguished by parameter list, and which one runs is chosen at compile time, from the arguments in the call. renew() and renew(int) are two different methods, not one method "overridden" with an extra parameter — @Override could never apply to the second one, because there's nothing declared in Renewable for it to replace.

§15.5

Implementing Multiple Interfaces — adopting more than one role at once

§11.3 fixed how many classes a class can extend at once: exactly one. Interfaces carry no such limit — a class can implement as many as it needs, each one a separate, independent capability the class is declaring it has:

interface Cancellable {
    void cancel();
}

class Subscription implements Renewable, Cancellable {
    @Override
    public void renew() {
        System.out.println("Subscription renewed for another year.");
    }

    @Override
    public void cancel() {
        System.out.println("Subscription cancelled.");
    }
}
Subscription sub = new Subscription();

Renewable   r = sub; // Subscription, viewed as something renewable
Cancellable c = sub; // the exact same object, viewed as something cancellable

Subscription now satisfies two separate contracts at once, and implementing one has nothing to do with implementing the other — Cancellable doesn't know Renewable exists, and neither interface changes what the other requires. Each is a distinct role Subscription is entitled to be treated as: r and c above hold the identical address, the same one object on the heap, just viewed through two unrelated declared types, each one showing only its own slice of what Subscription can do.

This is exactly what extends can't offer. A class fits into exactly one "is-a" lineage, because extends allows only one parent (§11.3) — but it can adopt as many "can-do" roles as it needs, because interfaces carry no state and no constructor for multiple parents to conflict over (§15.11 shows the same freedom one level up, for interfaces extending other interfaces).

§15.6

Interface-Typed Parameters — accepting any implementer

A method parameter can be declared with an interface type, exactly like any other type:

static void processRenewal(Renewable item) {
    item.renew();
}
processRenewal(new Subscription()); // Subscription renewed for another year.
processRenewal(new Passport());     // Passport renewed for another decade.

processRenewal accepts anything that implements Renewable — not just Subscription and Passport, but any class written later, by anyone, that adds implements Renewable to its own declaration. The parameter's declared type, Renewable, is the entire contract the method can rely on at compile time: it's guaranteed a renew() to call, and guaranteed nothing else. What actually arrives at runtime is still a genuine object of some concrete class — item can't hold an address unless something concrete was instantiated somewhere first (§4.1) — and that concrete, runtime type is what decides which renew() actually runs, exactly the dynamic dispatch already established in §11.5.

And exactly like any other reference parameter (§6.2), what's copied into item is only the address — never the object itself. Passing a Subscription to processRenewal doesn't duplicate it; item and the caller's original variable end up holding the identical address, both pointing at the one object on the heap.

§15.7

Anonymous Classes — an implementation with no name at all

Getting a real Renewable object onto the heap has exactly one dead end, and so far exactly one route around it. The dead end: new Renewable() is illegal — an interface has no body to run, so there's nothing for new to build. The route: declare a named class — Subscription, Passport — that implements Renewable and supplies a real body for every method, then instantiate that instead.

Sometimes a named class is more ceremony than the situation calls for — a Renewable needed exactly once, for one specific use, with no reason to ever exist as its own reusable type anywhere else. Java allows a third route: declare the class and create its one instance in the same expression, with no name at all.

Renewable oneOff = new Renewable() {
    @Override
    public void renew() {
        System.out.println("Renewed, just this once.");
    }
};

oneOff.renew(); // Renewed, just this once.

new Renewable() { ... } looks like it instantiates the interface directly — it doesn't. Java generates an unnamed class right there, implementing Renewable exactly the way Subscription does, and that single expression both declares it and creates its one and only instance. oneOff's declared type is still Renewable — everything about substitutability and the interface-as-type relationship from §15.3 applies exactly as before. The only thing missing is a name for the class, because nothing else in the program will ever need to refer to it.

When each one earns its keep

A named class pays for itself the moment more than one object needs the same implementation, or the implementation is substantial enough to deserve its own file. An anonymous class fits the opposite case: a single, throwaway implementation, used once, right where it's created — common enough in real Java code to be worth recognizing on sight, even though this tutorial won't lean on it further.

§15.8

Default Methods — an interface that carries behavior

Everything so far treats an interface as a pure contract — signatures only, no shared logic, matching how interfaces worked in Java's earliest versions. Modern Java allows more: a default method is a method written directly in the interface, with a real body, that every implementing class gets for free:

interface Renewable {
    void renew();

    default void remind() {
        System.out.println("Don't forget to renew!");
    }
}
Subscription sub = new Subscription();
sub.renew();  // Subscription renewed for another year.
sub.remind(); // Don't forget to renew! — never written in Subscription itself

Subscription never declared remind() — it inherited a working implementation directly from the interface, the same way a subclass inherits a concrete method from a superclass (§11.3). Passport gets the identical remind() for free too, with zero shared ancestor between the two classes. And exactly like an inherited class method, a default method can still be overridden — a class implementing Renewable is free to write its own remind() if the default wording doesn't fit.

This is what makes modern interfaces more than a contract

Before default methods existed, an interface could declare that something was possible, never how. Adding shared behavior meant a shared superclass — which only works when every class involved can agree on a single one, and Java's single-inheritance rule (§11.3) means that's not always possible. Default methods close that gap: real, shared, working code, distributed to any class that implements the interface, regardless of what — if anything — that class already extends.

§15.9

Default Method Collisions — when two interfaces disagree

§15.5 implemented two interfaces at once with no overlap between them — Renewable and Cancellable each contributed a method the other didn't have. Nothing stops two different interfaces from declaring a default method with the exact same name and parameter list, and a single class implementing both then has a real conflict to resolve:

interface Walk {
    default long getSpeed() {
        return 5;
    }
}

interface Run {
    default long getSpeed() {
        return 10;
    }
}

class Horse implements Walk, Run {
    @Override
    public long getSpeed() {
        return 20;
    }
}

Both Walk and Run offer a getSpeed(), and neither default wins automatically — Java refuses to guess which one Horse meant, so this doesn't compile at all unless Horse supplies its own getSpeed(), overriding both defaults at once with a single, unambiguous implementation. The override doesn't have to reuse either interface's version — it just has to exist, replacing the ambiguity with an actual decision.

Sometimes there's no override that could ever resolve it — not because the class hasn't tried hard enough, but because no single method could satisfy both interfaces at once:

interface Walk {
    default long getSpeed() {
        return 5;
    }
}

interface Run {
    default int getSpeed() { // a different return type
        return 10;
    }
}

class Horse implements Walk, Run { } // compile error — no override could ever fix this

Overriding a method requires the same return type, or a subtype of it — and int and long aren't related that way in either direction, despite one converting to the other in ordinary expressions (§9). Walk and Run aren't asking Horse to pick a winner here; they're demanding a single method that is somehow both an int-returning getSpeed() and a long-returning one. Nothing Horse could write satisfies both — this pair of interfaces simply cannot be implemented by the same class at all, override or no override.

§15.10

Static Methods — utilities that live on the interface itself

An interface can also declare static methods — utilities called on the interface's own name, never on an implementing object, the same way a class's static methods are called on the class itself (§6.1):

interface Renewable {
    void renew();

    default void remind() {
        System.out.println("Don't forget to renew!");
    }

    static boolean isRenewable(Object obj) {
        return obj instanceof Renewable;
    }
}
System.out.println(Renewable.isRenewable(new Subscription()));                // true
System.out.println(Renewable.isRenewable(new Car("Toyota", "Corolla", 2022))); // false

Renewable.isRenewable(...) is called on Renewable directly — there's no object being asked, no renew() being invoked, just a helper that happens to live in the interface because it's about the interface's own concept.

Being called on the interface itself, with no object involved, comes with the same restriction any static method has (§6.1): nothing inside it can depend on a particular object existing. A static interface method can call other static methods, and read public static final constants — anything that belongs to the interface itself, not to any one implementer. It cannot call an instance method, use this, or touch any per-object data, because at the point it runs, there may be no object at all:

interface Example {
    default void sayHi() {
        System.out.println("Hi");
    }

    static void test() {
        sayHi();      // compile error — cannot make a static reference to non-static method
        this.sayHi(); // compile error — cannot use this in a static context
    }
}

sayHi() is a default method — it only exists once some class has implemented Example and produced an actual object to call it on. test() is called on Example directly, with no such object anywhere in sight, so there's nothing for sayHi() to run on and nothing for this to refer to. The same divide already separates static and instance methods on ordinary classes (§6.1); an interface's static methods sit on the static side of it, same as ever.

§15.11

Interface Inheritance — one interface extending another

Interfaces can extend other interfaces, using the same keyword classes use to extend classes (§11.3) — with one difference: an interface can extend more than one other interface at once, since there's still no shared state or constructor for multiple parents to conflict over.

interface AutoRenewable extends Renewable {
    void enableAutoRenew();
}

AutoRenewable inherits everything Renewable declared — the abstract renew(), and the default remind() from §15.8 — and adds one more abstract method of its own, enableAutoRenew(). A class implementing AutoRenewable takes on the combined contract: every abstract method from both interfaces, inherited and newly added alike, needs a real implementation before that class can be instantiated.

class CloudSubscription implements AutoRenewable {
    @Override
    public void renew() {
        System.out.println("Cloud subscription renewed.");
    }

    @Override
    public void enableAutoRenew() {
        System.out.println("Auto-renew enabled.");
    }
}

CloudSubscription gets remind() for free, inherited through AutoRenewable from Renewable, exactly as if it had implemented Renewable directly — extends between interfaces doesn't create a new kind of relationship, just a longer chain of the same one.

An extending interface doesn't have to leave an inherited default method untouched, either. Facing a default method it inherits, AutoRenewable has three options:

  1. Leave it alone. This is what it did with remind() above — say nothing, and the inherited default is exactly what implementing classes get.
  2. Override it as a default. Provide a new default body under the same name, following the same overriding rules a class would (§11.5) — implementing classes now get this version instead, unless they override it again themselves. 
    interface AutoRenewable extends Renewable {
    void enableAutoRenew();

    @Override
    default void remind() {
        System.out.println("Auto-renew is on — no action needed.");
    }
}
  3. Redeclare it as abstract. Repeat the method with no body and no default — this withdraws the inherited implementation, and every implementing class is back to needing its own, exactly as if the method had never had a default at all. 
    interface AutoRenewable extends Renewable {
    void enableAutoRenew();
    void remind(); // no longer default — implementing classes must supply it themselves
}

That third option has a real, visible consequence, not just a theoretical one — withdrawing a default changes whether code compiles:

interface Mover {
    default void move() {
        System.out.println("Default moving behavior");
    }
}

interface Runner extends Mover {
    void move(); // redeclared abstract — withdraws Mover's default
}

class Car implements Mover { }          // fine — inherits Mover's default untouched
class MarathonMan implements Runner { } // compile error — must implement Runner's move()

Car implements Mover directly and never mentions move() at all — perfectly legal, since Mover's default is still standing. MarathonMan implements Runner instead, and Runner redeclared move() as abstract, which erased the default for anything going through Runner — so MarathonMan owes an implementation the exact same way it would for any interface method that was never given a default to begin with.

Every abstract method needs an implementer — with one exception

Any concrete class implementing an interface must provide a real body for every abstract method the interface declares, inherited or its own — this has held throughout the chapter, and it's the same rule §14 established for abstract classes. The one exception is symmetrical, too: a class declared abstract (§14.2) is allowed to leave some — or all — of those methods unimplemented. The obligation doesn't disappear; it simply passes down to whatever concrete subclass eventually extends that abstract class.

Implementing directly, with no detour, looks like every other example in this chapter:

interface Movable {
    void move();
}

// concrete
class Bike implements Movable {
    @Override
    public void move() {
        System.out.println("Cycling");
    }
}

An abstract class sitting in between can defer that same obligation instead of meeting it:

interface Movable {
    void move();
}

abstract class Vehicle implements Movable {
    // move() left unimplemented — legal, because Vehicle is abstract
}

class Bike extends Vehicle {
    @Override
    public void move() {
        System.out.println("Cycling");
    }
}

Vehicle is allowed to implement Movable without ever writing move() — being abstract is exactly what makes that legal (§14.2). The requirement hasn't gone away; it's simply been handed down one level, to Bike, the first concrete class in the chain. Either path — implementing an interface directly, or inheriting the obligation through an abstract class first — arrives at the same rule: some concrete class, somewhere, has to supply move() before an object can exist at all.

§15.12

The Strategy Pattern — swapping algorithms at runtime

§15.6 showed a method accepting anything that implements an interface. Take that one step further — store the interface-typed reference as a field, not just a parameter — and a class can hold onto a swappable piece of behavior, decided by whoever creates it, and changeable later. This combination has a name recognized across the industry: the Strategy pattern.

interface DiscountStrategy {
    double apply(double price);
}

class PercentageDiscount implements DiscountStrategy {
    private final double percent;

    PercentageDiscount(double percent) {
        this.percent = percent;
    }

    @Override
    public double apply(double price) {
        return price - (price * percent / 100);
    }
}

class FlatDiscount implements DiscountStrategy {
    private final double amount;

    FlatDiscount(double amount) {
        this.amount = amount;
    }

    @Override
    public double apply(double price) {
        return price - amount;
    }
}

PercentageDiscount and FlatDiscount aren't specific kinds of anything, and neither is a kind of the other — they're two different algorithms for the exact same operation, apply(double), entirely interchangeable with each other. That's the shape Strategy names: not "can this class do X," the way Renewable did, but "here is one particular way of doing X, packaged as an object in its own right."

A class that needs the operation done doesn't pick an algorithm itself — it holds a reference to whichever DiscountStrategy it was given, and defers to it:

class Order {
    private double total;
    private DiscountStrategy discount;

    Order(double total, DiscountStrategy discount) {
        this.total = total;
        this.discount = discount;
    }

    void setDiscount(DiscountStrategy discount) {
        this.discount = discount;
    }

    double finalPrice() {
        return discount.apply(total);
    }
}
Order order = new Order(100.0, new PercentageDiscount(10));
System.out.println(order.finalPrice()); // 90.0

order.setDiscount(new FlatDiscount(15));
System.out.println(order.finalPrice()); // 85.0

Order never mentions PercentageDiscount or FlatDiscount anywhere in its own declaration — only DiscountStrategy. Constructing it with one strategy, then swapping in a completely different one through setDiscount, changes what finalPrice() computes without changing a single line of Order itself, and without Order ever being told which strategy it's holding at any given moment.

Why this is worth naming

Nothing here is new syntax — it's the interface-as-type relationship from §15.3, interface-typed references from §15.6 held in a field instead of just a parameter, and the loose coupling named back in §15.3, combined into one recurring shape. Giving that shape a name — Strategy — means recognizing it on sight in unfamiliar code, and reaching for it deliberately: whenever "the same operation, done one of several interchangeable ways, chosen by whoever creates the object" shows up, an interface plus a held reference is usually the answer.

Scope of this chapter

Functional interfaces and lambda expressions, sealed interfaces, and generic interfaces are all left for later. The core shape carries over unchanged: an interface is a capability a class opts into, separate from what that class already is.
