Chapter 25

Input and Output

So far, every program in this tutorial has lived inside its own little world — variables, methods, and objects that already exist in memory. Real programs rarely stay that contained. This chapter is where data starts crossing the boundary between a program and everything outside it.

§25.1

The Problem — your program is not an island

So far, our programs have lived inside their own little world. We create variables, call methods, and work with objects that already exist in memory. Everything a program has needed, up to this point, was already there before the program started running.

But real programs rarely work that way. A banking application needs transactions from a database. A photo editor needs images from disk. A chat application needs messages from the network. At some point, data has to cross the boundary between a program and the outside world — and that boundary is exactly what I/O (input/output) is about.

The Idea of a Stream

Imagine filling a bucket with water from a pipe. The bucket doesn't receive an entire river all at once — it receives a continuous flow, arriving gradually. Data crossing the boundary into or out of a program often works the same way: a program receives or sends data gradually, not as one complete object handed over in an instant.

Input: program <- outside world

Output: program -> outside world

Java names these two directions directly:

  • an InputStream is a connection data flows in through;
  • an OutputStream is a connection data flows out through.
A stream is a connection, not the data itself

The single most important idea to hold onto before writing a line of I/O code: a stream is not the data — it's a connection through which data moves. A file on disk, a socket on the network, a database result set — none of these are a stream; a stream is simply the pipe a program reads them through, or writes them out through, one piece at a time.

§25.2

Bytes or Characters?

Imagine we are building a photo-sharing application. A user uploads a profile picture: profile-picture.jpg The program needs to read the file from disk and send it somewhere else. But what exactly are we reading? We are not reading "a picture" in the way a human sees it. The computer sees a sequence of bytes — something like: 

[137] [80] [78] [71] [13] [10] [26] [10] ...

Those numbers are the actual data stored in the file. Our program does not need to understand that they represent a picture. It just needs to move the bytes correctly.

Now imagine a different feature: loading a user's saved message. The file contains the text: Happy birthday! This time, the program does not just need the raw data. It needs to understand the text as characters. The same kind of stored bytes now need to be interpreted as: 

H a p p y (space) b i r t h d a y !

Now the distinction emerges naturally: the computer always stores bytes. Humans usually work with characters. The difference is whether the program cares about the raw representation, or the meaning of the text.

InputStream / OutputStream Reader / Writer
Works with Bytes Characters
Used for Images, videos, compressed files, network protocols Text files, configuration files, source code, JSON

Both pairs are still streams in the sense of §25.1 — a connection through which data moves, not the data itself. The difference is what unit that data moves in: raw bytes, or characters already decoded into text.

§25.3

Abstract vs. Concrete — InputStream vs. FileInputStream

InputStream says nothing about where the bytes actually come from. That's deliberate. Bytes might come from a file on disk, from a socket on the network, from a zip entry, from a chunk of memory — InputStream only promises that something can hand it bytes, one read() call at a time. It is an abstract class, in exactly the sense §12 introduced: it defines the contract, but has no idea how to fulfill it.

The class that actually knows how to fulfill it is a concrete subclass. Reading a profile picture from disk means reaching for a specific one: FileInputStream.

InputStream in = new FileInputStream("profile-picture.jpg");

Notice the shape of that line: the declared type is the abstract InputStream, but the object created is the concrete FileInputStream. This is the same "program to an interface, not an implementation" idea from §26.6 — the rest of the code that reads bytes from in never needs to know, or care, that they happen to be coming from a file.

The same split exists on the output side, and again on the character side. Four abstract types, each with its own family of concrete implementations that know exactly where the data is coming from or going to:

Bytes Characters
Input InputStream → FileInputStream Reader → FileReader
Output OutputStream → FileOutputStream Writer → FileWriter

FileInputStream, FileOutputStream, FileReader, and FileWriter are only one family among several — ByteArrayInputStream reads bytes already sitting in memory, Socket exposes an InputStream/OutputStream pair for network data, and so on. The abstract type is what the rest of the program is written against; the concrete type is chosen once, at the point where the stream is created, based on where the data actually lives.

§25.4

Reading and Writing — the core methods

I/O streams are all about reading and writing data, so it shouldn't be a surprise that the most important methods are read() and write().

Both InputStream and Reader declare a read() method to pull data from a stream — a byte at a time for InputStream, a character at a time for Reader. Likewise, both OutputStream and Writer declare a write() method to push a single unit of data onto the stream.

abstract int  read();   // InputStream — returns a byte, or -1 at end of stream
abstract void write(int b);   // OutputStream — writes a single byte

These are exactly the methods that make InputStream and OutputStream abstract in the first place, per §25.3: the abstract class only declares that some byte can be read or written; a concrete subclass like FileInputStream is what actually implements read() against a real file. Every other operation covered in this chapter — buffering, locating a file through Path — exists to make calling these two methods more convenient or more efficient. Neither replaces them.

§25.5

Buffering

Imagine asking someone to deliver rice one grain at a time. They walk to the storeroom, pick up a single grain, carry it to the kitchen, set it down — then turn around and walk all the way back for the next one. Technically, the rice gets delivered. But almost none of that effort is actually about the rice; nearly all of it is the walking. Do that a few thousand times and the trip itself becomes the bottleneck, not the grain.

Nobody delivers rice this way. Instead, we scoop up a bowlful in one trip and carry the whole bowl to the kitchen. The number of grains delivered hasn't changed — only the number of trips has, and that's what makes the difference.

The same tradeoff shows up in I/O. A stream created directly on a file or a socket typically talks to the operating system on every single read() or write() call, and each of those calls is expensive compared to the actual work of moving a byte:

Without buffering:

read byte
read byte
read byte
...

Buffering collects data into a chunk first, and lets the program work through that chunk in memory, only going back to the operating system once the chunk is exhausted:

With buffering:

read chunk
process chunk
...

This is exactly what BufferedInputStream and BufferedReader are for. Neither one changes what is being read — bytes are still bytes, characters are still characters — they only change how many round trips it takes to read them. And true to the pattern from §25.3, both are wrapped around another stream rather than replacing it:

InputStream in = new BufferedInputStream(new FileInputStream("profile-picture.jpg"));
Reader in = new BufferedReader(new FileReader("message.txt"));

FileInputStream and FileReader still own the connection to the actual file; BufferedInputStream and BufferedReader just sit in front of them, collecting bytes or characters into a chunk before handing them onward. The rest of the program still just sees an InputStream or a Reader — it never needs to know that a buffer is doing the heavy lifting underneath.

§25.6

Flushing

Write a document, hit save, and the laptop dies half a second later — did that last change actually make it to disk, or was it still sitting somewhere in between? Buffering, as §25.5 just showed, means a write doesn't necessarily land immediately; sometimes it's just waiting in memory for the bucket to fill.

Flushing is the act of forcing whatever is sitting in the buffer to be written to its destination — disk, network, console — immediately.

Why Flushing Is Needed

A buffer, as §25.5 introduced, accumulates data in memory and writes to the destination in chunks for efficiency. But that means data can sit in the buffer and never reach the destination if the program never flushes it:

BufferedWriter bw = new BufferedWriter(new FileWriter("file.txt"));

bw.write("important data");

// program crashes here — buffer never flushed — file.txt is empty!
A write() is not a promise it's on disk

Calling write() only hands data to the buffer sitting in memory — it says nothing about whether that data has actually reached the file, the socket, or wherever the stream is headed. Until a flush happens, "important data" above exists only in memory, and memory doesn't survive a crash.

When Does Flush Happen?

1. Automatically

a. When the buffer is full — once the buffer has no room left, it empties itself into the destination on its own, making room for more:

your code                buffer (memory)           destination (disk)
─────────                ───────────────           ──────────────────
write("Hello ")     →    [ Hello  ]                [ empty ]
write("World")      →    [ Hello World ]           [ empty ]
write("!!")         →    [ Hello World!! ]         [ empty ]
-- buffer full --
auto flush          →    [ empty ]            →    [ Hello World!! ]

b. When the stream is closed — close() calls flush() internally before shutting the stream down, so closing always flushes. The difference between the two is what happens to the stream afterward:

// flush() — empties the buffer, stream stays open for more writing
bw.flush();
bw.write("more data"); // still works

// close() — flushes AND closes, no more writing
bw.close();
bw.write("more data"); // IOException — stream closed

2. Explicit flush()

A program doesn't have to wait for either of those to happen. Calling flush() directly forces whatever is currently buffered out to the destination right now, without closing the stream — useful anywhere the program needs a guarantee that data has actually left memory, such as after logging an event:

BufferedWriter bw = new BufferedWriter(new FileWriter("log.txt"));

bw.write("Server started");
bw.newLine();
bw.flush(); // force to disk now, don't wait for the buffer to fill

bw.write("Request received");
bw.newLine();
bw.flush(); // again
§25.7

NIO.2 — the modern file API

Everything so far has been about streams — connections that data flows through. But before a program can open a stream on a file, it first has to say which file. That's a separate question, and it deserves its own story, centered on a single type: Path.

Introduce: What Is a Path?

The old way of naming a file looks like this:

File file = new File("notes.txt");

The modern way looks similar on the surface, but is built on a different idea:

Path path = Path.of("notes.txt");
A path is not a file

A Path is a description of where something exists — a location, expressed as a sequence of names, nothing more. It doesn't open anything, lock anything, or guarantee that the file is even there. It is to a file what an address is to a house: writing the address down doesn't mean the house exists, and it doesn't move anyone into it.

Sit with that for a moment against the line above: path doesn't represent a file — it represents where a file could be. notes.txt might not even exist. A Path can just as easily be built pointing at:

  • an existing file,
  • a non-existent file,
  • a directory,
  • or a symbolic link.

Path.of(...) doesn't check which of these it is, or whether it's any of them at all — it simply describes a location in the file system, nothing more.

That same "just a location" idea means a Path doesn't know, or care, whether it points to a file or a directory. It's nothing more than a sequence of name components — "some/folder/file.txt" or "mydir/" are both equally valid paths, and Path makes no distinction between them.

A .txt name doesn't make it a file

/data/zoo.txt could just as easily be a directory — nothing about a Path's text guarantees what it actually is on disk, or that anything is there at all. Whether a given Path refers to a file or a directory is something to check, with Files (later in this section), never something to assume from how the name is spelled.

Create: Path.of() vs. Paths.get()

Path is an interface — and an interface can never be instantiated with new. So how does a line like Path.of("notes.txt") actually produce an object?

When NIO.2 was first introduced, the standard way to hand back an instance of an interface was a separate, plural-named utility class holding the factory method, since the interface itself couldn't hold one. That's how Paths.get() was born, and for years it was the only way to create a Path:

// Java 7–10 — had to go through the Paths factory class
Path p = Paths.get("file.txt");

// Java 11+ — Path has its own factory method
Path p = Path.of("file.txt");

Java 8 changed the premise that pattern rested on, by letting interfaces carry actual method bodies — not just declarations — in two forms: default methods, run on an instance, and static methods, run on the interface type itself, exactly the way Path.of(...) is called on Path rather than on some separate factory class. (Java 9 later added a third form, private methods, for reusable logic internal to the interface — not covered elsewhere in this tutorial, but part of the same shift.) Java 11's Path.of() is a direct product of that shift: rather than adding another method to a Paths-style helper class, the factory method was placed directly on Path itself, as part of a broader effort across the language to reduce exactly this kind of "utility class pollution" — one interface, one place to look, instead of an interface plus a same-purpose-different-name class beside it. Paths still exists, but is largely a relic of the pattern Path.of() replaced.

So Path.of("notes.txt") hands back an object — but an object of what actual class? Not Path itself; an interface has no object of its own, only whatever concrete class implements it (§13). On Linux or macOS that's a class called sun.nio.fs.UnixPath; on Windows, it's sun.nio.fs.WindowsPath instead. Neither one is something a program is ever meant to reference, or even likely to encounter by name:

  • They are not part of the public Java API — the official Javadoc only documents exported public API, and these classes aren't in it.
  • They live in a package beginning with sun.*, which has never been part of the supported Java SE API — a naming convention that itself signals "implementation detail, not public contract."
  • Since the module system arrived in Java 9, they're harder to reach still: the java.base module doesn't export sun.nio.fs at all, so code outside the JDK can't normally access those classes even by name.

The only place to actually see them is the JDK's own source — OpenJDK ships files like src/java.base/unix/classes/sun/nio/fs/UnixPath.java and src/java.base/windows/classes/sun/nio/fs/WindowsPath.java, each implementing the same Path interface against a completely different underlying file system.

That invisibility is exactly the point, and it's the same "program to an interface" payoff §26.6 makes elsewhere: every line of code in this chapter has been written against Path, never against UnixPath or WindowsPath by name. If a future JDK rewrote the file system implementation from scratch, any program written against Path wouldn't need to change at all — only code that had reached past the interface, into sun.*, could possibly break.

Operate: Working with an Existing Path

Every operation on a Path follows the same rule: none of them change the Path they're called on. resolve() is the simplest example — it's easy to misread a line like p1.resolve("README.md") as changing p1, but it doesn't. It returns a new Path, leaving the original exactly as it was:

Path p1 = Path.of("/users/mk/dev/java");
Path p2 = p1.resolve("README.md");
// p1 is unchanged; p2 is a new object

System.out.println(p1); // /users/mk/dev/java
System.out.println(p2); // /users/mk/dev/java/README.md

That's the same immutable-value discipline §11 built up for classes like String: a Path can be handed to another method, stored, or shared across threads without any risk that something else silently changes what it points to. Path also plugs directly into streams (§21) through methods like Files.lines() and Files.walk(), which is not something File was ever built to support.

normalize() is a second operation with the same shape, and it earns its keep because a Path is only text — nothing stops that text from containing redundant pieces: a stray current-directory marker, a detour up and back down the tree, a doubled separator. normalize() cleans exactly that up, without ever touching the file system. Nothing is read, moved, or checked for existence — it's pure text processing on the Path itself, and it handles three kinds of redundancy:

Redundancy Example
A . ("current directory") component — doesn't change the effective path, so it's simply dropped /home/user/./documents → /home/user/documents
A .. ("parent directory") component — resolved by moving back up the tree, when possible /home/user/../documents → /home/documents
Doubled directory separators — condensed to one /home//user → /home/user 
Path p = Path.of("/home/user/../documents/./notes.txt");
Path cleaned = p.normalize();

System.out.println(p);       // /home/user/../documents/./notes.txt
System.out.println(cleaned); // /home/documents/notes.txt

A directory and a .. right after it don't just move up the tree — they cancel each other out completely, since going into armadillo and immediately back out leaves nothing behind:

var p1 = Path.of("./armadillo/../shells.txt");

System.out.println(p1); // ./armadillo/../shells.txt

// normalize: remove redundant elements from the path
System.out.println(p1.normalize()); // shells.txt

// notice immutability!
System.out.println(p1); // ./armadillo/../shells.txt

The leading ./ is dropped as a redundant current-directory marker, and armadillo/.. collapses to nothing at all — leaving just shells.txt. And exactly as with resolve(), p1 itself never changes: normalize() only ever hands back a new Path.

.. beyond the root

Resolving .. assumes there's a parent to move up to. Whether that assumption holds when .. would climb past the root — and what normalize() does about it — depends on the underlying file system and on whether the Path is absolute or relative to begin with.

Acting On a Path: Files

Everything above only describes a location, or reshapes that description — none of it touches the file system. Actually doing something with that location — reading it, copying it, deleting it — is a separate step, handled by the Files utility class:

String contents = Files.readString(path);
Files.copy(source, target);
Files.delete(path);

Files is built the same way as Collections or Arrays: nothing but static methods, a private constructor, and no reason to ever be instantiated. Splitting "where something is" (Path) from "what to do with it" (Files) is what makes this API so much more expressive than the old File class, which bundled both ideas into one object with a much smaller, clumsier set of operations.

Summary: Four Types, Four Jobs

Type Purpose
java.io.File The old I/O representation of a location in the file system. A concrete, mutable class that bundles "where" and "what to do with it" together.
java.nio.file.Path The NIO.2 representation of a location — a file or a directory. An interface, and effectively immutable: no operation changes a Path, every operation returns a new one.
java.nio.file.Paths A factory class for creating a Path from a string or URI — largely superseded once Path got its own static factory method.
java.nio.file.Files A utility class of static methods that operate on a Path — reading, writing, copying, deleting, and more.
§25.8

Closing Streams Safely — try-with-resources

Every concrete stream this chapter has introduced — FileInputStream, FileReader, BufferedWriter, and the rest — is a resource that has to be closed once a program is done with it, the same way §25.6 showed a buffer has to be flushed. And the same problem shows up: if an exception is thrown between opening the stream and closing it, a hand-written close() call never runs, and the file handle or socket leaks.

// Bad — close() never runs if reading throws
FileInputStream in = new FileInputStream("data.txt");
in.read();
in.close();

This is exactly the problem §14.9's try-with-resources exists to solve, and every stream in this chapter is already built to use it: InputStream, OutputStream, Reader, and Writer all implement Closeable, so any of their concrete subclasses can be declared directly inside a try's parentheses:

try (FileInputStream in = new FileInputStream("data.txt")) {
    in.read();
} // in.close() runs automatically, even if read() throws

Copying from one file to another needs two streams open at once — one to read, one to write — and both can be declared the same way, separated by a semicolon:

try (FileInputStream in = new FileInputStream("source.jpg");
     FileOutputStream out = new FileOutputStream("copy.jpg")) {
    in.transferTo(out);
} // out.close() runs first, then in.close() — reverse of declaration order

Nothing about I/O changes how try-with-resources behaves — §14.9 already covers the mechanics in full, including what happens when close() itself throws while unwinding from another exception. What's worth remembering here is narrower: every stream type this chapter has built up is a Closeable, which is exactly what makes it a safe fit for this syntax without any extra work.

§25.9

Comparing Files

a. Are They Physically the Same File?

A file system location can be written more than one way — a redundant ., a detour through .., a symbolic link — so two Path objects with different text can still point at the exact same file. That means equals() can't be trusted to answer "is this the same file," because equals() never looks at the file system at all — it only compares the path's text, element by element:

Path Value
p1 /users/mk/brics.csv
p2 /users/mk/./brics.csv
p3 /users/mk/dev/../brics.csv 
Path p1 = Path.of("/users/mk/brics.csv");
Path p2 = Path.of("/users/mk/./brics.csv");
Path p3 = Path.of("/users/mk/dev/../brics.csv");

System.out.println(p1.equals(p2)); // false

try {
    System.out.println(Files.isSameFile(p1, p2)); // true
} catch (IOException e) {
    e.printStackTrace();
}

All three paths above resolve to the exact same location once normalized, but p1.equals(p2) still returns false — equals() sees "/users/mk/brics.csv" and "/users/mk/./brics.csv" as different strings, full stop, regardless of what either one actually points to on disk.

Files.isSameFile(Path, Path) is the method built to answer the question equals() can't: it resolves path symbols, follows symbolic links, and checks the file system itself to determine whether two Path objects name the same location — despite the name, that includes two Paths that both refer to the same directory, not only files.

b. Are the Contents of the Files the Same?

Identity and content are independent questions — a path can name the right file without telling you what's inside it, and matching content doesn't require it to be the same file at all:

Question Method
Identity — is this the same file? Files.isSameFile(Path, Path)
Content — do they hold the same bytes? Files.mismatch(Path, Path)

Files.isSameFile() answers identity, as above — a common real-world use is detecting that two different-looking paths are actually symlinks pointing at the same underlying file.

Files.mismatch(Path, Path), introduced in Java 12, answers content instead: it compares two files byte by byte and reports the first position where they differ. It returns -1 if the files are identical; otherwise the returned index falls somewhere in the inclusive range from 0L up to the size, in bytes, of the smaller of the two files.

§25.10

Handling Methods That Declare IOException

Unlike some modern APIs that try to hide checked exceptions, java.nio.file.Files leans heavily into them: it's almost entirely composed of static methods that throw IOException. That design choice reflects a reality §14.4 already named — a checked exception is the compiler's way of forcing acknowledgment of a failure the caller can't ignore — and file system operations are exactly that kind of failure: they depend on hardware, permissions, and network state a program cannot fully control or predict. They are, in a word, brittle.

The old java.io.File API took the opposite approach. Methods like delete() or renameTo() returned a plain boolean — true or false. That was widely criticized, because a false told you the operation failed and nothing else: not whether the file was missing, locked, or permission-denied, just failure with no explanation. NIO.2's move to IOException fixes exactly that: a thrown exception carries a detailed message and a stack trace explaining what actually went wrong.

Methods that access or change files and directories — most of what Files offers — declare IOException for this reason. But not every method does. Files.exists() is a telling exception to the rule: it returns a plain boolean and declares nothing. If it threw an exception the moment a file didn't exist, it could never actually return false — the one answer the method exists to give.

A rule of thumb

If an NIO.2 method declares IOException, it usually requires the paths it operates on to already exist. A method whose whole job is to answer whether something exists — like Files.exists() — has no business throwing just because the answer is no.

Common causes of a Files method throwing IOException include:

  • Loss of communication to the underlying file system.
  • A file or directory is required, but does not exist.
  • A file or directory exists, but cannot be accessed or modified.
  • A file exists, but cannot be overwritten.
§25.11

Creating Files and Directories

Path.of() — no disk access

Worth repeating from §25.7: Path.of(...) simply builds a Java object describing a location. Nothing happens on disk when it runs — no file is created, no directory checked. Every method in this section is what actually reaches the file system; Path.of() never does.

NIO.2 keeps that same split for creation: distinct Files methods for distinct kinds of entries, rather than one method that guesses what you meant.

Why Files.createFile(Path), not Path.createFile()?

It's tempting to expect the method on Path itself — but that would blur exactly the split §25.7 built up. A Path describes a location; Files performs operations on that location. In a call like Files.createFile(path), the Path isn't the one doing anything — the file system is. Files is only the gateway to it.

Think of a city map with an address on it: 221B Baker Street. Does the address build the house? No — the address only identifies a location. If a house gets built there, it's the construction company doing the building, not the address. A Path is the address; Files is the service that performs operations at that address.

That analogy explains the design, but it's worth being honest about the rest of the answer too. If NIO.2 were being designed today, it's entirely possible we'd see something like path.createFile(), path.delete(), path.exists(), path.size() — instance methods, called directly on a Path. There's nothing fundamentally wrong with that design. The reason we don't have it is partly historical, and partly architectural.

The historical part: NIO.2 shipped in Java 7, in 2011. At that point, an interface couldn't hold a method with a body at all — every method on Path would have had to be pure declaration, with the real logic living somewhere else regardless. Default and static methods, the feature that would have made path.createFile() possible, didn't arrive until Java 8, a year later — by which point Path and Files were already public API, and splitting them apart was the only option on the table.

The architectural part is everything §25.7 and the callout above already cover: Path is an immutable value describing a location, and keeping every operation that touches the file system in one separate place keeps that value simple and side-effect-free, regardless of when the interface-methods feature happened to arrive.

1. Creating Files

To actually create a file on disk, use Files.createFile(Path):

Files.createFile(Path.of("/home/user/notes.txt"));

Two failure modes are worth knowing by name: it throws FileAlreadyExistsException if the file already exists, and NoSuchFileException if the parent directory doesn't.

2. Creating Directories

Directories have two creation methods, and the difference between them is exactly whether the parent folders are expected to already be there.

Files.createDirectory(Path) creates only the last directory in the path — a single step, and it assumes every segment before that last one already exists:

Path.of("/home/user/a/b/c")
//        ▲     ▲  ▲ ▲   ▲
//   must already exist  ┇
//                       ┖ only this gets created
Files.createDirectory(Path.of("/home/user/a/b/c"));

// if /home/user/a/b exists     — creates c
// if /home/user/a/b is missing — NoSuchFileException

It throws NoSuchFileException if any parent is missing — createDirectory() assumes everything except the last segment is already there — and FileAlreadyExistsException if the target directory already exists.

Files.createDirectories(Path) (plural) takes the opposite stance: it creates the target directory and every missing parent along the way, similar to mkdir -p on Unix. It's also idempotent — calling it on a path that already exists doesn't fail, it just silently succeeds:

Files.createDirectories(Path.of("/home/user/a/b/c"));

// creates a, b, and c — whichever of them don't already exist
// succeeds silently even if all three already exist

The singular/plural naming is a genuine clue to the contract: createDirectory() is a single step that trusts the caller to have the parent ready; createDirectories() is willing to do all the intermediate work itself.
