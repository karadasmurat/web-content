Chapter 26

Input and Output

So far, every program in this tutorial has lived inside its own little world — variables, methods, and objects that already exist in memory. Real programs rarely stay that contained. This chapter is where data starts crossing the boundary between a program and everything outside it.

§26.1

The Problem — your program is not an island

So far, our programs have lived inside their own little world. We create variables, call methods, and work with objects that already exist in memory. Everything a program has needed, up to this point, was already there before the program started running.

But real programs rarely work that way. A banking application needs transactions from a database. A photo editor needs images from disk. A chat application needs messages from the network. At some point, data has to cross the boundary between a program and the outside world — and that boundary is exactly what I/O (input/output) is about.

The Idea of a Stream

Imagine filling a bucket with water from a pipe. The bucket doesn't receive an entire river all at once — it receives a continuous flow, arriving gradually. Data crossing the boundary into or out of a program often works the same way: a program receives or sends data gradually, not as one complete object handed over in an instant. 

Input: program <- outside world

Output: program -> outside world

Java names these two directions directly:

  • an InputStream is a connection data flows in through;
  • an OutputStream is a connection data flows out through.
A stream is a connection, not the data itself

The single most important idea to hold onto before writing a line of I/O code: a stream is not the data — it's a connection through which data moves. A file on disk, a socket on the network, a database result set — none of these are a stream; a stream is simply the pipe a program reads them through, or writes them out through, one piece at a time.

§26.2

Bytes or Characters? — how Java represents, and moves, data

Imagine we are building a photo-sharing application. A user uploads a profile picture: profile-picture.jpg The program needs to read the file from disk and send it somewhere else. But what exactly are we reading? We are not reading "a picture" in the way a human sees it. The computer sees a sequence of bytes — something like: 

[137] [80] [78] [71] [13] [10] [26] [10] ...

Those numbers are the actual data stored in the file. Our program does not need to understand that they represent a picture. It just needs to move the bytes correctly.

Now imagine a different feature: loading a user's saved message. The file contains the text: Happy birthday! This time, the program still receives bytes off disk — exactly the same kind of raw data as the picture above, nothing about the file itself is different. What's different is what happens next: the program runs those bytes through a decoding scheme, one that defines which byte values stand for which characters, and gets text back out: 

H a p p y (space) b i r t h d a y !

Now the distinction emerges naturally: the computer always stores and moves bytes — that part never changes. Whether a program treats those bytes as opaque data or interprets them as characters, using some encoding scheme to say what each byte (or group of bytes) means, is a choice layered on top, not a difference in what's actually being read.

InputStream / OutputStream Reader / Writer
Works with Bytes Characters
Used for Images, videos, compressed files, network protocols Text files, configuration files, source code, JSON

Both pairs are still streams in the sense of §26.1 — a connection through which data moves, not the data itself. The difference is what unit that data moves in: raw bytes, or characters already decoded into text.

Abstract vs. Concrete: InputStream vs. FileInputStream

InputStream says nothing about where the bytes actually come from. That's deliberate. Bytes might come from a file on disk, from a socket on the network, from a zip entry, from a chunk of memory — InputStream only promises that something can hand it bytes, one read() call at a time. It is an abstract class, in exactly the sense §13 introduced: it defines the contract, but has no idea how to fulfill it.

The class that actually knows how to fulfill it is a concrete subclass. Reading a profile picture from disk means reaching for a specific one: FileInputStream. 

public class FileInputStream extends InputStream

The signature says it plainly: FileInputStream is an InputStream (§10), just one that knows specifically how to read bytes from a file. That inheritance relationship is what makes the next line legal at all: 

InputStream in = new FileInputStream("profile-picture.jpg");

Notice the shape of that line: the declared type is the abstract InputStream, but the object created is the concrete FileInputStream. This is the same "program to an interface, not an implementation" idea from §27.8 — the rest of the code that reads bytes from in never needs to know, or care, that they happen to be coming from a file.

The same split exists on the output side, and again on the character side. Four abstract types, each with its own family of concrete implementations that know exactly where the data is coming from or going to:

Bytes Characters
Input InputStream → FileInputStream Reader → FileReader
Output OutputStream → FileOutputStream Writer → FileWriter

FileInputStream, FileOutputStream, FileReader, and FileWriter are only one family among several — ByteArrayInputStream reads bytes already sitting in memory, Socket exposes an InputStream/OutputStream pair for network data, and so on. The abstract type is what the rest of the program is written against; the concrete type is chosen once, at the point where the stream is created, based on where the data actually lives.

What FileReader Actually Is

A file on disk is never inherently text rather than bytes, so reading one as characters still means reading bytes first. FileReader doesn't skip that step; it just performs it silently, underneath, on the caller's behalf.

The class that actually performs it is a bridge: InputStreamReader wraps any InputStream — a file, a socket, anything — and decodes its raw bytes into characters according to a specified Charset (UTF-8, for instance). Written out by hand for a file, that bridge looks like this: 

Reader reader = new InputStreamReader(new FileInputStream("data.txt"), StandardCharsets.UTF_8);

new FileReader("data.txt") is, in every way that matters, doing exactly that. In the JDK's own source it isn't even composition — it's inheritance: FileReader extends InputStreamReader directly, and its constructor simply builds the FileInputStream internally and hands it to the InputStreamReader machinery it inherits. The class hierarchy is exactly that chain: 

Reader -> InputStreamReader -> FileReader
Three layers, three jobs
  • FileInputStream — the bottom layer. Talks to the OS, reads raw bytes, knows nothing about characters or encoding. This is the class that actually issues the blocking read() call §26.6 walked through.
  • InputStreamReader — the bridge. Decodes whatever bytes an InputStream hands it into characters, using a given Charset. This is where "bytes on disk" becomes "text."
  • FileReader — the convenience. Hardcodes the first two steps together for the one specific case of "I want text, from a file," so the caller doesn't have to write both lines.

The practical difference between the two shows up in exactly one place: which charset gets used. Without one specified, new FileReader("data.txt") falls back to Charset.defaultCharset() — the JVM's default charset — and what that default actually is has itself changed. Before Java 11, there was no way to override it at all for a FileReader; Java 11 added FileReader(String, Charset) so a caller finally could.

JEP 400 moved the default itself

Being able to override the default only matters if the default is worth worrying about. Before Java 18, it was: the JVM's default charset was the platform's native encoding, tied to the operating system's locale, so the same FileReader call could read the same file differently on different machines — a classic bug that only surfaces after deployment, on a machine the code was never tested against. JEP 400, in Java 18, changed the default itself to always be UTF-8, regardless of platform or locale, unless explicitly overridden. On a modern JDK, the no-argument FileReader constructor is far less of a landmine than it used to be — "default" now almost always means UTF-8. Naming the charset explicitly, either through FileReader(String, Charset) or by writing the InputStreamReader chain out by hand, still removes the question entirely rather than relying on that default holding.

Reading and Writing: The Core Methods

I/O streams are all about reading and writing data, so it shouldn't be a surprise that the most important methods are read() and write().

Both InputStream and Reader declare a read() method to pull data from a stream — a byte at a time for InputStream, a character at a time for Reader. Likewise, both OutputStream and Writer declare a write() method to push a single unit of data onto the stream. 

abstract int  read();   // InputStream — returns a byte, or -1 at end of stream
abstract void write(int b);   // OutputStream — writes a single byte

These are exactly the methods that make InputStream and OutputStream abstract in the first place, per the abstract-vs-concrete split above: the abstract class only declares that some byte can be read or written; a concrete subclass like FileInputStream is what actually implements read() against a real file. Every other operation covered in this chapter — buffering, locating a file through Path — exists to make calling these two methods more convenient or more efficient. Neither replaces them.

The photo-sharing app from earlier in this section needs exactly this pair of methods for one of its simplest features: backing up an uploaded profile picture before any further processing touches it. Nothing about that task needs buffering, character decoding, or any of the convenience this chapter builds up later — just the raw contract, called directly: 

try (InputStream in = new FileInputStream("profile-picture.jpg");
     OutputStream out = new FileOutputStream("profile-picture-backup.jpg")) {

    int b;
    while ((b = in.read()) != -1) {
        out.write(b);
    }
} // both streams closed automatically — §15.8

in.read() hands back the next byte, or -1 the moment there's nothing left — the exact sentinel the abstract signature promised. The loop's condition is the read: each pass pulls one byte, and the loop itself ends the instant that byte turns out to be the end-of-stream marker rather than real data. out.write(b) pushes that same byte straight through to the backup file, one for one.

This is correct, and it's also the slow way — a multi-megabyte photo means millions of individual read() and write() calls, each one a potential round trip into the OS (§26.6). §26.3 picks up from exactly this loop to explain why, and §26.10 shows a built-in shortcut that skips writing it by hand entirely. Neither changes what's actually happening here — they just make it faster, or shorter to write.

§26.3

Buffering

Imagine asking someone to deliver rice, and they choose to bring it one grain at a time. They walk to the storeroom, pick up a single grain, carry it to the kitchen, set it down — then turn around and walk all the way back for the next one. Technically, the rice gets delivered. But almost none of that effort is actually about the rice; nearly all of it is the walking. Do that a few thousand times and the trip itself becomes the bottleneck, not the grain.

Nobody delivers rice this way. Instead, we scoop up a bowlful in one trip and carry the whole bowl to the kitchen. The number of grains delivered hasn't changed — only the number of trips has, and that's what makes the difference.

The same tradeoff shows up in I/O. A stream created directly on a file or a socket typically talks to the operating system on every single read() or write() call, and each of those calls is expensive compared to the actual work of moving a byte: 

Without buffering:

read byte
read byte
read byte
...

Buffering collects data into a chunk first, and lets the program work through that chunk in memory, only going back to the operating system once the chunk is exhausted: 

With buffering:

read chunk
process chunk
...

This is exactly what BufferedInputStream and BufferedReader are for. Neither one changes what is being read — bytes are still bytes, characters are still characters — they only change how many round trips it takes to read them. And true to the pattern from §26.2, both are wrapped around another stream rather than replacing it: 

InputStream in = new BufferedInputStream(new FileInputStream("profile-picture.jpg"));
Reader in = new BufferedReader(new FileReader("message.txt"));

FileInputStream and FileReader still own the connection to the actual file; BufferedInputStream and BufferedReader just sit in front of them, collecting bytes or characters into a chunk before handing them onward. The rest of the program still just sees an InputStream or a Reader — it never needs to know that a buffer is doing the heavy lifting underneath.

BufferedReader doesn't stop at making the existing read() cheaper, either — having a whole chunk of characters in memory at once lets it offer an operation Reader has no way to provide on its own: readLine(), which reads and returns one entire line of text in a single call, instead of the caller assembling one character at a time. 

BufferedReader reader = new BufferedReader(new FileReader("message.txt"));
String line = reader.readLine(); // "Happy birthday!" — a whole line, one call
A different kind of buffer, already there

FileReader already keeps an internal buffer of its own, underneath — but not for the reason just covered above. UTF-8 is a variable-width encoding: one character can take anywhere from one to four bytes. The character é, for instance, is two bytes — 0xC3 0xA9. A decoder handed only 0xC3 can't produce a character from it alone; that byte just signals "a two-byte character starts here, one more byte is coming." It has to hold onto 0xC3 and wait for 0xA9 before it can hand back é. So the InputStreamReader machinery FileReader inherits (above) reads bytes in chunks internally — not to cut down on round trips, but because a decoder that only ever saw one byte at a time couldn't decode multi-byte characters at all, not just slower. That buffer exists whether or not the caller ever wraps FileReader in anything else.

So does new BufferedReader(new FileReader("foo.in")) still make sense? Yes — for two reasons that have nothing to do with what the decoder already solved. First, readLine() only exists on BufferedReader; FileReader has no way to read a whole line at once, buffer or not. Second, code that calls read() one character at a time still pays for a full method call into the decoder on every single call, even though no OS round trip is involved — BufferedReader holds its own char[] and serves most of those calls straight out of it, cheaper than re-entering the decoder each time. BufferedReader is batching at the level the caller's code operates at — characters and lines — not re-solving the byte-level problem the decoder already handled.

§26.4

Flushing

Write a document, hit save, and the laptop dies half a second later — did that last change actually make it to disk, or was it still sitting somewhere in between? Buffering, as §26.3 just showed, means a write doesn't necessarily land immediately; sometimes it's just waiting in memory for the bucket to fill.

Flushing is the act of forcing whatever is sitting in the buffer to be written to its destination — disk, network, console — immediately.

Why Flushing Is Needed

A buffer, as §26.3 introduced, accumulates data in memory and writes to the destination in chunks for efficiency. But that means data can sit in the buffer and never reach the destination if the program never flushes it: 

BufferedWriter bw = new BufferedWriter(new FileWriter("file.txt"));

bw.write("important data");

// program crashes here — buffer never flushed — file.txt is empty!
A write() is not a promise it's on disk

Calling write() only hands data to the buffer sitting in memory — it says nothing about whether that data has actually reached the file, the socket, or wherever the stream is headed. Until a flush happens, "important data" above exists only in memory, and memory doesn't survive a crash.

When Does Flush Happen?

1. Automatically

a. When the buffer is full — once the buffer has no room left, it empties itself into the destination on its own, making room for more: 

your code                buffer (memory)           destination (disk)
─────────                ───────────────           ──────────────────
write("Hello ")     →    [ Hello  ]                [ empty ]
write("World")      →    [ Hello World ]           [ empty ]
write("!!")         →    [ Hello World!! ]         [ empty ]
-- buffer full --
auto flush          →    [ empty ]            →    [ Hello World!! ]

b. When the stream is closed — close() calls flush() internally before shutting the stream down, so closing always flushes. The difference between the two is what happens to the stream afterward: 

// flush() — empties the buffer, stream stays open for more writing
bw.flush();
bw.write("more data"); // still works

// close() — flushes AND closes, no more writing
bw.close();
bw.write("more data"); // IOException — stream closed

2. Explicit flush()

A program doesn't have to wait for either of those to happen. Calling flush() directly forces whatever is currently buffered out to the destination right now, without closing the stream — useful anywhere the program needs a guarantee that data has actually left memory, such as after logging an event: 

BufferedWriter bw = new BufferedWriter(new FileWriter("log.txt"));

bw.write("Server started");
bw.newLine();
bw.flush(); // force to disk now, don't wait for the buffer to fill

bw.write("Request received");
bw.newLine();
bw.flush(); // again
§26.5

Sockets — streams that reach across a network

§26.1 opened with three reasons a program needs I/O: a banking app needing a database, a photo editor needing disk, and a chat application needing messages from the network. Every concrete stream actually built since then has connected to something local — a file on disk. It's time the third case got its due.

A socket extends the exact same I/O model across a network. Instead of connecting a stream to a file, a socket connects a stream to another program — running on the same machine, or on a completely different one, anywhere reachable over a network. Opening a connection to a chat server looks like this: 

Socket socket = new Socket("chat.example.com", 5000);

InputStream in = socket.getInputStream();
OutputStream out = socket.getOutputStream();

Notice what socket itself is not — the same rule §26.1 opened with, extended one step further. A Socket names the connection; it is not a stream any more than a Path is a file. Once connected, it hands back two ordinary streams: an InputStream for whatever the chat server sends, and an OutputStream for whatever this program sends back.

Everything already covered in this chapter still applies, completely unmodified — these two streams wrap exactly the way a file's streams do: 

BufferedReader reader = new BufferedReader(new InputStreamReader(socket.getInputStream()));
PrintWriter writer = new PrintWriter(socket.getOutputStream(), true);

reader.readLine() reads a line arriving from the chat server no differently than reading a line from a file (§26.3). writer looks unbuffered — nothing in that line mentions a BufferedWriter — but it isn't: PrintWriter's constructor quietly wraps one around the socket's OutputStream internally, the exact buffering §26.3 already covered, just hidden behind a convenience constructor instead of written out by hand.

That hidden buffer is exactly why the second argument, true, isn't optional here. It enables autoFlush (§26.4): every println forces that buffer's contents out over the network immediately, rather than leaving them to sit until the buffer happens to fill. Without it, a chat message could sit in memory indefinitely, fully "sent" as far as the calling code can tell, while the other program — the one actually waiting for it — receives nothing at all. The only real difference from everything earlier in this chapter is what sits on the other end: another running program, not a file on disk.

Two-Way Communication

A file is read from, or written to, but never both through the same stream at once. A socket is different — both programs can send and receive at the same time, through the same connection, each one's output wired directly to the other's input: 

 Program A                          Program B

            ---- OutputStream ---->
            <---- InputStream -----

Program A's OutputStream is Program B's InputStream, and Program B's own OutputStream is Program A's InputStream — the same pipe, running in both directions, at once.

A Conversation, Not a Storage Device

A useful way to hold the difference in mind: a file stores data; a socket carries a conversation. Write to a file, and the bytes sit on disk until someone eventually reads them — there's no urgency, and no one waiting. Write to a socket, and the bytes are transmitted right away to another program that may act on them the instant they arrive.

That urgency is exactly why a socket read() blocks the same way a file read() does (§26.6): when nothing has arrived yet, the thread is parked until the other program actually sends something, because — unlike a file, which already has whatever it has — there may genuinely be nothing to read until the other side decides to speak.

Once a connection exists, learning sockets turns out to be less about learning a new way to move data, and more about learning where that data is coming from and where it's going. Every stream operation this chapter already built up — reading, writing, buffering, flushing — carries over unchanged.

§26.6

What Blocking Actually Means — where read() goes when it doesn't return

Every read() call so far in this chapter has quietly assumed the same thing: it eventually returns. But picture a single-threaded program that calls inputStream.read() and the data just isn't there yet — nothing has arrived from disk or the network. Is that thread spinning in a loop, burning a core checking over and over? Is the core it was running on now sitting idle, waiting alongside it? Neither. What's actually happening is worth seeing once, concretely.

Down Into the Kernel

read() isn't ordinary Java code underneath — it's a native method, one that hands off to the operating system itself (§23.2) through a system call. The OS checks whether the requested bytes are already sitting in a buffer it controls, already copied in from disk or the network card. If they are, they're copied into the Java array and read() returns immediately — no blocking at all. Blocking only starts when the answer is no.

Parked, Not Spinning

When the data isn't ready, the OS doesn't leave the thread checking in a loop. It takes the thread off the core entirely and marks it as waiting, handing that core to the scheduler (§23.3) to give to something else — another thread, another process, anything that's actually ready to run. The waiting thread consumes zero CPU cycles the entire time; it's parked, not polling.

Blocking wastes time, not cores

The core our thread was actually running on doesn't sit there waiting alongside it. The instant the thread is parked, the scheduler hands that exact core to something else — another thread, another process, whatever's next in line — and it goes right on doing useful work. The CPU was never the thing waiting. Our thread was. For however long the wait lasts, our single-threaded program makes zero progress — not because the hardware ran out of work to do, but because the one thread that could have made that progress is parked, and there's no second thread standing by to keep going in its place.

Waking Back Up

The wait ends the moment the hardware finishes delivering the data into that kernel buffer — disk and network controllers transfer it without needing the CPU's attention at all. Once it's there, the OS marks the thread runnable again, and the scheduler eventually gives it a core — not necessarily the one it started on. The thread resumes exactly where it left off, still inside the native read() call, which copies the bytes into the Java array and finally returns.

Only then does the very next line of Java code run. There's nothing speculative about it, and nothing else the JVM could have run in the meantime — a single-threaded program has no other code to jump to, so the entire program is simply paused for exactly as long as the OS took to get the data ready.

§26.7

File System Fundamentals — what a path actually describes

Every stream so far in this chapter has read from, or written to, somewhere — but a stream doesn't invent that "somewhere" on its own. Before a single byte can move, something has to bridge the program to wherever the data actually lives on disk. That bridge is the file system: the part of the operating system in charge of organizing, storing, and retrieving data.

One Program, Any File System

Windows, macOS, and Linux each manage storage with a different underlying file system — NTFS, APFS, ext4 — with different rules for how files are organized and addressed. A Java program never has to know which one it's talking to. The JVM connects to whatever file system the host operating system provides, and exposes it through one consistent API on top — the exact same code that reads a file on Windows reads the same way on macOS or Linux, with nothing in the program itself aware of the difference underneath. That's the same portability §0.9 already established for bytecode in general, applied here specifically to storage.

Bits and Bytes, Briefly

At the lowest level, a file system stores everything as bits — each one just a 0 or a 1. A single bit is far too fine-grained to be useful on its own, so bits are grouped into 8-bit units called bytes, and a byte is the actual granularity §26.2's InputStream and OutputStream read and write at — one byte at a time, or, far more efficiently, a whole array of them in a single call.

Absolute and Relative Paths

Which file does "notes.txt" actually mean? On its own, that depends entirely on where the program happens to be running from — its current working directory. A path that doesn't say otherwise is read relative to that starting point, which is exactly why Java distinguishes two kinds of path:

  • An absolute path is the full path from the root of the file system all the way down to the file or directory — nothing left to infer.
  • A relative path is the path from the current working directory to the file or directory — meaningless without already knowing where "current" is.
Starts with Example Meaning
A forward slash, / /bird/parrot.png Absolute — / itself is the root
A drive letter, C: C:\bird\info Absolute — the drive is the root (Windows)
Anything else bird/parrot.png Relative to the current working directory

Path Symbols

Both kinds of path can contain a path symbol — one of a small, reserved set of characters with special meaning to the file system, rather than naming an actual file or directory:

Symbol Description
. A reference to the current directory
.. A reference to the parent directory

These aren't just notation to recognize by sight — §26.9's normalize() resolves exactly these two symbols programmatically, collapsing a path like /home/user/../documents down to /home/documents without ever touching the actual file system.

§26.8

Introducing NIO — buffers, channels, and selectors

Every stream this chapter has used, from FileInputStream to a Socket's own streams, follows the same one-way model §26.1 opened with: an InputStream delivers bytes from a source, an OutputStream sends bytes to a destination. That model is simple, and it's been enough for everything covered up to this point.

As software evolved, though, demands emerged that the stream model wasn't built for: a high-performance server juggling thousands of simultaneous network connections, a large file that needs transferring with as little copying as possible, an application that wants finer control over exactly how data moves between the operating system and the JVM. To address exactly this, Java 4 introduced New I/O — NIO.

NIO, not NIO.2

This section is about the original NIO — the java.nio package, from Java 4 — built around the three abstractions below. §26.9's Path and Files are a separate, later addition from Java 7, nicknamed NIO.2 mostly because it arrived in the same package family. Despite the name, NIO.2 doesn't use a Buffer, a Channel, or a Selector anywhere — it's built entirely on ordinary objects, the same way everything else in this chapter is. The two are related in name and package only, not in how they work.

NIO doesn't replace streams — every stream class this chapter already covered still exists, unchanged. It introduces a different model alongside them, built from three core abstractions.

Three core abstractions
  • A Buffer is a region of memory that temporarily holds data.
  • A Channel is a connection through which data flows, similar in spirit to a stream.
  • A Selector is an object that lets a single thread monitor many channels at once for I/O events.

Unlike streams, which are generally one-way — an InputStream or an OutputStream, never both — many channels support reading and writing through the same object.

A typical NIO operation follows the same basic pattern every time: read data from a channel into a buffer, process what's in the buffer, then write from the buffer back out to a channel. 

           read()                write()

+---------+       +--------+       +---------+
| Channel | ----> | Buffer | ----> | Channel |
+---------+       +--------+       +---------+

Reading a file this way looks like this — the same profile-picture upload from §26.2, moved through a channel and a buffer instead of a plain InputStream: 

try (FileChannel channel = FileChannel.open(Path.of("profile-picture.jpg"))) {
    ByteBuffer buffer = ByteBuffer.allocate(1024);

    channel.read(buffer); // read: channel -> buffer
    buffer.flip();        // switch the buffer from filling to draining

    while (buffer.hasRemaining()) {
        byte b = buffer.get(); // process what's in the buffer
    }
}

Separating the transport — the channel — from the storage — the buffer — is what gives this model its flexibility: the same buffer can be drained into a different channel, refilled, and reused, none of which a stream's one-directional read()/write() pair was ever built to support.

The most significant idea NIO introduces, though, is non-blocking I/O. §26.6 showed a blocking read() parking its one thread until data arrives, doing nothing else in the meantime. A Selector sidesteps that entirely: one thread registers many channels with it and asks, in a single call, which of them are actually ready — for reading, for writing, for a new connection — reacting only to the ones that are, never parked waiting on any single one. That's exactly the shape a web server or a messaging system needs: thousands of connections, most of them idle at any given moment, handled without a thread sitting blocked on each one.

Most Code Never Touches a Channel Directly

For most everyday file operations, there's no need to reach for FileChannel at all — and that's one of the main reasons NIO.2, covered next, exists in the first place: to make common tasks far simpler than the buffer-and-channel pattern above requires. Instead of writing this: 

FileChannel channel = FileChannel.open(path, StandardOpenOption.READ);
// ...allocate a ByteBuffer, read into it, decode it...

a single call does the same job:

String text = Files.readString(path);
List<String> lines = Files.readAllLines(path);
Files.copy(source, target);

Files's methods are high-level convenience methods — they may use a Channel underneath, or a stream, or call the operating system some other way entirely. Which one is an implementation detail, not something calling code needs to know or ever sees: 

     Your code

Files.readString(path)
        │
        ▼
Files implementation
        │
        ▼
Channels / Streams / OS

The two abstractions from this section, Buffer and Channel, don't disappear — they're still doing the actual work, just one layer further down than most code ever needs to look.

§26.9

NIO.2 — the modern file API

Everything so far has been about streams — connections that data flows through. But before a program can open a stream on a file, it first has to say which file. That's a separate question, and it deserves its own story, centered on a single type: Path.

Introduce: What Is a Path?

The old way of naming a file looks like this:

File file = new File("notes.txt");

The modern way looks similar on the surface, but is built on a different idea: 

Path path = Path.of("notes.txt");
A path is not a file

A Path is a description of where something exists — a location, expressed as a sequence of names, nothing more. It doesn't open anything, lock anything, or guarantee that the file is even there. It is to a file what an address is to a house: writing the address down doesn't mean the house exists, and it doesn't move anyone into it.

Sit with that for a moment against the line above: path doesn't represent a file — it represents where a file could be. notes.txt might not even exist. A Path can just as easily be built pointing at:

  • an existing file,
  • a non-existent file,
  • a directory,
  • or a symbolic link.

Path.of(...) doesn't check which of these it is, or whether it's any of them at all — it simply describes a location in the file system, nothing more.

That same "just a location" idea means a Path doesn't know, or care, whether it points to a file or a directory. It's nothing more than a sequence of name components — "some/folder/file.txt" or "mydir/" are both equally valid paths, and Path makes no distinction between them.

A .txt name doesn't make it a file

/data/zoo.txt could just as easily be a directory — nothing about a Path's text guarantees what it actually is on disk, or that anything is there at all. Whether a given Path refers to a file or a directory is something to check, with Files (later in this section), never something to assume from how the name is spelled.

Create: Path.of() vs. Paths.get()

Path is an interface — and an interface can never be instantiated with new. So how does a line like Path.of("notes.txt") actually produce an object?

When NIO.2 was first introduced, the standard way to hand back an instance of an interface was a separate, plural-named utility class holding the factory method, since the interface itself couldn't hold one. That's how Paths.get() was born, and for years it was the only way to create a Path: 

// Java 7–10 — had to go through the Paths factory class
Path p = Paths.get("file.txt");

// Java 11+ — Path has its own factory method
Path p = Path.of("file.txt");

Java 8 changed the premise that pattern rested on, by letting interfaces carry actual method bodies — not just declarations — in two forms: default methods, run on an instance, and static methods, run on the interface type itself, exactly the way Path.of(...) is called on Path rather than on some separate factory class. (Java 9 later added a third form, private methods, for reusable logic internal to the interface — not covered elsewhere in this tutorial, but part of the same shift.) Java 11's Path.of() is a direct product of that shift: rather than adding another method to a Paths-style helper class, the factory method was placed directly on Path itself, as part of a broader effort across the language to reduce exactly this kind of "utility class pollution" — one interface, one place to look, instead of an interface plus a same-purpose-different-name class beside it. Paths still exists, but is largely a relic of the pattern Path.of() replaced.

So Path.of("notes.txt") hands back an object — but an object of what actual class? Not Path itself; an interface has no object of its own, only whatever concrete class implements it (§14). On Linux or macOS that's a class called sun.nio.fs.UnixPath; on Windows, it's sun.nio.fs.WindowsPath instead. Neither one is something a program is ever meant to reference, or even likely to encounter by name:

  • They are not part of the public Java API — the official Javadoc only documents exported public API, and these classes aren't in it.
  • They live in a package beginning with sun.*, which has never been part of the supported Java SE API — a naming convention that itself signals "implementation detail, not public contract."
  • Since the module system arrived in Java 9, they're harder to reach still: the java.base module doesn't export sun.nio.fs at all, so code outside the JDK can't normally access those classes even by name.

The only place to actually see them is the JDK's own source — OpenJDK ships files like src/java.base/unix/classes/sun/nio/fs/UnixPath.java and src/java.base/windows/classes/sun/nio/fs/WindowsPath.java, each implementing the same Path interface against a completely different underlying file system.

That invisibility is exactly the point, and it's the same "program to an interface" payoff §27.8 makes elsewhere: every line of code in this chapter has been written against Path, never against UnixPath or WindowsPath by name. If a future JDK rewrote the file system implementation from scratch, any program written against Path wouldn't need to change at all — only code that had reached past the interface, into sun.*, could possibly break.

Operate: Working with an Existing Path

Every operation on a Path follows the same rule: none of them change the Path they're called on. resolve() is the simplest example — it's easy to misread a line like p1.resolve("README.md") as changing p1, but it doesn't. It returns a new Path, leaving the original exactly as it was: 

Path p1 = Path.of("/users/mk/dev/java");
Path p2 = p1.resolve("README.md");
// p1 is unchanged; p2 is a new object

System.out.println(p1); // /users/mk/dev/java
System.out.println(p2); // /users/mk/dev/java/README.md

That's the same immutable-value discipline §12 built up for classes like String: a Path can be handed to another method, stored, or shared across threads without any risk that something else silently changes what it points to. Path also plugs directly into streams (§22) through methods like Files.lines() and Files.walk(), which is not something File was ever built to support.

normalize() is a second operation with the same shape, and it earns its keep because a Path is only text — nothing stops that text from containing redundant pieces: a stray current-directory marker, a detour up and back down the tree, a doubled separator. normalize() cleans exactly that up, without ever touching the file system. Nothing is read, moved, or checked for existence — it's pure text processing on the Path itself, and it handles three kinds of redundancy:

Redundancy Example
A . ("current directory") component — doesn't change the effective path, so it's simply dropped /home/user/./documents → /home/user/documents
A .. ("parent directory") component — resolved by moving back up the tree, when possible /home/user/../documents → /home/documents
Doubled directory separators — condensed to one /home//user → /home/user 
Path p = Path.of("/home/user/../documents/./notes.txt");
Path cleaned = p.normalize();

System.out.println(p);       // /home/user/../documents/./notes.txt
System.out.println(cleaned); // /home/documents/notes.txt

A directory and a .. right after it don't just move up the tree — they cancel each other out completely, since going into armadillo and immediately back out leaves nothing behind: 

var p1 = Path.of("./armadillo/../shells.txt");

System.out.println(p1); // ./armadillo/../shells.txt

// normalize: remove redundant elements from the path
System.out.println(p1.normalize()); // shells.txt

// notice immutability!
System.out.println(p1); // ./armadillo/../shells.txt

The leading ./ is dropped as a redundant current-directory marker, and armadillo/.. collapses to nothing at all — leaving just shells.txt. And exactly as with resolve(), p1 itself never changes: normalize() only ever hands back a new Path.

.. beyond the root

Resolving .. assumes there's a parent to move up to. Whether that assumption holds when .. would climb past the root — and what normalize() does about it — depends on the underlying file system and on whether the Path is absolute or relative to begin with.

Acting On a Path: Files

Everything above only describes a location, or reshapes that description — none of it touches the file system. Actually doing something with that location — reading it, copying it, deleting it — is a separate step, handled by the Files utility class: 

String contents = Files.readString(path);
Files.copy(source, target);
Files.delete(path);

Files is built the same way as Collections or Arrays: nothing but static methods, a private constructor, and no reason to ever be instantiated.

Splitting "where something is" (Path) from "what to do with it" (Files) isn't just a cleaner replacement for the old File class — it's what actually makes two things true at once that couldn't otherwise both hold.

First, Path isn't backed by one single class. Path.of(...) can hand back a UnixPath, a WindowsPath, or — since NIO.2 supports pluggable file systems — a path into a zip archive, an in-memory file system, or any custom provider. If reading, writing, and deleting lived as instance methods directly on Path, every one of those implementations, present and future, would have to reimplement that entire operation surface itself. Centralizing them in Files means there's exactly one place those operations are defined; Files simply delegates each call to whichever file system provider actually owns the Path it was handed. A brand-new file system gains the entire API for free just by implementing that provider interface, without Path itself changing at all.

Second, keeping operations off Path is what makes the earlier "a path is not a file" callout actually true, rather than just a nice description. The moment reading or deleting becomes a method on Path, holding a Path object means holding something that can fail with an IOException or delete a real file the instant a method is called on it — exactly the side effect a pure location description was supposed to be free of. Path stays an immutable value, the same discipline §12 built up for String: it describes a location without ever acting on it. Only once a Path is handed to Files does the program actually touch the file system. (If NIO.2 were being designed today, something like path.createFile() might well exist as an instance method — but it shipped in 2011, in Java 7, a full year before default and static methods made that possible at all. By the time the language could have supported it, Path and Files were already public API.)

NIO.2 keeps that same split for creating things, too — distinct Files methods for distinct kinds of entries, rather than one method that tries to guess what was meant.

Creating Files

To actually create a file on disk, use Files.createFile(Path): 

Files.createFile(Path.of("/home/user/notes.txt"));

Two failure modes are worth knowing by name: it throws FileAlreadyExistsException if the file already exists, and NoSuchFileException if the parent directory doesn't.

Creating Directories

Directories have two creation methods, and the difference between them is exactly whether the parent folders are expected to already be there.

Files.createDirectory(Path) creates only the last directory in the path — a single step, and it assumes every segment before that last one already exists: 

Path.of("/home/user/a/b/c")
//        ▲     ▲  ▲ ▲   ▲
//   must already exist  ┇
//                       ┖ only this gets created
Files.createDirectory(Path.of("/home/user/a/b/c"));

// if /home/user/a/b exists     — creates c
// if /home/user/a/b is missing — NoSuchFileException

It throws NoSuchFileException if any parent is missing — createDirectory() assumes everything except the last segment is already there — and FileAlreadyExistsException if the target directory already exists.

Files.createDirectories(Path) (plural) takes the opposite stance: it creates the target directory and every missing parent along the way, similar to mkdir -p on Unix. It's also idempotent — calling it on a path that already exists doesn't fail, it just silently succeeds: 

Files.createDirectories(Path.of("/home/user/a/b/c"));

// creates a, b, and c — whichever of them don't already exist
// succeeds silently even if all three already exist

The singular/plural naming is a genuine clue to the contract: createDirectory() is a single step that trusts the caller to have the parent ready; createDirectories() is willing to do all the intermediate work itself.

Comparing Files: Same File, or Same Content?

A file system location can be written more than one way — a redundant ., a detour through .., a symbolic link — so two Path objects with different text can still point at the exact same file. That means equals() can't be trusted to answer "is this the same file," because equals() never looks at the file system at all — it only compares the path's text, element by element:

Path Value
p1 /users/mk/brics.csv
p2 /users/mk/./brics.csv
p3 /users/mk/dev/../brics.csv 
Path p1 = Path.of("/users/mk/brics.csv");
Path p2 = Path.of("/users/mk/./brics.csv");
Path p3 = Path.of("/users/mk/dev/../brics.csv");

System.out.println(p1.equals(p2)); // false

try {
    System.out.println(Files.isSameFile(p1, p2)); // true
} catch (IOException e) {
    e.printStackTrace();
}

All three paths above resolve to the exact same location once normalized, but p1.equals(p2) still returns false — equals() sees "/users/mk/brics.csv" and "/users/mk/./brics.csv" as different strings, full stop, regardless of what either one actually points to on disk.

Files.isSameFile(Path, Path) is the method built to answer the question equals() can't: it resolves path symbols, follows symbolic links, and checks the file system itself to determine whether two Path objects name the same location — despite the name, that includes two Paths that both refer to the same directory, not only files.

Identity and content are independent questions, though — a path can name the right file without telling you what's inside it, and matching content doesn't require it to be the same file at all:

Question Method
Identity — is this the same file? Files.isSameFile(Path, Path)
Content — do they hold the same bytes? Files.mismatch(Path, Path)

Files.isSameFile() answers identity, as above — a common real-world use is detecting that two different-looking paths are actually symlinks pointing at the same underlying file.

Files.mismatch(Path, Path), introduced in Java 12, answers content instead: it compares two files byte by byte and reports the first position where they differ. It returns -1 if the files are identical; otherwise the returned index falls somewhere in the inclusive range from 0L up to the size, in bytes, of the smaller of the two files.

Summary: Four Types, Four Jobs

Type Purpose
java.io.File The old I/O representation of a location in the file system. A concrete, mutable class that bundles "where" and "what to do with it" together.
java.nio.file.Path The NIO.2 representation of a location — a file or a directory. An interface, and effectively immutable: no operation changes a Path, every operation returns a new one.
java.nio.file.Paths A factory class for creating a Path from a string or URI — largely superseded once Path got its own static factory method.
java.nio.file.Files A utility class of static methods that operate on a Path — reading, writing, copying, deleting, and more.
§26.10

Closing Streams Safely — try-with-resources

Every concrete stream this chapter has introduced — FileInputStream, FileReader, BufferedWriter, and the rest — is a resource that has to be closed once a program is done with it, the same way §26.4 showed a buffer has to be flushed. And the same problem shows up: if an exception is thrown between opening the stream and closing it, a hand-written close() call never runs, and the file handle or socket leaks. 

// Bad — close() never runs if reading throws
FileInputStream in = new FileInputStream("data.txt");
in.read();
in.close();

This is exactly the problem §15.8's try-with-resources exists to solve, and every stream in this chapter is already built to use it: InputStream, OutputStream, Reader, and Writer all implement Closeable, so any of their concrete subclasses can be declared directly inside a try's parentheses: 

try (FileInputStream in = new FileInputStream("data.txt")) {
    in.read();
} // in.close() runs automatically, even if read() throws

Copying from one file to another needs two streams open at once — one to read, one to write — and both can be declared the same way, separated by a semicolon: 

try (FileInputStream in = new FileInputStream("source.jpg");
     FileOutputStream out = new FileOutputStream("copy.jpg")) {
    in.transferTo(out);
} // out.close() runs first, then in.close() — reverse of declaration order

Nothing about I/O changes how try-with-resources behaves — §15.8 already covers the mechanics in full, including what happens when close() itself throws while unwinding from another exception. What's worth remembering here is narrower: every stream type this chapter has built up is a Closeable, which is exactly what makes it a safe fit for this syntax without any extra work.

§26.11

Handling Methods That Declare IOException

Unlike some modern APIs that try to hide checked exceptions, java.nio.file.Files leans heavily into them: it's almost entirely composed of static methods that throw IOException. That design choice reflects a reality §15.5 already named — a checked exception is the compiler's way of forcing acknowledgment of a failure the caller can't ignore — and file system operations are exactly that kind of failure: they depend on hardware, permissions, and network state a program cannot fully control or predict. They are, in a word, brittle.

The old java.io.File API took the opposite approach. Methods like delete() or renameTo() returned a plain boolean — true or false. That was widely criticized, because a false told you the operation failed and nothing else: not whether the file was missing, locked, or permission-denied, just failure with no explanation. NIO.2's move to IOException fixes exactly that: a thrown exception carries a detailed message and a stack trace explaining what actually went wrong.

Methods that access or change files and directories — most of what Files offers — declare IOException for this reason. But not every method does. Files.exists() is a telling exception to the rule: it returns a plain boolean and declares nothing. If it threw an exception the moment a file didn't exist, it could never actually return false — the one answer the method exists to give.

A rule of thumb

If an NIO.2 method declares IOException, it usually requires the paths it operates on to already exist. A method whose whole job is to answer whether something exists — like Files.exists() — has no business throwing just because the answer is no.

Common causes of a Files method throwing IOException include:

  • Loss of communication to the underlying file system.
  • A file or directory is required, but does not exist.
  • A file or directory exists, but cannot be accessed or modified.
  • A file exists, but cannot be overwritten.
