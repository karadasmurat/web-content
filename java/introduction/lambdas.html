Chapter 20

Functional Programming

Some languages let a function travel as casually as a number does — stored, passed, handed back, built on the spot. Java's answer looks different underneath but reaches the same place. This chapter opens with what that capability actually means, then builds Java's version of it from the ground up: §14's Strategy pattern, shrunk twice more, down to a single arrow.

§20.1

Functions as First-Class Citizens — what it means for behavior to be a value

In some programming languages, functions are first-class citizens. This means they are treated like any other value. A function can be assigned to a variable, passed as an argument to a method, returned from another function, or stored in a data structure.

Capability What it means for behavior specifically
Stored in a variable A variable can hold a piece of behavior, the way it already holds a number or a reference
Passed as an argument A method can receive a piece of behavior from its caller, to run later, on the caller's terms
Returned from a method A method can hand back a piece of behavior as its result, not just an ordinary value
Created at runtime A new piece of behavior can be built while the program is running, from data only available then — not just fixed in advance as source code
Not every language wraps behavior the same way

Java takes a different approach. Methods are not values and cannot exist independently of a class or object. You cannot assign a method directly to a variable or pass a method itself as an argument. Instead, Java represents behavior through objects.

§20.2

Java "Simulating" First-Class Functions

From its earliest versions, Java has supported passing behavior to methods. The traditional approach was to implement an interface, create an object, and pass that object to the code that needed the behavior.

§0.6 already stated Java's one unbending rule: there are no free-standing functions. Every method belongs to a class. So when Java eventually needed to support the four capabilities §20.1 just listed, it couldn't add a raw function value floating outside the object system — that would break the rule the language never bends anywhere else. Instead, it reused what already existed: objects, and the interfaces (§14) that describe what an object can do.

The trick is narrow and specific: wrap the behavior in an object whose type is an interface. A functional interface defines a single operation, and a lambda expression provides the implementation of that operation. The resulting object can then be passed, returned, or stored just like any other object.

It's worth keeping three things separate here, since the syntax is about to make them look like one: 

DiscountStrategy tenPercentOff = price -> 0.9 * price;

DiscountStrategy is the type — an interface. There's an actual object on the heap that tenPercentOff's address points to. The type and the value are visible in the source code; the object is what's actually running.

Put together: Java never added functions to the language. It added a compact syntax for creating single-method objects. Practically, the result covers the same four capabilities §20.1 opened with — store it, pass it, return it, build it on the spot — just always through that one extra layer of an object underneath.

Here is a clean explanation which separates the contract (the interface) from the behavior (the implementation), which is exactly how Java's type system works:

  1. Declare a functional interface which describes the contract for a single operation.
  2. Provide an implementation of that operation (using a named class, an anonymous class, or a lambda expression).
  3. Create an instance implementing the functional interface.
  4. Pass the reference to that object wherever the behavior is needed.
§20.3

The Problem — a rule that doesn't need a name

§14.11 built DiscountStrategy, Order, and two named implementations — PercentageDiscount and FlatDiscount — around the Strategy pattern: 

interface DiscountStrategy {
    double apply(double price); // returns the discounted price, not the discount itself
}

class PercentageDiscount implements DiscountStrategy {
    private final double rate;   // a fraction, e.g. 0.10 for 10% off

    PercentageDiscount(double rate) {
        this.rate = rate;
    }

    @Override
    public double apply(double price) {
        return (1 - rate) * price;
    }
}

class FlatDiscount implements DiscountStrategy {
    private final double amount;

    FlatDiscount(double amount) {
        this.amount = amount;
    }

    @Override
    public double apply(double price) {
        return price - amount;
    }
}

class Order {
    private double total;
    private DiscountStrategy discount;

    Order(double total, DiscountStrategy discount) {
        this.total = total;
        this.discount = discount;
    }

    void setDiscount(DiscountStrategy discount) {
        this.discount = discount;
    }

    double finalPrice() {
        return discount.apply(total);
    }
}
A problem worth fixing first: finalPrice() isn't a record, it's a live query

Look at what finalPrice() actually does: it reads the current discount field and recomputes, every single time it's called. Call it once, call setDiscount(...), then call it again, and the exact same Order reports two different prices for the exact same sale — with nothing in the object recording which answer was ever actually charged. That's precisely the trap §12.3 warned about: a reference whose meaning depends on when you ask. A real order is a business record — the kind of thing an invoice, a receipt, or an audit needs to point back to later and get one fixed, permanent answer from, the same way a Transaction (§12.4) never needs to represent an updated version of itself. Order, as written in §14.11, can't make that promise.

The fix is exactly §12.4's playbook: compute the result once, store it in a final field, and stop offering any way to change it after construction. Wanting a different discount no longer means mutating an existing order — it means constructing a new one, the same way ImmutablePoint.translate(...) (§12.4) returns a new point instead of moving the old one. And the specific fact worth freezing is discountAmount, not finalPrice — a real sales document lists the discount itself as its own line, the same way an invoice shows a subtotal, a discount, and a total as three separate figures: 

class Order {
    private final double total;
    private final double discountAmount;

    Order(double total, DiscountStrategy discount) {
        this.total = total;
        this.discountAmount = total - discount.apply(total);   // apply() returns the discounted price; subtract from total for the amount saved
    }

    double finalPrice() {
        return total - discountAmount;
    }
}

No setDiscount. discountAmount is computed once, from discount.apply(total), at the exact moment the order is placed — never again afterward. finalPrice() is now a derived value, not a stored one, but that's no longer a problem: it's derived from two final fields that can never change, so it returns the exact same answer no matter how many times it's called, no matter what happens anywhere else in the program afterward. This is the version of Order the rest of this chapter builds on.

With Order fixed, PercentageDiscount and FlatDiscount are ready to explore in other syntactic forms — starting with a mechanism already introduced once before, back in §14.6.

§20.4

Another Anonymous Class — the same route, a different interface

§14.6 introduced the anonymous class for exactly this shape: declare the implementation and create its one instance in the same expression, with no class name at all. Nothing about that mechanism was specific to Renewable — it applies to DiscountStrategy just as directly. Here are both discounts from §14.11 again, written as anonymous classes instead of top-level named ones: 

DiscountStrategy tenPercentOff = new DiscountStrategy() {
    @Override
    public double apply(double price) {
        return (1 - 0.10) * price;
    }
};

DiscountStrategy fifteenOff = new DiscountStrategy() {
    @Override
    public double apply(double price) {
        return price - 15;
    }
};

Order order1 = new Order(100.0, tenPercentOff);
Order order2 = new Order(100.0, fifteenOff);
System.out.println(order1.finalPrice()); // 90.0
System.out.println(order2.finalPrice()); // 85.0

Neither PercentageDiscount nor FlatDiscount needed to exist as its own top-level class for either of these to work — Java generates an unnamed class implementing DiscountStrategy right there, at each new DiscountStrategy() { ... } expression, and that single expression both declares the class and creates its one instance. tenPercentOff's declared type is still DiscountStrategy — everything about substitutability from §14.11 applies exactly as before.

It's shorter than declaring PercentageDiscount and FlatDiscount as their own top-level classes, but still not short. Writing out new DiscountStrategy() { @Override public double apply(double price) { return (1 - 0.10) * price; } } is a lot of ceremony — a class header, an annotation, a method signature repeated in full, braces upon braces — to say one thing: ninety percent of price. Everything from here shrinks exactly that ceremony, without changing what the code actually does.

§20.5

Functional Interfaces — the property that unlocks shrinking

Look closely at what DiscountStrategy actually declares: exactly one method, apply(double). Not two, not zero — one. That single fact has a name: an interface declaring exactly one abstract method — one method with no body, the kind that's been the only kind since §14.3 — is a functional interface. Default methods (§14.8) and static methods (§14.10) don't count against that total, because both already carry a body; only the unimplemented signatures do.

Renewable from §14.7 qualifies too — one abstract method, renew() — which is exactly why an anonymous class was enough to implement it in a single expression back there. An interface with two unrelated abstract methods, by contrast, would not qualify; there'd be no single method for a shorthand notation to unambiguously refer to.

Java lets a functional interface declare that fact explicitly, with an annotation: 

@FunctionalInterface
interface DiscountStrategy {
    double apply(double price); // returns the discounted price, not the discount itself
}

@FunctionalInterface changes nothing about how the interface behaves — DiscountStrategy was already a functional interface the moment it was written back in §14.11, annotation or not. What the annotation buys is a compiler check: if a second abstract method is ever added by mistake, breaking the single-method guarantee, the compiler reports an error right there, at the interface declaration, instead of the mistake surfacing later as a confusing error somewhere a lambda tries and fails to implement it.

Why the count matters

Everything in §20.6 and §20.7 depends entirely on an interface having exactly one abstract method. With exactly one, the compiler always knows which method any shorthand notation is standing in for — there's nothing else it could mean. With two or more, that guarantee disappears, and Java has no way to guess which one you intend. A functional interface isn't a special kind of interface, syntactically — it's an ordinary interface that happens to satisfy a count that makes a shorter notation unambiguous.

§20.6

Lambda Expressions — the anonymous class, minus everything inferable

Because DiscountStrategy is a functional interface, its one abstract method can be implemented with a lambda expression — a parameter list, an arrow, and a body, with the interface itself supplied nowhere at all: 

DiscountStrategy tenPercentOff = price -> (1 - 0.10) * price;

Compare this, piece by piece, against §20.4's anonymous class version of the exact same rule. Gone: new DiscountStrategy(), the @Override, the method name apply, the declared parameter type double, the return type, the braces, the return keyword. What's left is only what §20.4's version didn't already say some other way: the parameter's name, price, and the one expression computed from it, (1 - 0.10) * price.

None of what vanished was actually optional information — it's all still there, just inferred instead of written. tenPercentOff's declared type, DiscountStrategy, tells the compiler exactly which method is being implemented and what its signature is; that's called target typing. Knowing the target, the compiler already knows the method is named apply, takes one double, and returns a double — none of that needs to be repeated by the code that implements it.

A lambda body with more than one statement needs braces and an explicit return, exactly like an ordinary method body would: 

DiscountStrategy loyaltyDiscount = price -> {
    double discounted = price - 25;
    return Math.max(discounted, 0);   // never a negative price
};

§14.11's other named implementation shrinks the identical way, and both now work side by side with the one already declared above: 

DiscountStrategy fifteenOff = price -> price - 15;

Order order1 = new Order(100.0, tenPercentOff);
Order order2 = new Order(100.0, fifteenOff);
System.out.println(order1.finalPrice()); // 90.0
System.out.println(order2.finalPrice()); // 85.0

Notice what Order itself needed: nothing beyond the fix already made earlier in this section. Order was already written entirely in terms of the DiscountStrategy interface (§14.11), so it has no way to know, or care, whether the object behind that reference came from a named class, an anonymous class, or a lambda — a different Order per discount, frozen at construction, is all either one needs.

A lambda is still an object ← §2.2

A lambda expression is not a free-floating function — Java has no such thing. price -> (1 - 0.10) * price compiles to an instance of some class implementing DiscountStrategy, allocated on the heap exactly like every other object this tutorial has built, with tenPercentOff holding its address exactly like any other reference variable (§2.3). The lambda is Java generating that class and that one instance for you, from less syntax than an anonymous class needs — not a different mechanism, the same one, written more compactly.

§20.7

Method References — when the body is just another method's name

Sometimes a lambda's entire body is a call to a method that already exists, unchanged, with matching parameters: 

class Discounts {
    static double flatFifteen(double price) {
        return price - 15;
    }
}
DiscountStrategy fifteenOff = price -> Discounts.flatFifteen(price);

price is accepted, then immediately handed off to flatFifteen with nothing else done to it. Whenever a lambda's job is exactly that — forward every parameter, unchanged, to an existing method — a method reference names the method directly, with :: in place of a parameter list and a body: 

DiscountStrategy fifteenOff = Discounts::flatFifteen;

Read Discounts::flatFifteen as "use this method as the implementation." The compiler checks that flatFifteen's own signature — one double parameter, a double return — matches what DiscountStrategy.apply requires, exactly the same target-typing check a lambda goes through, and generates the connecting code automatically.

A method reference works the same way whether the method is static or belongs to a specific object:

Form Example
Static method Discounts::flatFifteen
Instance method, on one particular object existingOrder::finalPrice
Instance method, on whichever object is passed in String::toUpperCase
Constructor ArrayList::new

All four forms follow the identical rule: name the method (or constructor), and let the compiler match it against the target functional interface's single abstract method. A method reference is never anything a lambda couldn't already do — it's a lambda whose entire body was already just another method's name, written the short way.

§20.8

java.util.function — functional interfaces Java already wrote

DiscountStrategy is a functional interface built by hand, for one specific job. Most shapes a lambda needs to fit are common enough that Java ships them already, generic (§16) over whatever types are actually involved, in the java.util.function package:

Interface Method Shape
Function<T, R> R apply(T t) takes a T, returns an R
Predicate<T> boolean test(T t) takes a T, returns a yes/no answer
Consumer<T> void accept(T t) takes a T, returns nothing
Supplier<T> T get() takes nothing, produces a T

DiscountStrategy is, structurally, nothing more than a Function<Double, Double> wearing its own name: 

Function<Double, Double> tenPercentOff = price -> (1 - 0.10) * price;
System.out.println(tenPercentOff.apply(100.0)); // 90.0

Same lambda, same .apply(...) call — the only difference is which interface it's satisfying: a homemade one, or one the standard library already declared. Writing a dedicated interface like DiscountStrategy is still worth it when the name itself carries meaning in the domain; reaching for Function<T, R> instead is usually right when the shape is all that matters and a project-specific name would add nothing.

One of these interfaces closes a gap left open all the way back in §19: Comparator<T>, whose single abstract method, compare(T a, T b), decides ordering between two elements — exactly the piece §19.6 deferred to "later": 

List<String> names = new ArrayList<>(List.of("Priya", "Alice", "Marco"));

names.sort((a, b) -> a.compareTo(b));
// or, since that lambda is just forwarding to an existing method:
names.sort(String::compareTo);

System.out.println(names); // [Alice, Marco, Priya]

Nothing here is new mechanism — a lambda, then the identical logic shrunk to a method reference, exactly as §20.6 and §20.7 built them, now handed directly to List.sort as the ordering rule.

§20.9

Key Takeaways

  • The same shape — one interface, one algorithm needed — can be implemented five ways: a named class, an anonymous class (§14.7), a lambda expression, a method reference, or a standard java.util.function interface instead of a homemade one. All five compile to the same thing: an object implementing the target interface.
  • A functional interface is an interface with exactly one abstract method. That single-method count is what makes lambdas and method references unambiguous — the compiler always knows which method a shorthand expression is implementing.
  • A lambda expression is an anonymous class implementation with everything the compiler can already infer — the interface name, @Override, the method name, parameter types, the return type — left unwritten. It is still, underneath, an ordinary object (§2.2).
  • A method reference (Type::method) is a lambda whose entire body was already just a call to some other existing method, forwarding its parameters unchanged.
  • java.util.function ships generic, ready-made functional interfaces — Function, Predicate, Consumer, Supplier, and others like Comparator — for the shapes that come up often enough not to deserve a new interface every time.
