Chapter 5

How Memory Works

Every diagram since §2 has quietly relied on two regions of memory working differently from each other — a slot that holds a value directly, and a slot that holds an address pointing somewhere else. This chapter makes that model explicit: what the stack and the heap actually are, what a method call does to them, and what a return leaves behind.

§5.1

The Problem — two kinds of slot, never named

Every memory diagram since §2 has drawn two separate boxes side by side, labeled "Stack" and "Heap," without ever stopping to say what those words mean or why the split exists. A reference variable's diagram always shows an arrow leaving the left box and landing in the right one — never the other way around, never an arrow between two boxes on the same side. That's not an accident of how the diagrams happened to get drawn. It's a real fact about how the Java Virtual Machine organizes memory, and it's the reason a handful of behaviors that otherwise look arbitrary — why a local variable disappears the instant its method returns, why passing an object to a method lets that method change it, why passing an int never does — all follow from the exact same underlying model.

This chapter makes that model explicit. §5.2 names the two regions and what each one is for. §5.3 follows a chain of nested method calls to see the stack grow and shrink in real time. §5.4 puts both regions to work together in one full trace, watching an object get created, mutated, and handed back across a method boundary — the exact mechanics that every reference-type diagram in this book has been quietly assuming all along.

§5.2

Stack vs. Heap — two regions, two jobs

A running Java program keeps its memory in two separate regions, and every value the program ever touches lives in exactly one of them.

The Stack

Every thread (§23) gets its own stack — organized so that only the most recently started method call's variables are ever active at once, and every one of them disappears, automatically, the moment that call finishes. §5.3 opens that structure up call by call to see exactly how; what matters for now is simpler: every local variable, primitive or reference, lives in a slot that belongs to one specific method call, and that slot exists for exactly as long as that call is running — no longer.

The Heap

Every object ever created with new — regardless of which method created it — lives on the heap, one single region shared by every thread in the program. Unlike a local variable's slot, a heap object's lifetime has no fixed relationship to any particular method call: an object can be created inside a method and still be alive long after that method has returned, as long as something, somewhere, still holds its address. Nothing removes a heap object on a schedule the way a finished call's variables disappear. Java's garbage collector reclaims a heap object only once it becomes unreachable — no reference to it left anywhere in the program (§9.6 already showed this exact moment happening to a discarded String).

Where a Value Actually Lives

§2.3 already drew the distinction that matters most: a primitive variable holds its value directly, in whichever slot declares it; a reference variable holds an address, pointing to an object that lives somewhere else. What §2.3 left unsaid is where that "somewhere else" is, and where the variable's own slot lives in the first place — and the answer to both is now available. A local variable's slot — primitive or reference — is always a slot belonging to whichever method call is currently running (§5.3 shows exactly what that slot is part of). What a reference slot points at is always an object on the heap:

Stack
discountPercent = 20
order = 0x9000
Heap
Order @ 0x9000
id1042
total250.0
discountedfalse

discountPercent holds its value, 20, directly — there's nothing to follow, nowhere else to look. order holds an address, 0x9000, and the actual Order — its id, its total, its discounted flag — sits on the heap, reachable only by following that address. Both slots are on the stack; only one of them holds something that lives there too.

§5.3

The Call Stack — how nested method calls actually happen in memory

§4.5 opened a single stack frame to see where this and a method's parameters actually live. Real programs are rarely one call deep — a method calls another method, which calls another, and so on. This section follows one such chain from the moment main starts to the moment it ends, watching the stack grow with every call and shrink again, on its own, as each one finishes.

A Tower of Crates

Picture the loading dock behind a bookstore's stockroom. Orders arrive one at a time, and each one gets its own crate, set down on top of whatever's already stacked there — never underneath, never wedged into the middle. Whoever is working the dock can only ever reach the crate sitting on top; that's the one being handled right now. Everything below it is still open, still mid-task, still waiting — but it has to wait its turn until the crate above it is finished and lifted away. Once a crate is done, it comes off the top, and whatever was directly underneath is exposed again, exactly as it was left. The stack only ever changes at the top.

A running program keeps a stack exactly like that one, per thread — the diagrams below label it "Main Thread," since every Java program starts running on one, before any others (§23) ever exist. Every method call sets down a new crate — a stack frame — holding that call's own parameters and local variables. Every method return lifts the topmost crate off, and whatever called it picks up exactly where it left off.

Three Calls Deep

A payroll run turns a timesheet into a paycheck through three methods, each calling the next — and each asking for only what it actually needs, not everything the call before it had: 

public class Payroll {

    public static void main(String[] args) {
        double takeHomePay = calculatePaycheck(38, 24.00);
        System.out.println("Take-home pay: $" + takeHomePay);
    }

    static double calculatePaycheck(int hoursWorked, double hourlyRate) {
        double grossPay = hoursWorked * hourlyRate;
        double netPay = withholdTax(grossPay);
        return netPay;
    }

    static double withholdTax(double grossPay) {
        double taxableAtLowRate = Math.min(grossPay, 600.00);
        double taxableAtHighRate = grossPay - taxableAtLowRate;
        double tax = taxableAtLowRate * 0.10 + taxableAtHighRate * 0.25;
        return grossPay - tax - 25.00;
    }
}

main calls calculatePaycheck, which calls withholdTax — three crates deep at its peak. hoursWorked and hourlyRate play genuinely different roles — a count and a rate, multiplied to produce an amount — but withholdTax doesn't need either one individually; by the time the call reaches it, they've already done their job and collapsed into a single dollar figure. Follow the stack through the whole call, one crate at a time.

Step 1 — main starts running. One frame, and it's still empty: takeHomePay isn't declared until the next line executes.

Stack — Main Thread
main(String[]) frame
— nothing declared yet —

Step 2 — main calls calculatePaycheck(38, 24.00). A new crate goes on top; its parameters are filled in immediately, as part of the call itself:

Stack — Main Thread
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
main(String[]) frame
— nothing declared yet —

Step 3 — calculatePaycheck multiplies its two parameters into grossPay, then calls withholdTax(grossPay). A third crate lands on top, holding just one number — not hoursWorked and hourlyRate, which withholdTax was never given and doesn't need. Its parameter happens to share the name grossPay with the local variable one crate down, but that's just a coincidence of naming, not a connection:

Stack — Main Thread
withholdTax(double) frame
grossPay = 912.0
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
grossPay = 912.0
main(String[]) frame
— nothing declared yet —
Same name, different crates

calculatePaycheck's grossPay and withholdTax's grossPay share a name and, right now, a value — but they are not the same slot. Copying 912.0 into the new frame (§4.2's parameter-passing rule) is exactly as independent as copying it into any other local variable. If withholdTax reassigned its own grossPay, calculatePaycheck's copy, one crate down, wouldn't notice.

Step 4 — withholdTax works through a small ladder of rates: the first $600 of grossPay is taxed at 10%, everything above that at 25%, and a flat $25.00 insurance premium comes off on top. Along the way it declares locals of its own — taxableAtLowRate, taxableAtHighRate, tax — right there in its own crate. None of that matters to what happens next: once withholdTax returns, its entire crate — grossPay and every local it built up — is lifted off the top and discarded in one motion. The value it hands back, 749.0, lands in calculatePaycheck's frame as netPay:

Stack — Main Thread
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
grossPay = 912.0
netPay = 749.0
main(String[]) frame
— nothing declared yet —

Step 5 — calculatePaycheck hits return netPay;. Same thing happens one level up: its whole crate — hoursWorked, hourlyRate, grossPay, netPay — is lifted off. 749.0 lands in main as takeHomePay:

Stack — Main Thread
main(String[]) frame
takeHomePay = 749.0

Step 6 — main prints the total and reaches the end of its own body. Its crate is lifted off too — the last one. The tower is gone:

Stack — Main Thread
— empty —

Nobody had to clean anything up. No crate was ever reached into from the middle, and none was ever left behind — each one simply came off the top the moment its own method returned, in exactly the reverse order the calls went in: withholdTax first, then calculatePaycheck, then main last.

The one rule underneath it all

A stack only ever changes at the top. Calling a method pushes a new frame on top of whichever frame is currently running. Returning from a method pops that same frame — always the topmost one, never one further down. That single discipline is what lets withholdTax, calculatePaycheck, and main share one stack safely, no matter how deep the chain of calls goes.

§5.4

Copying this — one call, one new object

§5.3 traced a chain of calls built entirely from primitives — no object ever entered the picture, so the heap never needed to appear in a single diagram. Before combining stack and heap in full (§5.5), it's worth watching the smallest possible case on its own: one object, one method call with no arguments at all, one brand new object handed back. Every idea in the larger trace that follows is already present here, just without anything else going on around it. 

class TicketDispenser {
    private int ticketsIssued;

    Ticket takeNext() {
        ticketsIssued++;
        Ticket ticket = new Ticket(ticketsIssued);
        return ticket;
    }
}
class Ticket {
    private final int number;

    Ticket(int number) {
        this.number = number;
    }

    int number() {
        return number;
    }
}
class DeliCounter {
    public static void main(String[] args) {
        TicketDispenser dispenser = new TicketDispenser();
        Ticket ticket = dispenser.takeNext();

        System.out.println("Now serving: " + ticket.number());
    }
}

dispenser.takeNext() takes no arguments — the only thing that crosses into the new frame at all is the implicit this, a copy of the address dispenser already holds. Mid-call, both frames exist at once, and exactly one heap object exists so far:

Stack — Main Thread
takeNext() frame
this = 0xA000
main(String[]) frame — paused
dispenser = 0xA000
Heap
TicketDispenser @ 0xA000
ticketsIssued0

this and dispenser are two separate slots, in two separate frames, holding the identical copied address — the exact same fact §5.5's Order example leans on, here with nothing else around it to explain first. Inside the call, ticketsIssued++ mutates the object both slots point to, then new Ticket(ticketsIssued) puts a second object on the heap, reachable right now only through this frame's own local ticket:

Stack — Main Thread
takeNext() frame
this = 0xA000
ticket = 0xA050
main(String[]) frame — paused
dispenser = 0xA000
Heap
TicketDispenser @ 0xA000
ticketsIssued1
Ticket @ 0xA050
number1

return ticket; pops the frame. The address it held — 0xA050 — is copied out one instant before that frame disappears, into a brand new slot back in main. Two frames become one; two heap objects stay exactly two, both now reachable from the single frame left standing:

Stack — Main Thread
main(String[]) frame — resumed
dispenser = 0xA000
ticket = 0xA050
Heap
TicketDispenser @ 0xA000
ticketsIssued1
Ticket @ 0xA050
number1

Four variables took part in this call, across its whole lifetime, never more than three visible in memory at once: dispenser and ticket in main, this and the local ticket inside takeNext(). Only two of them ever coexisted with an object the other didn't already know about — and even then, both were copies of an address, never a second object. Two calls, zero arguments beyond this, and the entire mechanism §5.5 relies on is already fully in view.

§5.5

A Full Trace — stack and heap, together

§5.4 kept things minimal — one call, no arguments, one new object handed back. Real programs usually do more per call: mutate the object on the other end of a reference and create a brand new object before returning, both in the same method. This section traces exactly that, one line at a time, watching both regions move together.

An order gets a percentage discount applied to it:

class Order {
    private final int id;
    private double total;
    private boolean discounted;

    Order(int id, double total) {
        this.id = id;
        this.total = total;
    }

    Receipt applyDiscount(int discountPercent) {
        double discountAmount = total * discountPercent / 100.0;
        total -= discountAmount;
        discounted = true;

        Receipt receipt = new Receipt(id, discountAmount);
        return receipt;
    }

    double total() {
        return total;
    }
}
class Receipt {
    private final int orderId;
    private final double amountSaved;

    Receipt(int orderId, double amountSaved) {
        this.orderId = orderId;
        this.amountSaved = amountSaved;
    }

    double amountSaved() {
        return amountSaved;
    }
}
class Checkout {
    public static void main(String[] args) {
        int discountPercent = 20;
        Order order = new Order(1042, 250.00);

        Receipt receipt = order.applyDiscount(discountPercent);

        System.out.println(order.total());          // 200.0
        System.out.println(receipt.amountSaved());  // 50.0
    }
}

Step 1 — main declares

main starts running: one frame, pushed onto the call stack, program counter sitting at the top of it. Its first two lines declare a primitive and a reference. discountPercent stores its value, 20, directly in its own slot. order stores an address; the Order object itself — freshly built by new — lands on the heap, entirely separate from the frame that created it:

Stack — Main Thread
main(String[]) frame
discountPercent = 20
order = 0x9000
Heap
Order @ 0x9000
id1042
total250.0
discountedfalse

Step 2 — the call pushes a frame

order.applyDiscount(discountPercent) runs. A new frame goes on top, and two things are copied into it, exactly the way §4.2's parameter-passing rule always works: the address order holds is copied into the implicit this, and the value discountPercent holds is copied into the parameter of the same name. Nothing about main's own slots changes — this is a brand new slot, in a brand new frame, that happens to start out holding the same address order already had. Alongside the copied values, the frame also remembers a return address — the exact point back in main where execution resumes once this call is done:

Stack — Main Thread
applyDiscount(int) frame
↩ return to main, after this call
this = 0x9000
discountPercent = 20
main(String[]) frame — paused
discountPercent = 20
order = 0x9000
Heap
Order @ 0x9000
id1042
total250.0
discountedfalse

Both arrows land on the exact same box. this and order are two different slots, in two different frames, that happen to hold the identical address — no different from member1 and member2 aliasing the same String back in §9. main's frame sits directly underneath, completely untouched, holding its own discountPercent and order exactly as it left them — paused, not destroyed, waiting for the crate above it to come off.

Step 3 — the method body runs

applyDiscount computes discountAmount — a local primitive, alive only in this frame — then does two very different things with it. First, it reaches through this and changes a field on the object at the other end: total -= discountAmount. Second, it builds a brand new Receipt with new, and stores its address in a local reference, receipt:

Stack — Main Thread
applyDiscount(int) frame
↩ return to main, after this call
this = 0x9000
discountPercent = 20
discountAmount = 50.0
receipt = 0x9050
main(String[]) frame — paused
discountPercent = 20
order = 0x9000
Heap
Order @ 0x9000
id1042
total200.0
discountedtrue
Receipt @ 0x9050
orderId1042
amountSaved50.0

total changed from 250.0 to 200.0 — not because this changed, but because the field on the object this points to changed. That object is the exact same one main's order points to, so the moment execution eventually gets back there, order.total() will already reflect it — nothing needs to be handed back for that to be true. A second, brand new object appeared on the heap too: Receipt @ 0x9050, reachable right now only through this frame's own receipt slot.

What mutating doesn't include

If this method instead reassigned its own parameter — discountPercent = 0; — that would change nothing about main's copy, for the exact reason §5.3's "Same name, different crates" callout gave: this frame's discountPercent and main's discountPercent are two separate slots that merely started out holding the same value. Mutating the object this points to is visible outside this frame, because there's only one Order; reassigning a copied value never is, because there were two slots all along.

Step 4 — the method returns

return receipt; runs. The topmost frame is popped — gone, all at once. discountAmount, a plain double that only ever existed in that frame's own slot, is destroyed along with it; nothing else in the program ever held it. But the frame doesn't take the heap down with it: Order @ 0x9000 is still alive, because main's own order still points to it. And the value receipt held — the address 0x9050 — was copied out one instant before the frame disappeared, into a brand new slot back in main:

Stack — Main Thread
main(String[]) frame — resumed
discountPercent = 20
order = 0x9000
receipt = 0x9050
Heap
Order @ 0x9000
id1042
total200.0
discountedtrue
Receipt @ 0x9050
orderId1042
amountSaved50.0

Back to one frame, exactly as Step 1 started — except this one carries a new slot Step 1 never had. discountPercent and order are precisely as main left them before the call; order's object shows the discount already applied, because it was always the same object, never a copy. System.out.println(order.total()) and println(receipt.amountSaved()) now run against this exact picture: 200.0 and 50.0.

What survives a frame, and what doesn't

discountAmount never had a chance of outliving applyDiscount's frame — nothing outside that frame ever held it, primitive or not, so it simply ceased to exist the instant the frame popped. Receipt @ 0x9050 survived for a completely different reason: not because return values are special, but because its address got copied out — into main's new receipt slot — in the same instant the frame died, the same copy-on-return mechanics that handed 749.0 up the chain in §5.3. If applyDiscount had computed a Receipt and never returned it, that object would still exist for one instant after the frame popped — then become unreachable, exactly like the discarded String in §9.6, sitting on the heap with nothing left anywhere to point at it, waiting for the garbage collector to notice.

§5.6

Key Takeaways

  • Memory splits into two regions: the stack, one per thread, holding each method call's own frame; and the heap, one per program, holding every object ever created with new.
  • A stack frame is pushed the instant a method is called and popped the instant it returns — always the topmost one, never one further down. Everything the frame held, primitive or reference, disappears with it.
  • A primitive variable's slot holds its value directly, wherever that slot happens to live. A reference variable's slot holds an address, and the object it points to always lives on the heap, regardless of which frame's reference points at it.
  • Calling a method copies values into the new frame — the address in this, every argument — the same pass-by-value rule from §4.2, applied uniformly whether the value being copied is a primitive or an address.
  • Mutating a field through a copied reference is visible everywhere that shares the same address, because there's only ever one object. Reassigning a copied value — primitive or reference — is never visible anywhere else, because there were always two separate slots.
  • A heap object outlives the frame that created it only if its address escapes that frame before the frame is popped — returned, stored in a field, assigned to a variable further down the stack. Otherwise it becomes unreachable the instant the frame dies, left for the garbage collector to reclaim later.
