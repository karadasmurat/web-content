Chapter 6

Methods

From a plain function to an instance method — a single idea transformed step by step through four calling conventions.

A method call performs two implicit assignments: the argument is assigned to the parameter when the method is called, and the return value is assigned to the receiving variable when it returns. The rules are exactly those from §5 — primitive assignments copy the value; reference assignments copy the address. Nothing new is needed. This chapter introduces four forms of method, each building on the previous one.

§6.1

Static Methods — a function with a namespace

Why we start here ← §0

Java has no freestanding functions — every method must live inside a class. static is the way to say "this is a function; the class is just its namespace." We start here because a static method is exactly that: a plain function, no objects involved, no OO machinery. You don't create a ShapeUtils object to call it.

A static method belongs to the class. For now, simply think of it as a plain function that lives in the namespace of the class.

class ShapeUtils {
    static int calculatePerimeter(int width, int height) {
        return 2 * (width + height);
    }
}

You call it with the class name as a namespace qualifier:

int p = ShapeUtils.calculatePerimeter(10, 20);
// p = 60

The dot here is a namespace separator, not a dereference. ShapeUtils is not an object — no heap allocation occurred, no object exists. The class is simply a container for the function.

The call performs two primitive assignments — one per argument — and one for the return value:

// ShapeUtils.calculatePerimeter(10, 20) implicitly does:
//   width  = 10   ← primitive assignment
//   height = 20   ← primitive assignment
// then returns 60, and:
//   p = 60        ← primitive assignment
§6.2

Static Method with a Reference Parameter — the address is assigned to the parameter

Instead of two loose integers, we can pass the Rectangle that already holds them:

class ShapeUtils {
    static int calculatePerimeter(Rectangle rect) {
        return 2 * (rect.width + rect.height);
    }
}
Rectangle r = new Rectangle(10, 20);
int p = ShapeUtils.calculatePerimeter(r);
// p = 60

The call performs a reference assignment from the argument to the parameter — the same rule from §5: the address in r is copied into rect. No Rectangle is copied. Both r and rect now hold the same address and point to the same object.

Memory model ← §5

Calling calculatePerimeter(r): the address from r is copied into the parameter slot rect. No Rectangle is copied. Inside the method, rect.width dereferences that address and reads from the same object r points to.

The method can read and modify the object's fields through rect because they share the same address. But it cannot make r point to a different object — it only has a copy of the address, not the variable r itself.

§6.3

The Dot Operator Revisited — still a dereference

Look at the body of calculatePerimeter:

static int calculatePerimeter(Rectangle rect) {
    return 2 * (rect.width + rect.height);
            //  ^^^^^^^^^^  dereference
}

rect.width executes in three steps:

  1. Read the address stored in rect.
  2. Go to that address on the heap.
  3. Read the width slot.

This is the same operation described in §4.2 when we first introduced the dot operator. The fact that rect arrived as a parameter rather than being declared locally changes nothing. A reference variable is a reference variable regardless of how it was obtained.

Connection ← §4.2

In §4.2: r.width = 10 — dereference r, write to its width slot. Here: rect.width — dereference rect, read its width slot. Same mechanism, different variable name, different direction (write vs. read). The dot always means: follow the address, then access the member.

This consistency is worth pausing on. The dot operator has one meaning throughout Java — dereference — whether the variable is a local, a parameter, a field, or part of a chain like r.origin.x.

§6.4

Instance Methods — the object receives the call

So far, calculatePerimeter lived in ShapeUtils and received the rectangle as an explicit argument. An instance method is a method defined inside the class it operates on. Instead of being passed as an argument, the object is placed to the left of the dot at the call site.

Here is calculatePerimeter defined as an instance method on Rectangle:

class Rectangle {
    int width;
    int height;

    int calculatePerimeter() {
        return 2 * (this.width + this.height);
    }
}

And the call:

Rectangle r = new Rectangle(10, 20);
int p = r.calculatePerimeter();
// p = 60

The object to the left of the dot — r — is not just a target. Its address is assigned to an implicit reference parameter called this, available inside the method body. this.width dereferences this to read the width slot — exactly as a regular reference variable would.

Assignment ← §5

Calling r.calculatePerimeter() performs a reference assignment: r's address is copied into this. The same rule as any other reference assignment — this and r point to the same object.

Now that instance methods are introduced, the relationship to §6.2 becomes clear:

Static call (§6.2)
ShapeUtils
  .calculatePerimeter(rect);
Instance call
rect
  .calculatePerimeter();

Same computation, different calling convention. rect moves from the argument list to the left of the dot, and is renamed this inside the method. Nothing else changes.

The full progression ← §6.1 ← §6.2

§6.1: calculatePerimeter(width, height) — two primitives, no object.
§6.2: calculatePerimeter(rect) — one reference; the object carries the data.
§6.4: rect.calculatePerimeter() — same reference, moved left of the dot, renamed this.
Same computation. Evolving calling convention.

Summary — the mechanical picture

A variable holds a value or an address. Primitives hold values. Reference variables hold addresses. Assignment copies what the variable holds. The dot dereferences. new allocates, initializes, and returns an address. Static methods are functions in a class namespace. Instance methods are functions where one reference argument — this — is passed via the dot notation rather than the argument list. That is the mechanical foundation of object-oriented Java.
