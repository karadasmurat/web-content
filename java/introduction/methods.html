Chapter 4

Methods

From a plain function to an instance method — a single idea transformed step by step through four calling conventions.

§4.1

Why Methods — the idea before the syntax

A checkout program needs the packaging area for three different boxes, one per item in the order, before it can finalize shipping:

public class PackagingCalculator {

    public static void main(String[] args) {

        double width = 2;
        double height = 5;

        double area = width * height;

        System.out.println(area); // 10.0 — packaging for item 1

        width = 3;
        height = 4;

        area = width * height;

        System.out.println(area); // 12.0 — packaging for item 2

        width = 6;
        height = 9;

        area = width * height;

        System.out.println(area); // 54.0 — packaging for item 3
    }
}

The same three lines — reassign width and height, multiply, print — appear three times, differing only in which numbers go in. It works, but it doesn't stay working for free: find a bug in that formula (say, packaging actually needs 10% extra for padding) and there are three places to fix, not one, each one an easy place to fix two out of three and miss the third.

A method solves this by giving that block of logic a name. Write width * height once, call it calculateArea, and every one of those three places becomes a single call to that name instead of a repeated formula. One place to fix. One place to change. 

public class PackagingCalculator {

    static double calculateArea(double width, double height) {
        return width * height;
    }

    public static void main(String[] args) {

        double area = calculateArea(2, 5);

        System.out.println(area); // 10.0 — packaging for item 1

        area = calculateArea(3, 4);

        System.out.println(area); // 12.0 — packaging for item 2

        area = calculateArea(6, 9);

        System.out.println(area); // 54.0 — packaging for item 3
    }
}

The duplication is gone: three calls instead of three copies of the same formula. If packaging math needs to change tomorrow — that 10% padding, say — there's exactly one line to edit, and every call site picks up the fix automatically. This is the entire motivation for methods in one before-and-after: not a new capability the language didn't have, but a way of naming a piece of logic so it only has to be written, and fixed, once.

§4.2

What Is a Method?

That's what a method is: a named, reusable block of statements that performs a specific task — exactly what calculateArea just did for the checkout example. Giving code a name is the entire idea — everything else in this chapter is about the details of how Java lets you do that. The rest of this chapter follows that same calculateArea idea through a rectangle instead of the checkout example above, since a rectangle already has fields for a method to work with:

static int calculateArea(int width, int height) {
    return width * height;
}

calculateArea is the name. Everything between the braces is the task. From anywhere else in the program, that whole block can now be run just by writing its name.

Inputs and Outputs

Most methods need information to do their job, and most produce a result the caller wants back. Both are visible right in the method's own declaration:

int calculateArea(int width, int height) {
    return width * height;
}

width and height are parameters — the inputs the method needs, each with its own type, exactly like a variable declaration. The int before the method's name is the type of the return value — the output the method hands back to whoever called it, via the return statement.

Calling a Method

Writing a method doesn't run it — a method's body only runs when something calls it, by name, with a value for each parameter. A method call performs two implicit assignments:

  1. The argument is assigned to the parameter, when the method is called.
  2. The return value is assigned to the receiving variable, when the method returns.

The rules are exactly those from §3 — primitive assignments copy the value; reference assignments copy the address. Nothing new is needed. This chapter introduces four forms of method, each building on the previous one:

int a = calculateArea(10, 20);

Execution jumps into the method's body, with width holding 10 and height holding 20 — exactly the two primitive assignments §3 already described. The body runs top to bottom, hits return, and execution jumps back to right after the call, carrying the returned value with it. a ends up holding 200. Nothing about the rest of the program paused or restarted — it simply picked up again exactly where it left off.

Methods Calling Other Methods

A method's body is just statements — and a method call is a statement, so a method can call another method, the same way main calls anything else:

static int calculateArea(int width, int height) {
    return width * height;
}

static void printArea(int width, int height) {
    int a = calculateArea(width, height); // one method, calling another
    System.out.println(a);
}

This is how real programs are actually built: not as one long block of statements, but as many small methods, each doing one focused thing, calling each other to assemble larger behavior out of smaller pieces. The rest of this chapter is about the different forms that "smaller piece" can take in Java.

Overloading

Nothing requires a method name to be used only once. calculateArea works for a rectangle because it takes a width and a height — but a square only has one measurement to give, and Java lets a second method share the exact same name, as long as its parameter list is different:

static int calculateArea(int width, int height) {
    return width * height;
}

static int calculateArea(int side) {
    return side * side; // a square — one parameter instead of two
}
int rectangle = calculateArea(10, 20); // 200 — matches the two-parameter version
int square    = calculateArea(5);      // 25 — matches the one-parameter version

This is overloading: multiple methods sharing one name, distinguished entirely by parameter list — different number of parameters, different types, or both. Which one actually runs is decided at compile time, purely from the arguments written at the call site: calculateArea(10, 20) has two arguments, so only the two-parameter version is even a candidate; calculateArea(5) has one, so only the other version is. Two arguments never magically "become" one — each call site matches exactly one overload, decided before the program ever runs.

§4.3

Outputs — what a method hands back

Imagine asking a student to do a calculation. That request can go one of two ways.

Version 1: "Do this calculation." The student works it out — then writes the answer on a piece of paper, mentions it to a friend, or jots it down in a notebook. Something happened, but nothing came back to you. If you want the answer, you'd have to go find it yourself.

Version 2: "Do this calculation, and tell me the result." The student can still jot it in a notebook, or mention it to a friend, too — nothing stops that. But this time there's one more requirement: the student must also tell the answer directly back to you.

Same math, same effort. The only difference is whether the result comes back to whoever asked. Methods work exactly the same way.

Some Methods Answer a Question

Some methods perform an action. Others answer a question.

So far, we've used methods that perform an action:

printer.printInvoice();
alarm.start();
logger.log("Application started");

Each one does something out in the world — printing, starting, logging — but none of them hand anything back to whoever called them. The caller sends the request and moves on.

Many methods exist for a different reason, though: not just to do something, but to compute a value the caller actually wants handed back.

Imagine asking a calculator: "What is 7 + 5?" You don't expect it to store the result somewhere without showing it to you. You expect it to give you the answer.

Methods can do the same thing:

int result = Calculator.add(7, 5);

The add() method performs a calculation and gives back the value 12. We say that the method returns a value.

Returning a Value

Some methods are asked to produce an answer. return is simply how they hand that answer back. Inside the method, the return statement tells Java: "The work is finished. Give this value back to whoever called me."

What to do with the result

The caller receives the returned value and decides what to do with it.

A returned value can be stored using a variable:

int result = add(7, 5);

It can be used immediately:

System.out.println(add(7, 5));

or as part of a larger expression:

int doubled = add(7, 5) * 2;

or ignored completely:

add(7, 5);

Although ignoring the result usually isn't very useful.

Methods That Don't Return Anything

Not every method has an answer to give. Some methods simply perform an action:

static void greet(String name) {
    System.out.println("Hello, " + name);
}

The return type is void, which means: this method doesn't return a value. It finishes its work and control simply returns to the caller.

A Mental Model

I like to summarize it like this: calling a method is like asking someone to perform a task. Sometimes you ask them to do something. Sometimes you ask them to tell you something.

§4.4

Functions to Objects — the evolution of a method

static keyword

static isn't new — §2.6 used it on a field, to say "this belongs to the class itself, not to any object." Here it says the same thing about a method: calculateArea method belongs to the class ShapeUtils, the same way Rectangle.count did.

Why Begin with static Methods

Every method Java lets you write eventually belongs to a class — there's no such thing as a global, freestanding function, the way there is in some other languages. The rest of this chapter starts with static methods anyway, specifically because they're the closest thing Java has to that plain, freestanding function: called through the class's name. That makes them the simplest possible starting point. We start here because a static method is exactly that: a plain function, no objects involved, no OO machinery. You don't create a ShapeUtils object to call it.

4.4.1 Two primitive parameters

class ShapeUtils {
    static int calculateArea(int width, int height) {
        return width * height;
    }
}

The class is simply a container for the function.
No ShapeUtils object is required for this call. You call it with the class name as a namespace qualifier. 

int a = ShapeUtils.calculateArea(10, 20);
// a = 200

The call performs two primitive assignments — one per argument — and one for the return value:

// ShapeUtils.calculateArea(10, 20) implicitly does:
//   width  = 10   ← primitive assignment
//   height = 20   ← primitive assignment
// then returns 200, and:
//   a = 200       ← primitive assignment

Nothing here is a new rule — it's §3's ordinary primitive assignment, happening three times in a row: once per argument, once more for the return value. A method call isn't a separate mechanism layered on top of assignment; it's assignment, wearing parentheses. Keep that in view, because 4.4.2 changes exactly one thing — what kind of value gets copied — and nothing else about how a call works.

4.4.2 One reference variable as a parameter

Instead of two loose integers, we can pass a reference to the Rectangle object which already holds width and height fields:

class ShapeUtils {
    static int calculateArea(Rectangle rect) {
        return rect.width * rect.height;
    }
}
Rectangle r = new Rectangle(10, 20);
int a = ShapeUtils.calculateArea(r);
// a = 200

The call performs a reference assignment from the argument to the parameter — the same rule from §3: the address in r is copied into rect. No Rectangle is copied. Both r and rect now hold the same address and point to the same object.

Memory model ← §3

Calling calculateArea(r): the address from r is copied into the parameter slot rect. No Rectangle is copied. Inside the method, rect.width dereferences that address and reads from the same object r points to.

The method can read and modify the object's fields through rect because they share the same address. But it cannot make r point to a different object — it only has a copy of the address, not the variable r itself.

The Dot Operator Revisited — still a dereference

Look at the body of calculateArea:

static int calculateArea(Rectangle rect) {
    return rect.width * rect.height;
            //  ^^^^^^^^^^  dereference
}

rect.width executes in three steps:

  1. Read the address stored in rect.
  2. Go to that address on the heap.
  3. Read the width slot.

This is the same operation described in §2.3 when we first introduced the dot operator. The fact that rect arrived as a parameter rather than being declared locally changes nothing. A reference variable is a reference variable regardless of how it was obtained.

Connection ← §2.3

In §2.3: r.width = 10 — dereference r, write to its width slot. Here: rect.width — dereference rect, read its width slot. Same mechanism, different variable name, different direction (write vs. read). The dot always means: follow the address, then access the member.

This consistency is worth pausing on. The dot operator has one meaning throughout Java — dereference — whether the variable is a local, a parameter, a field, or part of a chain like r.origin.x.

4.4.3 An Instance Method: Behavior that operates on an object

So far, we used static methods, which belongs to the class itself. They simply performed a task using inputs passed to them explicitly. But many operations are naturally tied to a specific object. A method that belongs to an object is called an instance method. area calculation is an operation on a particular rectangle.

Here it is defined directly on Rectangle:

class Rectangle {
    int width;
    int height;

    int area() {
        return this.width * this.height;
    }
}

And the call makes the relationship explicit: The object to the left of the dot — rect — is the target to perform area operation. Each invocation can operate on a different object. 

Rectangle rect = new Rectangle(10, 20);
int a = rect.area();
// a = 200
Assignment ← §3

Calling r.area() performs a reference assignment: r's address is copied into this. The same rule as any other reference assignment — this and r point to the same object.

Now that instance methods are introduced, the relationship to 4.4.2 becomes clear:

Static call (4.4.2)
ShapeUtils
  .calculateArea(rect);
Instance call (4.4.3)
rect
  .area();

Same computation, different calling convention. rect moves from the argument list to the left of the dot, is renamed this inside the method, and the method itself drops "calculate" from its name now that it's asking the object about itself rather than being handed a stranger to inspect.

Summary — the mechanical picture

A variable holds a value or an address. Primitives hold values. Reference variables hold addresses. Assignment copies what the variable holds. The dot dereferences. new allocates, initializes, and returns an address. Static methods are functions in a class namespace. Instance methods are functions where one reference argument — this — is passed via the dot notation rather than the argument list. That is the mechanical foundation of object-oriented Java.

Key insight

Same calculation, same values, across all three forms — the contrast is deliberate. The difference isn't in what gets computed; it's in where the responsibility for the data lives. In 4.4.1, the caller is responsible for gathering and passing every number. In 4.4.3, the object is responsible for knowing its own data, and the caller just asks for the result. That shift — from passing data to functions, to asking objects to do things with data they already hold — is the core OO idea the rest of this tutorial builds on: behavior moves closer to the data it operates on.

§4.5

The "Hidden" Parameter — how this works under the hood

Writing high-level object-oriented code, 4.4.3's rect.area() feels like the method lives inside the object — as if every Rectangle carries its own private copy of area() along with it. It doesn't. There is exactly one copy of area()'s code in memory, shared by every Rectangle that will ever exist. The computer still needs a way to know which object's width and height that one shared copy of code should read on any given call — and this is the answer to that question.

The Compiler's Transformation

When you call an instance method, the compiler silently rewrites both the method's own signature and the call site to pass the object along as an explicit, ordinary argument. Here's what you actually write:

public class Player {

    int health = 100;

    public void takeDamage(int amount) {
        health -= amount;  // implicitly this.health
    }
}

And roughly what the compiler treats it as, underneath — pseudocode, not real Java syntax; you can never actually write a method this way yourself:

// The method becomes a plain function with an extra parameter at index 0
void Player_takeDamage(Player this, int amount) {
    this.health -= amount;
}

// The call site passes the object's address as that extra argument
Player_takeDamage(myPlayer, 20);

this isn't magic, and it isn't stored anywhere on the object itself — it's an ordinary parameter, filled in by the compiler instead of by you, carrying the address of whichever object sat to the left of the dot at the call site. That's the whole trick behind "the method lives inside the object": one shared block of code, plus one hidden argument telling it which object's fields to read and write on this particular call.

The Mechanics of an Instance Method Call

class Player {
    int health = 100;
    public void takeDamage(int amount) {
          health = health - amount; // implicitly this.health
    }
}
          
public class Game {
          
    public static void main(String[] args) {
        Player myPlayer = new Player();
        myPlayer.takeDamage(10); // instance method will operate on the object: 'myPlayer'
    }
}

Every instance method call does exactly two things, structurally:

  1. Pushes a new stack frame onto the calling thread's stack.
  2. Initializes that frame's parameters by copying values into it:
    1. this — the implicit first parameter. The dot operator is what produces it, wired up automatically from the object sitting to its left, rather than written explicitly by the caller.
    2. Every explicit parameter — copies of whatever the caller passed, in declaration order.

Picture it mid-call, right after myPlayer.takeDamage(20) runs but before the method body executes its first line:

Stack — Main Thread
takeDamage(int) frame
this   = 0x9000
amount = 20
main(String[]) frame
myPlayer = 0x9000
Heap
Player @ 0x9000
health100

Two frames, one thread, stacked on top of each other exactly the way §4.1's "methods calling other methods" description implied — takeDamage's frame sits on top of main's because main is still waiting for it to return. this, in the new frame, holds the exact same address myPlayer already held one frame down — copied in in the same call, the same way amount was. Only one Player object exists on the heap; both frames just happen to be holding its address at the same time, each in their own slot.

this and a method's explicit parameters are the same mechanism under that hood — both are just parameters, both initialized the same way: by copying a value into the new frame when the call happens. this isn't a special case Java bolts on for instance methods; it's the same parameter-passing rule §4.2's "Calling a Method" already established, with one parameter the dot notation fills in automatically instead of one you write inside the parentheses yourself.

§4.6

Key Takeaways

  • A method groups related statements into a reusable unit.
  • Methods improve readability and reduce duplicated code.
  • Parameters allow methods to receive information.
  • Return values allow methods to produce results.
  • Good programs are built from many small, focused methods.
§4.7

Appendix: The Call Stack — how nested method calls actually happen in memory

§4.5 opened a single stack frame to see where this and a method's parameters actually live. Real programs are rarely one call deep — a method calls another method, which calls another, and so on. This appendix follows one such chain from the moment main starts to the moment it ends, watching the stack grow with every call and shrink again, on its own, as each one finishes.

A Tower of Crates

Picture the loading dock behind a bookstore's stockroom. Orders arrive one at a time, and each one gets its own crate, set down on top of whatever's already stacked there — never underneath, never wedged into the middle. Whoever is working the dock can only ever reach the crate sitting on top; that's the one being handled right now. Everything below it is still open, still mid-task, still waiting — but it has to wait its turn until the crate above it is finished and lifted away. Once a crate is done, it comes off the top, and whatever was directly underneath is exposed again, exactly as it was left. The stack only ever changes at the top.

A running program keeps a stack exactly like that one, per thread (§4.5 already named it "Main Thread"). Every method call sets down a new crate — a stack frame — holding that call's own parameters and local variables. Every method return lifts the topmost crate off, and whatever called it picks up exactly where it left off.

Three Calls Deep

A payroll run turns a timesheet into a paycheck through three methods, each calling the next — and each asking for only what it actually needs, not everything the call before it had:

public class Payroll {

    public static void main(String[] args) {
        double takeHomePay = calculatePaycheck(38, 24.00);
        System.out.println("Take-home pay: $" + takeHomePay);
    }

    static double calculatePaycheck(int hoursWorked, double hourlyRate) {
        double grossPay = hoursWorked * hourlyRate;
        double netPay = withholdTax(grossPay);
        return netPay;
    }

    static double withholdTax(double grossPay) {
        double taxableAtLowRate = Math.min(grossPay, 600.00);
        double taxableAtHighRate = grossPay - taxableAtLowRate;
        double tax = taxableAtLowRate * 0.10 + taxableAtHighRate * 0.25;
        return grossPay - tax - 25.00;
    }
}

main calls calculatePaycheck, which calls withholdTax — three crates deep at its peak. hoursWorked and hourlyRate play genuinely different roles — a count and a rate, multiplied to produce an amount — but withholdTax doesn't need either one individually; by the time the call reaches it, they've already done their job and collapsed into a single dollar figure. Follow the stack through the whole call, one crate at a time.

Step 1 — main starts running. One frame, and it's still empty: takeHomePay isn't declared until the next line executes.

Stack — Main Thread
main(String[]) frame
— nothing declared yet —

Step 2 — main calls calculatePaycheck(38, 24.00). A new crate goes on top; its parameters are filled in immediately, as part of the call itself:

Stack — Main Thread
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
main(String[]) frame
— nothing declared yet —

Step 3 — calculatePaycheck multiplies its two parameters into grossPay, then calls withholdTax(grossPay). A third crate lands on top, holding just one number — not hoursWorked and hourlyRate, which withholdTax was never given and doesn't need. Its parameter happens to share the name grossPay with the local variable one crate down, but that's just a coincidence of naming, not a connection:

Stack — Main Thread
withholdTax(double) frame
grossPay = 912.0
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
grossPay = 912.0
main(String[]) frame
— nothing declared yet —
Same name, different crates

calculatePaycheck's grossPay and withholdTax's grossPay share a name and, right now, a value — but they are not the same slot. Copying 912.0 into the new frame (§4.2's parameter-passing rule) is exactly as independent as copying it into any other local variable. If withholdTax reassigned its own grossPay, calculatePaycheck's copy, one crate down, wouldn't notice.

Step 4 — withholdTax works through a small ladder of rates: the first $600 of grossPay is taxed at 10%, everything above that at 25%, and a flat $25.00 insurance premium comes off on top. Along the way it declares locals of its own — taxableAtLowRate, taxableAtHighRate, tax — right there in its own crate. None of that matters to what happens next: once withholdTax returns, its entire crate — grossPay and every local it built up — is lifted off the top and discarded in one motion. The value it hands back, 749.0, lands in calculatePaycheck's frame as netPay:

Stack — Main Thread
calculatePaycheck(int, double) frame
hoursWorked = 38
hourlyRate = 24.0
grossPay = 912.0
netPay = 749.0
main(String[]) frame
— nothing declared yet —

Step 5 — calculatePaycheck hits return netPay;. Same thing happens one level up: its whole crate — hoursWorked, hourlyRate, grossPay, netPay — is lifted off. 749.0 lands in main as takeHomePay:

Stack — Main Thread
main(String[]) frame
takeHomePay = 749.0

Step 6 — main prints the total and reaches the end of its own body. Its crate is lifted off too — the last one. The tower is gone:

Stack — Main Thread
— empty —

Nobody had to clean anything up. No crate was ever reached into from the middle, and none was ever left behind — each one simply came off the top the moment its own method returned, in exactly the reverse order the calls went in: withholdTax first, then calculatePaycheck, then main last.

The one rule underneath it all

A stack only ever changes at the top. Calling a method pushes a new frame on top of whichever frame is currently running. Returning from a method pops that same frame — always the topmost one, never one further down. That single discipline is what lets withholdTax, calculatePaycheck, and main share one stack safely, no matter how deep the chain of calls goes.
