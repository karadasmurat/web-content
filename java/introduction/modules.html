Chapter 23

Java Modules

Packages (§10) organize code inside a single compilation unit, but they were never designed to protect one JAR from another — public means public to anyone who happens to have that JAR on the classpath, whether they were meant to see it or not. This chapter introduces the module system: a way to name a group of packages, declare exactly what it depends on, and declare exactly what it's willing to share with the outside world.

§23.1

The Problem — the classpath doesn't know or care

Picture a small business's Java codebase split into two JARs: orders.jar, containing everything related to placing and tracking orders, and reporting.jar, which reads order data to generate sales reports. reporting.jar depends on orders.jar — it calls into classes that live there.

On the classpath, that dependency is never actually written down anywhere. Both JARs get thrown onto one flat list at launch:

java -cp orders.jar:reporting.jar:some-unrelated-tool.jar com.store.reporting.Main

Three problems fall directly out of that flatness:

  • No real dependency declaration. Nothing about reporting.jar says "I need orders.jar specifically." If it's missing from the classpath, nothing fails until some line of code actually tries to use a class that isn't there — a NoClassDefFoundError, potentially minutes into a long-running job, not a clear error at startup.
  • No real encapsulation across JARs. A public class in orders.jar is public to everyone — reporting.jar, yes, but also some-unrelated-tool.jar, or literally any other code that ever ends up sharing a classpath with it. §10's public was never designed to draw a line around "the code I actually intend to expose."
  • No protection against split packages. Nothing stops two different JARs from both contributing classes to the exact same package name — whichever one the classpath happens to resolve first silently wins, with no warning that the other one's classes were shadowed.

Every one of these is a problem of the classpath not knowing anything about structure — it just sees a bag of class files. The module system exists to put that structure back.

§23.2

What Is a Module

That's exactly the structure the classpath was missing — a way for orders.jar and reporting.jar to describe themselves, instead of just being handed to the JVM as an undifferentiated pile of classes with no declared dependencies and no declared boundaries.

A module is a named, self-describing group of packages. "Self-describing" is the key word: a module declares, up front, exactly which of its packages are visible to the outside world, and exactly which other modules it needs in order to compile and run. Both declarations live in one file — module-info.java — at the root of the module's source.

orders.jar and reporting.jar, reframed as modules, become com.store.orders and com.store.reporting — module names conventionally following the same reverse-domain convention as package names (§10.1), since a module is often built from one root package of the same name.

§23.3

module-info.java

Every module's source tree gets exactly one module-info.java, sitting at the root, above every package it contains:

com.store.orders/
├── module-info.java
└── com/store/orders/
    ├── api/
    │   └── OrderService.java
    └── internal/
        └── OrderValidator.java

At its simplest, the file just names the module:

module com.store.orders {
}

An empty body compiles, but it's not useful yet — nothing is exported, and nothing is declared as a dependency. §23.4 and §23.5 fill that body in.

§23.4

exports

com.store.orders has two packages: com.store.orders.api, meant for other modules to call into, and com.store.orders.internal, holding validation logic that's orders's own business. exports is how the module draws that line explicitly:

module com.store.orders {
    exports com.store.orders.api;
}

Only com.store.orders.api is listed. com.store.orders.internal isn't mentioned at all — and that omission is the whole point. §23.6 comes back to exactly what it buys.

§23.5

requires

com.store.reporting calls into com.store.orders.api, so it says so, explicitly, in its own module-info.java:

module com.store.reporting {
    requires com.store.orders;
}

This is the declaration §23.1 pointed out was missing entirely from the classpath. With it in place, launching com.store.reporting without com.store.orders present fails immediately, with a clear error naming the missing module — not a NoClassDefFoundError surfacing later, deep inside whatever line of code first happened to need a class from it.

§23.6

Strong Encapsulation

OrderValidator, in the un-exported com.store.orders.internal package, is declared public in the ordinary sense §10.5 covered:

package com.store.orders.internal;

public class OrderValidator {
    public static boolean isValid(String customer, double amount) {
        return customer != null && !customer.isBlank() && amount > 0;
    }
}

On the classpath, that public would mean exactly what §10.5 described: accessible from anywhere, including com.store.reporting, whether or not that was ever intended. As a module, it doesn't — com.store.orders.internal was never named in an exports clause, so code in com.store.reporting that tries to reference OrderValidator directly fails to compile, with an error naming the package as inaccessible from outside its module. public alone is no longer enough to reach it from outside; the module has to have chosen to expose the package it lives in.

This is strong encapsulation, and it's the module system's central payoff: a boundary the classpath structurally could not enforce, because the classpath never had any concept of "outside this JAR" to begin with. com.store.orders.api.OrderService, meanwhile, works exactly as before — it's public, and its package is exported, so com.store.reporting reaches it without any trouble.

§23.7

The Module Path

Named modules run from the module path (--module-path, or -p), the module-aware counterpart to the classpath (-cp) used throughout the rest of this tutorial:

java --module-path out --module com.store.reporting/com.store.reporting.Main

Code still built the ordinary classpath way doesn't stop working — it runs as part of what's called the unnamed module: one large, unstructured module with no exports restrictions and no declared requires, behaving exactly like the classpath always has. Modules and classpath JARs can even coexist in the same launch. The strong encapsulation of §23.6 only applies to code that has actually opted in by declaring itself a named module in the first place.

§23.8

requires transitive

Suppose OrderService's own public methods return a type that itself lives in some third module — a shared com.store.money module defining a Money type, say. Any module that calls OrderService ends up holding a Money value, and so needs com.store.money on its own module path too, even though it never called into com.store.money directly:

module com.store.orders {
    requires com.store.money;
    exports com.store.orders.api;
}

Without anything more, every module that requires com.store.orders would also have to separately write requires com.store.money itself, just to compile against a return type it never asked for by name. requires transitive passes that requirement along automatically:

module com.store.orders {
    requires transitive com.store.money;
    exports com.store.orders.api;
}

Now com.store.reporting's own requires com.store.orders is enough — it implicitly gets com.store.money along with it, because orders already declared that dependency as part of its own public API, not just an internal implementation detail.

§23.9

When (Not) to Use Modules

None of this is free. A one-off script, a small personal tool, or a single-team application with no real internal boundaries to enforce gains little from writing and maintaining module-info.java files and carefully-scoped exports lists — the plain classpath, and the unnamed module §23.7 described, remain the default for most day-to-day Java code, including everything else in this tutorial.

Modules earn their keep at a different scale: a large codebase, several teams, or a library shipped to consumers who shouldn't be able to reach into its internals no matter how convenient it might be for them to try. Strong encapsulation across a boundary like that is worth real ceremony; a small application drawing that same hard line around itself usually isn't solving a problem it actually has.

§23.10

Key Takeaways

  • The classpath treats every JAR as one flat bag of classes — no declared dependencies, no real encapsulation between JARs, and no protection against two JARs quietly sharing a package name.
  • A module is a named group of packages, self-described by a single module-info.java at the root of its source.
  • exports lists exactly which packages are visible outside the module; everything else stays invisible no matter how public its classes are declared.
  • requires declares a dependency explicitly, so a missing module fails fast at launch instead of surfacing as a NoClassDefFoundError deep in a running program.
  • Strong encapsulation is the module system's central payoff: a public class in a package that was never exported is genuinely unreachable from outside its module — something the classpath alone could never enforce.
  • Named modules run from the module path; classpath code keeps working unchanged as the unnamed module, with none of the module system's restrictions.
  • requires transitive passes a dependency along automatically to anything that depends on the module declaring it, when that dependency's types are part of the module's own exported API.
  • Modules pay off at the scale of large codebases, multiple teams, or libraries with real internals to protect — not every small application needs the ceremony.
