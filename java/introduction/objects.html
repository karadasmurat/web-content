Chapter 4

Objects

Creating objects, reaching them through references, pointing objects at other objects, and giving them their initial state — the four mechanics that make OO work.

§4.1

Instantiation — what new does

The class is the blueprint. An object is an instance of that blueprint — a concrete chunk of memory laid out according to the class's field declarations. You create one with new:

new Rectangle();

new does three things in sequence:

  1. Allocates a chunk of memory on the heap — a region of memory managed separately from local variables, designed to hold data whose lifetime isn't tied to any single method call.
  2. Zeroes all fields. For our Rectangle: width = 0, height = 0.
  3. Returns the address of the newly allocated chunk.

The object now exists on the heap. But the address returned by new hasn't been stored anywhere — so we have no way to reach it. We need a variable that can hold an address. That's §4.2.

Memory model

The heap is a pool of memory that persists independently of any method call. Objects live there. Local variables live on the stack, which grows and shrinks as methods are called and return. An object on the heap survives as long as at least one reference variable points to it; when none do, the garbage collector reclaims it.

§4.2

Reference Variables — a variable that holds an address

In §1 we said a variable holds either a value or an address. Now we need the second kind. A reference variable holds an address — the location of an object on the heap.

Rectangle r = new Rectangle();

Read this as: allocate a Rectangle on the heap, and store its address in the reference variable r. The variable r does not contain a Rectangle. It contains a number — the memory address where that Rectangle lives.

Memory model

r is a slot that holds an address, say 0x4a2f. At address 0x4a2f on the heap: a Rectangle object with its own width and height slots (both zero for now).

To access the object's fields, you use the dot operator. The dot means: go to the address stored in the variable on my left, then access the member named on my right.

Rectangle r = new Rectangle();
r.width  = 10;  // go to the object r points to; set its width slot to 10
r.height = 20;  // go to the object r points to; set its height slot to 20

System.out.println(r.width); // dereference r, read its width slot → 10

Every use of the dot operator is a dereference: follow the address, then act on what's there.

Aliasing

When you assign one reference variable to another, you copy the address — not the object.

Rectangle r  = new Rectangle();
r.width = 10;

Rectangle r2 = r;   // r2 now holds the same address as r

r2.width = 99;      // dereferences r2, modifies the same object

System.out.println(r.width); // 99 — r and r2 point to the same object

Both r and r2 hold the same address. They are two names for one object. This is called aliasing.

Contrast with primitives ← §2

int b = a copies the value — two independent slots. Rectangle r2 = r copies the address — two names for the same object. The difference flows directly from value vs. address (§1).

§4.3

Reference Fields — objects pointing to objects

In §3 we constrained Rectangle to primitive fields only. Now that reference variables are understood, we can lift that constraint.

A field can be of any type — including a class. A field of class type is a reference field: a slot inside the object that holds an address (or null).

A rectangle naturally has two things beyond its dimensions: a position (where its top-left corner sits) and a border (how its edge is drawn). Both are structured values — each made of several primitives — so each becomes its own class:

class Point {
    int x;
    int y;
}

class Border {
    int     thickness;
    boolean visible;
}

Now Rectangle can hold references to both:

class Rectangle {
    int    width;
    int    height;
    Point  origin;  // reference field — holds an address, not a Point itself
    Border border;  // reference field — holds an address, not a Border itself
}

When a Rectangle is created, both reference fields are initialized to null — the slots exist but point to nothing yet.

Rectangle r = new Rectangle();
r.width  = 10;
r.height = 20;
// r.origin is null, r.border is null

Point p = new Point();
p.x = 5;
p.y = 8;
r.origin = p;    // copy p's address into r's origin slot

Border b = new Border();
b.thickness = 2;
b.visible   = true;
r.border = b;    // copy b's address into r's border slot
Memory model

On the heap: a Rectangle with width=10, height=20, origin=0x4a2f, border=0x7c10. At 0x4a2f: a Point with x=5, y=8. At 0x7c10: a Border with thickness=2, visible=true. The Rectangle does not contain the Point or the Border — it contains their addresses.

The dot operator chains naturally across reference fields. Each dot is one dereference:

System.out.println(r.origin.x);      // follow r → Rectangle; follow origin → Point;  read x         → 5
System.out.println(r.border.visible); // follow r → Rectangle; follow border → Border; read visible   → true
Connection ← §4.2

origin and border are reference variables that happen to live inside an object rather than in a method. The rules are identical: each holds an address, the dot dereferences it, and aliasing applies. If you write Border b2 = r.border, both b2 and r.border point to the same Border object.

§4.4

Constructors — initialization before the reference is returned

After new Rectangle(), all fields are zeroed. That's rarely what you want. A constructor is initialization code that new runs after allocation, before the address is returned to the caller.

class Rectangle {
    int width;
    int height;

    Rectangle(int w, int h) {   // constructor: same name as class, no return type
        width  = w;
        height = h;
    }
}

Now you pass initial values directly to new:

Rectangle r = new Rectangle(10, 20);
// r.width  is 10
// r.height is 20

The sequence new Rectangle(10, 20) executes as:

  1. Allocate a Rectangle on the heap; zero its fields.
  2. Run the constructor body with w=10, h=20.
  3. Return the address.

By the time r receives the address, the object is already initialized. The caller never sees the zeroed state.

Scope for now

One constructor, no chaining, no inheritance — just the core idea. The essential point: new = allocate + initialize + return address.

If you define no constructor, Java provides one automatically — it takes no arguments and leaves all fields at their zero values. The moment you define your own, the automatic one goes away.
