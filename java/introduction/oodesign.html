Chapter 27

Object-Oriented Design

Every mechanism up to this point — classes, references, inheritance, interfaces, generics, collections — is a tool. This chapter is about judgment: how to decide what a class should be responsible for, how objects should relate to one another, and why some technically-correct designs still turn into a mess six months later. One running example, a small library catalog, carries the whole chapter.

§27.1

The Problem — a class that knows too much

Say we're modeling books for a small library system. A first, entirely reasonable-looking attempt:

class Book {
    String title;
    String author;
    String isbn;
    int    pageCount;
    String format;      // "Hardcover", "Paperback", "E-book"
}

This compiles, and for a single book it works fine. The trouble shows up the moment reality does what it actually does: the same book — same title, same author, same content — routinely exists as more than one physical or digital product. Clean Code by Robert C. Martin is a hardcover, a paperback, and an e-book, each with its own ISBN and its own page count, but there's still only one title and one author to speak of.

Model that with the class above, and each edition needs its own Book object:

Book hardcover = new Book();
hardcover.title = "Clean Code";
hardcover.author = "Robert C. Martin";
hardcover.isbn = "978-0132350884";
hardcover.pageCount = 464;
hardcover.format = "Hardcover";

Book ebook = new Book();
ebook.title = "Clean Code";       // typed again
ebook.author = "Robert C. Martin"; // typed again
ebook.isbn = "978-0132766789";
ebook.pageCount = 464;
ebook.format = "E-book";

title and author are copied by hand into every edition. Nothing in the compiler enforces that both copies agree — a typo in one, a correction applied to the other, and the two "same" books silently disagree about their own title. This isn't a syntax problem or a bug in any single method. It's a shape problem: one class is carrying two different ideas — a book, the work, and an edition, one particular printing of it — squeezed into a single set of fields with no way to say which is which.

Design vs. mechanics

Everything so far in this tutorial has asked "how do I make Java do this?" This chapter asks a different question: "given that Java can do several things, which one should this class actually be?" That's object-oriented design (OOD) — the practice of deciding what classes exist, what each one owns, and how they refer to each other, independent of any single language feature.

§27.2

Cohesion — one reason to change

Book, as first written, has at least two separate reasons someone might need to edit it: something about the underlying work changed (the author's name was printed wrong), or something about one particular printing changed (a paperback edition was released with a new ISBN). Those are unrelated changes, arriving from unrelated sources, both landing on the same class.

A class with a single, clear reason to change is cohesive. A class carrying several unrelated reasons to change is doing more than one job — and the more jobs it does, the more often some change touches it, and the harder it gets to reason about what any one field actually means. Cohesion isn't a Java feature; it's a property of the design, and it's the first thing worth checking whenever a class's field list starts feeling like a grab bag.

The fix is to split the two ideas into two classes — one for the work itself, one for a specific printing of it:

class Book {
    private final String title;
    private final String author;

    Book(String title, String author) {
        this.title  = title;
        this.author = author;
    }

    String title()  { return title; }
    String author() { return author; }
}

Book now has exactly one reason to change: the work's own identity — its title, its author. Nothing about ISBNs, page counts, or formats belongs here at all.

Connection ← §12.3

Book is built the way §12 built Point: private final fields, set once in the constructor, read through accessor methods, no setters. A book's title and author aren't the kind of state that should ever silently change out from under whoever's holding a reference to it — immutability isn't just a nice-to-have here, it's what makes Book safe to share across every edition that refers to it, which is exactly what §27.3 does next.

§27.3

Composition, or "Has-A" — building BookEdition out of Book

The second idea — one specific printing — becomes its own class, BookEdition. It doesn't repeat title and author; instead, it holds a reference to the Book it's an edition of:

class BookEdition {
    private final Book   book;
    private final String isbn;
    private final int    pageCount;
    private final String format;

    BookEdition(Book book, String isbn, int pageCount, String format) {
        this.book      = book;
        this.isbn      = isbn;
        this.pageCount = pageCount;
        this.format    = format;
    }

    Book   book()      { return book; }
    String isbn()      { return isbn; }
    int    pageCount() { return pageCount; }
    String format()    { return format; }
}

Now the three editions of Clean Code share one Book, instead of each carrying its own disconnected copy of the title and author:

Book cleanCode = new Book("Clean Code", "Robert C. Martin");

BookEdition hardcover = new BookEdition(cleanCode, "978-0132350884", 464, "Hardcover");
BookEdition paperback = new BookEdition(cleanCode, "978-0132760800", 464, "Paperback");
BookEdition ebook      = new BookEdition(cleanCode, "978-0132766789", 464, "E-book");

System.out.println(hardcover.book() == ebook.book()); // true — the very same Book object

Three BookEdition objects, three distinct ISBNs — and exactly one Book underneath all of them. Correct the title once, on the one shared Book, and every edition reflects it, because every edition was never holding its own copy in the first place — just a reference to the one that exists.

Connection ← §2.4

This is nothing more than a reference field, the same mechanism §2.4 introduced with Rectangle's origin and border. BookEdition doesn't contain a Book — it holds an address pointing at one, on the heap, possibly shared with other objects that hold the same address. Naming that relationship is what's new here: whenever one class holds a reference to another as a field, that's composition — a "has-a" relationship, as opposed to the "is-a" relationship inheritance models (§10). A BookEdition has a Book; it is not, itself, a kind of Book.

Composition is also how coupling gets managed deliberately. BookEdition depends on Book — it has to, to do its job — but the dependency runs in one direction only. Book has no field, no method, no import that mentions BookEdition at all; it has no idea editions exist. That asymmetry matters: Book can be reused, tested, or changed on its own terms, entirely unaware of anything built on top of it.

§27.4

Tell, Don't Ask — encapsulation, revisited

With book() exposed, it's tempting to reach straight through a BookEdition to assemble a description from the outside:

// works, but every caller has to know BookEdition's internal shape
String description = edition.book().title() + " by " + edition.book().author()
        + " (" + edition.format() + ", " + edition.pageCount() + " pages, ISBN "
        + edition.isbn() + ")";

Every piece of code that ever wants a description now needs to know that a BookEdition holds a Book, that a Book exposes title() and author(), and the exact order and punctuation the description is assembled in. Change any of that internal shape — say, Book gains a separate publisher field that should appear in the description — and every one of those call sites needs to change with it.

The fix is to ask the object to do the work, instead of asking it for its parts and doing the work yourself — a habit compact enough to have its own name, Tell, Don't Ask:

class BookEdition {
    // ...fields and constructor as before...

    String describe() {
        return "%s by %s (%s, %d pages, ISBN %s)"
                .formatted(book.title(), book.author(), format, pageCount, isbn);
    }
}
System.out.println(edition.describe());
// Clean Code by Robert C. Martin (Hardcover, 464 pages, ISBN 978-0132350884)

Now there's exactly one place that knows how a description is built, and it's the one class actually holding the data. Every caller just says describe yourself — a single method call, unaffected by whatever BookEdition's fields look like internally, today or after the next change.

Connection ← §11.4

§11 framed encapsulation as hiding a field behind private so its invariants can't be broken from outside. Tell, Don't Ask is the same idea applied one level up: hide not just the data, but the logic that uses it, behind a method that does the whole job. A getter that just hands back a field is sometimes unavoidable — book() still exists above — but reaching for one by default, everywhere, is exactly the reflex this section pushes back on.

§27.5

Where a Method "Belongs" — an object's own state vs. the wider system

Tell, Don't Ask (§27.4) says to give an object a method that does the whole job, instead of reaching in and doing it from outside. That's good advice — right up until "the whole job" turns out to need things the object was never actually holding. Picture a Notification meant to represent the same content no matter how it eventually reaches someone — shown in an in-app notification list today, emailed as a daily digest tomorrow, pushed to a phone next quarter. Whoever added send() bolted one specific delivery mechanism onto a class that was never supposed to know delivery existed:

class Notification {
    private final String recipient;
    private final String subject;
    private final String body;

    Notification(String recipient, String subject, String body) {
        this.recipient = recipient;
        this.subject = subject;
        this.body = body;
    }

    void send() {
        SmtpClient client = new SmtpClient("smtp.mailprovider.com", 587);
        client.connect();
        client.authenticate("notifications@example.com", loadPassword());
        client.sendMessage(recipient, subject, body);
        client.disconnect();
    }
}

Look at what send()'s body is actually made of. recipient, subject, and body — the three fields Notification actually owns — show up exactly once each, just to be handed off. Everything else in that method is about something Notification doesn't own at all: a hostname, a port, credentials, a network connection. The method is named after the object it sits on, but almost none of its work is about that object's own state. Renaming the class to EmailNotification wouldn't fix this — it would just make the mistake honest. The actual problem isn't the name; it's that the in-app notification list and the eventual push channel now have to drag SMTP along with them, because whatever they read as a Notification comes bundled with a method that only makes sense for one of the three places it's used.

A rule of thumb

A method belongs on a class when its body is built mostly out of that class's own fields — reading them, combining them, occasionally calling other methods on things it already holds a reference to. A method whose body is mostly about something else entirely — a network client, a file system, a database connection, another service — is describing a collaboration, not a behavior. The object supplying the data doesn't have to be the object doing the work; nothing says it has to import a networking class just to fulfill a method with its own name on it.

§27.2 already named this shape: two reasons to change, wearing one class. A notification's content changes for content reasons — a new field, a different greeting, a formatting fix — decided by whoever owns what a notification says. How a notification actually gets delivered changes for entirely different reasons — switching providers, adding retries, adding a push-notification channel alongside email — decided by whoever owns how messages move. Cramming both into send() means a change to either reason edits the same class, for no reason connected to the other.

Split the object that holds the data from the object that does the delivering, and name the delivery capability as its own contract (§14):

final class Notification {
    private final String recipient;
    private final String subject;
    private final String body;

    Notification(String recipient, String subject, String body) {
        this.recipient = recipient;
        this.subject = subject;
        this.body = body;
    }

    String recipient() { return recipient; }
    String subject()   { return subject; }
    String body()      { return body; }
}
interface NotificationSender {
    void send(Notification notification);
}

class SmtpNotificationSender implements NotificationSender {
    private final SmtpClient client;

    SmtpNotificationSender(SmtpClient client) {
        this.client = client;
    }

    @Override
    public void send(Notification notification) {
        client.connect();
        client.authenticate("notifications@example.com", loadPassword());
        client.sendMessage(notification.recipient(), notification.subject(), notification.body());
        client.disconnect();
    }
}
Notification reminder = new Notification(
        "user@example.com", "Reminder", "Your appointment is tomorrow.");

NotificationSender sender = new SmtpNotificationSender(smtpClient);
sender.send(reminder);

Notification is now a plain, immutable value (§12) — three fields, three getters, nothing that reaches outside itself, and nothing about it changes no matter how the message eventually gets delivered. NotificationSender owns the entire delivery concern, coded to an interface rather than a concrete class (§27.8): swap SmtpNotificationSender for a push-notification sender, or a fake one in a test, and Notification never has to change, because it was never the thing doing the sending in the first place.

But couldn't send() just be abstract?

An abstract Notification with an abstract send(), overridden by EmailNotification, would compile fine — and it's tempting, since §13 already builds exactly that shape elsewhere. It doesn't actually fix anything here, though: the SMTP details just move from a method on Notification to a method on a subclass of Notification, still one inheritance chain carrying both "what a notification says" and "how it gets delivered." A content fix to the base class can still ripple into every delivery-specific subclass, and an SMTP retry-policy change still lives inside something that is-a Notification.

The sharper problem is that content and channel vary independently — the same reminder needs to go out by email and, eventually, by push. With inheritance, one object is one concrete subclass, so reaching two channels means two separate objects duplicating identical content. With NotificationSender, it's the same Notification handed to two different senders — nothing duplicated, because content and delivery were never the same axis of variation to begin with. That's the real test past §27.7's is-a check: when two concerns can be mixed and matched freely, composition lets you cross them; inheritance forces one subclass per combination.

Connection ← §27.4

This isn't a rejection of Tell, Don't Ask — sender.send(reminder) is still telling an object to do a whole job, not reaching in and doing it from outside. The refinement is about which object gets told. Tell-Don't-Ask says don't make the caller assemble the behavior out of parts it reached in and grabbed; it never said the object being called has to be the one whose name is most obviously attached to the job. send() was never Notification's job to begin with — it belonged to whatever object actually holds the network connection.

§27.6

Query vs. Command — what is this method actually doing?

Two methods on the BankAccount from §12:

double getBalance() { return balance; }
void deposit(double amount) { balance += amount; }

Calling getBalance() ten times in a row is harmless — the balance comes back the same every time, no matter how many times it's asked. Calling deposit(100) ten times moves a thousand dollars. Nothing about either body needed reading to know this — the method's own name gave it away, because every instance method falls into exactly one of two roles.

Command-Query Separation (CQS)

Bertrand Meyer named this split in his work on the Eiffel language, and it's the one hard boundary in this section — everything else below is descriptive vocabulary layered on top of it:

  • Query — reads state, changes nothing. Safe to call any number of times, in any order, interleaved with anything else — a pure function of the object's state and its arguments.
  • Command — changes state. Every call counts; calling it twice is a different thing from calling it once.

The finer labels below name different points on that line, not different categories sitting beside it. Two methods can land in genuinely ambiguous territory between two of them, and that's fine — the labels are vocabulary for a conversation, not a classification to enforce.

Queries: one shape, several outputs

Every query has the same underlying shape: state and arguments go in, an answer comes out, nothing changes. What differs from one query to the next is only how much work happens along the way, and what shape the answer takes:

double getBalance() {                    // a direct projection of one field
    return balance;
}

boolean isOverdrawn() {                   // same shape — output happens to be boolean
    return balance < 0;
}

double calculateInterest(double rate) {   // same shape — real computation this time
    return balance * rate;
}

All three are pure functions of state, and the differences between them are differences of degree, not of kind. Informally: a reader for a trivial field projection like getBalance(); a predicate for a boolean-returning query like isOverdrawn() (the is/has/can naming convention usually gives this one away before the return type does); and simply a query that does real work, like calculateInterest(). Where exactly getBalance() ends and calculateInterest() begins on that spectrum doesn't have a principled answer, and it doesn't need one — all three are queries either way, and that fact is what actually constrains how they're allowed to be called.

Commands: mutators, and one deliberate exception

A command changes state. Under strict CQS it returns void — its entire purpose is the side effect, nothing else:

void deposit(double amount) {
    balance += amount;
}

A method that both mutates state and hands back a computed value is doing two jobs in one call. That's often a warning sign — a method that reads like a query but quietly mutates underneath is the single most common source of surprising bugs in this territory. But one shape breaks CQS on purpose, by design: a builder mutates state and returns this, purely to enable chaining. The returned value is never new information — it's always just the object itself:

Order addItem(Item item) {
    items.add(item);
    return this;
}

// order.addItem(a).addItem(b).addItem(c);

Roles that cut across the line

A few roles aren't defined by whether a method mutates — they can land on either side, depending on what they're built around:

  • Delegator — does little itself, just forwards to another object's method. Whether it's a query or a command depends entirely on what it delegates to.
  • Lifecycle / contract method — behavior is fixed by a contract with the JVM, a collection, or a framework, not by domain logic. equals() and hashCode() are queries by contract; close() (§15.8) is a command by contract. These aren't designed freely — they satisfy a spec.
  • Orchestrator — coordinates several queries and commands in sequence to perform one multi-step operation, often the method a caller actually calls, with everything else underneath it as private support. Its own status is just whatever the sum of its parts adds up to: if it mutates anything along the way, it's a command, full stop, even if it also returns something at the end.
Role Query or command What distinguishes it
Reader Query Trivial projection of one field
Predicate Query Output is boolean
Computed query Query Real computation before returning
Mutator Command Pure side effect, returns void
Builder Command Mutates, returns this for chaining
Delegator Either Forwards to another object's method
Lifecycle / contract Either Behavior fixed by a spec, not domain logic
Orchestrator Either Sum of whatever it coordinates
Connection ← §27.4

Tell, Don't Ask pushed back on reaching for a getter and doing the caller's own work outside the object. Query vs. Command names the fact underneath that habit: a getter is a query by definition, and every extra query a class exposes is one more piece of internal shape callers can come to depend on. Preferring a command like describe() over a chain of readers isn't only about hiding structure — it's about keeping the query surface small enough that "is this safe to call twice?" stays an easy question to answer.

Connection ← §12.4

Transaction, the immutable class §12.4 built, has exactly two methods — getType() and getAmount() — and both are queries. That's not a coincidence: an immutable object has no state left to mutate, so past its own constructor it can never expose a command at all. Every method call on an immutable object is automatically safe to repeat, interleave, or share across threads (§12.6) — the entire category of "is this safe to call twice?" questions this section has been raising simply doesn't arise.

§27.7

Composition over Inheritance — why BookEdition doesn't extend Book

Inheritance (§10) was the tutorial's first tool for reusing another class's fields and methods, so it's worth confronting directly: why not simply write class BookEdition extends Book, and inherit title and author for free?

// tempting, but wrong here
class BookEdition extends Book {
    private final String isbn;
    private final int    pageCount;
    private final String format;

    BookEdition(String title, String author, String isbn, int pageCount, String format) {
        super(title, author);
        this.isbn      = isbn;
        this.pageCount = pageCount;
        this.format    = format;
    }
}

Two problems, and neither is about syntax:

The "is-a" test fails. §10.1 grounded inheritance in a taxonomy — extends should read as "a more specific kind of." A hardcover is a kind of physical object, sure, but is it a more specific kind of Book, the way a Dog is a more specific kind of Animal? Not really — a hardcover isn't a specialized book, it's a printing of a book. The relationship is "belongs to," not "is a kind of," and §27.3 already gave that relationship its correct name: composition.

The original problem comes right back. Each BookEdition subclass instance still carries its own inherited copy of title and author — inheritance doesn't create sharing, it copies the parent's field layout into every subclass object. Constructing a hardcover and an e-book "edition" of the same work still means passing "Clean Code" and "Robert C. Martin" into two separate constructor calls, with no single shared Book object underneath either one. Exactly the duplication §27.1 identified — inheritance just hides it behind extends instead of fixing it.

A rule of thumb

When one object needs the data or behavior of another, composition — a reference field — should be the default choice. Reach for inheritance only when the relationship genuinely is "is-a-more-specific-kind-of," and the two classes are meant to share one common notion of identity, substitutable for each other wherever the parent type is expected (§10.5's polymorphism). "I want to reuse some fields" is not, by itself, a reason to inherit — composition reuses the exact same fields without committing to a taxonomy that doesn't actually hold.

§27.8

Program to an Interface, Not an Implementation

The library catalog will eventually hold more than books — perhaps magazines, or audiobooks, later. Every one of those needs to be described, the same way a BookEdition does. Rather than writing catalog code that only understands BookEdition specifically, name the capability itself as an interface (§14):

interface Describable {
    String describe();
}

class BookEdition implements Describable {
    // ...fields and constructor as before...

    @Override
    public String describe() {
        return "%s by %s (%s, %d pages, ISBN %s)"
                .formatted(book.title(), book.author(), format, pageCount, isbn);
    }
}

Nothing about describe()'s body changed — only its declaration now carries @Override, satisfying a contract named Describable instead of just happening to have a method with that name.

Connection ← §14.11, ← §19.3

This is interface-typed programming exactly as §14.11 described it, now paired with the collections framework from §19: a variable, parameter, or — as below — a List, declared in terms of what an object can do (Describable) rather than what it concretely is (BookEdition). The catalog that follows never needs to change when a new kind of describable item shows up; it only ever asks for describe().

§27.9

SOLID — the same five ideas, with names

Five heuristics from this chapter have a well-known industry shorthand: SOLID. Three of the five letters are ideas this chapter already built, under different names — worth knowing, since the vocabulary shows up everywhere outside this book. The other two are new, and small enough to cover here directly.

Letter Principle Where it already happened
S Single Responsibility Named cohesion — §27.2's "one reason to change," reinforced in §27.5.
O Open/Closed Below.
L Liskov Substitution §27.7's rule of thumb — a subtype should be "substitutable... wherever the parent type is expected."
I Interface Segregation Below.
D Dependency Inversion Below.

O — Open for Extension, Closed for Modification

A well-designed class should be extendable without editing its own source. §27.8 already delivered this without naming it: Library depends only on Describable, so the day a MagazineIssue starts implementing it, Library gains a new capability without a single line of Library itself changing. It's open to that extension, and closed — untouched — against the modification that used to be required to support it.

I — Interface Segregation

A class shouldn't be forced to implement a method it has no honest answer for. Suppose Describable had grown a second method along the way:

interface Describable {
    String describe();
    String isbn();
}

Fine for BookEdition — every book has one. The moment a MagazineIssue tries to implement Describable too, isbn() has nothing truthful to return: magazines aren't sold by ISBN. Returning null, or throwing, both just relocate the problem to whoever calls isbn() next. The fix isn't a bigger interface — it's a narrower one, split along the line where "every describable thing" stops and "every book" begins:

interface Describable {
    String describe();
}

interface Identifiable {
    String isbn();
}

class BookEdition implements Describable, Identifiable { /* ... */ }
class MagazineIssue implements Describable { /* no isbn() to fake */ }

Library still only asks for Describable, so nothing about §27.8 changes. What changes is that MagazineIssue is never forced to answer a question that was never its to answer in the first place.

D — Dependency Inversion

"Program to an interface" (§27.8) says how to use a dependency. Dependency Inversion says something stronger, about who gets to define the interface in the first place. Picture a high-level scheduling policy that decides when reminders (§27.5) go out, built the direct way:

class ReminderScheduler {
    private final SmtpNotificationSender sender = new SmtpNotificationSender(smtpClient);

    void sendDueReminders(List<Notification> due) {
        for (Notification reminder : due) {
            sender.send(reminder);
        }
    }
}

ReminderScheduler is high-level policy — it decides when, not how. But as written, it depends directly on SmtpNotificationSender, a low-level detail (§27.5) that knows about hostnames and connections. Every low-level decision down there — switching providers, adding retries — now has a direct line back up into the class that's only supposed to be deciding schedules.

NotificationSender already exists as an abstraction (§27.5). Dependency Inversion says ReminderScheduler should depend on that, not on any concrete sender:

class ReminderScheduler {
    private final NotificationSender sender;

    ReminderScheduler(NotificationSender sender) {
        this.sender = sender;
    }

    void sendDueReminders(List<Notification> due) {
        for (Notification reminder : due) {
            sender.send(reminder);
        }
    }
}

and — this is the part "program to an interface" alone doesn't say — NotificationSender belongs conceptually to the high-level side. It's shaped around what a scheduler needs ("send this notification"), not around what SMTP happens to offer. SmtpNotificationSender depends on it, supplying an implementation that fits a contract someone else defined. Before, the dependency arrow ran from high-level policy straight down into a low-level detail; now both the policy and the detail point at the abstraction between them. That flip — the interface belonging to the consumer, not the implementer — is the "inversion" in the name.

"Belongs to the high-level side" is easy to assert and easy to miss in practice, because an interface still has to be written by someone, and it's tempting to let whoever writes the low-level code shape it around what they already have. Imagine NotificationSender had been designed that way instead — grown out of SmtpClient's own steps rather than out of what a scheduler actually needs:

// shaped around SMTP, not around "send a notification"
interface NotificationSender {
    void connect(String host, int port);
    void authenticate(String username, String password);
    void deliver(String to, String subject, String body);
    void disconnect();
}

This still satisfies "program to an interface" — ReminderScheduler could still depend on it instead of a concrete class. But look at what it forces on anything else that wants to implement it. A push-notification service has no host, no port, and nothing resembling a login step; a PushNotificationSender would have to fake connect() and disconnect() as empty methods just to satisfy a shape that only ever made sense for SMTP. The interface didn't remove the coupling to SMTP — it just spread SMTP's vocabulary onto every future implementation, forever, because the low-level detail got to define the contract.

NotificationSender as actually written — void send(Notification notification) — has no such residue, because it was never derived from any one implementation's steps. It was derived from the one sentence ReminderScheduler would use to describe the job on its own terms: send this notification. Every implementation, present or future, adapts to that sentence instead of the other way around — which is the entire test for whether an interface was actually inverted, or just extracted.

ReminderScheduler decides the shape of what it depends on — but notice it never decides which sender actually shows up. Nowhere in its own code does it write new SmtpNotificationSender(...); that would just reintroduce the exact coupling the interface was built to remove. Somebody still has to construct a real SmtpNotificationSender and hand it to a real ReminderScheduler — just not either of those two classes themselves:

public static void main(String[] args) {
    NotificationSender sender = new SmtpNotificationSender(smtpClient);
    ReminderScheduler scheduler = new ReminderScheduler(sender);

    scheduler.sendDueReminders(dueReminders);
}

main() here is doing a job neither collaborator is allowed to do itself: choosing concrete classes and wiring them together. That one spot — often main(), sometimes a small dedicated setup routine in a larger program — is usually called the composition root: the single place a program is allowed to write new SmtpNotificationSender(...) and new ReminderScheduler(...) in the same breath. Every other class in the program, on both sides of the abstraction, only ever sees NotificationSender — never the concrete type actually behind it.

Two Different Reasons to Reach for an Interface

NotificationSender and DiscountStrategy (§27.10) are built out of the exact same shape — an interface, a consumer holding a reference to it, one or more implementations supplied from outside. Lining them up side by side is what actually separates Dependency Inversion from the Strategy pattern it so often gets confused with:

DiscountStrategy NotificationSender
What varies Interchangeable business logic — several ways to compute the same kind of answer from the same kind of input. Interchangeable infrastructure — several ways to reach an external system that isn't part of the domain at all.
Who chooses which implementation A business rule — which promotion is active right now. Deployment or configuration — which provider this environment happens to use.
Usual name for the move Strategy pattern. "Abstracting the dependency" — Dependency Inversion, without necessarily reaching for a named pattern at all.

PercentageDiscount and a flat-amount discount both answer the exact same question — given this price, what's the discounted price? — just with different arithmetic. That's what makes it Strategy: the interface exists to let business rules pick between several interchangeable pieces of logic. SmtpNotificationSender isn't answering a question with alternate math; it's the only piece of this program that knows how to talk to a mail server at all. The interface exists so that ReminderScheduler — a policy about when, not how — never has to know that mail servers exist.

Both are Dependency Inversion: a high-level piece of code depending on an abstraction it owns, rather than a low-level detail. Strategy is simply the name people reach for when the thing being swapped is business logic. When it's infrastructure instead, the same move usually just gets called what it is — abstracting the dependency — without needing a pattern name attached at all.

§27.10

Putting It Together — a small Library

Every idea from this chapter lands in one small class. A Library holds a growing collection of anything describable, without knowing or caring what concrete types those things are:

import java.util.ArrayList;
import java.util.List;

class Library {
    private final List<Describable> holdings = new ArrayList<>();

    void add(Describable item) {
        holdings.add(item);
    }

    void printCatalog() {
        for (Describable item : holdings) {
            System.out.println(item.describe());
        }
    }
}
Book cleanCode = new Book("Clean Code", "Robert C. Martin");
Book effectiveJava = new Book("Effective Java", "Joshua Bloch");

Library library = new Library();
library.add(new BookEdition(cleanCode, "978-0132350884", 464, "Hardcover"));
library.add(new BookEdition(cleanCode, "978-0132766789", 464, "E-book"));
library.add(new BookEdition(effectiveJava, "978-0134685991", 412, "Paperback"));

library.printCatalog();
// Clean Code by Robert C. Martin (Hardcover, 464 pages, ISBN 978-0132350884)
// Clean Code by Robert C. Martin (E-book, 464 pages, ISBN 978-0132766789)
// Effective Java by Joshua Bloch (Paperback, 412 pages, ISBN 978-0134685991)

Trace what each class is actually responsible for: Book owns a work's identity and nothing else — cohesive, immutable, and entirely unaware anything called BookEdition exists. BookEdition owns everything specific to one printing, reaches into its Book only through its own describe() method, and exposes that capability through Describable rather than its own concrete name. Library depends on nothing but that one interface, so it works unchanged the day a MagazineIssue or an AudiobookEdition starts implementing Describable too. Four small, single-purpose classes, wired together by composition and an interface, do what one overloaded Book class couldn't.

A Second Case Study: Product and Price

Every idea in this chapter came from one running example. Here's a second, smaller case, to check the same instincts transfer to a domain that has nothing to do with books. A beginner modeling a store's catalog might reasonably write this first:

class Product {
    String name;
    double price;
    double discount;
}

It compiles, and for a single product with no promotions running, it even works. The trouble is the same shape problem §27.1 opened with: name answers "what is this product?" — it changes when the catalog is corrected, rarely, by whoever manages product data. price and discount answer a completely different question, "what does this cost right now?" — and that changes constantly, on its own schedule, driven by a pricing or marketing team who has no reason to ever touch name. Two unrelated reasons to change, sitting in one class — the same failure of cohesion §27.2 named, just wearing a different domain's clothes.

There's a second problem Book's version didn't have to face quite this directly: nothing anywhere actually knows how to turn price and discount into what a customer actually pays. That arithmetic ends up rewritten, slightly differently, at every call site that needs a final price:

double finalPrice = product.price - (product.price * product.discount / 100); // written out, again, wherever it's needed

That's Tell, Don't Ask (§27.4) failing before the object even had a chance: outside code is asking for two raw numbers and doing the business's own math itself, instead of telling something "give me the price," and letting the answer come back already computed.

A single Price class holding one amount and one discount would fix the symptom, but it would quietly keep the real disease: a store's prices aren't one number that occasionally gets overwritten, and a promotion isn't a permanent property of a product either. A price is valid for a period — a laptop might cost $1200 starting January 1st, then $1100 once a supplier's costs drop in March. A promotion is its own separate event, run by a marketing team on its own calendar, layered temporarily on top of whatever base price happens to be in effect — Black Friday doesn't replace the laptop's price, it just discounts it for a few days. Squeezing both of those into one mutable field would only move the cohesion problem, not solve it.

Both a Price and a Promotion need the exact same thing: a start, an end, and a way to ask "does this moment fall inside?" That's a fourth class in its own right, ValidityPeriod, rather than two copies of the same three lines. It's built on Instant (§25.4), Java's type for a single, time-zone-independent point on the timeline — exactly what a validity window needs when the business rule behind it is global, the way this online store's promotions are:

class ValidityPeriod {
    private final Instant start;
    private final Instant end; // null = open-ended, still in effect

    ValidityPeriod(Instant start, Instant end) {
        this.start = start;
        this.end   = end;
    }

    boolean contains(Instant moment) {
        boolean started  = !moment.isBefore(start);
        boolean notEnded = end == null || !moment.isAfter(end);
        return started && notEnded;
    }
}
Why Instant here — but not always

The right type isn't fixed; it depends on what the business actually means by "starts on July 1." A single, physical store announcing "our sale starts when we open on July 1" genuinely means a LocalDate — the store has one time zone, and that date, interpreted in the store's own zone, is the whole meaning. There'd be nothing wrong with a ValidityPeriod built on LocalDate start, end for that business. But Product here is modeling an online store — "the discount is active from 00:00 UTC on July 1" means the same instant for every shopper, everywhere, regardless of their own time zone. Instant is correct because this business rule is global by nature, not because Instant is universally the "right" choice for every validity period — a later, region-aware version of this same catalog might need a ValidityPeriod per market instead, each built on whichever type actually matches how that market's promotions are meant.

A Price now knows an amount and the period it's valid for:

class Price {
    private final double amount;
    private final ValidityPeriod period;

    Price(double amount, ValidityPeriod period) {
        this.amount = amount;
        this.period = period;
    }

    boolean isActiveAt(Instant moment) {
        return period.contains(moment);
    }

    double amount() {
        return amount;
    }
}

A Promotion knows nothing about amounts at all — it pairs a ValidityPeriod of its own with a DiscountStrategy, the exact interface §14.11 already built for "one of several interchangeable ways to discount a price":

class Promotion {
    private final DiscountStrategy discount;
    private final ValidityPeriod period;

    Promotion(DiscountStrategy discount, ValidityPeriod period) {
        this.discount = discount;
        this.period   = period;
    }

    boolean isActiveAt(Instant moment) {
        return period.contains(moment);
    }

    double apply(double amount) {
        return discount.apply(amount);
    }
}

Product doesn't hold one price and one discount — it holds every price it has ever had, and every promotion it has ever run, and answers one question: what does this cost at a given moment?

import java.time.Instant;
import java.util.ArrayList;
import java.util.List;

class Product {
    private final String name;
    private final List<Price> priceHistory = new ArrayList<>();
    private final List<Promotion> promotions = new ArrayList<>();

    Product(String name) {
        this.name = name;
    }

    void setPrice(Price price) {
        priceHistory.add(price);
    }

    void addPromotion(Promotion promotion) {
        promotions.add(promotion);
    }

    double priceAt(Instant moment) {
        double base = basePriceAt(moment);
        for (Promotion promotion : promotions) {
            if (promotion.isActiveAt(moment)) {
                return promotion.apply(base);
            }
        }
        return base;
    }

    private double basePriceAt(Instant moment) {
        for (Price price : priceHistory) {
            if (price.isActiveAt(moment)) {
                return price.amount();
            }
        }
        throw new IllegalStateException(name + " has no price in effect at " + moment);
    }
}
Product laptop = new Product("Laptop");

ValidityPeriod sinceLaunch = new ValidityPeriod(Instant.parse("2024-01-01T00:00:00Z"), null);
laptop.setPrice(new Price(1200.00, sinceLaunch));

ValidityPeriod blackFriday = new ValidityPeriod(Instant.parse("2024-11-24T00:00:00Z"),
                                                 Instant.parse("2024-11-30T23:59:59Z"));
laptop.addPromotion(new Promotion(new PercentageDiscount(0.10), blackFriday));

System.out.println(laptop.priceAt(Instant.parse("2024-06-15T00:00:00Z"))); // 1200.0 — no promotion active then
System.out.println(laptop.priceAt(Instant.parse("2024-11-27T12:00:00Z"))); // 1080.0 — Black Friday promotion active

Product now has-a list of Prices and a list of Promotions (§27.3), not one of each — because "what did this cost historically" and "what promotions have run" are both things a real catalog needs to answer, not just "what does it cost right now." Every caller still just tells laptop a moment and asks for the price; nothing outside Product ever touches priceHistory or promotions directly, and nothing outside Promotion ever needs to know whether it's a percentage or a flat amount off — that question stayed exactly where §14.11 already put it.

The same lesson, from the other direction ← §13

§13's MotorVehicle deliberately kept rental pricing off the vehicle hierarchy entirely, for this exact reason: pricing is a business rule, not a fact about what a vehicle is, so it has no business living on the class that represents the vehicle itself. Product and Price is what that same instinct looks like followed all the way through, on the other side of the same problem — two classes instead of one, exactly because "what this thing is" and "what it costs" are two different reasons to change.

§27.11

Key Takeaways

  • A class should have one reason to change (cohesion). If a class's fields split naturally into unrelated groups, that's usually two classes wearing one name.
  • Composition — one class holding a reference to another — models "has-a" relationships without duplicating state, and keeps the dependency running in one direction only.
  • Tell, Don't Ask: give an object a method that does the whole job, instead of pulling its parts out and doing the job yourself from the outside.
  • A method belongs on the class whose own state it's actually built out of. A method whose body is mostly about something the class doesn't own — a network client, a file system, another service — is a collaboration wearing one object's name; give that job to whichever object actually holds what it needs.
  • Every method is a Query (reads, never mutates) or a Command (mutates) — never comfortably both. A method whose name promises one but does the other is a recurring source of surprising bugs.
  • Prefer composition to inheritance by default. Reach for extends only when the relationship is genuinely "is-a-more-specific-kind-of" and the subtype should be usable anywhere the supertype is expected.
  • Program to an interface, not a concrete class, wherever code needs to work with a capability rather than one specific implementation of it — it's what let Library stay unchanged as new describable types show up.
  • None of this is Java-specific. Every idea in this chapter — cohesion, composition, encapsulation, favoring composition over inheritance, coding to an interface — applies in any object-oriented language; Java's classes, interfaces, and references are simply the vocabulary this tutorial used to build them.
