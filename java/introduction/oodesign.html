Chapter 26

Object-Oriented Design

Every mechanism up to this point — classes, references, inheritance, interfaces, generics, collections — is a tool. This chapter is about judgment: how to decide what a class should be responsible for, how objects should relate to one another, and why some technically-correct designs still turn into a mess six months later. One running example, a small library catalog, carries the whole chapter.

§26.1

The Problem — a class that knows too much

Say we're modeling books for a small library system. A first, entirely reasonable-looking attempt:

class Book {
    String title;
    String author;
    String isbn;
    int    pageCount;
    String format;      // "Hardcover", "Paperback", "E-book"
}

This compiles, and for a single book it works fine. The trouble shows up the moment reality does what it actually does: the same book — same title, same author, same content — routinely exists as more than one physical or digital product. Clean Code by Robert C. Martin is a hardcover, a paperback, and an e-book, each with its own ISBN and its own page count, but there's still only one title and one author to speak of.

Model that with the class above, and each edition needs its own Book object:

Book hardcover = new Book();
hardcover.title = "Clean Code";
hardcover.author = "Robert C. Martin";
hardcover.isbn = "978-0132350884";
hardcover.pageCount = 464;
hardcover.format = "Hardcover";

Book ebook = new Book();
ebook.title = "Clean Code";       // typed again
ebook.author = "Robert C. Martin"; // typed again
ebook.isbn = "978-0132766789";
ebook.pageCount = 464;
ebook.format = "E-book";

title and author are copied by hand into every edition. Nothing in the compiler enforces that both copies agree — a typo in one, a correction applied to the other, and the two "same" books silently disagree about their own title. This isn't a syntax problem or a bug in any single method. It's a shape problem: one class is carrying two different ideas — a book, the work, and an edition, one particular printing of it — squeezed into a single set of fields with no way to say which is which.

Design vs. mechanics

Everything so far in this tutorial has asked "how do I make Java do this?" This chapter asks a different question: "given that Java can do several things, which one should this class actually be?" That's object-oriented design (OOD) — the practice of deciding what classes exist, what each one owns, and how they refer to each other, independent of any single language feature.

§26.2

Cohesion — one reason to change

Book, as first written, has at least two separate reasons someone might need to edit it: something about the underlying work changed (the author's name was printed wrong), or something about one particular printing changed (a paperback edition was released with a new ISBN). Those are unrelated changes, arriving from unrelated sources, both landing on the same class.

A class with a single, clear reason to change is cohesive. A class carrying several unrelated reasons to change is doing more than one job — and the more jobs it does, the more often some change touches it, and the harder it gets to reason about what any one field actually means. Cohesion isn't a Java feature; it's a property of the design, and it's the first thing worth checking whenever a class's field list starts feeling like a grab bag.

The fix is to split the two ideas into two classes — one for the work itself, one for a specific printing of it:

class Book {
    private final String title;
    private final String author;

    Book(String title, String author) {
        this.title  = title;
        this.author = author;
    }

    String title()  { return title; }
    String author() { return author; }
}

Book now has exactly one reason to change: the work's own identity — its title, its author. Nothing about ISBNs, page counts, or formats belongs here at all.

Connection ← §11.3

Book is built the way §11 built Point: private final fields, set once in the constructor, read through accessor methods, no setters. A book's title and author aren't the kind of state that should ever silently change out from under whoever's holding a reference to it — immutability isn't just a nice-to-have here, it's what makes Book safe to share across every edition that refers to it, which is exactly what §26.3 does next.

§26.3

Composition, or "Has-A" — building BookEdition out of Book

The second idea — one specific printing — becomes its own class, BookEdition. It doesn't repeat title and author; instead, it holds a reference to the Book it's an edition of:

class BookEdition {
    private final Book   book;
    private final String isbn;
    private final int    pageCount;
    private final String format;

    BookEdition(Book book, String isbn, int pageCount, String format) {
        this.book      = book;
        this.isbn      = isbn;
        this.pageCount = pageCount;
        this.format    = format;
    }

    Book   book()      { return book; }
    String isbn()      { return isbn; }
    int    pageCount() { return pageCount; }
    String format()    { return format; }
}

Now the three editions of Clean Code share one Book, instead of each carrying its own disconnected copy of the title and author:

Book cleanCode = new Book("Clean Code", "Robert C. Martin");

BookEdition hardcover = new BookEdition(cleanCode, "978-0132350884", 464, "Hardcover");
BookEdition paperback = new BookEdition(cleanCode, "978-0132760800", 464, "Paperback");
BookEdition ebook      = new BookEdition(cleanCode, "978-0132766789", 464, "E-book");

System.out.println(hardcover.book() == ebook.book()); // true — the very same Book object

Three BookEdition objects, three distinct ISBNs — and exactly one Book underneath all of them. Correct the title once, on the one shared Book, and every edition reflects it, because every edition was never holding its own copy in the first place — just a reference to the one that exists.

Connection ← §2.4

This is nothing more than a reference field, the same mechanism §2.4 introduced with Rectangle's origin and border. BookEdition doesn't contain a Book — it holds an address pointing at one, on the heap, possibly shared with other objects that hold the same address. Naming that relationship is what's new here: whenever one class holds a reference to another as a field, that's composition — a "has-a" relationship, as opposed to the "is-a" relationship inheritance models (§9). A BookEdition has a Book; it is not, itself, a kind of Book.

Composition is also how coupling gets managed deliberately. BookEdition depends on Book — it has to, to do its job — but the dependency runs in one direction only. Book has no field, no method, no import that mentions BookEdition at all; it has no idea editions exist. That asymmetry matters: Book can be reused, tested, or changed on its own terms, entirely unaware of anything built on top of it.

§26.4

Tell, Don't Ask — encapsulation, revisited

With book() exposed, it's tempting to reach straight through a BookEdition to assemble a description from the outside:

// works, but every caller has to know BookEdition's internal shape
String description = edition.book().title() + " by " + edition.book().author()
        + " (" + edition.format() + ", " + edition.pageCount() + " pages, ISBN "
        + edition.isbn() + ")";

Every piece of code that ever wants a description now needs to know that a BookEdition holds a Book, that a Book exposes title() and author(), and the exact order and punctuation the description is assembled in. Change any of that internal shape — say, Book gains a separate publisher field that should appear in the description — and every one of those call sites needs to change with it.

The fix is to ask the object to do the work, instead of asking it for its parts and doing the work yourself — a habit compact enough to have its own name, Tell, Don't Ask:

class BookEdition {
    // ...fields and constructor as before...

    String describe() {
        return "%s by %s (%s, %d pages, ISBN %s)"
                .formatted(book.title(), book.author(), format, pageCount, isbn);
    }
}
System.out.println(edition.describe());
// Clean Code by Robert C. Martin (Hardcover, 464 pages, ISBN 978-0132350884)

Now there's exactly one place that knows how a description is built, and it's the one class actually holding the data. Every caller just says describe yourself — a single method call, unaffected by whatever BookEdition's fields look like internally, today or after the next change.

Connection ← §10.4

§10 framed encapsulation as hiding a field behind private so its invariants can't be broken from outside. Tell, Don't Ask is the same idea applied one level up: hide not just the data, but the logic that uses it, behind a method that does the whole job. A getter that just hands back a field is sometimes unavoidable — book() still exists above — but reaching for one by default, everywhere, is exactly the reflex this section pushes back on.

§26.5

Composition over Inheritance — why BookEdition doesn't extend Book

Inheritance (§9) was the tutorial's first tool for reusing another class's fields and methods, so it's worth confronting directly: why not simply write class BookEdition extends Book, and inherit title and author for free?

// tempting, but wrong here
class BookEdition extends Book {
    private final String isbn;
    private final int    pageCount;
    private final String format;

    BookEdition(String title, String author, String isbn, int pageCount, String format) {
        super(title, author);
        this.isbn      = isbn;
        this.pageCount = pageCount;
        this.format    = format;
    }
}

Two problems, and neither is about syntax:

The "is-a" test fails. §9.1 grounded inheritance in a taxonomy — extends should read as "a more specific kind of." A hardcover is a kind of physical object, sure, but is it a more specific kind of Book, the way a Dog is a more specific kind of Animal? Not really — a hardcover isn't a specialized book, it's a printing of a book. The relationship is "belongs to," not "is a kind of," and §26.3 already gave that relationship its correct name: composition.

The original problem comes right back. Each BookEdition subclass instance still carries its own inherited copy of title and author — inheritance doesn't create sharing, it copies the parent's field layout into every subclass object. Constructing a hardcover and an e-book "edition" of the same work still means passing "Clean Code" and "Robert C. Martin" into two separate constructor calls, with no single shared Book object underneath either one. Exactly the duplication §26.1 identified — inheritance just hides it behind extends instead of fixing it.

A rule of thumb

When one object needs the data or behavior of another, composition — a reference field — should be the default choice. Reach for inheritance only when the relationship genuinely is "is-a-more-specific-kind-of," and the two classes are meant to share one common notion of identity, substitutable for each other wherever the parent type is expected (§9.5's polymorphism). "I want to reuse some fields" is not, by itself, a reason to inherit — composition reuses the exact same fields without committing to a taxonomy that doesn't actually hold.

§26.6

Program to an Interface, Not an Implementation

The library catalog will eventually hold more than books — perhaps magazines, or audiobooks, later. Every one of those needs to be described, the same way a BookEdition does. Rather than writing catalog code that only understands BookEdition specifically, name the capability itself as an interface (§13):

interface Describable {
    String describe();
}

class BookEdition implements Describable {
    // ...fields and constructor as before...

    @Override
    public String describe() {
        return "%s by %s (%s, %d pages, ISBN %s)"
                .formatted(book.title(), book.author(), format, pageCount, isbn);
    }
}

Nothing about describe()'s body changed — only its declaration now carries @Override, satisfying a contract named Describable instead of just happening to have a method with that name.

Connection ← §13.11, ← §18.3

This is interface-typed programming exactly as §13.11 described it, now paired with the collections framework from §18: a variable, parameter, or — as below — a List, declared in terms of what an object can do (Describable) rather than what it concretely is (BookEdition). The catalog that follows never needs to change when a new kind of describable item shows up; it only ever asks for describe().

§26.7

Putting It Together — a small Library

Every idea from this chapter lands in one small class. A Library holds a growing collection of anything describable, without knowing or caring what concrete types those things are:

import java.util.ArrayList;
import java.util.List;

class Library {
    private final List<Describable> holdings = new ArrayList<>();

    void add(Describable item) {
        holdings.add(item);
    }

    void printCatalog() {
        for (Describable item : holdings) {
            System.out.println(item.describe());
        }
    }
}
Book cleanCode = new Book("Clean Code", "Robert C. Martin");
Book effectiveJava = new Book("Effective Java", "Joshua Bloch");

Library library = new Library();
library.add(new BookEdition(cleanCode, "978-0132350884", 464, "Hardcover"));
library.add(new BookEdition(cleanCode, "978-0132766789", 464, "E-book"));
library.add(new BookEdition(effectiveJava, "978-0134685991", 412, "Paperback"));

library.printCatalog();
// Clean Code by Robert C. Martin (Hardcover, 464 pages, ISBN 978-0132350884)
// Clean Code by Robert C. Martin (E-book, 464 pages, ISBN 978-0132766789)
// Effective Java by Joshua Bloch (Paperback, 412 pages, ISBN 978-0134685991)

Trace what each class is actually responsible for: Book owns a work's identity and nothing else — cohesive, immutable, and entirely unaware anything called BookEdition exists. BookEdition owns everything specific to one printing, reaches into its Book only through its own describe() method, and exposes that capability through Describable rather than its own concrete name. Library depends on nothing but that one interface, so it works unchanged the day a MagazineIssue or an AudiobookEdition starts implementing Describable too. Four small, single-purpose classes, wired together by composition and an interface, do what one overloaded Book class couldn't.

A Second Case Study: Product and Price

Every idea in this chapter came from one running example. Here's a second, smaller case, to check the same instincts transfer to a domain that has nothing to do with books. A beginner modeling a store's catalog might reasonably write this first:

class Product {
    String name;
    double price;
    double discount;
}

It compiles, and for a single product with no promotions running, it even works. The trouble is the same shape problem §26.1 opened with: name answers "what is this product?" — it changes when the catalog is corrected, rarely, by whoever manages product data. price and discount answer a completely different question, "what does this cost right now?" — and that changes constantly, on its own schedule, driven by a pricing or marketing team who has no reason to ever touch name. Two unrelated reasons to change, sitting in one class — the same failure of cohesion §26.2 named, just wearing a different domain's clothes.

There's a second problem Book's version didn't have to face quite this directly: nothing anywhere actually knows how to turn price and discount into what a customer actually pays. That arithmetic ends up rewritten, slightly differently, at every call site that needs a final price:

double finalPrice = product.price - (product.price * product.discount / 100); // written out, again, wherever it's needed

That's Tell, Don't Ask (§26.4) failing before the object even had a chance: outside code is asking for two raw numbers and doing the business's own math itself, instead of telling something "give me the price," and letting the answer come back already computed.

A single Price class holding one amount and one discount would fix the symptom, but it would quietly keep the real disease: a store's prices aren't one number that occasionally gets overwritten, and a promotion isn't a permanent property of a product either. A price is valid for a period — a laptop might cost $1200 starting January 1st, then $1100 once a supplier's costs drop in March. A promotion is its own separate event, run by a marketing team on its own calendar, layered temporarily on top of whatever base price happens to be in effect — Black Friday doesn't replace the laptop's price, it just discounts it for a few days. Squeezing both of those into one mutable field would only move the cohesion problem, not solve it.

Both a Price and a Promotion need the exact same thing: a start, an end, and a way to ask "does this moment fall inside?" That's a fourth class in its own right, ValidityPeriod, rather than two copies of the same three lines. It's built on Instant (§24.4), Java's type for a single, time-zone-independent point on the timeline — exactly what a validity window needs when the business rule behind it is global, the way this online store's promotions are:

class ValidityPeriod {
    private final Instant start;
    private final Instant end; // null = open-ended, still in effect

    ValidityPeriod(Instant start, Instant end) {
        this.start = start;
        this.end   = end;
    }

    boolean contains(Instant moment) {
        boolean started  = !moment.isBefore(start);
        boolean notEnded = end == null || !moment.isAfter(end);
        return started && notEnded;
    }
}
Why Instant here — but not always

The right type isn't fixed; it depends on what the business actually means by "starts on July 1." A single, physical store announcing "our sale starts when we open on July 1" genuinely means a LocalDate — the store has one time zone, and that date, interpreted in the store's own zone, is the whole meaning. There'd be nothing wrong with a ValidityPeriod built on LocalDate start, end for that business. But Product here is modeling an online store — "the discount is active from 00:00 UTC on July 1" means the same instant for every shopper, everywhere, regardless of their own time zone. Instant is correct because this business rule is global by nature, not because Instant is universally the "right" choice for every validity period — a later, region-aware version of this same catalog might need a ValidityPeriod per market instead, each built on whichever type actually matches how that market's promotions are meant.

A Price now knows an amount and the period it's valid for:

class Price {
    private final double amount;
    private final ValidityPeriod period;

    Price(double amount, ValidityPeriod period) {
        this.amount = amount;
        this.period = period;
    }

    boolean isActiveAt(Instant moment) {
        return period.contains(moment);
    }

    double amount() {
        return amount;
    }
}

A Promotion knows nothing about amounts at all — it pairs a ValidityPeriod of its own with a DiscountStrategy, the exact interface §13.11 already built for "one of several interchangeable ways to discount a price":

class Promotion {
    private final DiscountStrategy discount;
    private final ValidityPeriod period;

    Promotion(DiscountStrategy discount, ValidityPeriod period) {
        this.discount = discount;
        this.period   = period;
    }

    boolean isActiveAt(Instant moment) {
        return period.contains(moment);
    }

    double apply(double amount) {
        return discount.apply(amount);
    }
}

Product doesn't hold one price and one discount — it holds every price it has ever had, and every promotion it has ever run, and answers one question: what does this cost at a given moment?

import java.time.Instant;
import java.util.ArrayList;
import java.util.List;

class Product {
    private final String name;
    private final List<Price> priceHistory = new ArrayList<>();
    private final List<Promotion> promotions = new ArrayList<>();

    Product(String name) {
        this.name = name;
    }

    void setPrice(Price price) {
        priceHistory.add(price);
    }

    void addPromotion(Promotion promotion) {
        promotions.add(promotion);
    }

    double priceAt(Instant moment) {
        double base = basePriceAt(moment);
        for (Promotion promotion : promotions) {
            if (promotion.isActiveAt(moment)) {
                return promotion.apply(base);
            }
        }
        return base;
    }

    private double basePriceAt(Instant moment) {
        for (Price price : priceHistory) {
            if (price.isActiveAt(moment)) {
                return price.amount();
            }
        }
        throw new IllegalStateException(name + " has no price in effect at " + moment);
    }
}
Product laptop = new Product("Laptop");

ValidityPeriod sinceLaunch = new ValidityPeriod(Instant.parse("2024-01-01T00:00:00Z"), null);
laptop.setPrice(new Price(1200.00, sinceLaunch));

ValidityPeriod blackFriday = new ValidityPeriod(Instant.parse("2024-11-24T00:00:00Z"),
                                                 Instant.parse("2024-11-30T23:59:59Z"));
laptop.addPromotion(new Promotion(new PercentageDiscount(0.10), blackFriday));

System.out.println(laptop.priceAt(Instant.parse("2024-06-15T00:00:00Z"))); // 1200.0 — no promotion active then
System.out.println(laptop.priceAt(Instant.parse("2024-11-27T12:00:00Z"))); // 1080.0 — Black Friday promotion active

Product now has-a list of Prices and a list of Promotions (§26.3), not one of each — because "what did this cost historically" and "what promotions have run" are both things a real catalog needs to answer, not just "what does it cost right now." Every caller still just tells laptop a moment and asks for the price; nothing outside Product ever touches priceHistory or promotions directly, and nothing outside Promotion ever needs to know whether it's a percentage or a flat amount off — that question stayed exactly where §13.11 already put it.

The same lesson, from the other direction ← §12

§12's MotorVehicle deliberately kept rental pricing off the vehicle hierarchy entirely, for this exact reason: pricing is a business rule, not a fact about what a vehicle is, so it has no business living on the class that represents the vehicle itself. Product and Price is what that same instinct looks like followed all the way through, on the other side of the same problem — two classes instead of one, exactly because "what this thing is" and "what it costs" are two different reasons to change.

§26.8

Key Takeaways

  • A class should have one reason to change (cohesion). If a class's fields split naturally into unrelated groups, that's usually two classes wearing one name.
  • Composition — one class holding a reference to another — models "has-a" relationships without duplicating state, and keeps the dependency running in one direction only.
  • Tell, Don't Ask: give an object a method that does the whole job, instead of pulling its parts out and doing the job yourself from the outside.
  • Prefer composition to inheritance by default. Reach for extends only when the relationship is genuinely "is-a-more-specific-kind-of" and the subtype should be usable anywhere the supertype is expected.
  • Program to an interface, not a concrete class, wherever code needs to work with a capability rather than one specific implementation of it — it's what let Library stay unchanged as new describable types show up.
  • None of this is Java-specific. Every idea in this chapter — cohesion, composition, encapsulation, favoring composition over inheritance, coding to an interface — applies in any object-oriented language; Java's classes, interfaces, and references are simply the vocabulary this tutorial used to build them.
