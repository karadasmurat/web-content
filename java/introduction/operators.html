Chapter 5

Operators

Operators compute new values from existing ones. You've been reading them silently since §2 — width * height, i < numbers.length — this chapter names them properly, and adds the ones that produce true/false instead of numbers.

§5.1

Arithmetic Operators

+ - * / % — addition, subtraction, multiplication, division, and remainder. All five already appeared in earlier chapters without being named:

int width  = 10;
int height = 5;

int area   = width * height;  // 50
int perim  = 2 * (width + height); // 30
int diff   = width - height;  // 5

% (remainder) is the newest-looking one: 7 % 3 is 1 — what's left over after dividing 7 by 3 as many whole times as it goes. It's how you test "is this number even" (n % 2 == 0) without a division at all.

Integer division truncates

7 / 2 is 3, not 3.5 — when both operands are int, / performs whole-number division and discards the remainder entirely (that's what % recovers). Make either operand a double — 7 / 2.0 — and you get 3.5. This is exactly the primitive-type distinction from §2: the operator doesn't decide the answer's type, the operands' types do.

Type promotion: which type wins

Mix an int and a long in the same expression — 1L - 1 — and Java doesn't complain about the mismatch the way it would if you tried to store a String in an int variable. It just quietly produces a result. But a result of what type?

This is entirely a question about values — primitives hold their data directly (§2). The value 1 is the same value whether it's sitting in an int, a long, or a double — those are just differently-sized containers for it. So when an arithmetic operator's two operands are different primitive types, Java doesn't error out over the mismatch; it simply moves the value in the smaller container into a same-sized container as the other, so the operation ends up happening between two values of one shared type. The short version: the type with more storage room wins, and that's the type the result comes out as.

var res0 = 1 - 1;     // 0,   int
var res1 = 1L - 1;    // 0,   long
var res2 = 1L - 1L;   // 0,   long
var res3 = 1L - 1.0;  // 0.0, double
var res4 = 1 - 1.0;   // 0.0, double

res1: long vs. int — long has more room, so the int becomes a long and the result is long. res3: long vs. double — double has more room than long, so this time the long is the one that gets promoted, all the way to double.

Reading the literals

The L suffix on 1L makes that specific literal a long instead of the default int — with no suffix, 1 is always int. A decimal point, as in 1.0, already makes a literal a double by default (a trailing D, as in 1.0D, says the same thing explicitly, but is rarely needed). var — from §1.5 — just means each resN's type is inferred from whatever type comes out of its own expression; the comments show what that inferred type actually is for each line.

§5.2

Unary Operators

Every operator seen so far takes two operands — width * height, a == b. A unary operator takes exactly one, applied directly in front of it:

int temperature = 5;
int belowZero    = -temperature; // -5 — unary minus, negates the value

boolean isWide   = true;
boolean isNarrow = !isWide;      // false — unary NOT, inverts the boolean

- here isn't the subtraction operator from §5.1 wearing a disguise — with only one operand to its right and none to its left, it can only mean "negate this," never "subtract this from something." ! is exactly the NOT already used in §5.4's logical operators, just applied on its own rather than combined with && or ||. Both take a single value and produce a new one — nothing is stored anywhere, nothing about temperature or isWide themselves changes.

Unary plus

+ has a unary form too — +temperature — but it doesn't do anything a bare temperature wouldn't already do on its own; it exists mostly for symmetry with unary minus. In real code it's essentially never written.

Increment and Decrement

Increment (++) and decrement (--) are unary in the same sense as - and ! above — one operand, no second operand needed to make sense of them. They're different from - and ! in one important way, though: they don't just compute a new value and hand it back, they also write that new value into the variable's own slot, the same way += does (§5.5 covers that connection in full). What's specific to ++ and -- is that they come in two forms, and which one you use changes what the expression itself evaluates to:

int i = 5;

int a = i++; // postfix: use i's current value first, then increment
// a = 5, i = 6

int b = ++i; // prefix: increment first, then use the new value
// b = 7, i = 7

i++ (postfix) hands back the value i had before incrementing — a gets 5, the old value, even though i itself is 6 by the time the statement finishes. ++i (prefix) increments first, then hands back the already-updated value — b and i agree, both 7. Used on its own line, as a standalone statement, the two are indistinguishable — i++; and ++i; do the exact same thing to i. The difference only shows up the moment the expression's own value is used for something else, the way a and b just did.

§5.3

Comparison Operators

== != < > <= >= — these don't compute a number. They compute a boolean: true or false, one of §2's three primitives.

int a = 10;
int b = 20;

System.out.println(a == b); // false
System.out.println(a < b);  // true
System.out.println(a != b); // true

== always compares values — it never looks any deeper than what's sitting directly in the two variables' own slots. What counts as "the value" depends entirely on what kind of variable it is (§1):

  1. Primitive variables: the value is the primitive value itself, held directly. == compares that value.
  2. Reference variables: the value is an address (§2.3). == compares whether two reference variables hold the exact same address — whether they point to the exact same object in memory.

On primitives, that means == is exactly what it sounds like: same value. On reference variables, it means something more specific — and this is where §2.3 and §3 pay off.

Rectangle r1 = new Rectangle(10, 20);
Rectangle r2 = new Rectangle(10, 20);
Rectangle r3 = r1;

System.out.println(r1 == r2); // false
System.out.println(r1 == r3); // true
Stack
r1 = 0x9000
r3 = 0x9000
r2 = 0x9100
Heap
Rectangle @ 0x9000
width10
height20
Rectangle @ 0x9100
width10
height20

Two separate Rectangle objects, identical field values, at two different addresses — r1 and r3 both hold 0x9000 and point to the same one; r2 holds 0x9100, a different object entirely. == only ever looks at what's in the variable's own slot. (The Stack column above is ordered r1, r3, r2 rather than declaration order, purely so the two arrows sharing a target don't have to cross the one that doesn't.)

Same value, different address ← §2.3

A reference variable holds an address, not an object. == on two reference variables compares those addresses — nothing about the objects' fields is inspected. r1 and r2 are two separate Rectangles that happen to hold identical width/height values, at two different heap addresses — r1 == r2 is false. r3 = r1 copied r1's address ← §3, so r1 and r3 hold the exact same address — r1 == r3 is true.

This was already the story with String ← §8

§8 covered this exact split for String: == compares addresses, always; .equals(...) is a method a class can replace with its own version, to compare content instead. Rectangle hasn't replaced its inherited equals, so r1.equals(r2) is false right now, for the same reason r1 == r2 is. The mechanism for replacing an inherited method belongs to a later chapter (§9); the operator itself, here and always, only ever compares addresses.

Equality vs. Identity

Arrays make the same point even more starkly, since two arrays can hold the exact same elements, in the exact same order, and still be two entirely separate objects:

int[] a1 = { 1, 2, 3 };
int[] a2 = { 1, 2, 3 };

System.out.println(a1 == a2);              // false
System.out.println(Arrays.equals(a1, a2)); // true

a1 == a2 is false for the same reason r1 == r2 was — two new expressions, two separate heap allocations, two different addresses, regardless of what ended up inside either one. Arrays.equals(...) is a utility method that actually walks both arrays and compares their elements one by one — content, not address. == never does that work itself; something has to ask for it explicitly.

class Book {
    String ISBN;

    public Book(String ISBN) {
        this.ISBN = ISBN;
    }
}

Book b1 = new Book("978-123");
Book b2 = new Book("978-123");

Book b3 = null;
Book b4 = null;

System.out.println(b1 == b2); // false
System.out.println(b3 == b4); // true

b1 and b2 are built from identical ISBNs, but two separate new Book(...) calls still mean two separate objects at two separate addresses — b1 == b2 is false, the same identity-not-content rule as every other reference type. b3 == b4 is the interesting one: both variables hold null, and null is a single value (§2.3), not an address pointing at two different somethings. Two slots holding that exact same value compare equal, the same way two int slots holding 0 would — there's no object on the heap for either one to disagree about.

Identity vs. equality, named

This distinction has names worth using precisely: == on reference types tests identity — are these two variables looking at the exact same object? .equals(...) (or a helper like Arrays.equals(...)) tests equality — do these two objects (or arrays) represent the same content, even if they're not the same object? Java never confuses the two for you; picking the right one is always the caller's job.

§5.4

Logical Operators

&& || ! — AND, OR, and NOT. They combine or invert boolean values, and only ever operate on booleans:

int width  = 10;
int height = 20;

boolean isWide  = width > height;
boolean isSquare = width == height;

System.out.println(isWide || isSquare); // false
System.out.println(!isWide);            // true
Short-circuit evaluation

&& and || don't always evaluate both sides. a && b skips b entirely if a is already false — the whole expression can't become true no matter what b is. a || b skips b if a is already true, for the same reason in reverse. This isn't just an optimization: it's what makes a pattern like rect != null && rect.width > 0 safe. If rect is null, the left side is false, && short-circuits, and rect.width — which would throw the NullPointerException from §2.3 — never runs.

Short-Circuiting

"Skips b entirely" is easy to read past without quite believing — it's worth seeing proof that the right-hand side of an expression really can go completely unevaluated, not just unnecessary. Give each side a visible side effect, and watch which ones actually run:

public boolean isRaining() {
    System.out.println("Checking rain..");
    return true;
}

public boolean isSnowing() {
    System.out.println("Checking snow..");
    return true;
}
if (isRaining() || isSnowing()) {
    System.out.println("Better stay at home.");
} else {
    System.out.println("Let's go out!");
}
Checking rain..
Better stay at home.

isSnowing()'s own println never fires — not skipped as an optimization detail happening invisibly somewhere, but genuinely never called at all. isRaining() alone already returned true, which makes the whole || expression true no matter what isSnowing() would have returned, so Java never bothers calling it. This is exactly why short-circuiting matters beyond raw efficiency: if isSnowing() had a side effect of its own — logging, a network call, throwing an exception — none of that would happen either. The right-hand side of || and && isn't just a value that might go unused; it's code that may never run.

§5.5

Compound Assignment and Increment

Updating a variable based on its own current value is common enough to have shorthand:

int total = 0;
total += 10;   // same as: total = total + 10;   → 10
total -= 3;    // same as: total = total - 3;    → 7
total *= 2;    // same as: total = total * 2;    → 14

int i = 0;
i++;           // same as: i = i + 1;            → 1
i--;           // same as: i = i - 1;            → 0

Every one of these is still just an assignment — §3's rule applies unchanged, it copies a (newly computed) value into the variable's slot. Nothing new is happening underneath; it's purely shorter to write. i++ will show up constantly in §7's loops — it's the standard way to move a counter forward by one.

Scope of this chapter

The operators here are the ones the rest of this tutorial needs. Bitwise operators (&, |, ^, <<, >>), the ternary operator (?:), and the fine print of operator precedence are left for later.
