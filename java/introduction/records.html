Chapter 16

Records

§11 built a data carrier by hand — fields, constructor, getters — and that was already the leanest version. This chapter shows what was still missing, and the one-line declaration that supplies all of it at once.

§16.1

The Problem — the boilerplate a data carrier still needs

A class built to hold onto a couple of related values — a point's x and y, say — keeps its fields private (§10.3) and exposes them through accessor methods, a getter per field. That's already the ordinary shape any class takes: private state, plus a way to read it from outside. But a class meant to represent a plain value like this has to answer a few more questions correctly before it's actually safe to use, and none of those answers come for free.

Every class inherits an equals(Object) method whose default behavior is to compare addresses — the same rule == follows (§8.6 already established exactly this for String, before it overrides the default). Left alone, two separately-constructed points with identical coordinates would compare unequal, because they're two different objects, at two different addresses, even though the values they represent are the same. Fixing that means overriding equals — and once equals is overridden, hashCode has to be overridden too, with logic that agrees with it, or the object silently misbehaves the moment it's placed in a hash-based collection. And printing an object without its own toString prints whatever Object's default produces — a class name and a hash, not the coordinates anyone actually wants to see.

class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }
    public int getY() { return y; }

    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof Point)) return false;
        Point other = (Point) obj;
        return x == other.x && y == other.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }

    @Override
    public String toString() {
        return "Point[x=" + x + ", y=" + y + "]";
    }
}

Every line added here is entirely predictable from the two fields alone — nothing about equals, hashCode, or toString for a simple coordinate pair requires a human decision. And yet Java requires all of it spelled out by hand, for every class shaped like this one, with real, silent consequences for getting it wrong: forget hashCode after overriding equals, and a HashSet of points quietly stops finding duplicates it should have caught.

§16.2

record — one declaration, everything generated

A record declares its fields once, as part of the type's own header, and the compiler generates everything §16.1 had to write by hand:

record Point(int x, int y) { }

x and y, declared right there in the header, are called the record's components — the vocabulary used for this chapter's rest. This single line produces a class with: a private final field for each component; a constructor taking all of them and assigning them, in order; public accessor methods; and working equals, hashCode, and toString implementations, matching the ones written by hand in §16.1 exactly.

Point p1 = new Point(3, 4);
Point p2 = new Point(3, 4);

System.out.println(p1.x());        // 3
System.out.println(p1 == p2);      // false — different addresses, same as any two objects
System.out.println(p1.equals(p2)); // true — same coordinates
System.out.println(p1);            // Point[x=3, y=4]
The accessor is named x(), not getX()

A record's generated accessors take the field's own name directly, with no get prefix — x() and y(), not getX() and getY(). This is a deliberate, different convention from the hand-written getters in §11.3 and §16.1, not an inconsistency to work around.

An ordinary class (§10.3) can keep a private field entirely to itself — state that exists inside the object, used by its own methods, but never exposed to any caller at all. A record cannot do this. record Point(int x, int y) declares everything there is to know about the type, right there in the header — every field is listed, and the compiler generates a public accessor for every single one. There is no way to add a field to a record that isn't in the header, and so no way to give it a field the outside world can't see.

A record is transparent

Looking at record Point(int x, int y) tells you the complete state of any Point that will ever exist — nothing is hidden, because nothing can be hidden. This is a real trade-off, not a free upgrade over an ordinary class: it's exactly right for a pure, transparent bundle of values, and exactly wrong the moment a type needs internal state that isn't part of its public identity.

§16.3

Implicitly Final — nothing left to add, nothing left to override

Could someone write class ColoredPoint extends Point? A record says no — flatly, always, whether or not that would have been a reasonable thing to want. That's deliberate, not a limitation nobody got around to lifting.

A record is implicitly final — a keyword on a class that blocks it from ever being extended, the mirror image of abstract (§12.2): where abstract forces a class to be extended before it can produce an object, final forbids extending it at all. Point is exactly this kind of dead end, whether written as a record or not — there is no meaningful subclass of "a pair of coordinates," and a record makes that guarantee automatic rather than a matter of discipline.

The generated fields are implicitly private and final as well, exactly matching §11.3's two ingredients for immutability — except now neither one can be forgotten. There is no way to declare a setter that actually reassigns x or y, because nothing outside the generated constructor is ever allowed to touch those fields at all. Everything §11 achieved through careful, disciplined class design, a record achieves by construction.

§16.4

Shallow Immutability — final protects the reference, not what it points to

§11.4 already showed the gap this creates for an ordinary hand-written immutable class: final only locks a field's own slot — the address it holds — never what that address points to. A record's generated fields are just final fields (§16.3), so they inherit the exact same gap:

record Person(String name, List<String> hobbies) { }
Person p = new Person("Alice", new ArrayList<>(List.of("reading")));
p.hobbies().add("gaming"); // compiles and runs fine

The hobbies field is final in exactly the sense §16.3 described: it can never be reassigned to point at a different List. But the generated accessor, p.hobbies(), hands back that exact same List object — the live one, not a defensive copy — so anything holding p can call p.hobbies() and mutate it directly, entirely through the public accessor itself, no reflection or workaround required. hobbies the field is never reassigned by that .add(...) call; the one ArrayList it has always pointed to is just modified in place, through a reference that final was never protecting from writes to begin with.

A record is not "an immutable data container"

The accurate description is narrower: a record is a transparently shallow-immutable data container — every component reference is guaranteed final, but whether the object each one points to is itself immutable depends entirely on that component's own type. Person above is only genuinely, fully immutable if List<String> itself never changes after construction — which an ordinary ArrayList does not guarantee at all.

Java could have required every record component to be an immutable type, but that would rule out List, Map, and a long list of other everyday types as record components, making records far less useful in practice. Instead, Java enforces only what §16.3 already promised — the reference itself can't be reassigned — and leaves deep immutability to the programmer: passing in an already-immutable collection (List.copyOf(...), say) is what actually closes this gap, not the record keyword by itself.

§16.5

Compact Constructors — validation without repeating the parameter list

Sometimes a value needs validating before it's accepted — a negative coordinate might be nonsense for whatever Point represents in a given program. An ordinary constructor would restate the full parameter list and assign every field by hand, exactly like §16.1's hand-written version did. A record instead allows a compact constructor: the validation only, with no parameter list and no assignment statements at all.

record Point(int x, int y) {
    Point {
        if (x < 0 || y < 0) {
            throw new IllegalArgumentException("Coordinates cannot be negative");
        }
    }
}
Point p = new Point(-1, 4); // throws IllegalArgumentException

The compact constructor's body runs first, given x and y exactly as passed to new Point(...) — and once it finishes without throwing, the compiler still performs the ordinary field assignments automatically, the same assignments §16.2's plain record Point(int x, int y) { } always generated. Nothing about declaring a compact constructor removes that generated behavior; it only inserts a checkpoint in front of it.

§16.6

Records Can Do More — additional methods, implemented interfaces

A record's body isn't limited to a compact constructor — it can declare ordinary methods too, exactly like any class:

record Point(int x, int y) {
    double distanceFromOrigin() {
        return Math.sqrt(x * x + y * y);
    }
}
Point p = new Point(3, 4);
System.out.println(p.distanceFromOrigin()); // 5.0

distanceFromOrigin() is a method written the ordinary way, with full access to x and y — a record only automates the constructor, the accessors, and equals/hashCode/toString; everything else about writing a class still applies. A record can also implement an interface (§13.3), exactly the way any class does — the generated pieces and any interface's required methods coexist without conflict.

§16.7

When to Reach for One — a record is a shape, not a replacement

A record fits exactly the shape §11 was already building toward: a fixed bundle of values, set once at construction, never reassigned, compared by content rather than address. The moment a type needs mutable state, a meaningful subclass, or fields beyond what the constructor receives, it no longer fits that shape — and an ordinary class, written out in full, is still the right tool.

Scope of this chapter

Generic records, records used with switch and pattern matching, and manually overriding a single generated accessor while keeping the rest are all left for later. The core shape carries over unchanged: a record is the data carrier §11 already described — transparent, shallow-immutable by construction (§16.4), with the boilerplate written for you.
