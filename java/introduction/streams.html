Chapter 21

Streams

Every collection query so far has meant writing a loop by hand: one to filter, another to transform, a third to total something up — each one restating how to iterate right alongside what to compute. This chapter introduces the Stream API: a way to describe a pipeline of operations over a collection's elements — filter, transform, collect, aggregate — declaratively, using exactly the functional interfaces (§19) already in hand.

§21.1

The Problem — the same iteration, rewritten every time

Picture a small online store's order history, one Order per completed or pending purchase:

record Order(String customer, double amount, OrderStatus status) { }

enum OrderStatus { PENDING, SHIPPED, CANCELLED }

A perfectly ordinary business question: how much has actually gone out the door — the total value of every order that's shipped so far? That's a deliberately narrower question than "total revenue": a PENDING order might still count as a sale, and a CANCELLED one might never have been fulfilled at all, but neither one has actually shipped yet. With a List<Order> and nothing but what earlier chapters have already covered, answering it means a hand-written loop:

double shippedValue = 0;
for (Order order : orders) {
    if (order.status() == OrderStatus.SHIPPED) {
        shippedValue += order.amount();
    }
}

This works. But the next question — how many distinct customers have a shipped order? — means writing an entirely new loop, restating the same iteration machinery from scratch:

Set<String> shippedCustomers = new HashSet<>();
for (Order order : orders) {
    if (order.status() == OrderStatus.SHIPPED) {
        shippedCustomers.add(order.customer());
    }
}
int distinctCustomers = shippedCustomers.size();

Every new question about the same list means another loop, another local variable to accumulate into, another if restating "shipped" — the actual question (shipped value? customer count? largest order?) buried in nearly identical boilerplate each time. What changes between these two loops is genuinely small: what to check, and what to do with a match. What doesn't change — the mechanics of walking the list one element at a time — is written out in full, every single time regardless.

The real problem: "how," not just repetition

Rewriting the loop each time is a symptom, not the actual problem underneath it. Both loops above are imperative: they tell the computer exactly how to get the answer — start at zero, look at each element in order, check a condition, mutate a variable, move to the next one. Every line executes in a fixed, explicit sequence the code itself dictates, and nothing about that sequence is negotiable once it's written — there's no room for the JVM to notice, say, that a filter and a sum could be fused into a single pass instead of conceptually two.

A stream pipeline answers the same question declaratively — stating what result is wanted, not the step-by-step mechanics of producing it:

double shippedValue = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .mapToDouble(Order::amount)
        .sum();

Nothing here says "start a loop" or "keep a running total in a variable" — it says what's wanted: the shipped ones, their amounts, summed. §21.4 onward builds this syntax up piece by piece, but the shift in mindset matters more than the syntax itself: how the elements actually get walked, and whether that walk can be optimized, is now the Java runtime's responsibility, not something spelled out by hand in every single query written against this list.

§21.2

What Is a Stream — a pipeline, not a container

Picture a warehouse holding every item at once, indefinitely — and a conveyor belt threaded through it once, carrying each item past a sequence of stations before it's gone for good. A List is the warehouse. A Stream is the belt.

A Stream<T> is not a collection (§18) — it doesn't hold elements the way a List or a Set does, and it has no storage of its own. A Stream describes a sequence of operations to run over a source of elements — a List<Order>, an array, another collection entirely — without itself owning any of the data.

Two consequences follow directly from that:

  • A stream doesn't change the collection it's built from — running one over orders never adds, removes, or reorders anything in orders itself.
  • A stream is used exactly once. Once its elements have passed through to a result, that same Stream object can't be run again — a genuinely finished conveyor belt, not a warehouse to revisit.

The rest of this chapter is about what runs on that belt, and what comes out the other end.

Why a Stream has no get(index)

A natural question, coming from List: why can't a Stream be indexed directly, the way orders.get(3) reaches straight into a List<Order>? Two reasons, and both trace straight back to what a stream actually is.

The first is the abstraction §21.1 just built toward: filter and map describe what should happen to the data, not how to walk it — and "index 3" is an inherently how-shaped idea. It only means something if there's a fixed, numbered position for every element to sit in, the way a List's backing array guarantees. A pipeline offers no such guarantee: after a filter has discarded some orders and a map has reshaped the rest, "the third one" no longer corresponds to anything stable — there's no array underneath a stream for a number to index into.

The second reason is what that same abstraction makes possible: parallelism. Calling parallelStream() instead of stream() lets the same pipeline split its source into chunks and run them across multiple CPU cores at once — the exact "optimize the walk however's fastest" freedom an imperative loop's fixed, line-by-line sequence never allows. But once the source is split across threads, "index 5" stops meaning anything at all — which thread's chunk is it even in? Processing order across threads isn't guaranteed to match the source's original order. Retrofitting indexed access on top of that would mean every thread constantly synchronizing on a shared position counter — exactly the kind of coordination overhead parallelism exists to avoid in the first place. A Stream gives up indexing precisely so it's free to make that trade when it's worth making.

§21.3

Creating a Stream — starting the pipeline

Every Collection (§18.2) has a stream() method built in — the ordinary way a pipeline begins:

Stream<Order> orderStream = orders.stream();

Two other sources come up often enough to be worth naming directly: a fixed handful of values, and an existing array.

Stream<String> names = Stream.of("Ava", "Noah", "Mia");
Stream<Order> fromArray = Arrays.stream(orderArray);

None of these three actually do anything yet — creating a Stream is just naming a source. §21.8 comes back to exactly why nothing runs until much later.

Generating a stream instead of sourcing one

Every source so far has come from data that already exists — a List, an array, some literal values. Sometimes there's no source to point at yet: say the warehouse system needs to pre-assign a batch of order confirmation numbers before any of those orders exist. Stream.iterate(seed, UnaryOperator) builds a stream by repeatedly feeding each value back into a function to produce the next one:

Stream<Integer> confirmationNumbers = Stream.iterate(10230, n -> n + 1);

Stream.generate(Supplier) covers a different shape of need — each element independent of the last, rather than derived from it. A batch of fresh shipment tracking codes, say, each one pulled from a generator with no relationship to the one before it:

Stream<String> trackingCodes = Stream.generate(TrackingCodeGenerator::next);

Both of these are, deliberately, infinite — neither has any built-in stopping point, so calling one without limit(n) attached is a mistake that never finishes running:

// Dangerous — no limit(), so the terminal operation below never returns
Stream.iterate(10230, n -> n + 1).forEach(System.out::println);
// Correct — limit(n) turns the infinite stream into a finite one before anything runs
List<Integer> nextFiveConfirmationNumbers = Stream.iterate(10230, n -> n + 1)
        .limit(5)
        .collect(Collectors.toList());

It's worth pausing on why Stream.iterate(10230, n -> n + 1) is even legal to write, given that it describes a sequence with no end. An int[] of unlimited length would exhaust memory just being built — but this line doesn't build anything yet. Confirmation number 10231 doesn't exist anywhere until something actually asks for it; the stream isn't a container holding infinitely many numbers, it's a rule for producing the next one, applied only when a later step in the pipeline pulls a value forward. limit(5) is what tells that rule to stop being asked after five pulls — without it, whatever terminal operation comes next just keeps asking, forever. §21.8 covers exactly why a stream can get away with generating elements one at a time, on demand, like this.

§21.4

Intermediate Operations — describing the pipeline

An intermediate operation takes a Stream and returns another Stream, describing one more step in the pipeline without running any of it yet.

filter: keeping only what matches

filter(Predicate) (§19.8's Predicate<T>) keeps only elements matching a condition, discarding the rest:

Stream<Order> shipped = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED);

Every order in orders passes through the same Predicate; only the ones it returns true for continue on to whatever comes next in the pipeline.

Filtering with a predicate written for a supertype

filter's actual declaration is more permissive than "a Predicate<Order>" suggests:

Stream<T> filter(Predicate<? super T> predicate)

Notice the wildcard: ? super T, not a plain Predicate<T> — §15.9 covered exactly this wildcard; here it is doing real work in the standard library.

Suppose the business tracks a property shared across several kinds of records, each implementing one common interface:

interface Priced {
    double amount();
}

record Order(String customer, double amount, OrderStatus status) implements Priced { }
record Invoice(String reference, double amount, LocalDate dueDate) implements Priced { }

A rule like "is this over some threshold" doesn't care whether it's looking at an Order or an Invoice — only that whatever it's given can answer amount(). Writing the rule once, against the shared interface, makes it reusable everywhere Priced applies:

Predicate<Priced> isHighValue = priced -> priced.amount() > 100;

Without the wildcard, filter would only accept a Predicate<Order> exactly — a Predicate<Priced> would be rejected outright, even though a predicate that knows how to test any Priced obviously knows how to test the narrower Order it's actually handed. Predicate<? super T> is precisely what makes passing it in legal:

List<Order> highValueOrders = orders.stream()
        .filter(isHighValue)
        .collect(Collectors.toList());

The reasoning is the same producer/consumer split §15.9 already named, just running in the opposite direction: filter only ever hands elements to the predicate — it never receives anything back from it except true/false — so any predicate built for a supertype of Order is guaranteed to already know how to handle an actual Order. Requiring an exact Predicate<Order> would reject that perfectly safe, more reusable predicate for no real reason.

map: transforming, one-to-one

map(Function) transforms each element into something else, one-to-one — here, an Order into just its amount:

Stream<Double> shippedAmounts = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .map(Order::amount);

Order::amount is a method reference (§19.7) — shorthand for the lambda order -> order.amount(), meaning exactly the same thing with less noise. Because every intermediate operation returns a Stream, calls chain directly, one step flowing into the next: filter narrows down to the orders that matter, then map reshapes each surviving order into just the number this particular question needs.

sorted() and distinct() round out the common set — sorting elements and dropping duplicates, respectively — each one just another link added to the same chain.

limit and skip: slicing a stream

limit(n) cuts a stream off after its first n elements; skip(n) does the opposite, discarding the first n elements and keeping everything after them. A common, very concrete reason to want both together: paging through an order history, ten orders at a time.

int pageNumber = 3;
int pageSize = 10;

List<Order> page = orders.stream()
        .skip((pageNumber - 1) * pageSize) // skip the first 20 — pages 1 and 2
        .limit(pageSize)                   // take the next 10 — page 3
        .collect(Collectors.toList());

skip(20) discards orders 1 through 20, and limit(10) then keeps only the next 10 that follow — orders 21 through 30, exactly the slice page 3 needs. Change pageNumber and the same two lines serve any page, without a single hand-written loop counting positions.

limit is what's called a short-circuiting operation (§21.8): once n elements have made it through, it stops pulling more from upstream entirely, rather than continuing to walk the rest of the stream just to discard them. skip isn't short-circuiting on its own — it still has to walk past the elements it's discarding — but paired with limit downstream, as above, the pipeline as a whole still stops early once it has what it needs.

This is also exactly what makes limit the tool that turns one of §21.3's infinite streams finite. Applying skip and limit together to Stream.iterate(10230, n -> n + 1) reads the same way it did for a bounded list — skip past a few, then take a handful:

Stream.iterate(10230, n -> n + 1)
        .skip(3)
        .limit(5)
        .forEach(System.out::println); // 10233 10234 10235 10236 10237
§21.5

Terminal Operations — running the pipeline

A terminal operation is what actually runs everything described so far, producing a real result instead of another Stream — and it's the only thing that makes a pipeline do anything at all.

Primitive streams: IntStream, LongStream, DoubleStream

Every stream built so far has held a reference type — Stream<Order>, Stream<String>. Adding up every shipped order's amount — a double — by reaching for an ordinary map would produce a Stream<Double>: every single amount boxed into a Double object, then unboxed again each time a later step reads its value. That's real, repeated overhead across an entire order history, for a value that was a plain primitive to begin with (§2).

Java sidesteps it with three specialized stream types, one for each primitive worth optimizing around:

Stream type Primitive Boxed equivalent
IntStream int Stream<Integer>
LongStream long Stream<Long>
DoubleStream double Stream<Double>

mapToDouble (alongside mapToInt and mapToLong) is how a Stream<T> converts into one of these mid-pipeline, whenever the next step is a number rather than an object. Beyond skipping the boxing cost, these three types carry numeric-specific terminal operations a plain Stream<T> doesn't have at all — sum(), average(), max(), min() — which is exactly what makes answering §21.1's original question a one-line pipeline:

double shippedValue = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .mapToDouble(Order::amount)
        .sum();

Getting several statistics at once

A dashboard summarizing shipped orders rarely needs just the total — the count, the average, the smallest, and the largest are usually wanted together. Calling sum(), then average(), then max() on the same pipeline doesn't work: §21.2 already established that a stream is used exactly once, so a second terminal operation on the same source throws an exception rather than quietly re-running the pipeline. summaryStatistics() exists precisely for this — one pass, one terminal operation, every common statistic at once:

DoubleSummaryStatistics shippedStats = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .mapToDouble(Order::amount)
        .summaryStatistics();

System.out.println(shippedStats.getCount());   // how many shipped
System.out.println(shippedStats.getSum());     // shippedValue, computed once
System.out.println(shippedStats.getAverage());
System.out.println(shippedStats.getMin());
System.out.println(shippedStats.getMax());

summaryStatistics() only exists on the primitive streams from the table above — IntSummaryStatistics, LongSummaryStatistics, and DoubleSummaryStatistics are its three matching result types. A plain Stream<T> has no equivalent, for the same reason it has no sum(): without narrowing to a specific numeric primitive first, there's no single well-defined way to add two elements together.

Compare this to §21.1's hand-written loop: the same two ideas — which orders count, what to total — are still exactly what this code says, but the iteration itself is no longer written out by hand at all. count() answers "how many," forEach(Consumer) (§19.8) runs a side effect per element, and anyMatch/allMatch/noneMatch each take a Predicate and answer a yes-or-no question about the whole stream:

boolean hasCancelledOrders = orders.stream()
        .anyMatch(order -> order.status() == OrderStatus.CANCELLED);

reduce is the general-purpose terminal operation underneath several of these — combining every element into one result, one pairwise step at a time — but sum(), count(), and the *Match family cover the overwhelming majority of real questions without ever needing to reach for it directly.

Crossing back over: boxed and mapToObj

mapToInt, mapToLong, and mapToDouble each cross from a Stream<T> into the matching primitive stream. Two more methods make the same crossing in reverse — necessary because IntStream, LongStream, and DoubleStream don't support collect(Collector) the way a Stream<T> does, so getting a primitive stream's results into an ordinary List means converting back first.

boxed() is the plain version of that crossing: every element goes back into its wrapper type, with nothing else about it changed. Distinct shipped amounts, gathered into a List<Double>:

List<Double> distinctShippedAmounts = orders.stream()
        .mapToDouble(Order::amount)
        .distinct()
        .boxed()
        .collect(Collectors.toList());

DoubleStream has no Collectors-based collect() to call directly — boxed() is what turns it back into an ordinary Stream<Double>, at which point collect(Collectors.toList()) works exactly as it does everywhere else in this chapter.

mapToObj makes the same crossing, but transforms each primitive along the way rather than just wrapping it — useful when a primitive stream is really just an intermediate step toward building something else. A fresh batch of confirmation code strings, built straight from a range of numbers:

List<String> confirmationCodes = IntStream.rangeClosed(1, 5)
        .mapToObj(n -> "ORD-" + (10230 + n))
        .collect(Collectors.toList());
// ["ORD-10231", "ORD-10232", "ORD-10233", "ORD-10234", "ORD-10235"]

IntStream.rangeClosed(1, 5) is itself a primitive stream of the plain integers 1 through 5 — no Order or business object in sight yet. mapToObj is what turns each one into the actual String the pipeline is after, crossing from IntStream to Stream<String> in the same step that does the formatting.

§21.6

Collecting Results — collect() and Collectors

Not every pipeline ends in a single number or a yes/no answer — often, what's needed back is an actual collection. collect(Collector) is the terminal operation for exactly that — but a Collector itself is a small, general-purpose recipe for building up one result from a stream of elements: something to start with, a rule for folding the next element in, a rule for combining partial results. Writing one by hand for "gather these into a List" or "join these into one comma-separated String" every time would mean re-implementing the same handful of recipes in every codebase that ever calls collect().

Common Collectors

In practice, a small set of these recipes comes up over and over — enough that Java ships them ready-made, rather than leaving every developer to reimplement the same ones by hand. Collectors is that class: a library of pre-built Collectors, covering the overwhelming majority of what collect() is ever actually called with.

List<String> shippedCustomers = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .map(Order::customer)
        .collect(Collectors.toList());

Collectors.joining(", ") concatenates a stream of Strings directly, with a separator between each:

String customerList = shippedCustomers.stream()
        .collect(Collectors.joining(", "));
// "Ava Patel, Noah Kim, Mia Chen"

Collectors.groupingBy(Function) answers a different shape of question entirely — not "which ones match," but "how are they organized" — splitting a stream into a Map (§18.6) keyed by whatever the given function returns:

Map<OrderStatus, List<Order>> byStatus = orders.stream()
        .collect(Collectors.groupingBy(Order::status));

List<Order> pendingOrders = byStatus.get(OrderStatus.PENDING);

Every order in orders ends up in exactly one list, keyed by its own status — one pipeline replacing what would otherwise be three separate hand-written loops, one per status value.

§21.7

Streams and Optional<T> — when a result might not exist

Some questions about a stream don't have a guaranteed answer — the largest order in an empty list, the first shipped order when nothing has shipped yet. §20 already built the type for exactly this shape of problem, and several terminal operations return it directly rather than risking null (§20.2) or throwing:

Optional<Order> firstShipped = orders.stream()
        .filter(order -> order.status() == OrderStatus.SHIPPED)
        .findFirst();

findFirst() returns an empty Optional<Order> if no order matches, rather than null or an exception — the caller is compiler-nudged (§20.4) to actually handle the empty case, exactly the same guarantee Optional was built to provide everywhere else. max(Comparator) and min(Comparator) work the same way, each returning an Optional since an empty stream has no largest or smallest element to hand back:

Optional<Order> largestOrder = orders.stream()
        .max(Comparator.comparingDouble(Order::amount));

largestOrder.ifPresent(order ->
        System.out.println("Largest order: " + order.customer()));

ifPresent(Consumer) (§20.5) runs only if a value actually exists — no explicit if, no risk of dereferencing an order that was never there.

§21.8

Laziness and Pipelines — nothing runs until it has to

Every intermediate operation in this chapter so far has been described as building a pipeline, not running one — and that's meant literally. filter and map don't touch a single element the instant they're called; they just record what should happen, later, when something finally asks for a result:

Stream<Order> pipeline = orders.stream()
        .filter(order -> {
            System.out.println("Checking " + order.customer());
            return order.status() == OrderStatus.SHIPPED;
        });
System.out.println("Pipeline built — nothing printed above this line yet");

Nothing prints until a terminal operation is actually added and run — building the pipeline is just assembling a description of work, not performing it. This laziness is what makes findFirst() genuinely efficient rather than just convenient: given a thousand orders, it doesn't run filter across all of them and then grab the first survivor — it walks the source one element at a time, running every intermediate step on just that one element, and stops completely the instant a match is found. This is short-circuiting, and it's exactly why findFirst(), anyMatch(), and limit() can be dramatically cheaper than processing an entire stream to completion, even though nothing in how the pipeline was written suggested it should stop early.

Tracing a pipeline, element by element

The natural assumption, coming from loops, is that filter runs over the entire stream first, handing off a fully-filtered batch for map to then run over in a second pass. That's not what happens. Adding a trace to both steps reveals the real order of events:

List<Order> orders = List.of(
        new Order("Ava", 40.0, OrderStatus.SHIPPED),
        new Order("Noah", 25.0, OrderStatus.PENDING),
        new Order("Mia", 60.0, OrderStatus.SHIPPED));

List<Double> shippedAmounts = orders.stream()
        .filter(order -> {
            System.out.println("Filtering " + order.customer());
            return order.status() == OrderStatus.SHIPPED;
        })
        .map(order -> {
            System.out.println("Mapping " + order.customer());
            return order.amount();
        })
        .collect(Collectors.toList());

System.out.println(shippedAmounts);
Filtering Ava
Mapping Ava
Filtering Noah
Filtering Mia
Mapping Mia
[40.0, 60.0]

Ava is filtered and mapped before Noah is even looked at. Noah fails the filter and never reaches map at all — there's no second pass over "whatever survived filtering," because there was never a batch of survivors sitting around in the first place. Each order runs through the entire pipeline, one step immediately after the next, before the next order starts at the top. This is exactly what makes the laziness above possible: nothing is waiting for a full pass over one stage before the next stage can begin.

This is also exactly what §21.3's Stream.iterate was leaning on. An infinite stream has no "full pass" to wait for in the first place — there's no batch of confirmation numbers sitting in memory, finished or otherwise. Each number is produced right when the next pipeline step pulls it forward, the same one-at-a-time flow this section just traced through orders, and limit(5) simply stops the pulling after the fifth one. A stream with no end and a stream with a thousand elements are handled by exactly the same mechanism; the only difference is when something finally tells it to stop.

§21.9

When (Not) to Use Streams — readability over reflex

A stream pipeline isn't automatically the better choice just because it exists. A single, simple pass over a list — printing every order, say — often reads more clearly as a plain loop than as a pipeline built purely to imitate one:

// Needless — a stream adds a pipeline for a job with no filtering or transforming to describe
orders.stream().forEach(order -> System.out.println(order.customer()));
// Just as clear, and doesn't reach for machinery this simple task never needed
for (Order order : orders) {
    System.out.println(order.customer());
}

Streams earn their keep specifically when there's a real chain to describe — filter, then transform, then aggregate — collapsing several loops' worth of bookkeeping into one readable pipeline, the way §21.1's shipped-value calculation did. A single unconditional pass isn't that; it's just a loop wearing stream syntax.

A more serious anti-pattern: using map or forEach to mutate state outside the stream, rather than to compute or act on a value:

// Bad — map() is for transforming values, not for producing side effects
List<Double> totals = new ArrayList<>();
orders.stream().map(order -> {
    totals.add(order.amount()); // mutating something outside the pipeline
    return order.amount();
});

map's entire contract is transforming one value into another — reaching into a variable outside the pipeline to mutate it defeats the laziness §21.8 just described and makes the pipeline's actual behavior depend on side effects a reader has no reason to expect from a method named map. collect(Collectors.toList()) exists precisely so building a result list never requires this:

// Good — the result comes from the pipeline itself, no external mutation
List<Double> totals = orders.stream()
        .map(Order::amount)
        .collect(Collectors.toList());
§21.10

Key Takeaways

  • A Stream describes a pipeline of operations over a source's elements — it holds no data of its own, doesn't modify its source, and can only be run once.
  • stream(), Stream.of(...), and Arrays.stream(...) are the common ways to start a pipeline; none of them run anything by themselves.
  • Intermediate operations (filter, map, sorted, distinct, limit) each return a new Stream and describe one more pipeline step, lazily.
  • Terminal operations (forEach, count, sum, reduce, the *Match family, collect) are what actually run the pipeline and produce a real result.
  • Collectors.toList(), joining(), and groupingBy() are the common ways to gather a stream's results back into an ordinary collection.
  • Operations like findFirst(), max(), and min() return Optional<T> (§20) rather than risking null or an exception on an empty stream.
  • Nothing in a pipeline runs until a terminal operation triggers it, and short-circuiting operations like findFirst() and anyMatch() can stop processing long before the source is exhausted.
  • Reach for a stream when there's a real chain of filtering, transforming, or aggregating to describe — not as a reflexive replacement for every loop, and never for a pipeline step whose real job is mutating something outside it.
