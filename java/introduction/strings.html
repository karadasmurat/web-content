Chapter 8

Strings

Text has been sitting in plain sight since §2 — char, and every String literal since. This chapter reopens both: what a char actually can and can't hold, and what a String really is once you know that.

§8.1

Writing Systems and Unicode — a universal catalogue

One word of caution before this chapter properly begins: "character" here means something more general than either of Java's own terms for it, char and Character, both still a section away. Used on its own, a character just means a symbol belonging to some writing system — a letter, a digit, a punctuation mark, an emoji — independent of any programming language, or any notion of how it might be stored in memory.

Unicode assigns a unique number — called a code point — to every character, in that general sense, in every writing system in the world. The letter A, for instance, is code point 65.

Unicode conventionally writes these numbers in hexadecimal rather than decimal, prefixed with U+ — hex represents large numbers more compactly, and code points run well past a million, so the saving is worth it. 65 in hex is 41, so 'A' is written U+0041:

  • The letter A is code point 65 — U+0041 in hex.
  • The character € is code point 8,364 — U+20AC in hex.
  • The emoji 😀 is code point 128,512 — U+1F600 in hex.

That's all Unicode does: define a universal catalogue of characters, and give each one a number. Nothing about that catalogue says how many bits it takes to store one of those numbers, or how a programming language ought to represent one in memory — that's a separate question entirely, and exactly where char comes in, next.

§8.2

char — Java's representation of a code point

§2 defined char in one line: a single 16-bit Unicode character. The 16-bit part is not incidental — recall §2's rule: a slot built from N bits holds exactly 2N distinct patterns, no more. 16 bits give 65,536 of them. When Java was designed, in the mid-1990s, Unicode itself was believed to top out at roughly that many characters — every writing system in existence, once and for all, fitting inside one char. Sizing char at 16 bits wasn't a guess; it was believed to be the entire, final answer.

It wasn't. Unicode has since grown past 1.1 million possible code points — the actual number the standard assigns to a character ('A' is code point U+0041). Historic scripts, thousands of emoji, rare CJK ideographs have all been added since. 65,536 values — what a single char can hold — covers only the Basic Multilingual Plane, a subset of modern Unicode, not the whole thing.

char c = 'A';       // fine — always 2 bytes, code point U+0041, well inside the BMP
char emoji = '🙂';  // compile error — code point U+1F642 exceeds a single char's capacity

'A' fits with room to spare — U+0041 is a small number, one of 65,536 a char can hold. '🙂''s code point, U+1F642, is well over a million past that ceiling — there is no single char value that means it. The compiler doesn't approximate or truncate; it simply refuses.

Java's fix: a code point outside that 16-bit range is represented as a surrogate pair — two char values together, standing in for the one code point neither could hold alone.

String emoji = "😀";
System.out.println(emoji.length()); // 2

One visible character. Two chars. That's not a bug — 😀's code point is U+1F600, far outside the 16-bit range, so it takes a surrogate pair to represent.

A char isn't always a character

The name is a historical relic. A char is really a single UTF-16 code unit — one 16-bit storage slot — not a guarantee of one visible, user-perceived character. Most everyday text (Latin letters, digits, common punctuation) needs only one char per character, which is why the distinction stayed invisible for so long. It stops being invisible the moment text includes anything outside the Basic Multilingual Plane.

§8.3

Character — the wrapper class for char

A primitive slot has no methods — no indirection, nothing to call (§2). Java pairs every primitive type with a wrapper class: an ordinary class that boxes the primitive value inside an object and gives it a toolkit of methods to go with it. char's is Character; int's is Integer; double's is Double; boolean's is Boolean. This chapter only needs Character — the others follow the same idea.

Character's everyday methods classify or transform a single char, one at a time — no loop required yet to use them:

String username = "player1";
char first = username.charAt(0);

System.out.println(Character.isLetter(first)); // true
System.out.println(Character.isDigit(first));   // false
System.out.println(Character.toUpperCase(first)); // 'P'

These are static methods (§6.1) — called on the class itself, Character.isLetter(...), not on an object.

Character also has methods built specifically for the code-point problem §8.2 just raised — this is the payoff:

int codePoint = 0x1F600; // 😀's actual Unicode code point

System.out.println(Character.isSupplementaryCodePoint(codePoint)); // true — outside the BMP
System.out.println(Character.charCount(codePoint));                // 2 — needs a surrogate pair

char[] pair = Character.toChars(codePoint);
System.out.println(pair.length); // 2

Character.charCount(0x1F600) returning 2 is exactly why "😀".length() was 2 back in §8.2 — the same fact, confirmed directly from the code point itself, rather than just observed on a String.

§8.4

Strings Are Sequences of char — indexing and length

A String is a sequence of characters — but the methods it exposes work specifically in terms of char code units (§8.2), which is exactly why .length() counts UTF-16 units, not "characters" in the everyday sense. Reaching a specific one uses charAt, not brackets:

String make = "Toyota";

System.out.println(make.length());   // 6
System.out.println(make.charAt(0));  // 'T'
System.out.println(make.charAt(5));  // 'a'
A method, not a field ← §7

§7's arrays exposed their size as a field — temperatures.length, no parentheses. A String's size is a method — make.length(), with parentheses — and a specific character comes from charAt(i), a method call, not make[i], a bracket. A String is a reference type holding a sequence, much like an array, but it is not an array, and the two don't share syntax.

§8.5

Immutability — every "change" is a new String

An array's slots can be overwritten in place (§7). A Rectangle's fields can be reassigned (§5). A String cannot — once created, its characters never change. Every method that looks like it modifies a String actually builds and returns a brand new one, leaving the original untouched:

String model = "corolla";
String upper = model.toUpperCase();

System.out.println(model); // corolla — unchanged
System.out.println(upper); // COROLLA — a new String, a new address

model still holds the address of the original, all-lowercase String after this runs. toUpperCase() didn't reach in and rewrite it — it allocated a new String object, uppercased, and returned its address, which upper then holds. Forgetting this is a common mistake: model.toUpperCase(); alone, with no assignment, computes a new String and immediately discards it — model itself never changes, no matter how many methods you call on it.

§8.6

== vs .equals() — address vs content

A String variable holds an address, exactly like any reference variable (§4.2). == never looks past that address — it compares only what's sitting in the two variables' own slots, for any reference type, Strings included. new forces two genuinely separate objects, even with identical content:

String a = new String("Toyota");
String b = new String("Toyota");

System.out.println(a == b);      // false — two separate objects
System.out.println(a.equals(b)); // true — same characters

a and b hold two different addresses — new allocated a separate object each time (§4.1) — so a == b is false, the same rule that applies to any two independently-constructed objects. .equals(...) asks a different question entirely: not "same address," but "same characters, in the same order." String answers that question by actually comparing content, character by character — which is precisely why it gives the opposite answer here.

Why String can answer that question

String is a class like any other (§3) — and like any class, it can define its own version of a method it inherits, replacing the inherited behavior with its own. Every class inherits an equals(Object) method whose default behavior is to compare addresses, same as ==. String replaces that default with one that compares characters instead. The mechanics of how a class does that — and the word for it — belong to a later chapter (§11); for now, the important fact is simply that String has already done it, which is why .equals(...) means "same content" specifically for Strings, not address, the way the default would otherwise give you.

Always .equals(), never ==, for String content

String literals are handled specially by Java — identical literals are frequently, though not always reliably, the exact same object underneath, which can make == appear to work by accident. It is not a rule you can depend on. Comparing String content always means .equals(...); == on Strings should be read as a deliberate address comparison, not a shortcut for it.

§8.7

Concatenation with +

+ has appeared on Strings informally since early examples. Formally: + between two Strings joins them into a new one; + between a String and anything else converts the other side to its String form first, then joins:

String greeting = "Hello, " + "world" + "!";
System.out.println(greeting); // Hello, world!

String withNumber = "Count: " + 5;
System.out.println(withNumber); // Count: 5

Like every other String "modification," concatenation builds a brand-new String (§8.5) — greeting is a fresh object, never one of the three literals mutated in place. This is also why the same + symbol means something so different here than it did for ints and doubles in §2 — the operator itself isn't doing arithmetic at all when a String is on either side; it's building and returning a new sequence of characters.

Scope of this chapter

split, String.format, text blocks, and regular expressions all build on what's here, and are left for later. So is the mechanism behind String's own equals() override — §11 covers overriding in general, and is worth revisiting with this chapter in mind. StringBuilder — building a String across many steps without discarding one candidate per step — gets its own explanation in §13.6, once immutability itself has a name.
