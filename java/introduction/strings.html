Chapter 8

Strings

Primitive values let us represent individual characters. But what if we want to represent a person's name?

Java represents text using the String class:

String name = "Alice";

One word of caution before this chapter properly begins: Used on its own, a character just means a symbol belonging to some writing system — a letter, a digit, a punctuation mark, an emoji — independent of any programming language, or any notion of how it might be stored in memory. The general idea of a character is something more general than either of Java's own terms for it, char and Character, both still a section away.

§8.1

Writing Systems and Unicode — a universal catalogue of characters

Type an emoji into a text message, a French accent into a Word document, and a Korean name into a form — and somehow all three just work, stored, displayed, and sent around the world without anyone having to think about it. That only works because every character in every writing system has already been given a single, agreed-on number, once, for every computer to share.

Unicode is that agreement: it assigns a unique number — called a code point — to every character, in that general sense, in every writing system in the world. The letter A, for instance, is code point 65.

Unicode conventionally writes these numbers in hexadecimal rather than decimal, prefixed with U+ — hex represents large numbers more compactly, and code points run well past a million, so the saving is worth it. 65 in hex is 41, so 'A' is written U+0041:

  • The letter A is code point 65 (U+0041)
  • The character € is code point 8364 (U+20AC)
  • The emoji 😀 is code point 128512 (U+1F600)

That's all Unicode does: define a universal catalogue of characters, and give each one a number. Nothing about that catalogue says how many bits it takes to store one of those numbers, or how a programming language ought to represent one in memory — that's a separate question entirely, and exactly where char comes in, next.

§8.2

char — Java's representation of a Unicode character

§2 defined char in one line: a single 16-bit Unicode character. With 16 bits of size, a char can represent 65K different characters. When Java was designed, in the mid-1990s, Unicode itself was believed to top out at roughly that many characters — every writing system in existence, once and for all, fitting inside one char. Sizing char at 16 bits wasn't a guess; it was believed to be the entire, final answer.

The first 65,536 code points — the actual numbers the standard assigns to a character (for example, char 'A' is code point 65 / U+0041) — make up what Unicode calls the Basic Multilingual Plane, or BMP. That's exactly what a single char can represent. Historic scripts, thousands of emoji, and rare CJK ideographs have all been added since, though — Unicode has since grown past 1.1 million possible code points. This is far beyond what a single char can hold. 

char c = 'A';       // fine — always 2 bytes, code point U+0041, well inside the BMP
char emoji = '🙂';  // compile error — code point U+1F642 exceeds a single char's capacity

'A' fits with room to spare — U+0041 is a small number, one of 65,536 a char can hold. '🙂''s code point, U+1F642, is well over a million past that ceiling — there is no single char value that means it. The compiler doesn't approximate or truncate; it simply refuses.

Java's fix: a code point outside that 16-bit range is represented as a surrogate pair — two char values together, standing in for the one code point neither could hold alone.

String emoji = "😀";
System.out.println(emoji.length()); // 2

One visible character. Two chars.

That's not a bug — 😀's code point is U+1F600, far outside the 16-bit range, so it takes a surrogate pair to represent.

A char isn't always a character

The name is a historical relic. A char is really a single UTF-16 code unit — one 16-bit storage slot — not a guarantee of one visible, user-perceived character. Most everyday text (Latin letters, digits, common punctuation) needs only one char per character, which is why the distinction stayed invisible for so long. It stops being invisible the moment text includes anything outside the Basic Multilingual Plane, like an emoji.

§8.3

Character — the wrapper class for char

A char holds a letter — but what if you need to ask is this a digit? or make this uppercase? A primitive alone can't answer that: its value is not an address to follow, and there's no method to call on it, just the raw value sitting in its slot (§2).

Primitive types are simple, lightweight, and efficient. They represent values directly, which makes them ideal for everyday calculations. But they also have one important limitation. They are not objects. That means an int doesn't have methods.

Java's answer is to pair every primitive type with a wrapper class: an ordinary class that boxes the primitive value inside an object and gives it a toolkit of methods. For every primitive type, Java provides a corresponding class. char's is class Character.

Why have both?

At first glance, it may seem strange that Java has both char and Character. The reason is simple: they solve different problems.
  • Use primitive types when you simply need a value.
  • Use wrapper classes when
    • an object is required by the API (ie, you're using collections: List, Map, etc. - Chp. 18).
    • you need behavior related to primitive values. ie, Character.isLetter('A')
Wrapper classes don't exist because primitive types are insufficient—they exist because sometimes Java needs an object that represents a primitive value.

Character's methods classify or transform a single char:

char first = 'A';

System.out.println(Character.isLetter(first)); // true
System.out.println(Character.isDigit(first));   // false
System.out.println(Character.toUpperCase(first)); // 'P'

These are static methods (§4.1) — called on the class itself, Character.isLetter(...), not on an object.

Character also has methods built specifically for the code-point problem §8.2 just raised — this is the payoff:

int codePoint = 0x1F600; // 😀's actual Unicode code point

System.out.println(Character.isSupplementaryCodePoint(codePoint)); // true — outside the BMP
System.out.println(Character.charCount(codePoint));                // 2 — needs a surrogate pair

char[] pair = Character.toChars(codePoint);
System.out.println(pair.length); // 2

Character.charCount(0x1F600) returning 2 is exactly why "😀".length() was 2 back in §8.2 — the same fact, confirmed directly from the code point itself, rather than just observed on a String.

§8.4

String as a Sequence of Characters

Every piece of text used so far — a name, a message, a username — has been written the same way: letters between double quotes. It reads like a built-in type, the same way int or boolean are. But it isn't. Java represents text using the String class — String is actually a class, which means creating one is really object creation (§2.2), and Java gives you two distinct ways to do it.

Creating Strings

1. String Literals

The way every example so far has done it — text in double quotes, assigned directly:

String name = "Alice";

There's no new anywhere in this line, but it still creates a String object — "Alice" doesn't stop being an object just because writing it looks as simple as writing a primitive literal like 5 or true. Java is doing the object creation here too; a literal is just a second way of asking for it, not an exception to §2.2's rule that a new object is what a class actually gives you.

Behind this ordinary-looking assignment, Java maintains a string pool. The next section looks at what that actually is.

The String Pool

Write "Toyota" twice in the same program — does Java actually create two separate objects, one for each line? It seems like it should, given §2.2 already established that new allocates a fresh object every time it runs. But a string literal never uses new, and that turns out to matter.

The string pool is a dedicated region — part of the heap, but managed on its own — that holds exactly one copy of each distinct literal a program uses. The first time the compiler sees a given literal, it puts a new String object in the pool for it. Every later appearance of that same literal, anywhere in the program, doesn't create a new object at all — it just hands back the address already in the pool: 

String a = "Toyota";
String b = "Toyota";

System.out.println(a == b); // true — same pooled object, same address
Stack
a = 0x5000
b = 0x5000
Heap — String Pool
String @ 0x5000
value"Toyota"

a and b were never assigned to each other, and neither one was involved in the other's declaration — yet they hold the same address, because they wrote the identical literal. That's the pool's entire job: identical literals collapse to one object, automatically, with no new anywhere in sight.

2. The new Keyword

Explicit construction, exactly like any other class (§2.2):

String name = new String("Alice");

new always allocates a fresh object on the heap — it never checks the pool, even when an identical literal is already sitting in it:

String c = new String("Toyota");

System.out.println(a == c); // false — new forces a separate object, pool or not

new doesn't check other new calls either — two separate calls, even with the exact same argument, still produce two separate objects:

String member1 = new String("Bob");
String member2 = new String("Bob");

System.out.println(member1 == member2); // false — two separate objects, identical content
Stack
member1 = 0x7f10
member2 = 0x7f88
Heap
String @ 0x7f10
value"Bob"
String @ 0x7f88
value"Bob"

member1 and member2 hold two different addresses, pointing to two distinct String objects that just happen to hold identical content. Same characters, exactly like a and c above — still a separate object every time new runs.

Why this matters for ==

This is why the member1/member2 example above used new String("Bob") twice, instead of two plain "Bob" literals — two literals would have been pooled to the same object, making member1 == member2 come back true and defeating the point. new is how you deliberately opt out of pooling to get a genuinely separate object. §8.7 revisits this exact pair to contrast == against .equals().

A Sequence of Characters

Whichever way it's created, a String is a sequence of characters underneath.

What exactly is stored inside?

Modern Java (9+) doesn't store a char[]. With compact strings, a String stores a byte[] and encoding instead. This makes common Latin-1 characters one byte per character — a plain memory optimization . Every method a String exposes, charAt, length(), and the rest, still works specifically in terms of char. What's actually stored is an implementation detail; what the type promises to you is always char.

The methods it exposes work specifically in terms of char, which is exactly why .length() counts UTF-16 units, not "characters" in the everyday sense.

Reaching a specific one uses charAt, not brackets:

String make = "Toyota";

System.out.println(make.length());   // 6
System.out.println(make.charAt(0));  // 'T'
System.out.println(make.charAt(5));  // 'a'
A method, not a field ← §6

§6's arrays exposed their size as a field — temperatures.length, no parentheses. A String's size is a method — make.length(), with parentheses — and a specific character comes from charAt(i), a method call, not make[i], a bracket. A String is a reference type holding a sequence, much like an array, but it is not an array, and the two don't share syntax.

§8.5

Text Blocks — multi-line String literals

§8.4 covered two distinct ways to create a String — a literal and new. Text blocks are a third piece of syntax, but not a third mechanism: this section exists to show that a text block evaluating to a fixed value is still just a literal underneath, handled by the compiler exactly the same way.

An ordinary literal can't span multiple lines — every line break needs an explicit \n, and joining several lines means concatenating with + (§8.8):

String name = "J. R. R. Tolkien";

Since Java 15, a text block — delimited by triple quotes """ — lets you write the same value with source formatting instead of escape sequences. A text block that evaluates to the exact same characters is still just a String, created differently:

// Using a string literal
String name = "J. R. R. Tolkien";

// Using a text block - same characters as name
String name1 = """
        J. R. R. Tolkien""";

Java strips the "incidental" leading whitespace common to every line (the indentation used to line the text block up with the surrounding code) — what's left is the content itself, not the source indentation.

Text blocks and the String Pool

A text block isn't a separate creation mechanism running alongside literals and new — when it evaluates to a plain, fixed value, like this one does, the compiler treats it exactly like a literal: the resulting String is interned into the string pool at compile time, the same pool §8.4 just introduced. Identical content means the identical pooled object, regardless of which syntax produced it:

// Both name and name1 refer to the same object in the string pool
System.out.println(name == name1); // true

Where a text block genuinely differs is trailing whitespace — specifically, whether the closing """ sits on its own line: 

// content becomes "J. R. R. Tolkien\n" — different from name
String name2 = """
        J. R. R. Tolkien
        """;

System.out.println(name1 == name2); // false

name1's closing delimiter sits right after the text, so its content is exactly "J. R. R. Tolkien" — identical to name, hence pooled together. name2's closing delimiter sits on its own line, which contributes one trailing \n to the content — a genuinely different sequence of characters, and therefore a genuinely different pooled object. == hasn't changed meaning at all here; it's still comparing addresses (§8.7) — it's the content, and therefore which pool entry gets created, that changed.

§8.6

Immutability — every "change" is a new String

An object's fields, taken together, are a representation: a snapshot, held in memory, of some condition at a moment in time. Every class considered so far has quietly assumed that representation can change:

  • An array's slots can be overwritten in place (§6).
  • A Rectangle's width can be reassigned (§3).
  • A BankAccount's balance can be reassigned after a deposit.

Java has a name for that assumption: a mutable object allows its state to change after it's created.

String breaks that assumption entirely. An immutable object never changes its state after creation — if something needs to change, a new object gets created instead. Every method that looks like it modifies a String actually builds and returns a brand new one, leaving the original untouched.

Call model.toUpperCase(), capture the result, and print model itself right afterward:

String model = "corolla";
String upper = model.toUpperCase();
                  
System.out.println(model); // corolla — unchanged
System.out.println(upper); // COROLLA — a new String, a new address

model is still lowercase. "Uppercase" sounds like an action performed on model — so why does model come out unchanged? Because toUpperCase() never touched model at all. It built an entirely new String and returned it, leaving the original exactly as it started.

Stack
model = 0x6a10
upper = 0x6b40
Heap
String @ 0x6a10
value"corolla"
String @ 0x6b40
value"COROLLA"
Memory model

Two separate heap allocations, two separate String objects. model and upper hold different addresses — toUpperCase() never touched the object model points to; it built a brand new one and handed upper its address.

model still holds the address of the original, all-lowercase String after this runs. toUpperCase() didn't reach in and rewrite it — it allocated a new String object, uppercased, and returned its address, which upper then holds. Forgetting this is a common mistake: model.toUpperCase(); alone, with no assignment, computes a new String and immediately discards it — model itself never changes, no matter how many methods you call on it.

§8.7

== vs .equals() — address vs content

When comparing text, is it a must that 2 references point to the exact same object in the memory, or is it ok if 2 different objects which contain the same sequence of characters?

In everyday terms the answer is simple: two pieces of text are "equal" when they hold the same sequence of characters — that's the only sense in which a person would ever compare names or words. But in a running program, two String objects can be separate instances, sitting at separate addresses, while still holding that same sequence of characters. They can be the same in one sense — state, the characters they contain — and different in another — identity, which object, at which address, you actually have.

Java gives you a distinct operator for each question:

  1. == always compares values stored in the variables themselves.
    1. Primitive variables contain actual data directly.
      == compares whether two primitive variables contain the exact same data as the value.
    2. Reference variables contain an "address" as the value.
      == compares whether two reference variables point to the exact same object in memory.
  2. .equals(...) makes it possible to compare state stored in the actual object, asking them: "Do you represent the same thing?"

A String variable holds an address, exactly like any reference variable (§2.3). == never looks past that address — it compares only what's sitting in the two variables' own slots, for any reference type, Strings included. new forces two genuinely separate objects, even with identical content:

String member1 = new String("Bob");
String member2 = new String("Bob");
String copy    = member1;

System.out.println(member1 == member2);      // false — two separate objects
System.out.println(member1.equals(member2)); // true — same characters

System.out.println(member1 == copy);         // true — the very same object

member1 and member2 hold two different addresses — new allocated a separate object each time (§2.2) — so member1 == member2 is false, the same rule that applies to any two independently-constructed objects. .equals(...) asks a different question entirely: not "same address," but "same characters, in the same order." String answers that question by actually comparing content, character by character — which is precisely why it gives the opposite answer here. copy, on the other hand, was never given its own object — String copy = member1; just copies member1's address (§2.3), the same aliasing rule that applies to any reference type. copy and member1 aren't two members who happen to match; they're one member with two name tags, so member1 == copy is true.

Stack
member1 = 0x7f10
copy = 0x7f10
member2 = 0x7f88
Heap
String @ 0x7f10
value"Bob"
String @ 0x7f88
value"Bob"

member1 and member2 hold different addresses, pointing to two distinct String objects on the heap — as separate as two people who happen to share a name. Knowing both members are named "Bob" doesn't make them the same member; the same is true here: knowing both objects hold the characters "Bob" doesn't make them the same object. == answers "same object?" the way you'd ask "same person?"; .equals(...) answers "same characters?" the way you'd ask "same name?" — two different questions, two different answers. copy is a third, different case: it holds 0x7f10, the exact same address as member1, not a lookalike address to a lookalike object. There's no third "Bob" on the heap — just one member with a second name tag pointing at him, which is why member1 == copy comes back true while member1 == member2 does not.

Why String can answer that question

String is a class like any other (§2) — and like any class, it can define its own version of a method it inherits, replacing the inherited behavior with its own. Every class inherits an equals(Object) method whose default behavior is to compare addresses, same as ==. String replaces that default with one that compares characters instead. The mechanics of how a class does that — and the word for it — belong to a later chapter (§9); for now, the important fact is simply that String has already done it, which is why .equals(...) means "same content" specifically for Strings, not address, the way the default would otherwise give you.

Always .equals(), never ==, for String content

String literals are handled specially by Java — identical literals are frequently, though not always reliably, the exact same object underneath, which can make == appear to work by accident. It is not a rule you can depend on. Comparing String content always means .equals(...); == on Strings should be read as a deliberate address comparison, not a shortcut for it.

§8.8

Concatenation with +

+ has appeared on Strings informally since early examples. Formally: + between two Strings joins them into a new one; + between a String and anything else converts the other side to its String form first, then joins:

String greeting = "Hello, " + "world" + "!";
System.out.println(greeting); // Hello, world!

String withNumber = "Count: " + 5;
System.out.println(withNumber); // Count: 5

Like every other String "modification," concatenation builds a brand-new String (§8.6) — greeting is a fresh object, never one of the three literals mutated in place. This is also why the same + symbol means something so different here than it did for ints and doubles in §2 — the operator itself isn't doing arithmetic at all when a String is on either side; it's building and returning a new sequence of characters.

Scope of this chapter

String.format and regular expressions build on what's here, and are left for later. So is the mechanism behind String's own equals() override — §9 covers overriding in general, and is worth revisiting with this chapter in mind. StringBuilder — building a String across many steps without discarding one candidate per step — gets its own explanation in §11.6, once immutability itself has a name.

§8.9

Important String Methods

Every method below follows the same rule §8.6 already established: none of them modify the String they're called on. Each one reads it and returns something new — a new String, or a simple value describing what was found. The running example is a customer's email address, arriving from a form exactly as a user typed it:

String input = "  Alice.Smith@Example.com  ";

Cleaning Up

strip() removes leading and trailing whitespace — the older trim() does the same for ordinary spaces, but strip() is Unicode-aware and is the one to reach for now:

String cleaned = input.strip();
System.out.println(cleaned); // "Alice.Smith@Example.com" — no leading/trailing spaces

replace(...) swaps every occurrence of one character or sequence for another, returning the result as a new String:

String noDots = cleaned.replace('.', ' ');
System.out.println(noDots); // "Alice Smith@Example.com"

Searching and Testing

contains(...), startsWith(...), and endsWith(...) each answer a yes/no question about content, returning a boolean:

System.out.println(cleaned.contains("@"));                 // true
System.out.println(cleaned.toLowerCase().endsWith("example.com")); // true
System.out.println(cleaned.startsWith("Bob"));              // false

indexOf(...) answers a related but more specific question: not just whether something appears, but where — the position of its first occurrence, or -1 if it never appears at all:

int at = cleaned.indexOf('@');
System.out.println(at); // 11
System.out.println(cleaned.indexOf('Z')); // -1 — never found

Extracting Parts

substring(start) and substring(start, end) extract a piece of the sequence — start is inclusive, end is exclusive, the same half-open convention arrays use for ranges (§6). Paired with indexOf, they split a value into meaningful pieces: 

String username = cleaned.substring(0, at);   // characters before '@'
String domain   = cleaned.substring(at + 1);  // everything after '@'

System.out.println(username); // "Alice.Smith"
System.out.println(domain);   // "Example.com"

split(...) takes this further, cutting a String into an array everywhere a separator appears — useful when there isn't just one delimiter to work with, but several:

String sku = "SKU-1029-RED";
String[] parts = sku.split("-");

System.out.println(parts.length);  // 3
System.out.println(parts[0]);      // "SKU"
System.out.println(parts[2]);      // "RED"

Checking for Emptiness

isEmpty() asks whether a String has zero characters; isBlank() asks the more useful question in practice — whether it has no characters that aren't whitespace:

String blank = "   ";

System.out.println(blank.isEmpty()); // false — it does contain characters (three spaces)
System.out.println(blank.isBlank()); // true — none of them are non-whitespace
Why this distinction matters

A form field containing only spaces isn't "empty" by isEmpty()'s definition — it holds three chars, same as any other three-character String (§8.4). Validating user input with isEmpty() alone would let " " through as if it were meaningful content. isBlank() is almost always the right check for "did the user actually type something."
