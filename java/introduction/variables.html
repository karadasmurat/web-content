Chapter 1

Variables and Values

§1.1

Values

Imagine part of your program is responsible for tracking a person's age. Somewhere, that age has to actually exist as data the program can work with — not as an idea, but as something concrete: 

20

That's the essence of it: a program works with information — a person's age, a product price, a user's name, the result of a calculation. Whatever form it takes, Java has a name for it — a value is a piece of information a program can work with.

By itself, 20 is just a number. It becomes useful once a program has a way to refer back to it — which is exactly what a variable gives it.

§1.2

Variables in Java

A variable is a named location in memory that stores a value. We use a human-friendly label that lets us refer to a value later.

Memory model

When a program runs, it needs memory to store the information it works with. You can think of memory as a large collection of storage locations, each capable of holding a value. Imagine memory as a large wall of letterboxes. Each letterbox has a label, and the label is how your program refers to the value stored inside.

A variable is a named location that stores a value. The variable name is the label your program uses to refer to that location, and the variable's type determines what kind of value it can store.

Instead of writing the number 20 everywhere the program needs it, the program uses the name — the same System.out.println(...) that §0.6 introduced as Java's way of printing to the console, just handed a variable this time instead of fixed text: 

System.out.println(age); // 20

Java looks up what age refers to and prints 20 — the name and the value are two different things, connected by the variable.

Three different pieces of information:

  1. Type: what kind of data this variable can hold, and how large its slot needs to be.
  2. Name : how the rest of the program refers to it.
  3. Value: the actual information stored there.

1. Type

Every value has a type. A type is a contract, like a promise: "I will hold values of this type only, and nothing else." Java classifies every type into one of two broad categories: primitive types and reference types.

This chapter covers primitive types — §1.3 onward. Reference types wait until there's an object to actually point at, once classes are on the table. Notice what wasn't said just now: not one word about what a reference type actually is. That's deliberate — for now, it's enough to know the other branch of this tree exists at all.

2. Naming Variables

A variable name is simply a human-friendly label that lets us refer to a value later. The computer could work with anonymous locations in memory, but people write and maintain software—so we give important values meaningful names.

Good variable names make programs easier to understand. Prefer names that say what the value actually is: 

int customerAge;
double accountBalance;

over names that say nothing at all:

int x;
double a;

Java convention uses camelCase for variable names — the first word lowercase, every following word capitalized, no underscores: 

numberOfStudents
totalPrice
userName

None of this changes what the code does — a differently-named variable holds the exact same value the exact same way. What it changes is whether the next person reading the code (often you, months later) can tell what a variable is for without tracing back through the program to find out.

3. Value

The value is the actual data sitting in that named, typed slot at a given moment — 20, for an int age. Type and name are fixed the moment a variable is declared and never change again for that variable's entire lifetime. The value is the one piece of the three that's allowed to: the same slot can hold 20 now and 21 a moment later, still the same variable, same type, same name throughout.

That difference is worth sitting with, because it's exactly what the next section is about: the specific moments a variable's value gets set, and reset.

Declaration, Assignment, and Reassignment

"Declare," "assign," and "reassign" get used almost interchangeably in casual conversation, but each one names a genuinely distinct moment in a variable's life. Four of them, in fact — and it's worth seeing all four side by side, using the same variable throughout.

1. Declaration — creating the slot, with a type and a name, but no value yet: 

int age;

The slot exists the moment this line runs, but it holds nothing yet — the variable is declared, not initialized.

2. Assignment — giving that empty slot its first value, using =: 

age = 20;

This is the moment age actually starts holding something. Nothing about this line redeclares age — its type and name were already fixed in step 1; this line only fills the slot in.

3. Declare and assign, combined — declaration and assignment don't have to be two separate lines. §1.2's very first example, int age = 20;, is exactly steps 1 and 2 folded into one statement: a type, a name, and a value, all at once. 

int age = 20;

4. Reassignment — giving an already-initialized variable a different value: 

int age = 20;
age = 25;

The variable name stays the same — age is still age — but the value it refers to has changed, from 20 to 25. Nothing about this line creates a new variable; it just overwrites what's already sitting in the slot age names. Syntactically, this line looks identical to step 2 — age = 20; reads exactly like age = 25; did. The only difference is whether the slot already had something in it beforehand, which is precisely why "assignment" and "reassignment" are worth keeping as two separate words even though Java itself never distinguishes them.

§1.3

Primitive Types — the simplest kind of variable

Picture a labeled box with the actual data sitting directly inside it. Open the box, and the value is right there. That's what a primitive variable is: the slot has the value, sitting there. Java has eight of these primitive types, and a variable's data type determines which kind of value its box may hold.

The word value is going to show up constantly from here on, so it's worth being precise about what it means the moment it's said of a primitive variable: the data itself, occupying the slot, with nothing standing between the variable and what it holds. 

int     width  = 10;    // slot named 'width'  contains 10
double  price  = 3.99;  // slot named 'price'  contains 3.99
boolean active = true;  // slot named 'active' contains true

All eight primitive types:

Type Description Size Value range
byte Whole number 8-bit -128 to 127
short Whole number 16-bit -32,768 to 32,767
int Whole number 32-bit -2,147,483,648 to 2,147,483,647
long Whole number, for values bigger than int can hold 64-bit -9,223,372,036,854,775,808 to 9,223,372,036,854,775,807
float Decimal number, lower precision 32-bit ~ ±3.4 × 1038, about 6–7 significant decimal digits
double Decimal number 64-bit ~ ±1.8 × 10308, about 15–16 significant decimal digits
boolean Exactly one of two values 1 bit's worth of information true or false — nothing else
char A single Unicode character 16-bit 0 to 65,535 (one UTF-16 code unit — §8 covers what that means)

Worth reinforcing here, since it's easy to reduce a type to nothing but its size: types are about meaning, not just storage. An int isn't just 32 bits — that's how the promise §1.2 called a contract gets kept, not what the contract actually says. What it says is that this slot represents a whole number. The bits are the mechanism; the meaning is the point.

Every one of these types also has a default value — what a field of that type starts out holding, before any constructor runs. §2.5 covers default values in full, once fields and constructors are actually on the table.

Type as a contract

We can say "age is an int." Better to say it the way the contract actually reads: the variable age is allowed to store integer values. That phrasing is what makes the next two lines make sense: 

int age = 20;      // fine — an integer value, exactly what was promised
int age = "Alice"; // compile error — not allowed

The compiler refuses the assignment. Why? Because age made a promise: "I only store integers." "Alice" isn't one, so its rejected even before the program runs. This is exactly what static typing means — the promise is checked, and enforced, at compile time, not discovered as a surprise later.

What "N-bit" actually means

A bit is a single binary digit — 0 or 1, the smallest unit of storage there is. A slot built from N bits can hold exactly 2N distinct patterns, and not one more — that's the entire rule, for every type above. 8 bits give 28 = 256 distinct patterns, which is exactly byte's range: 256 values, split to run from -128 to 127.

That ceiling matters more than it looks like it should — enough to be worth its own chapter (§8) once char is asked to represent something bigger than 65,536 possibilities actually turns out to be.

§1.4

Literals

Every example so far has written a value straight into the code — 20, 3.99, true. Nothing computed it, nothing read it from anywhere else; it's just typed directly where it's needed. Java has a name for exactly this: a fixed value spelled out in the code itself is called a literal. A literal's own syntax is what tells the compiler which type it is, without needing a declared variable anywhere nearby:

Kind Example Notes
Decimal integer 42, -7 Base 10 — the default form, int
Hexadecimal integer 0x2A Prefixed 0x, base 16 — 0x2A is the same value as 42
Binary integer 0b101010 Prefixed 0b, base 2 — also 42
Long 10000000000L Trailing L forces 64-bit, overriding the 32-bit int default
Double 3.14 The default for any decimal literal — a trailing D is legal but redundant
Float 3.14f Trailing f forces 32-bit, overriding the double default
Boolean true, false Exactly these two words — nothing else is a boolean literal
Character 'A', '\n', '\u0041' Single quotes; escape sequences like \n and Unicode escapes like \u0041 are both legal — '\u0041' is the same character as 'A'

Each of those literals is typically assigned straight into a variable of the matching type: 

int   decimal = 42;
int   hex     = 0x2A;
int   binary  = 0b101010;
long  big     = 10000000000L;
double price  = 3.14;
float ratio   = 3.14f;
boolean done  = true;
char  letter  = 'A';
char  newline = '\n';
char  alsoA   = '\u0041';

Notice each literal's own form already matches the declared type on its left — 0x2A and 0b101010 are both just 42 written differently, and a variable declared int accepts either one without any conversion, because the value they produce is identical. Get the pairing wrong — say, assigning 3.14 to an int without a suffix telling the compiler otherwise — and it's a compile error, not a rounding surprise at runtime.

Underscores for readability

A numeric literal can use underscores to group digits — 1_000_000 is exactly the same int as 1000000. The compiler strips the underscores entirely; they exist purely to make a long literal easier for a person to read at a glance.

§1.5

The var Keyword

Look back at int age = 20; for a moment. The literal 20 already makes it obvious this is a whole number — so does writing int just ahead of it actually tell Java anything it couldn't have figured out on its own? Often, it doesn't. Modern Java lets you skip that repetition using var: 

var age = 20;

Java determines from the value on the right that age is an int — but the type is still fixed once that line compiles, exactly as if it had been written out. This is called local variable type inference: var changes what you have to type, not what Java actually does: 

var age = 20;

age = "twenty"; // compile error — age is still an int underneath
Recommendation

Learn traditional declarations first — int age = 20; — since they keep the type visible while you're still learning what each type means. Reach for var later, once it's the type itself that's obvious and repeating it would just be noise.

§1.6

Common Beginner Mistakes

Using a variable before it's ever been given a value:

int age;

System.out.println(age); // compile error — variable age might not have been initialized

Trying to store the wrong type of value:

int age = "twenty"; // compile error — incompatible types

Confusing the variable name with the value it holds:

age = 20;

Here age is not the value — it's the name that refers to the value. §1.2 already drew this distinction; it's worth keeping straight as the rest of the tutorial builds on it.

§1.7

What actually goes inside a variable?

Every variable holds a value. A piece of information. Simple enough, right?

Imagine two boxes sitting side by side. The first box contains an actual number: age +----+ | 25 | +----+ The value itself is stored there.

The second box contains a note — an address. The real thing is somewhere else, but can be found following that address.

Both boxes contain a value , but the values are fundamentally different. In Java, a variable holds one of two kinds of values:

  • A primitive value:The variable stores The actual data itself, stored directly in the slot. This is what happens with types such as int, and boolean.
  • A reference value: The variable stores a reference to an object. The object itself exists separately, and the reference allows the program to access it. This is what reference variables do — coming soon.
Java is statically typed

Because Java is statically typed, every variable declaration includes a type, and that type determines which of the two applies: a primitive type means the variable holds a value directly, a reference type means it holds an address. The value case is fully behind you now — the address case is still ahead; §2.3 is where it's actually built out, once there's a class, and an object of it, to point at (§2).

§1.8

Key Takeaways

  • A value is information a program can work with; a variable gives it a name to refer to it later.
  • A Java variable declaration has three parts: a type, a name, and a value.
  • Java has eight primitive types; a primitive variable's slot holds its value directly, with no indirection.
  • Every value written directly into source code is a literal — its own syntax tells the compiler which type it is.
  • Every variable holds either a value directly (primitives, just covered) or an address pointing elsewhere (references, §2.3 — still ahead).
  • A variable's value can change after declaration; its name and type stay fixed.
  • var lets Java infer a local variable's type — the type itself never becomes optional.
§1.9

Appendix: Number Systems — binary, decimal, and hex

§1.3's "N-bit" callout and §1.4's 0x2A / 0b101010 literals both leaned on binary and hex without stopping to explain them. This appendix is that missing background — optional on a first read, worth returning to once those two sections have raised the question.

Why Base 10 Isn't Special

Write the number 247, and nobody thinks twice about what it means — but the meaning isn't in the digits alone, it's in their position. That 2 isn't worth 2; it's worth 200, because of where it sits:

Place (Power) Hundreds (10²) Tens (10¹) Ones (10⁰)
Digit 2 4 7
Digit value 200 40 7

200 + 40 + 7 = 247.

Each position is worth ten times the position to its right, because decimal has ten available digits — 0 through 9 — before it has to roll over into the next position. That's the entire idea behind a base: how many distinct digits exist before counting has to carry into the next column. Nothing about place-value notation actually requires the base to be ten — it's just the base humans happened to standardize on, almost certainly because of finger-counting. A computer has no fingers, and settles on a different base for a very different reason.

Binary: Base 2

A bit, as §1.3 already said, is a single binary digit — 0 or 1, nothing else. Binary is base 2: only two digits exist before a column has to carry, so each position is worth twice the one to its right instead of ten times:

Power 2⁵ 2⁴ 2³ 2² 2¹ 2⁰
Digit 1 0 1 0 1 0
Digit value 32 0 8 0 2 0

That's 101010 in binary — exactly the digits behind 0b101010 from §1.4. 32 + 8 + 2 = 42. Converting the other direction works the same way any base does: read each column's digit times its place value, and add them up.

Hexadecimal: Base 16

Binary is precise but long-winded — writing out 32 bits one digit at a time is tedious and easy to mis-copy. Hexadecimal exists to make binary shorter to write, not to replace it: each hex digit stands in for exactly 4 bits, so a number twice as long in binary is only half as long in hex. Base 16 needs 16 distinct digits before a column carries, and decimal only supplies 10 of them — so hex borrows six letters to fill the rest:

Decimal 0 1 2 ... 9 10 11 12 13 14 15
Hex digit 0 1 2 ... 9 A B C D E F

0x2A from §1.4 reads the same way as any other base — each column times its place value:

Power 16¹ 16⁰
Digit 2 A
Digit value 32 10

A stands for 10, so 2A in hex is (2 × 16) + 10 — 42 again, the same value 0b101010 and plain 42 already named. All three are the same number; only the notation changed.

Why hex lines up with bits so neatly

4 bits give exactly 24 = 16 distinct patterns — precisely the range one hex digit covers, no more and no less. That's not a coincidence; it's why hex was chosen as binary's shorthand in the first place. A byte (8 bits, §1.3) always fits in exactly 2 hex digits; a 32-bit int always fits in exactly 8. Decimal has no such clean relationship to bit widths, which is exactly why hex — not decimal — is what shows up whenever a raw bit pattern needs to be written out compactly, from memory addresses to color codes.
