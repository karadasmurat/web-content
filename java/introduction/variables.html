Chapter 1

Variables

A variable is a named slot in memory, and it holds exactly one of two things: a value, stored directly, or an address, pointing elsewhere. That distinction drives everything that follows.

§1.1

Values

Programs work with information — a person's age, a product price, a user's name, the result of a calculation. A value is a piece of information a program can work with:

20

By itself, 20 is just a number. It becomes useful once a program has a way to refer back to it — which is exactly what a variable gives it.

§1.2

Giving Values Names

Say a program needs to keep track of a person's age:

age = 20;

This statement creates a name, age, and connects it to the value 20. Instead of writing the number 20 everywhere the program needs it, the program uses the name:

System.out.println(age); // 20

Java looks up what age refers to and prints 20 — the name and the value are two different things, connected by the variable.

§1.3

Variables in Java

Your program runs in a computer that has memory: a vast array of numbered slots, each holding a small piece of data. A variable is your way of naming one of those slots so you can refer to it by name rather than by its raw address.

Java is a statically typed language — every variable has a type, declared up front, that describes what kind of value it can hold. A full declaration names all three parts at once:

int age = 20;
  • Type (int) — what kind of data this variable can hold, and how large its slot needs to be.
  • Name (age) — how the rest of the program refers to it.
  • Value (20) — the actual information stored there.

The type and name can also be declared without a value yet, filled in later:

int count;
boolean active;

The slot exists the moment the declaration runs, but it has no value in it yet — the variable is declared, not initialized. Trying to read count or active before assigning one is a compile error (§1.8), not a runtime surprise; Java refuses to let a program read a slot it never filled.

Memory model

Think of a variable as a labeled slot. The label is the variable name. The slot has a fixed size determined by the type. What the slot actually contains — a value directly, or an address pointing elsewhere — is §1.5's subject.

§1.4

Naming Variables

Good variable names make programs easier to understand. Prefer names that say what the value actually is:

int customerAge;
double accountBalance;

over names that say nothing at all:

int x;
double a;

Java convention uses camelCase for variable names — the first word lowercase, every following word capitalized, no underscores:

numberOfStudents
totalPrice
userName

None of this changes what the code does — a differently-named variable holds the exact same value the exact same way. What it changes is whether the next person reading the code (often you, months later) can tell what a variable is for without tracing back through the program to find out.

§1.5

What a Variable Holds

Every variable in Java holds exactly one of two things:

  • A value — the data itself, stored directly in the slot. This is what primitives do (§2).
  • An address — the location of some data stored elsewhere in memory. This is what reference variables do (§4.2).
Java is statically typed

Because Java is statically typed, every variable declaration includes a type, and that type determines which of the two applies: a primitive type means the variable holds a value directly, a reference type means it holds an address. That distinction is the central thread running through the next several chapters.

§1.6

Changing a Variable's Value

A variable can usually be assigned a new value after it's created:

int score = 10;

score = 20;

The variable name stays the same — score is still score — but the value it refers to has changed. Nothing about the second line creates a new variable; it overwrites what's already in the slot score names.

§1.7

The var Keyword

Modern Java also supports local variable type inference using var:

var age = 20;

Java determines from the value on the right that age is an int — but the type is still fixed once that line compiles, exactly as if it had been written out. var changes what you have to type, not what Java actually does:

var age = 20;

age = "twenty"; // compile error — age is still an int underneath
Recommendation

Learn traditional declarations first — int age = 20; — since they keep the type visible while you're still learning what each type means. Reach for var later, once it's the type itself that's obvious and repeating it would just be noise.

§1.8

Common Beginner Mistakes

Using a variable before it's ever been given a value:

int age;

System.out.println(age); // compile error — variable age might not have been initialized

Trying to store the wrong type of value:

int age = "twenty"; // compile error — incompatible types

Confusing the variable name with the value it holds:

age = 20;

Here age is not the value — it's the name that refers to the value. §1.2 already drew this distinction; it's worth keeping straight as the rest of the tutorial builds on it.

§1.9

Key Takeaways

  • A value is information a program can work with; a variable gives it a name.
  • A Java variable declaration has three parts: a type, a name, and a value.
  • Every variable holds either a value directly (primitives, §2) or an address pointing elsewhere (references, §4.2).
  • A variable's value can change after declaration; its name and type stay fixed.
  • var lets Java infer a local variable's type — the type itself never becomes optional.
